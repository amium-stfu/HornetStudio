# TODO

## Title
Clarify transport handling for the ItemBroker.

## Problem
The current MQTT-based setup exposes ItemBroker data through one concrete interface. Future transports such as pipes, HTTP, or WebSocket need a clear model so transport diagnostics do not get mixed with the shared item namespace.

## Impact
Without a clear transport model, future adapters may create separate item views or duplicate system topic structures, making diagnostics and item visibility harder to understand.

## Suggested Fix
Define the ItemBroker as the central source of truth and treat transports as access paths that expose the same item namespace. Keep transport-specific status and metrics under system topics only, for example `$SYS/mqtt/status` or a future `$SYS/pipe/status`.

## Priority
Medium

## Notes
- Do not introduce a generic `transport` topic level until multiple transport implementations exist or the shared structure is actively needed.
- Transport-specific diagnostics should not contain or own the actual item tree.

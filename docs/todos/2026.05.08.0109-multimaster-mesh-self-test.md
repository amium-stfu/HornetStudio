# TODO

## Title
Define a real multimaster mesh self-test.

## Problem
The current multimaster demo self-test starts one local MQTT server and simulates three logical nodes as clients on that shared broker. This validates MQTT integration, but it does not validate the intended multimaster topology where each device owns its own server/broker and connects to peers through clients.

## Impact
The current test can pass while missing failures that only appear with multiple brokers, separate ports, peer-to-peer synchronization, startup ordering, reconnect behavior, ownership conflicts, or echo-loop prevention.

## Suggested Fix
Design and plan a mesh-style self-test where three local nodes each start their own `MqttItemServerHost` on separate ports, for example `18831`, `18832`, and `18833`. Each node should publish to its local server and connect as a remote client to the other two nodes. The test should verify baseline visibility, cross-node writes, runtime item creation, runtime updates, and remote writes across the separated server topology.

High

# IMPLEMENTATION HANDOFF

## Goal

Update the ItemBroker MQTT transport topic layout so the main item topic carries `Item.Properties["meta"]` as JSON, item values are published at `/value`, item properties are published as direct child topics, and an empty `BaseTopic` removes the MQTT topic prefix.

## Scope

Included:

- MQTT topic mapping changes in the shared MQTT mapper.
- MQTT adapter publish and incoming write mapping changes.
- Client-side subscription filter adjustment for empty base topics.
- Focused tests for mapper and adapter behavior.
- Documentation updates only where existing docs describe the old topic layout.

Explicitly NOT included:

- UI redesign.
- Item domain model changes.
- Project or namespace rename work.
- New runtime dependencies.
- Release packaging or version changes.

## Starting Point

- `src/Amium.Item.Server.Mqtt/MqttTopicMapping.cs` contains `MqttItemTopicMapper`.
- The mapper currently requires non-empty `BaseTopic`.
- The mapper currently publishes item value at the item topic.
- Non-value properties currently use a `properties` topic segment.
- `src/Amium.Item.Server.Mqtt/MqttItemServerAdapter.cs` publishes snapshots and changes through `PublishParameterTopicAsync`.
- `src/Amium.Item.Client.Mqtt/MqttItemTopicMapper.cs` exposes `ItemSubscriptionTopic` as `"{BaseTopic}/#"`.
- Default meta value should be `"{}"` when missing or empty.

## Tasks

1. Update `MqttItemTopicMapper` to accept empty base topics.
2. Add topic mapping semantics for reserved logical parameters:
   - `meta` maps to `<item-topic>`.
   - `value` maps to `<item-topic>/value`.
   - other properties map to `<item-topic>/<property-name>`.
3. Remove generation and parsing dependency on the `properties` segment for new topics.
4. Keep broker health topic handling working with and without a base topic.
5. Update `MqttItemServerAdapter` snapshot publishing:
   - publish main item topic with meta JSON.
   - publish `/value` with `Item.Value`.
   - publish direct child property topics.
   - skip `meta`, `name`, and `path` as child property topics.
6. Update value changes to publish `/value`.
7. Update property changes to publish direct child topics, except reserved/skipped properties.
8. Update incoming MQTT mapping:
   - `<item-topic>/value` updates item value.
   - `<item-topic>/<prop>` updates property.
   - `<item-topic>` maps to `meta` or is rejected/ignored according to the implementation decision.
9. Update client subscription topic:
   - non-empty base topic: `<baseTopic>/#`.
   - empty base topic: `#`.
10. Add or update tests for the new topic layout.
11. Run build/tests.

## File-Level Changes

- `src/Amium.Item.Server.Mqtt/MqttTopicMapping.cs` -> Own optional base-topic normalization, main-topic meta mapping, `/value` mapping, direct child property mapping, and reverse topic parsing.
- `src/Amium.Item.Server.Mqtt/MqttItemServerAdapter.cs` -> Publish meta/value/property topics using the new mapping, filter redundant properties, and handle incoming writes under the new semantics.
- `src/Amium.Item.Client.Mqtt/MqttItemTopicMapper.cs` -> Return `#` for empty base-topic subscriptions.
- `src/Amium.Item.Server.Tests/Program.cs` or relevant MQTT test file -> Add focused mapper/adapter assertions for new behavior.
- Existing Markdown docs mentioning old MQTT topics -> Update only if found.

## Implementation Order

1. Change mapper construction so empty `baseTopic` is valid and `BaseTopic` returns an empty string.
2. Change `ToTopic`:
   - normalize path to slash-separated topic segments.
   - prepend base segments only when present.
   - append no parameter segment for `meta`.
   - append `value` for `value`.
   - append normalized property name for other properties.
3. Change `TryMapTopic`:
   - strip optional base segments when configured.
   - reject empty topics.
   - map relative topic with one or more segments to `meta` by default.
   - map final segment `value` to property `value`.
   - map other final segments to direct property names when there is a parent item path.
4. Preserve/update broker health handling.
5. Refactor adapter publish calls minimally around the new mapper behavior.
6. Add a small helper in the adapter only if needed for reserved property filtering and meta payload formatting.
7. Update tests.
8. Run verification commands.

## Technical Constraints

- Follow existing C# style, nullable reference handling, and XML documentation rules.
- Use named parameters where calls have ambiguous or multiple parameters.
- Do not introduce new dependencies.
- Keep changes inside MQTT transport/client/tests/docs unless a direct compile issue requires otherwise.
- Write user-visible errors and diagnostics in English.
- Preserve retained MQTT publication behavior.
- Avoid broad refactoring.

## Acceptance Criteria

- With `BaseTopic = "hornet"`, `edm1.pressure` maps to:
  - `hornet/edm1/pressure` for `meta`.
  - `hornet/edm1/pressure/value` for `value`.
  - `hornet/edm1/pressure/unit` for `unit`.
- With `BaseTopic = ""`, `edm1.pressure` maps to:
  - `edm1/pressure` for `meta`.
  - `edm1/pressure/value` for `value`.
  - `edm1/pressure/unit` for `unit`.
- Snapshot publishing emits meta, value, and direct child properties.
- Snapshot publishing does not emit `/properties/...`, `/meta`, `/name`, or `/path`.
- Missing/empty meta publishes `"{}"`.
- Incoming MQTT writes still update writable values/properties correctly under the new layout.
- Empty base topic subscription filter is `#`.

## Out of Scope

- Release version generation.
- Git tags.
- Publish output under `Release`.
- Runtime FOSS notice changes, because no new dependency should be added.
- Editor UX changes unrelated to MQTT topic behavior.

## Risks / Watchouts

- Direct property mapping makes ambiguous topics possible: `<item>/value` is reserved for `Item.Value`, so a normal item property named `value` cannot be represented as a child property.
- Main-topic writes may conflict with broker-owned retained meta messages. Decide and test the desired behavior explicitly.
- Empty base-topic mapping must avoid accidentally treating broker health topics or unrelated MQTT topics as writable item topics.
- Existing external MQTT consumers expecting `/properties/...` will break.

## Notes

- Prefer case-insensitive filtering for `meta`, `name`, and `path`.
- Default meta should be exactly `"{}"` when no usable meta payload exists.
- If an existing property already stores JSON as a string, publish it as-is when non-empty.
- If an existing property stores a structured object, format it through existing payload formatting unless a local JSON formatter already exists in the project.

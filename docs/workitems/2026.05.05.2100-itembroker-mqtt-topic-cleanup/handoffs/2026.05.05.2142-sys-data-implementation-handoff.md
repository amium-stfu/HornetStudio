# IMPLEMENTATION HANDOFF

## Goal

Replace the confusing ItemBroker health/runtime MQTT topic hierarchy with a concise system-data layout under `$SYS`, while keeping normal item topics outside the system namespace.

## Scope

Included:

- Publish general service status under `$SYS/status`.
- Publish runtime metrics under `$SYS/metrics`.
- Publish MQTT server diagnostics under `$SYS/mqtt/status`.
- Remove redundant `item-broker`, `runtime/health`, and empty `heartbeat` topics from the system data output.
- Add or update focused tests for topic paths and basic payload behavior.
- Update documentation that describes the old system topic layout.

Explicitly NOT included:

- Adding pipe, HTTP, WebSocket, or any other transport.
- Adding a generic `transport` topic level.
- Changing the regular item namespace.
- Changing item value/property topic behavior beyond avoiding system-topic conflicts.
- Adding runtime dependencies.
- Release versioning, tags, or publish output.

## Starting Point

- The active discussion identified `Health.cs` as the main health/system-data area.
- The current visible topic layout includes `broker/mqtt/...` and `runtime/health/item_broker/...`.
- The desired structure is:

```text
$SYS/status
$SYS/metrics
$SYS/mqtt/status
```

- Normal item topics should remain parallel to `$SYS`, for example:

```text
edm1/pressure
edm1/temperature
```

- `$SYS` is used as the MQTT-compatible system namespace because `#sys` is not a valid MQTT publish topic.

## Tasks

1. Inspect the existing health contracts and publishers.
2. Identify where current system topics are generated.
3. Define the new status topic fields:
   - `$SYS/status/state`
   - `$SYS/status/uptimeSeconds`
   - `$SYS/status/startedAtUtc`
   - `$SYS/status/lastUpdatedUtc`
4. Define the initial metrics topic fields:
   - `$SYS/metrics/itemCount`
   - `$SYS/metrics/memoryWorkingSetBytes`
   - `$SYS/metrics/memoryManagedHeapBytes`
   - `$SYS/metrics/cpuUsagePercent`
5. Define the initial MQTT status topic fields:
   - `$SYS/mqtt/status/state`
   - `$SYS/mqtt/status/clientCount`
   - `$SYS/mqtt/status/endpoint`
   - `$SYS/mqtt/status/lastError`
6. Remove or stop publishing obsolete topics:
   - `broker/mqtt/status`
   - `broker/mqtt/uptime`
   - `broker/mqtt/heartbeat`
   - `runtime/health/item_broker/...`
7. Remove empty heartbeat publication when status updates already include `uptimeSeconds` and `lastUpdatedUtc`.
8. Add metric collection with existing .NET APIs only.
9. Keep CPU usage process-scoped if implemented; defer or document if it cannot be measured cleanly.
10. Add or update tests for the new topic names and removed obsolete topics.
11. Update relevant Markdown documentation.
12. Run build/tests.

## File-Level Changes

- `src/Amium.Item.Server/Health.cs` -> Adjust health/system data contracts and field names to the new status and metrics structure.
- `src/Amium.Item.Server.Mqtt/MqttItemServerAdapter.cs` -> Publish the new `$SYS/...` topics and stop publishing obsolete health/runtime topics.
- `src/Amium.Item.Server/Program.cs` -> Wire metric/status publishing if the service currently owns runtime state or intervals.
- `src/Amium.Item.Server.MqttDemoClient/Program.cs` -> Update demo subscriptions or display logic only if it explicitly references the old health topics.
- `src/Amium.Item.Server/Contracts.cs` -> Update only if public contracts directly encode the old health structure.
- Relevant test project files -> Add assertions for new system topic paths and removed old paths.
- Relevant docs under `docs/` -> Update old health/runtime topic references.

## Implementation Order

1. Read `Health.cs`, `MqttItemServerAdapter.cs`, service startup, and existing tests to locate current health publishing boundaries.
2. Decide whether `$SYS/status` and `$SYS/metrics` are represented as grouped objects internally or as individual publications externally.
3. Update health/system contracts with minimal public API change.
4. Update the MQTT publisher to emit the target system paths.
5. Remove old heartbeat/runtime publication paths.
6. Add metric values using existing APIs:
   - `Environment.WorkingSet` or `Process.WorkingSet64` for working set.
   - `GC.GetTotalMemory(forceFullCollection: false)` for managed heap.
   - Broker item registry/count API for item count.
   - Process CPU sampling only if an existing timer/update loop can calculate a delta.
7. Update demo or client code only where old topic names are hard-coded.
8. Update tests.
9. Update documentation.
10. Run verification.

## Technical Constraints

- Use existing project patterns and dependencies.
- Do not introduce new runtime dependencies.
- Keep changes minimal and directly related to system data topics.
- Use English for code, XML documentation, comments, UI text, and diagnostics.
- Document changed public classes and public methods with XML documentation.
- Prefer named parameters for ambiguous or multi-parameter calls.
- Do not introduce a generic transport abstraction for this work.
- Keep normal item topics outside `$SYS`.

## Acceptance Criteria

- System status is published under `$SYS/status/...`.
- Metrics are published under `$SYS/metrics/...`.
- MQTT server diagnostics are published under `$SYS/mqtt/status/...`.
- Normal items such as `edm1/pressure` remain outside the system namespace.
- No `$SYS/item-broker/...` topics are produced.
- No `runtime/health/item_broker/...` topics are produced.
- No empty heartbeat topic is required.
- `uptimeSeconds` and `lastUpdatedUtc` provide liveness.
- Metrics include item count, working set memory, managed heap memory, and CPU usage if feasible.
- No new dependencies are added.

## Verification

- Run `dotnet build HornetStudio.sln --no-restore`.
- Run the relevant ItemBroker tests if present.
- Manually inspect the MQTT demo output or retained topics to confirm the new topic structure.
- Confirm obsolete health/runtime topics are absent.

## Out of Scope

- Release publish.
- Version number generation.
- Git tags.
- FOSS notice changes, because no dependency should be added.
- Multi-transport design.
- Item namespace restructuring.

## Risks / Watchouts

- `$SYS` topics may require clients to subscribe explicitly to `$SYS/#`, because broad `#` subscriptions do not necessarily include `$` topics.
- CPU usage requires sampling over time; a single instantaneous process value is not meaningful.
- If old health topics are consumed externally, this is a breaking topic contract change.
- Metrics update frequency should avoid noisy retained-topic churn.
- Avoid treating `$SYS` topics as writable item topics when item base topic is empty.

- Prefer process CPU usage over host CPU usage.
- Defer `loadPercent` until a documented formula exists.
- Future transports should expose the same item namespace and only add transport-specific diagnostics under system topics.

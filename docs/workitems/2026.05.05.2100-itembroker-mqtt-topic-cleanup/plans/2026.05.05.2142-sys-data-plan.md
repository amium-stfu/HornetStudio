# ItemBroker System Data Topic Plan

## Goal

Define a clearer system-data topic structure for the ItemBroker MQTT server so runtime state, metrics, and MQTT server diagnostics are separated from normal item topics.

## Target Topic Layout

System data should live in a dedicated system namespace:

```text
$SYS/status
$SYS/metrics
$SYS/mqtt/status
```

Normal item topics remain outside that namespace:

```text
edm1/pressure
edm1/temperature
```

`$SYS` is not the item root topic and must not be combined with an `item-broker` segment. It is a parallel system namespace beside the regular item namespace.

## Scope

Included:

- Replace the current confusing health/runtime hierarchy with a flatter system-data layout.
- Remove redundant `item-broker` topic segments from system data.
- Remove `heartbeat` if regular status or uptime updates already provide liveness.
- Decide which runtime and metric values should be published first.
- Keep MQTT-specific diagnostics separate from general broker status and metrics.
- Update relevant tests and documentation if existing behavior is covered.

Not included:

- Implementing additional transports such as pipe, HTTP, or WebSocket.
- Introducing a generic `transport` topic level.
- Changing the regular item topic namespace.
- Changing ownership or synchronization rules for items.
- Adding new runtime dependencies.
- Release or publish work.

## Current Problem

The current health layout mixes multiple concerns:

```text
broker/mqtt/status
broker/mqtt/uptime
broker/mqtt/heartbeat
runtime/health/item_broker/transport/mqtt
runtime/health/item_broker/uptime
runtime/health/item_broker/heartbeat
```

This creates redundant state, unclear ownership, and unnecessary nesting. The `heartbeat` entries add little value if uptime or last-update timestamps are published regularly.

## Proposed Structure

### General Status

```text
$SYS/status/state
$SYS/status/uptimeSeconds
$SYS/status/startedAtUtc
$SYS/status/lastUpdatedUtc
```

Purpose:

- Describe whether the ItemBroker service is running.
- Provide liveness through periodically updated `uptimeSeconds` and `lastUpdatedUtc`.
- Avoid a separate empty heartbeat payload.

Recommended state values:

```text
Starting
Running
Degraded
Unhealthy
Stopping
Stopped
```

### Metrics

```text
$SYS/metrics/itemCount
$SYS/metrics/memoryWorkingSetBytes
$SYS/metrics/memoryManagedHeapBytes
$SYS/metrics/cpuUsagePercent
```

Optional later metrics:

```text
$SYS/metrics/messagesReceivedTotal
$SYS/metrics/messagesPublishedTotal
$SYS/metrics/messagesReceivedPerSecond
$SYS/metrics/messagesPublishedPerSecond
$SYS/metrics/errorCount
$SYS/metrics/loadPercent
```

`loadPercent` should only be added when the formula is explicitly defined. Until then, prefer raw metrics such as CPU, memory, item count, queue length, message rate, and latency.

### MQTT Server Diagnostics

```text
$SYS/mqtt/status/state
$SYS/mqtt/status/clientCount
$SYS/mqtt/status/endpoint
$SYS/mqtt/status/lastError
```

Purpose:

- Describe the concrete MQTT server interface that is currently running.
- Avoid generic `transport/mqtt` naming until multiple transport implementations exist.
- Keep MQTT diagnostics out of the normal item namespace.

## Design Decisions

- Use `$SYS` as the MQTT-compatible system namespace.
- Do not add an `item-broker` path segment under `$SYS`.
- Do not make `$SYS` the root of normal items.
- Do not introduce `transport` yet.
- Treat future transports as access paths to the same ItemBroker item namespace.
- Use system topics only for diagnostics, status, and metrics.
- Remove empty heartbeat topics when uptime and last-updated values are available.

## MQTT Note

In standard MQTT topic filters, `#` is a wildcard character and cannot be used inside a publish topic name. The MQTT-compatible system topic structure is:

```text
$SYS/status
$SYS/metrics
$SYS/mqtt/status
```

This plan uses `$SYS` because the originally discussed `#sys` topic root is not a valid MQTT publish topic.

## Sequencing

1. Inspect the current `Health.cs` contracts and existing MQTT health publishing code.
2. Identify the current retained system topics and their payload shapes.
3. Replace the current hierarchy with the target `$SYS/status`, `$SYS/metrics`, and `$SYS/mqtt/status` groups.
4. Remove `heartbeat` publication if `uptimeSeconds` and `lastUpdatedUtc` are updated periodically.
5. Add minimal metric collection for item count, process working set, managed heap size, and CPU usage if feasible without new dependencies.
6. Keep CPU measurement simple and documented, or defer it if it cannot be implemented accurately with the existing runtime APIs.
7. Update tests or add focused assertions for the new system topic paths.
8. Update documentation that mentions the old health/runtime topic structure.
9. Run the relevant build and test commands.

## Open Questions

- Should system-topic values be individual retained scalar topics, one retained JSON object per group, or both?
- What update interval should be used for metrics?
- Should `lastError` be retained until cleared, or published as an empty value after recovery?
- Should CPU be process CPU usage only, or host CPU usage? Recommended: process CPU usage.

## Verification Plan

- Confirm normal item topics such as `edm1/pressure` are still published outside `$SYS`.
- Confirm no `$SYS/item-broker/...` topics are published.
- Confirm no `runtime/health/...` topics are published after the change.
- Confirm no empty `heartbeat` topic is required for liveness.
- Confirm retained status and metrics update while the service is running.
- Run `dotnet build HornetStudio.sln --no-restore`.

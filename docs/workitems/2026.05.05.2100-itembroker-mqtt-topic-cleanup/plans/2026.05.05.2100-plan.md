# ItemBroker MQTT Topic Cleanup Plan

## Goal

Simplify the ItemBroker MQTT topic layout so item identity, values, and properties are easier to consume and no redundant item metadata is published as separate property topics.

## Target Topic Layout

```text
[baseTopic/]<item-path>         = Item.Properties["meta"] as JSON, default "{}"
[baseTopic/]<item-path>/value   = Item.Value
[baseTopic/]<item-path>/<prop>  = Item.Properties[prop]
```

Property exclusions:

- `meta` is mapped to the main item topic and must not be published as `/meta`.
- `name` is not published because it is derivable from the topic.
- `path` is not published because it is equal to the item topic path.

Base topic behavior:

- Empty `BaseTopic` means no MQTT topic prefix.
- Non-empty `BaseTopic` is normalized and prepended to all broker-owned item topics.

## Current Behavior

The MQTT mapper currently requires a non-empty base topic and publishes item values directly at the item topic:

```text
hornet/edm1/pressure = Item.Value
hornet/edm1/pressure/properties/name = Item.Name
hornet/edm1/pressure/properties/path = Item.Path
```

Non-value properties are currently nested under a `properties` segment. This makes topics longer and publishes redundant fields that can already be inferred from the MQTT topic.

## Scope

Included:

- Change MQTT topic mapping for item snapshots, value updates, property updates, removals, and incoming MQTT writes.
- Make `BaseTopic` optional.
- Publish the item main topic as JSON meta.
- Publish `Item.Value` as `/value`.
- Publish item properties as direct child topics.
- Filter redundant properties `meta`, `name`, and `path` from child property publishing.
- Update focused tests for mapper and adapter behavior.
- Update relevant documentation if existing MQTT topic behavior is documented.

Not included:

- Renaming ItemBroker projects or namespaces.
- Changing the in-memory broker domain model.
- Changing editor UI behavior unless it directly depends on topic previews or default base-topic handling.
- Adding new runtime dependencies.
- Release or publish work.

## Architecture

The MQTT transport boundary should remain the only place where MQTT topic shape is decided.

- `MqttItemTopicMapper` owns bidirectional path/property/topic mapping.
- `MqttItemServerAdapter` owns translating broker messages to retained MQTT publications and MQTT writes back to broker messages.
- Client-side mapper wrappers should delegate to the shared mapper and only adapt subscription filters.
- Existing item and broker models should remain unchanged.

The new layout should treat the main item topic as a reserved meta topic. The logical broker parameter name for this mapping can be `meta`, but the external MQTT topic should be the item path itself.

## Sequencing

1. Adjust mapper rules and optional base topic handling.
2. Adjust adapter publishing rules for snapshots and value/property changes.
3. Adjust incoming MQTT write mapping so `<item-topic>/value` updates the item value and `<item-topic>/<prop>` updates a property.
4. Decide whether incoming writes to the main item topic are ignored or mapped to the `meta` property. Recommended: map to `meta` only if the item is writable, otherwise reject like other properties.
5. Update client subscription filter behavior for empty base topics.
6. Update or add focused tests.
7. Run relevant builds/tests.

## Decisions

- Default meta payload is `"{}"` when the item has no `meta` property or the property value is empty.
- `meta` payload is JSON by contract.
- `meta`, `name`, and `path` are reserved for publishing and are not emitted as direct child property topics.
- Keep retained MQTT output behavior.
- Do not add JSON libraries unless existing project dependencies already provide one; prefer existing .NET APIs.

## Open Questions

- Should incoming main-topic publishes update the broker `meta` property, or should the adapter ignore main-topic writes as broker-owned metadata?
- Should `name` and `path` filtering be case-insensitive? Recommended: yes, matching existing path/property normalization behavior.
- Should malformed `meta` JSON be passed through as text, normalized to `"{}"`, or rejected? Recommended: preserve valid JSON payloads and default only for missing/empty values.

## Verification Plan

- Mapper tests:
  - Empty base topic maps `edm1/pressure` to path `edm1.pressure` and property `meta`.
  - Empty base topic maps `edm1/pressure/value` to property `value`.
  - Non-empty base topic maps `hornet/edm1/pressure/unit` to property `unit`.
  - Old `properties` topics are no longer generated.

- Adapter publish tests:
  - Snapshot publishes main meta topic, `/value`, and direct property topics.
  - Snapshot does not publish `/meta`, `/name`, or `/path`.
  - Value change publishes `/value`.
  - Property change publishes direct property topic.

- Build/test:
  - `dotnet build HornetStudio.sln --no-restore`
  - Relevant ItemBroker test project, if present and runnable in the current environment.

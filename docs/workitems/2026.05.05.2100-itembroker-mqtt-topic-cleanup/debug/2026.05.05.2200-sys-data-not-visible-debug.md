# DEBUG REPORT

## Problem
System data was not visible in MQTT Explorer after moving health data to the new system topic structure.

## Expected Behavior
MQTT should expose system data beside normal item topics:

```text
$SYS/status/state
$SYS/metrics/itemCount
$SYS/mqtt/status/state
edm1/pressure
edm1/temperature
```

## Actual Behavior
Only normal item topics such as `edm1/pressure` and `edm1/temperature` were visible.

## Error Messages / Logs
No runtime error was reported in the visible MQTT Explorer view.

## Relevant Code
- `MqttItemServerHost` publishes health items under internal broker paths such as `sys.status.state`.
- `MqttItemServerAdapter` subscribed only to `MqttItemServerOptions.SubscriptionRootPath`, which defaults to `runtime`.
- The previous external topic root `#sys` was not MQTT-compatible because `#` is a wildcard character in topic filters.

## Attempted Fixes
- Added a dedicated adapter subscription for `ItemServerHealthPaths.Root` when the main item subscription does not already cover it -> system broker items can now reach the MQTT publisher.
- Changed external system topics from `#sys/...` to `$SYS/...` -> MQTT-compatible publish topics.
- Updated tests and documentation to use `$SYS/...`.

## Current Hypothesis
- The missing MQTT output was caused by the adapter not subscribing to `sys.*` health items.
- Even after fixing the subscription, literal `#sys/...` would remain invalid for strict MQTT clients and brokers.

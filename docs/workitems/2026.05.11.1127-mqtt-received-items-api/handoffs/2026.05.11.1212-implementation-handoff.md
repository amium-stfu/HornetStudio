# IMPLEMENTATION HANDOFF

## Goal

Expose received MQTT item state through a simple `MqttItemClient.ReceivedItems` API, rename the current remote-item registry concept to received-item terminology, and support explicit manual seeding for development without connected hardware.

Intended user-facing access:

```csharp
var temperature = client.ReceivedItems["edm1.temperature"];
```

Intended development seeding:

```csharp
client.ReceivedItems.Seed("edm1.temperature", value: 23.5);
```

The indexer must return the live received `Item` instance stored by the registry.

## Scope

- Rename `MqttRemoteItemRegistry` to `MqttReceivedItemRegistry`.
- Rename related change event types to received-item terminology.
- Add public `MqttItemClient.ReceivedItems`.
- Add live path lookup API to the registry.
- Add explicit `Seed(...)` API for manually adding or updating received-item state.
- Keep existing compatibility APIs unless a compile-time conflict requires a minimal adjustment.
- Update tests and documentation affected by the new API.

Explicitly not included:

- Do not create missing items through the indexer.
- Do not add automatic publish/write behavior when a returned live item is mutated.
- Do not publish seeded values to MQTT.
- Do not add seeded-vs-received metadata unless required by implementation constraints.
- Do not change `Amium.Item/Item.cs`.
- Do not introduce new dependencies.
- Do not remove existing `RemoteItemsChanged` or `GetRemoteItemSnapshots()` in this change.

## Starting Point

- `amium_item/src/Amium.Item.Client/MqttRemoteItemRegistry.cs` reconstructs received item trees from MQTT topics.
- `MqttItemClientSession.RemoteItems` owns the registry and calls `RemoteItems.Apply(mapping, payload)` for received non-write MQTT item messages.
- `MqttItemClient` subscribes to registry changes and exposes `RemoteItemsChanged` plus `GetRemoteItemSnapshots()`.
- `GetRemoteItemSnapshots()` currently hides paths published by the local client.
- Existing samples, mesh code, host code, tests, and docs use `RemoteItemsChanged` and/or `GetRemoteItemSnapshots()`.
- Development without hardware currently requires real MQTT messages or custom test setup to populate the received registry.

## Tasks

1. Rename the registry file and types from remote terminology to received terminology.
2. Add live item lookup by full item path to the registry.
3. Add explicit manual seeding by full item path.
4. Add explicit snapshot lookup methods for callers that do not want live references.
5. Expose `ReceivedItems` on `MqttItemClient`.
6. Wire `MqttItemClientSession` and `MqttItemClient` to use the same received-item registry instance.
7. Preserve existing public behavior for `RemoteItemsChanged` and `GetRemoteItemSnapshots()`.
8. Update tests and documentation for the new API and seeding behavior.
9. Run focused validation.

## Technical Constraints

- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public classes and public methods need XML documentation.
- Prefer existing project patterns and libraries.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Do not introduce new dependencies.
- Keep refactoring tightly scoped to the received-item API, seeding, and rename.
- Use named parameters where they improve readability, especially with multiple or ambiguous arguments.

## Acceptance Criteria

- Application code can access a received live item with `client.ReceivedItems["edm1.temperature"]`.
- Application code can seed development data with `client.ReceivedItems.Seed("edm1.temperature", value: 23.5)`.
- `Seed(...)` works before hardware or MQTT messages are available.
- The registry indexer never creates missing received items.
- Unknown paths fail explicitly through `KeyNotFoundException`.
- `TryGet` supports safe lookup without exceptions.
- The live lookup API returns the stored registry item instance.
- Snapshot APIs remain available for clone-based access.
- Existing `RemoteItemsChanged` and `GetRemoteItemSnapshots()` callers continue to compile.
- MQTT receive behavior remains unchanged.
- Later MQTT messages can update seeded items.
- Local mutation of a received live item does not automatically publish to MQTT.
- Seeded values are not automatically published to MQTT.

## Out of Scope

- Removing deprecated remote terminology from all public APIs.
- Changing item path validation rules.
- Changing `Item` mutation behavior.
- Adding automatic publishing from live received item mutation.
- Adding new UI features.
- Adding persisted mock profiles or hardware simulation configuration.

## Risks / Watchouts

- `MqttItemClient.ReceivedItems` must not become stale when sessions are recreated.
- If `MqttItemClientSession` creates its own registry, seeded items may be disconnected from actual receive handling. Prefer injecting or sharing the client-level registry.
- Returning live items means callers can mutate registry state. This is intentional and must be documented clearly.
- MQTT receive updates may occur while application code reads or mutates live items.
- Existing snapshot behavior hides locally published paths in `MqttItemClient.GetRemoteItemSnapshots()`. Do not accidentally change that behavior.
- Direct registry root snapshot methods may not apply the client-level local-path hiding filter.
- `Seed(...)` should validate paths through `ItemPath.Normalize(...)` and should not silently accept non-canonical paths.

## Notes

- Prefer `MqttReceivedItemRegistry` over `MqttItemReceivedRegistry` because it reads naturally in English.
- Keep the user-facing term `ReceivedItems` for direct item access.
- Use `Seed(...)` instead of making the indexer auto-create items. This keeps read access predictable while supporting hardware-free development explicitly.
- Keep the compatibility term `RemoteItemsChanged` for now unless a separate breaking-change plan explicitly removes it.

# IMPLEMENTATION HANDOFF

## Goal

Simplify the received-item user workflow by replacing the already implemented `ReceivedItems.Seed(item)` API with a clearer standard workflow based on `ReceivedItems.Register(item)`.

Target user workflow:

```csharp
var temperature = new Item("temperature", value: 0.0, path: "edm1");
client.ReceivedItems.Register(temperature);
```

After registration, the application keeps using the same `temperature` instance. When MQTT later receives data for `edm1.temperature`, that registered live item is updated.

## Scope

- Replace the implemented `Seed(Item)` API with `Register(Item)`.
- Keep the existing `MqttReceivedItemRegistry` naming and received-item concept.
- Keep simple live access by path through `client.ReceivedItems["edm1.temperature"]`.
- Keep `TryGet(...)` and snapshot access if already implemented.
- Preserve compatibility APIs such as `RemoteItemsChanged` and `GetRemoteItemSnapshots()`.
- Update tests that currently assert seed behavior to assert register behavior.
- Update existing documentation that currently describes `Seed(...)`.
- Add a new user-focused Markdown documentation file in the `.Client` project area.

Explicitly not included:

- Do not keep `Seed(...)` as a public alias unless a compiler/API compatibility issue makes that explicitly necessary.
- Do not add `Register(string path, object? value)` or path/value overloads.
- Do not create missing items through the indexer.
- Do not add automatic MQTT publish/write behavior when a registered or received live item is mutated.
- Do not change `amium_item/src/Amium.Item/Item.cs`.
- Do not introduce new dependencies.

## Starting Point

- The previous seed plan has already been implemented.
- Current code contains:

## Required Client Documentation Content

The new `.Client` Markdown documentation must cover only user-relevant APIs:

- `MqttItemClient` constructors:
  - `MqttItemClient(string clientId, string host = ..., int port = ..., string baseTopic = ...)`
  - `MqttItemClient(MqttItemClientOptions options)`
- Connection lifecycle:
  - `ConnectAsync(...)`
  - `DisconnectAsync(...)`
  - `DisposeAsync()`
  - `IsConnected`
- Publishing and writing:
  - `PublishSnapshotAsync(...)`
  - `UpdateValueAsync(...)`
  - `UpdatePropertyAsync(...)`
  - `RequestWriteAsync(...)`
- Periodic publishing:
  - `AddPeriodicValue(...)`
  - `AddPeriodicProperty(...)`
  - `AddPeriodicSnapshot(...)`
- Receiving items:
  - `ReceivedItems`
  - `ReceivedItems.Register(Item item)` as the standard workflow
  - `ReceivedItems["path"]`
  - `ReceivedItems.TryGet(...)`
  - `ReceivedItems.ItemChanged` or the actual public change event name after implementation
- Existing compatibility:
  - `RemoteItemsChanged`
  - `GetRemoteItemSnapshots()`
- Diagnostics:
  - `Diagnostic`

The documentation should include a compact example showing the intended standard workflow:

```csharp
var client = new MqttItemClient(clientId: "demo_client");
var temperature = new Item("temperature", value: 0.0, path: "edm1");

client.ReceivedItems.Register(temperature);
await client.ConnectAsync();
```

The documentation must state:

- `Register(Item)` does not publish anything.
- Registered items are live instances.
- Incoming MQTT messages update the registered item when the path matches.
- Mutating a registered/received item locally does not publish automatically.
- Publishing/writing remains explicit through the client methods.

## Acceptance Criteria

- Standard user workflow is:

```csharp
var temperature = new Item("temperature", value: 0.0, path: "edm1");
client.ReceivedItems.Register(temperature);
```

- `ReceivedItems.Seed(...)` is no longer part of the public user API.
- `Register(Item)` uses `item.Path` as the single source of truth.
- `Register(Item)` stores the provided instance directly.
- Registered items are updated by later MQTT receive handling for the same path.
- `client.ReceivedItems["edm1.temperature"]` returns the live registered/received item.
- `TryGet(...)` supports safe dynamic lookup.
- Existing `RemoteItemsChanged` and `GetRemoteItemSnapshots()` callers continue to compile.
- Existing receive behavior remains unchanged.
- A new `.Client` Markdown documentation file exists and focuses only on application-facing APIs.
- Existing documentation no longer recommends `Seed(...)`.

## Out of Scope

- Removing deprecated remote terminology from all public compatibility APIs.
- Changing item path validation rules.
- Changing `Item` mutation behavior.
- Adding automatic publishing from live received item mutation.
- Adding path/value registration overloads.
- Adding UI features.
- Adding persisted mock profiles or hardware simulation configuration.

## Risks / Watchouts

- Because `Seed(...)` was already implemented, stale docs/tests may remain if only source code is renamed.
- If external consumers already used `Seed(...)`, removing it is a breaking API change. The current project direction accepts this to keep the API clear.
- `MqttItemClient.ReceivedItems` must not become stale when sessions are recreated.
- Returning/storing live items means callers can mutate registry state through their original reference. This is intentional and must be documented clearly.
- Existing snapshot behavior hides locally published paths in `MqttItemClient.GetRemoteItemSnapshots()`. Do not accidentally change that behavior.

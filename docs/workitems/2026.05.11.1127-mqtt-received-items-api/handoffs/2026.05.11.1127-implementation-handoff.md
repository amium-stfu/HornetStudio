# IMPLEMENTATION HANDOFF

## Goal

Expose received MQTT item state through a simple `MqttItemClient.ReceivedItems` API and rename the current remote-item registry concept to received-item terminology.

The intended user-facing access pattern is:

```csharp
var temperature = client.ReceivedItems["edm1.temperature"];
```

The indexer must return the live received `Item` instance stored by the registry.

## Scope

- Rename `MqttRemoteItemRegistry` to `MqttReceivedItemRegistry`.
- Rename related change event types to received-item terminology.
- Add public `MqttItemClient.ReceivedItems`.
- Add live path lookup API to the registry.
- Keep existing compatibility APIs unless a compile-time conflict requires a minimal adjustment.
- Update tests and documentation affected by the new API.

Explicitly not included:

- Do not add automatic publish/write behavior when a returned live item is mutated.
- Do not change `Amium.Item/Item.cs`.
- Do not introduce new dependencies.
- Do not remove existing `RemoteItemsChanged` or `GetRemoteItemSnapshots()` in this change.

## Starting Point

- `amium_item/src/Amium.Item.Client/MqttRemoteItemRegistry.cs` reconstructs received item trees from MQTT topics.
- `MqttItemClientSession.RemoteItems` owns the registry and calls `RemoteItems.Apply(mapping, payload)` for received non-write MQTT item messages.
- `MqttItemClient` subscribes to registry changes and exposes `RemoteItemsChanged` plus `GetRemoteItemSnapshots()`.
- `GetRemoteItemSnapshots()` currently hides paths published by the local client.
- Existing samples, mesh code, host code, tests, and docs use `RemoteItemsChanged` and/or `GetRemoteItemSnapshots()`.

## Tasks

1. Rename the registry file and types from remote terminology to received terminology.
2. Add live item lookup by full item path to the registry.
3. Add explicit snapshot lookup methods for callers that do not want live references.
4. Expose `ReceivedItems` on `MqttItemClient`.
5. Wire `MqttItemClientSession` and `MqttItemClient` to use the same received-item registry instance.
6. Preserve existing public behavior for `RemoteItemsChanged` and `GetRemoteItemSnapshots()`.
7. Update tests and documentation for the new API.
8. Run focused validation.

## File-Level Changes

  - Rename file to `MqttReceivedItemRegistry.cs`.
  - Rename:
    - `MqttRemoteItemRegistry` -> `MqttReceivedItemRegistry`
    - `MqttRemoteItemChangeKind` -> `MqttReceivedItemChangeKind`
    - `MqttRemoteItemChangedEventArgs` -> `MqttReceivedItemChangedEventArgs`
  - Add live indexer `public ItemModel this[string path] { get; }`.
  - Add `public bool TryGet(string path, out ItemModel? item)`.
  - Keep snapshot-returning root API. If method names are changed, preserve compatibility with a forwarding method where practical.
  - Add XML documentation explaining live-reference semantics.

- `amium_item/src/Amium.Item.Client/MqttItemClientSession.cs`
  - Change registry type to `MqttReceivedItemRegistry`.
  - Prefer received terminology internally.
  - Keep existing behavior for MQTT receive handling and diagnostics.

- `amium_item/src/Amium.Item.Client/MqttItemClient.cs`
  - Add `public MqttReceivedItemRegistry ReceivedItems { get; }` or expose the session-backed received registry through a stable property.
  - Ensure the property is usable by application code after connection.
  - Preserve `RemoteItemsChanged`.
  - Preserve `GetRemoteItemSnapshots()` semantics, including hiding locally published paths.
  - Update event handler signatures to the renamed event args and change kind.

- `amium_item/tests/Amium.Item.Server.Tests/Program.cs`
  - Rename direct registry tests to `MqttReceivedItemRegistry`.
  - Add coverage for:
    - Indexer returns an existing live item by full path.
    - Indexer throws `KeyNotFoundException` for an unknown path.
    - `TryGet` returns `false` for an unknown path.
    - Live lookup returns the stored instance, not a clone.
    - Snapshot lookup returns a clone when snapshot API exists.

- `amium_item/docs/amium-itembroker-usage.md`
  - Add or update usage examples to show `client.ReceivedItems["edm1.temperature"]`.
  - Document that indexer access returns a live item and does not publish changes automatically.

## Implementation Order

1. Rename the registry file and symbols.
2. Add registry path traversal helpers for full-path lookup without creating missing items.
3. Add live indexer and `TryGet`.
4. Add explicit snapshot methods or clarify existing snapshot method names.
5. Update `MqttItemClientSession` references.
6. Add `MqttItemClient.ReceivedItems` and keep existing compatibility APIs.
7. Update tests.
8. Update documentation.
9. Build and run focused self-tests.

## Technical Constraints

- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public classes and public methods need XML documentation.
- Prefer existing project patterns and libraries.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Do not introduce new dependencies.
- Keep refactoring tightly scoped to the received-item API and rename.
- Use named parameters where they improve readability, especially with multiple or ambiguous arguments.

## Acceptance Criteria

- Application code can access a received live item with `client.ReceivedItems["edm1.temperature"]`.
- The registry indexer never creates missing received items.
- Unknown paths fail explicitly through `KeyNotFoundException`.
- `TryGet` supports safe lookup without exceptions.
- The live lookup API returns the stored registry item instance.
- Snapshot APIs remain available for clone-based access.
- Existing `RemoteItemsChanged` and `GetRemoteItemSnapshots()` callers continue to compile.
- MQTT receive behavior remains unchanged.
- Local mutation of a received live item does not automatically publish to MQTT.

## Out of Scope

- Removing deprecated remote terminology from all public APIs.
- Changing item path validation rules.
- Changing `Item` mutation behavior.
- Adding automatic publishing from live received item mutation.
- Adding new UI features.

## Risks / Watchouts

- `MqttItemClient.ReceivedItems` must point at the active session registry. If sessions are recreated on reconnect, avoid exposing a stale registry instance.
- Returning live items means callers can mutate registry state. This is intentional and must be documented clearly.
- MQTT receive updates may occur while application code reads or mutates live items.
- Existing snapshot behavior hides locally published paths in `MqttItemClient.GetRemoteItemSnapshots()`. Do not accidentally change that behavior.
- Direct registry root snapshot methods may not apply the client-level local-path hiding filter.

# IMPLEMENTATION HANDOFF

## Goal

Fix UDL demo modules not appearing under `Received Items` after the strict snake_case path migration, while preserving the intended architecture where demo modules simulate real runtime items and the widget body reads runtime/attach data only.

## Scope

Included:

- Align UDL runtime, status, and attach-option path handling with snake_case.
- Keep demo and hardware item discovery on the same runtime-item path.
- Add read-side compatibility for legacy mixed-case UDL paths where needed.
- Update focused tests or add tests for runtime-relative path extraction and attach-option discovery if existing test structure supports it.

Explicitly NOT included:

- Reworking demo modules into a separate body data source.
- Changing `Item.Value`, `read`, or `write` semantics.
- Reintroducing `value` as a primary item property.
- Widget visual redesign.
- Release/version/publish work.

## Starting Point

- `SimulatedHostUdlClient` creates demo runtime items and the message counter increases, so the simulation loop is active.
- `UdlClientControl.RebuildAttachSectionRows()` builds `ReceivedItems` from `GetAttachOptions(item)`.
- `GetAttachOptions(item)` combines registry attach-option paths with `EnumerateClientItems().Select(GetRelativeRuntimePath)`.
- `GetRelativeRuntimePath(Item item)` currently searches for `Runtime` followed by `UdlClient`.
- UDL widget status paths are visible in the tree as snake_case segments such as `status`, `item_count`, and `message_counter`.
- Mixed-case hard-coded path segments remain in UDL widget code.

## Tasks

1. Inspect and document actual paths produced for demo runtime items after connection.
2. Identify all UDL widget path literals that build or parse runtime/status/attach-option paths.
3. Define canonical UDL path shapes:
   - `runtime.udl_client.<client_name>`
   - `studio.<folder_name>.<client_name>.status`
   - `studio.<folder_name>.<client_name>.status.attach_options`
4. Update new path construction to use canonical snake_case path segments.
5. Update runtime-relative path extraction to recognize both:
   - `runtime.udl_client.<client_name>`
   - `runtime.UdlClient.<client_name>`
6. Update attach-option discovery to recognize both canonical and legacy status/attach-option prefixes.
7. Keep `ReceivedItems` sourced from runtime/attach options only.
8. Verify demo module creation, connection, received item display, and attach behavior.
9. Run the solution build and focused tests.

## File-Level Changes

- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` -> centralize or locally normalize UDL runtime/status/attach-option path construction and parsing; fix `GetRelativeRuntimePath`, `GetAttachOptions`, `PublishAttachOptionItems`, `RemovePublishedRuntimeItems`, and related status base helpers.
- `src/HornetStudio.Editor/Widgets/Dialogs/AttachItemsEditorDialogWindow.axaml.cs` -> align UDL attach-option discovery with the same canonical and compatibility prefixes.
- `src/HornetStudio.Editor/Widgets/UdlClient/SimulatedHostUdlClient.cs` -> update `_itemsPath` only if it still writes mixed-case runtime base paths after the path model is confirmed.
- `src/HornetStudio.Editor.Tests` or existing relevant test project -> add/update focused tests if path helper logic is testable without UI startup.
- Documentation -> update only if existing docs describe UDL runtime/status/attach paths with old mixed-case segments.

## Implementation Order

1. Search UDL widget/editor code for hard-coded path segments:
   - `UdlClient`
   - `Status`
   - `AttachOptions`
   - `runtime.UdlClient`
2. Inspect actual runtime item paths by adding controlled diagnostics or using existing debug logging around `PublishClientItems()` / `LogRuntimeRootItems()`.
3. Implement minimal helper logic in `UdlClientControl.axaml.cs` or a small existing-nearby helper if reuse with the attach dialog is necessary.
4. Change new path construction to canonical snake_case.
5. Change path recognition to accept canonical plus legacy forms.
6. Update `AttachItemsEditorDialogWindow.axaml.cs` to use matching recognition logic.
7. Add or update focused tests if practical.
8. Run build and relevant tests.
9. Manually verify the demo workflow.

## Technical Constraints

- Chat answers must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Keep changes minimal and directly related to path handling.
- Prefer existing project helpers such as `TargetPathHelper` where appropriate.
- Do not introduce new dependencies.
- Do not change the architectural data flow from runtime items to `ReceivedItems`.
- Avoid broad refactoring of the UDL widget.
- Preserve hardware mode behavior.

## Acceptance Criteria

- A newly created demo module appears under `Received Items` after connecting in demo mode.
- `item_count` and `message_counter` still update.
- The demo module can be attached from `Received Items`.
- Attached demo modules resolve as live runtime items.
- Hardware runtime items still resolve through the same path extraction logic.
- New UDL widget status and attach-option paths use canonical snake_case segments.
- Legacy mixed-case paths are tolerated where needed for discovery.
- No `value` compatibility path is introduced as part of this fix.

## Verification

- `dotnet build HornetStudio.sln --no-restore`

  - Enable demo mode on a UDL widget.
  - Add demo module `m300`.
  - Connect.

## Out of Scope

- Changing UDL demo configuration serialization.
- Changing `Item` read/write channel behavior.
- Changing MQTT or ItemBroker semantics.
- UI layout redesign.
- Release publishing or version updates.

## Risks / Watchouts

- Client-name normalization must not accidentally strip or rename valid client names.
- Registry paths may already be normalized by lower layers; avoid relying on exact original casing.
- Compatibility reads should not cause duplicate attach options when both old and new paths exist.
- `GetRelativeRuntimePath` must skip exactly the runtime base and client name, returning `m300` for `runtime.udl_client.udl1.m300`.
- Path separators may appear as dots or slashes depending on source; use existing path splitting helpers.

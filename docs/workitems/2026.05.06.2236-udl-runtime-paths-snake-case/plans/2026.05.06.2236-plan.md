# UDL Runtime Paths Snake Case Plan

## Problem

UDL demo modules are created and the simulated client is active, but the widget body does not show the modules under `Received Items`. The status tree still shows a connected client and an increasing `message_counter`, so the simulation itself is likely running.

The most likely cause is inconsistent path handling after the strict snake_case path migration. UDL widget code still contains hard-coded mixed-case path segments such as `UdlClient`, `Status`, and `AttachOptions`, while registry paths now appear as snake_case segments such as `udl_client`, `status`, and `attach_options`.

## Scope

Included:

- Analyze and align UDL runtime, status, and attach-option path handling.
- Keep demo modules as real runtime item simulation, not as a separate UI data source.
- Define canonical UDL widget path segments for new writes.
- Define compatibility reads for old mixed-case paths only where needed.
- Cover the UDL widget body, attach dialog options, runtime item publishing, and runtime-relative path extraction.

Not included:

- Changing the demo module architecture.
- Changing `Item` `read` / `write` behavior.
- Reintroducing `value` as a primary item property.
- UI redesign.
- Release/version work.

## Current Behavior

- `SimulatedHostUdlClient` can run and emit frames.
- `UdlClientControl` publishes status values under the widget status path.
- `ReceivedItems` is built from `GetAttachOptions(item)`.
- `GetAttachOptions(item)` combines registry attach options and runtime item paths.
- `GetRelativeRuntimePath(item)` searches for `Runtime` followed by `UdlClient`.
- Several path helpers still build paths with mixed-case segments.

## Target Behavior

- Demo runtime items appear under `Received Items` exactly like hardware runtime items.
- Removing the demo flag later should not require changing attachments or exposure concepts.
- New UDL widget runtime/status/attach paths use one consistent snake_case representation.
- Existing mixed-case paths can still be read where that prevents avoidable breakage.

## Proposed Path Model

Use these canonical path shapes for new UDL widget paths:

- Runtime base: `runtime.udl_client.<client_name>`
- Status base: `studio.<folder_name>.<client_name>.status`
- Attach options base: `studio.<folder_name>.<client_name>.status.attach_options`

Compatibility path recognition may accept:

- `runtime.UdlClient.<client_name>`
- `studio.<folder_name>.<client_name>.Status`
- `studio.<folder_name>.<client_name>.Status.AttachOptions`

New writes should prefer the canonical snake_case shape.

## Architecture

Keep the widget body data flow unchanged:

1. Demo or hardware client produces runtime `Item` instances.
2. `UdlClientControl` enumerates runtime items.
3. Runtime item paths are converted to relative attach identities.
4. Root attach identities populate `ReceivedItems`.
5. Selected identities are attached into the UI folder context.

Do not add `UdlDemoModuleDefinitions` as a separate `ReceivedItems` source. That would make demo mode behave differently from hardware mode and weaken the intended "remove demo flag later" workflow.

## Main Decisions

- The fix should be path-normalization focused.
- Prefer small local helpers in the existing UDL widget area over a new global abstraction.
- If shared logic is needed between `UdlClientControl` and `AttachItemsEditorDialogWindow`, extract only the minimal path helper required.
- Compatibility should be read-side only unless existing code already writes legacy paths for another reason.
- Debug logging should be temporary or controlled through existing UDL debug logging; avoid noisy permanent logs.

## Implementation Sequence

1. Inspect actual runtime paths produced by `SimulatedHostUdlClient` after demo module creation.
2. Compare those paths with prefixes used by:
   - `GetAttachOptions`
   - `PublishAttachOptionItems`
   - `RemovePublishedRuntimeItems`
   - `GetRelativeRuntimePath`
   - attach dialog option discovery
3. Define or reuse a small UDL path helper for canonical and compatibility prefixes.
4. Update new path construction to canonical snake_case.
5. Update relative runtime path extraction to recognize both canonical and legacy UDL runtime markers.
6. Update attach-option discovery to recognize canonical and legacy status/attach-option paths.
7. Verify that demo modules appear under `Received Items` while connected.
8. Verify attaching a received demo module still attaches the runtime item into the UI folder context.
9. Run the build and focused tests.

## Verification Strategy

- Manual demo check:
  - Enable demo mode.
  - Add module `m300`.
  - Connect.
  - Confirm `item_count > 0`.
  - Confirm `message_counter` increments.
  - Confirm `Received Items` contains `m300`.
  - Attach `m300`.
  - Confirm attached item resolves as live.
- Code/search check:
  - Search remaining `UdlClient`, `Status`, and `AttachOptions` path literals in UDL widget code.
  - Confirm each remaining mixed-case path is either legacy compatibility or non-path UI text.
- Build check:
  - `dotnet build HornetStudio.sln --no-restore`

## Risks

- Existing saved layouts may still contain old attached paths.
- Registry keys may already be normalized by lower layers, so double-normalization must not change client names unexpectedly.
- Changing new write paths can affect any code that reads status or attach options from old paths only.
- Hardware mode must continue to produce the same attach identities as demo mode.

## Open Questions

- Should old mixed-case status/attach-option paths be removed after canonical paths are written, or only ignored?
- Are external tools reading `studio.<folder>.<client>.Status.AttachOptions` directly?
- Does `TargetPathHelper` already provide a canonical snake_case conversion that should be used instead of adding local string logic?

# PLAN

## Topic
Finalize a slim `Amium.Item.Server` public API around snapshot registration, unified updates, and protected item ownership.

## Goal
Define a public broker API that stays easy to use while enforcing stable ownership rules. Item snapshots should register ownership, later changes should use a unified update model, and existing owners must not be overwritten implicitly.

## Problem Framing
The simplified public direction of:
- `PublishSnapshotAsync(...)` for item registration
- `UpdateValueAsync(...)`
- `UpdateParameterAsync(...)`

solves the user-facing publish-versus-write confusion, but it introduces a new critical requirement: ownership must be reliable. If any later snapshot can silently replace an existing owner for the same item path, the whole ownership model becomes unstable and unsafe.

Without explicit owner protection, the system risks:
- accidental ownership takeover
- typo-driven or conflicting publishers replacing valid owners
- hard-to-diagnose update routing behavior
- a public API that looks simple but behaves unpredictably

## Agreed Direction
- Keep one public main API.
- Use snapshot publication to register or refresh an item plus its owner.
- Use unified update methods for later value and parameter changes.
- Protect existing owners from implicit overwrite.
- Allow owner change only through an explicit and intentional mechanism.

## Public API Direction
The preferred public operations should converge on:
- `SubscribeAsync(...)`
- `PublishSnapshotAsync(Item item, owner: ..., retained: true)`
- `UpdateValueAsync(Item item, retained: false)`
- `UpdateParameterAsync(Item item, string parameterName, retained: false)`
- `RemoveAsync(...)`

The standard user model remains:
- snapshot establishes the known item and owner
- later updates use `Update...`
- broker internally decides how to process the update based on sender and ownership

## Ownership Rules
- If no owner exists for an item path:
  - the first valid snapshot registers the owner
- If the same owner publishes the snapshot again:
  - allow it as a refresh/update of the same owned item
- If a different owner publishes a snapshot for an already owned item path:
  - reject it or return an explicit ownership conflict
- Do not silently overwrite an existing owner

## Ownership Transfer Direction
Owner change should not happen implicitly through normal snapshot publication. It should require an explicit path such as:
- unregister/remove by the current owner
- a dedicated transfer or claim mechanism
- another deliberately defined ownership-reset flow

The implementation plan should treat owner transfer as explicit behavior, not as a side effect of republishing snapshots.

## Naming Decisions Captured
- Public value/parameter changes should use `UpdateValueAsync` and `UpdateParameterAsync`.
- `PublishSnapshotAsync` remains the explicit full-state and ownership-registration operation.
- The public API should not force standard users to choose between `Publish...` and `Write...` for later value changes.

## Retention Direction
- Snapshot publishing should default to retained behavior.
- Update operations should default to non-retained behavior unless implementation confirms a strong reason to expose retained updates publicly.
- Ownership protection is more important than adding broad retention flexibility.

## Interface Direction
- If the current `IItemServer` shape does not match this simplified and ownership-safe model, it should be removed or fully redefined.
- There should not be two equal public entry models.
- Internal low-level publish/write routing may remain, but it must not be the standard public programming model.

## Scope
Included:
- snapshot-plus-update public API design
- owner registration and owner conflict protection
- unified update behavior for standard callers
- documentation, demos, wrappers, and tests aligned to the new model

Excluded:
- unrelated broker feature work
- transport expansion
- ownership transfer feature implementation unless required to make the design coherent

## Major Decisions To Validate During Implementation
- The exact form of the `owner` argument on `PublishSnapshotAsync`
- How ownership conflict is reported: exception, result object, ack, or rejection response
- Whether `RemoveAsync(...)` can release ownership and under which conditions
- Whether `SubscribeAsync(...)` and `RemoveAsync(...)` remain item-based, path-based, or mixed for practical reasons

## Suggested Sequence
1. Inventory current public broker, session, wrapper, and demo methods that rely on old publish/write semantics.
2. Define the new public contract around snapshot registration plus unified updates.
3. Add explicit ownership rules and conflict behavior to the design.
4. Refactor broker internals so snapshots register owners safely and updates are resolved using sender identity.
5. Ensure non-owner snapshot publication on an owned path is rejected clearly.
6. Update demos, tests, wrappers, and docs to the ownership-safe public model.

## Success Criteria
- Standard callers use snapshot registration and unified update methods.
- Existing owners cannot be overwritten implicitly by another snapshot publisher.
- Ownership conflicts are explicit and deterministic.
- `UpdateValueAsync` and `UpdateParameterAsync` stay simpler for users than explicit public publish/write split semantics.
- The public API remains easy to explain and harder to misuse.

# PLAN

## Topic
Refine `Amium.Item.Server` to a single slim public API centered on existing `Item` instances instead of free-form path strings.

## Goal
Define a minimal public broker API that prevents accidental ghost paths, keeps one preferred way per operation, and makes publish versus write semantics explicit.

## Problem Framing
The previous API direction still left a core risk: public methods that accept arbitrary `string path` values allow callers to publish or write to paths that do not correspond to a real known `Item`. That creates room for accidental broker state, typo-driven paths, and ambiguous ownership.

If the intended domain model is "operations act on existing items that already carry their own canonical path", then a free-form path-based public API is too permissive for the main usage path.

## Agreed Direction
- Keep one public main API.
- Base the normal public API on `Item`, not `string path`.
- Use `Item` to reduce ghost paths and typo-driven broker entries.
- Keep publish and write semantics clearly separate.
- Remove overlapping or ambiguous publish methods.

## Public API Direction
The preferred public operations should converge on:
- `SubscribeAsync(...)`
- `PublishSnapshotAsync(Item item, bool retained = true)`
- `PublishValueAsync(Item item, bool retained = false)`
- `PublishParameterAsync(Item item, string parameterName, bool retained = false)`
- `WriteValueAsync(Item item)`
- `WriteParameterAsync(Item item, string parameterName)`
- `RemoveAsync(Item item)` or another equally item-centered removal path to avoid reintroducing free-form public path entry

## Naming Decisions Captured
- `PublishItemAsync` should be removed from the public API.
- `PublishSnapshotAsync(Item item, bool retained = true)` becomes the explicit full-state publish path.
- `PublishValueAsync(Item item, bool retained = false)` becomes the explicit value-delta publish path.
- `PublishParameterAsync(Item item, string parameterName, bool retained = false)` becomes the explicit parameter-delta publish path.
- `WriteValueAsync(Item item)` and `WriteParameterAsync(Item item, string parameterName)` are the preferred write names.

## Retention Direction
- Retention should be visible only on publish operations.
- Snapshot publishing should default to `retained = true`.
- Value and parameter delta publishing should default to `retained = false`.
- Write operations should not expose `retained`, because write requests are commands, not retained state.

## Interface Direction
- If the current `IItemServer` only fits a transport-oriented or path-oriented design, it should be removed or fully redefined as the new item-centered public API.
- There should not be two equal public entry models.
- If path-based or message-based contracts are still required for infrastructure, adapters, or tests, they should not remain the normal public programming model.

## Scope
Included:
- redesign of the public broker API around `Item`
- removal of `PublishItemAsync`
- alignment of publish and write names with the new item-based surface
- review of remove and subscribe flows for consistency with the no-ghost-path goal
- updates to docs, demos, tests, and wrappers

Excluded:
- unrelated broker feature work
- transport protocol expansion
- deep item model redesign outside what this API change needs

## Major Decision To Validate During Implementation
The main open architecture check is whether subscribe and remove can also be made item-centered cleanly, or whether they still need a narrower path-based exception. The implementation should prefer consistency with the "no ghost paths" rule and only keep path-based access if there is a clear unavoidable use case.

## Suggested Sequence
1. Inventory all public broker, session, and wrapper methods that currently accept free-form paths.
2. Define the minimal item-centered contract for publish, write, and remove operations.
3. Decide how subscribe should fit the item-centered rule without harming valid monitoring scenarios.
4. Remove `PublishItemAsync` and migrate callers to explicit snapshot or delta operations.
5. Refactor implementation details so normal callers no longer construct transport-shaped messages or provide paths manually.
6. Update demos, tests, and documentation to the single item-centered public path.

## Success Criteria
- Normal public broker usage no longer requires free-form path strings for publish and write operations.
- `PublishItemAsync` is removed from the main public API.
- `PublishSnapshotAsync(Item item, bool retained = true)` is the explicit full-state publish method.
- `PublishValueAsync` and `PublishParameterAsync` use `Item` and retained defaults appropriate to deltas.
- `WriteValueAsync` and `WriteParameterAsync` are explicit and do not expose retention.
- The public API reduces the chance of accidental ghost paths.

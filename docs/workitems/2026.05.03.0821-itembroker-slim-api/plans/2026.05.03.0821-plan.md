# PLAN

## Topic
Slim down the public `Amium.Item.Server` API so users work with one clear broker surface instead of transport-shaped message construction.

## Goal
Define a single preferred public API that exposes only the operations users actually need, while moving message-shaped transport concerns out of the normal usage path.

## Problem Framing
The current broker surface is centered around `Item...Message` records. That keeps the internals explicit, but it makes normal usage verbose and exposes transport metadata such as `Timestamp`, `CorrelationId`, and repeated `SourceClientId` handling even when callers do not make a meaningful choice about them.

This creates three problems:
- the default usage path is heavier than the domain requires
- transport concerns are mixed into the public programming model
- there are too many chances for inconsistent usage patterns

## Target Direction
- Keep exactly one preferred public broker API.
- Make the public API operation-oriented, not message-construction-oriented.
- Preserve the semantic distinction:
  - `Publish...` means a state change already happened
  - `Write...` means a state change is requested
- Prefer minimal user inputs:
  - `path`
  - changed `value`
  - optional `parameterName`
  - optional subscription options only when defaults are not enough
- Do not require users to pass `Timestamp`, `CorrelationId`, or repeated `SourceClientId` in the standard path.

## Proposed Public Shape
The intended main surface should center around methods such as:
- `SubscribeAsync`
- `PublishSnapshotAsync`
- `PublishValueAsync`
- `PublishParameterAsync`
- `WriteValueAsync`
- `WriteParameterAsync`
- `RemoveAsync`

The API should prefer one dominant style. Because the broker already routes and stores by canonical path, `path` should remain the primary public identifier.

## Architecture Direction
- Replace or reshape the current `IItemServer` if its current message-centric form is no longer needed.
- Avoid two equal public entry points.
- Message records may remain as internal or infrastructure contracts if still needed by implementation details, adapters, or tests.

## Scope
Included:
- public broker API design cleanup
- naming cleanup around publish vs write semantics
- reduction of user-visible transport metadata
- call-site alignment in the current repository

Excluded:
- new broker features unrelated to API simplification
- retention or routing behavior changes unless required by the new API shape
- new transport protocols

## Major Decisions To Carry Into Implementation
- There should be one public main API.
- If the current `IItemServer` shape is only a technical transport surface, it should be removed or fully repurposed.
- `WriteValueAsync` and `WriteParameterAsync` are the preferred command-style names.
- `PublishValueAsync` and `PublishParameterAsync` remain event/state-update style names.

## Suggested Sequence
1. Inventory all current public broker entry points and all current call sites.
2. Design the minimal public contract and decide whether `IItemServer` keeps its name.
3. Refactor the implementation to use the new contract as the only public path.
4. Move message-shaped APIs behind the public surface or reduce their visibility if possible.
5. Update demos, tests, and documentation to the new usage pattern.

## Success Criteria
- A normal caller can publish, write, subscribe, and remove without constructing message records.
- The public API presents one obvious way per operation.
- Publish and write semantics are clearly separated by naming and behavior.
- Existing in-repo callers compile against the new shape with simpler code.

# PLAN

## Topic
Align HornetStudio and BrokerWidget with the slim ItemBroker API.

## Problem
The slim ItemBroker API refactor was implemented for the broker, MQTT client/session layers, and demo clients, but HornetStudio still publishes BrokerWidget items mainly as repeated snapshots. Published widget items no longer react reliably because the new broker model expects snapshot publication to establish ownership and later changes to use `UpdateValueAsync` or `UpdateParameterAsync`.

## Scope
Included:
- HornetStudio BrokerWidget publishing flow.
- Host-side broker client facade used by the editor.
- MQTT remote client facade used by Hornetstudio.
- Write-back and writable item behavior around broker-owned items.
- Focused tests or verification for widget publish/update behavior.

Excluded:
- A broad ItemBroker redesign.
- New transports beyond the existing MQTT path.
- Unrelated BrokerWidget UI changes.
- Release/versioning work.

## Current Structure
- `InMemoryItemServer` requires `PublishSnapshotAsync` with a source client id to register ownership.
- Owner updates should use `UpdateValueAsync` or `UpdateParameterAsync`.
- `MqttItemServerAdapter` maps incoming MQTT topic publishes to broker updates.
- `BrokerClientControl.HostItemBrokerPublisher` currently republishes snapshots for initial, interval, and changed local items.
- `IHostItemBrokerClient`, `HostItemBrokerClient`, and `MqttRemoteItemClient` currently expose snapshot publish and subscribe, but not update methods.

## Target Structure
- Initial BrokerWidget publication registers ownership with `PublishSnapshotAsync`.
- Later local value/parameter changes from the same widget flow through `UpdateValueAsync` or `UpdateParameterAsync`.
- Structural changes that introduce new broker paths publish snapshots for those new paths.
- Host-side client facades expose the same minimal publish/update/subscribe vocabulary as the new broker API.
- Writable remote updates keep a clear separation between owner updates and non-owner write requests.

## Main Decisions
1. Keep the existing MQTT value/parameter topic layout for this pass.
2. Extend HornetStudio client facades to expose update methods rather than bypassing them.
3. Track published broker paths in the BrokerWidget publisher so it can decide between first snapshot registration and later updates.
4. Preserve existing BrokerWidget UI behavior and only change broker API usage.
5. Treat full snapshot refresh as owner registration/structure synchronization, not as the default path for every value change.

## Planned Sequence
1. Audit the exact publish/update surfaces used by `BrokerClientControl`, `HostItemBrokerClient`, `MqttRemoteItemClient`, and `MqttItemClientSession`.
2. Extend `IHostItemBrokerClient` and implementations with `UpdateValueAsync` and `UpdateParameterAsync`.
3. Add matching methods to `MqttRemoteItemClient` and forward them to `MqttItemClientSession`.
4. Refactor `HostItemBrokerPublisher` so initial publish registers snapshots and later local changes send updates where possible.
5. Keep snapshot publication for new paths or structural refreshes that need owner registration.
6. Review writable handling so write-back does not accidentally echo or replace ownership.
7. Add or update focused tests for BrokerWidget publish/update decisions and MQTT client facade behavior where feasible.
8. Build the affected projects and document any remaining limitations.

## Acceptance Criteria
- BrokerWidget initial published items are registered through `PublishSnapshotAsync`.
- BrokerWidget value changes after registration use `UpdateValueAsync`.
- BrokerWidget parameter changes after registration use `UpdateParameterAsync` where the changed parameter can be identified.
- New child paths or structural additions still register ownership through snapshots.
- HornetStudio host-side broker client exposes the new API vocabulary.
- The existing MqttDemoWinForms behavior remains intact.
- Relevant builds/tests pass or any blockers are documented.

## Verification Plan
- Build `src/Hornetstudio.Editor/Hornetstudio.Editor.csproj`.
- Build or test `src/Amium.Item.Server.Tests/Amium.Item.Server.Tests.csproj` if still applicable.
- Run `src/Hornetstudio.Editor.Tests/Hornetstudio.Editor.Tests.csproj` if the current test harness supports it.
- Manually verify BrokerWidget publish flow against the demo MQTT broker:
  - connect BrokerWidget
  - publish a local item
  - change the local value
  - confirm the remote side reacts
  - verify writable updates still apply back to local items

## Risks
- The MQTT topic layout does not explicitly distinguish snapshot registration from update messages, so ownership registration may still require careful client-side sequencing.
- Repeated full snapshots may hide missing update handling if retained topics are observed directly.
- Parameter-level update detection may be limited when the local registry event does not expose the changed parameter clearly.
- Echo suppression must remain intact for writable items.

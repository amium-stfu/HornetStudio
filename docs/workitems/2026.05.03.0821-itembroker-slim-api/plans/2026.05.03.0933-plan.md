# PLAN

## Topic
Simplify `Amium.ItemBroker` to a single role-light public API centered on item snapshots plus item updates.

## Goal
Define a clean public broker API where users do not need to decide between `Publish` and `Write` during normal value changes. Ownership should be established when an item snapshot is published, and later changes should flow through a unified update API.

## Problem Framing
The previous direction still required users to understand and actively choose between two concepts:
- `Publish...` when acting as the owner
- `Write...` when requesting a change as a non-owner

This is semantically sound, but it increases mental overhead. When one client publishes an item and another client subscribes to the same item, it is not obvious from the API alone whether a later value change should use publish or write. That makes the public model harder to learn and easier to misuse.

## Agreed Direction
- Keep one public main API.
- Remove the need for users to choose between `PublishValue...` and `WriteValue...` in the standard path.
- Establish ownership when publishing a snapshot.
- Use a unified update API for later value and parameter changes:
  - `UpdateValue`
  - `UpdateParameter`
- Pass sender information internally on update calls so the broker can decide how to route or apply the change.

## Public API Direction
The preferred public operations should converge on:
- `SubscribeAsync(...)`
- `PublishSnapshotAsync(Item item, owner: ..., retained: true)`
- `UpdateValueAsync(Item item, retained: false)`
- `UpdateParameterAsync(Item item, string parameterName, retained: false)`
- `RemoveAsync(...)`

The user-facing semantics should be:
- Snapshot registers or refreshes the known item and its owner.
- Update changes the item state from the caller's point of view.
- The broker internally decides whether the update is owner-authoritative or must be treated like a routed write/request.

## Ownership Direction
- Ownership should be explicit at snapshot registration time.
- Later updates should always carry sender/client identity internally.
- The broker becomes responsible for deciding:
  - owner update -> apply as direct state update
  - non-owner update -> route as request, reject, or otherwise handle according to broker rules

## Naming Decisions Captured
- Public `PublishValue...` and `WriteValue...` should not both remain as the normal main API.
- Public `PublishParameter...` and `WriteParameter...` should not both remain as the normal main API.
- `UpdateValueAsync` and `UpdateParameterAsync` are the new preferred public names for normal item changes.
- `PublishSnapshotAsync` remains the explicit item registration/full-state operation.

## Retention Direction
- Snapshot publishing should default to retained behavior.
- Update operations should default to non-retained behavior unless there is a confirmed need to expose retained updates publicly.
- Retention should stay minimal in the public API and not complicate command-style semantics.

## Interface Direction
- If the current `IItemBroker` shape no longer matches this user-facing model, it should be removed or fully redefined.
- There should not be two equal public entry models.
- Low-level `Publish` and `Write` concepts may remain internally if the broker still needs them for routing and implementation, but they should no longer define the standard public usage path.

## Scope
Included:
- redesign of the public broker API around snapshot plus update operations
- ownership registration through snapshot publishing
- simplification of later item changes into unified update operations
- updates to demos, wrappers, docs, and tests

Excluded:
- unrelated broker feature work
- transport protocol expansion
- major policy redesign beyond what unified updates require

## Major Decisions To Validate During Implementation
- How explicit the `owner` parameter on `PublishSnapshotAsync` should be in the final contract.
- Whether `UpdateValueAsync` and `UpdateParameterAsync` should expose retained flags or keep retention fully internal.
- Whether `SubscribeAsync` and `RemoveAsync` stay item-based, path-based, or mixed for practical reasons.

## Suggested Sequence
1. Inventory current public broker, session, wrapper, and demo methods that expose publish/write split semantics.
2. Define the new public contract around snapshot registration plus unified update operations.
3. Decide how owner identity is supplied and stored during snapshot publication.
4. Refactor broker internals so update calls always include sender identity and are resolved according to ownership.
5. Remove overlapping public publish/write value and parameter APIs from the standard path.
6. Update demos, tests, wrappers, and docs to use the new snapshot plus update model.

## Success Criteria
- Standard callers no longer choose between public publish versus write methods for normal value and parameter changes.
- `PublishSnapshotAsync` remains the explicit full-state and ownership-registration operation.
- `UpdateValueAsync` and `UpdateParameterAsync` become the preferred public change operations.
- Ownership and sender handling move behind the API boundary.
- The public API becomes simpler to explain and harder to misuse.

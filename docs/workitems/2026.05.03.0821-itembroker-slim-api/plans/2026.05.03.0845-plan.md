# PLAN

## Topic
Consolidate `Amium.Item.Server` into one slim public API with one preferred way per operation.

## Goal
Define a minimal public broker surface that removes ambiguous or duplicate entry points and keeps transport-oriented details out of normal user code.

## Problem Framing
The broker API still exposes multiple concepts that overlap or are too technical for standard usage:
- message-shaped public operations
- multiple publish shapes with similar meaning
- technical metadata visible in common call sites
- unclear boundary between command-style writes and event-style publishes

The clearest example is the coexistence of snapshot publishing and higher-level item publishing:
- `PublishSnapshotAsync(...)`
- `PublishItemAsync(...)`

These names are too close while their behavior is not identical. That increases ambiguity for users and works against the agreed design goal: as much as necessary, as little as possible.

## Agreed Direction
- Keep one public main API.
- Remove duplicate or ambiguous public paths where possible.
- Preserve strict semantic separation:
  - `Publish...` reports a state that already changed
  - `Write...` requests a state change
- Prefer path-oriented operations as the dominant public style.
- Keep item snapshot publishing explicit.

## Public API Direction
The preferred public operations should converge on:
- `SubscribeAsync(...)`
- `PublishSnapshotAsync(item, retained: true)`
- `PublishValueAsync(path, value)`
- `PublishParameterAsync(path, parameterName, value)`
- `WriteValueAsync(path, value)`
- `WriteParameterAsync(path, parameterName, value)`
- `RemoveAsync(path)`

## Naming Decisions Captured
- `WriteValueAsync` is preferred over `WriteItemValueAsync` and any `Request`-based naming for the normal public API.
- `WriteParameterAsync` is the matching parameter command.
- `PublishItemAsync` should be removed from the public API.
- `PublishSnapshotAsync(item, retained: true)` should be the explicit snapshot path.

## Retention Direction
- Retention should stay minimal in the public surface.
- For snapshots, a visible default of `retained: true` is acceptable because retained snapshots are a core and understandable behavior.
- Retention should not be added as a broad `bool` across all publish operations without a strong need.

## Interface Direction
- If the current `IItemServer` only exists as a transport-shaped surface, it should be removed or fully redefined as the new slim public API.
- There should not be two equal public entry models.
- Low-level message contracts may remain only where implementation, adapters, or tests still require them.

## Scope
Included:
- simplification of the public broker API
- removal of overlapping public publish methods
- alignment of demos, client session, and docs with the new main API
- clarification of write vs publish semantics

Excluded:
- unrelated broker feature work
- transport protocol additions
- broad policy redesign unrelated to the new API surface

## Suggested Sequence
1. Inventory current public methods and all current usages of `PublishItemAsync`, `PublishSnapshotAsync`, `WriteAsync`, and direct message construction.
2. Redefine the broker contract to the minimal public operation set.
3. Remove `PublishItemAsync` from the public programming model and migrate callers to explicit snapshot/value/parameter operations.
4. Refactor implementation details so timestamps, correlation defaults, and source client handling stay out of the normal path as much as possible.
5. Update demos, tests, and documentation to use the single preferred API path.

## Success Criteria
- There is one obvious public way per operation.
- `PublishItemAsync` is no longer part of the public main API.
- Snapshot publishing is explicit through `PublishSnapshotAsync(item, retained: true)`.
- Write commands are explicit through `WriteValueAsync` and `WriteParameterAsync`.
- Normal callers no longer need transport-shaped message construction.

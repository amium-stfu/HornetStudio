# IMPLEMENTATION HANDOFF

## Goal
Refactor `Amium.ItemBroker` so the main public API operates on existing `Item` instances instead of free-form path strings, reducing accidental ghost paths and making the API smaller and safer.

## Scope
Included:
- shift the public broker API from path-based publish/write calls to item-based calls
- remove `PublishItemAsync` from the public programming model
- standardize explicit publish methods:
  - `PublishSnapshotAsync(Item item, bool retained = true)`
  - `PublishValueAsync(Item item, bool retained = false)`
  - `PublishParameterAsync(Item item, string parameterName, bool retained = false)`
- standardize explicit write methods:
  - `WriteValueAsync(Item item)`
  - `WriteParameterAsync(Item item, string parameterName)`
- evaluate `RemoveAsync` and `SubscribeAsync` for alignment with the item-centered rule
- migrate repository callers, docs, and tests

Excluded:
- unrelated broker feature additions
- protocol or transport expansion
- broad redesign of `Amium.Items` beyond what the API change requires

## Starting Point
- Existing workitem: `docs/workitems/2026.05.03.0821-itembroker-slim-api/`
- Earlier plan versions assumed a path-centered public API.
- Chat direction has changed: the main public API should prefer `Item` to avoid ghost paths from arbitrary strings.
- Current docs still show path-based examples, for example in `docs/amium-itembroker-usage.md`.
- Current implementations and wrappers likely still expose path-based publish helpers and `PublishItemAsync`.

## Tasks
1. Audit all current public broker, client-session, and wrapper APIs that accept `string path` for normal publish/write usage.
2. Define the new item-centered main contract and decide whether `IItemBroker` is removed or repurposed as that single public interface.
3. Remove `PublishItemAsync` from the public main API and replace usages with explicit snapshot/value/parameter publish operations.
4. Refactor publish methods so they derive the target path from `item.Path` and validate that the item provides a usable canonical path.
5. Implement retained defaults only on publish methods:
   - snapshot default `true`
   - value/parameter default `false`
6. Keep write methods free of retention parameters.
7. Review `RemoveAsync` and `SubscribeAsync`:
   - prefer item-centered usage where it fits the domain
   - only keep path-based public usage if there is a clear required scenario that cannot be modeled safely with `Item`
8. Update demos, docs, tests, and wrappers to the final item-centered API.

## File-Level Changes
- `src/Amium.ItemBroker/Contracts.cs` -> redefine the public contract around item-based publish/write methods.
- `src/Amium.ItemBroker/InMemoryItemBroker.cs` -> adapt implementation to resolve broker paths from `Item` and support retained defaults on publish operations.
- `src/Amium.ItemBroker/Messages.cs` -> keep only as internal/infrastructure contracts where still required.
- `src/Amium.ItemBroker.Client/IItemBrokerClientSession.cs` -> align the session surface with the new item-based API.
- `src/Amium.ItemBroker.Client/ItemBrokerClientSession.cs` -> remove `PublishItemAsync`, update publish/write calls to item-based forms, and preserve clear snapshot versus delta behavior.
- `src/Amium.ItemBroker.Mqtt.Client/*` -> adjust remote/session wrappers that currently expose item-plus-path or path-based publish APIs.
- `src/Amium.ItemBroker.MqttDemoWinForms/Controllers/*` -> migrate demos to the item-based API.
- `docs/amium-itembroker-usage.md` -> rewrite examples to the new item-centered surface.
- affected tests in broker, host, and editor projects -> update call sites and assertions.

## Implementation Order
1. Finalize the target public contract.
2. Implement item-based publish/write operations in the broker and session layers.
3. Remove overlapping public methods such as `PublishItemAsync`.
4. Update callers, wrappers, and demos.
5. Update tests and documentation.

## Technical Constraints
- Keep the public API minimal and offer one obvious way per operation.
- Publish methods and write methods must remain semantically distinct.
- Do not expose `retained` on write operations.
- Validate or normalize `item.Path` internally so callers do not pass redundant path data.
- Keep code, comments, documentation, and identifiers in English.
- Avoid unrelated structural changes.

## Acceptance Criteria
- Standard publish and write calls use `Item` instead of public free-form `string path` parameters.
- `PublishItemAsync` is removed from the public main API.
- `PublishSnapshotAsync(Item item, bool retained = true)` is the explicit snapshot method.
- `PublishValueAsync(Item item, bool retained = false)` and `PublishParameterAsync(Item item, string parameterName, bool retained = false)` exist and are used consistently.
- `WriteValueAsync(Item item)` and `WriteParameterAsync(Item item, string parameterName)` exist and do not expose retention.
- Updated repository code no longer encourages ghost paths through normal publish/write usage.

## Verification
- Build the affected broker, client, and demo projects.
- Run the affected tests.
- Review documentation and examples to confirm the item-based API is the single preferred public path.
- Manually verify that current demo workflows still publish, subscribe, and write correctly after migration.

## Out of Scope
- New retention modes or transport features.
- Redesign of item subscription semantics beyond what is necessary to fit the final chosen API.
- Broader item model changes unrelated to the broker API simplification.

## Risks / Watchouts
- Some scenarios may genuinely require path-based subscription or remote targeting even if publish/write become item-based.
- Existing wrappers may currently rely on explicit path overrides for repathing or external remapping.
- If `item.Path` is missing or inconsistent, the new API must fail clearly instead of silently creating unexpected broker state.

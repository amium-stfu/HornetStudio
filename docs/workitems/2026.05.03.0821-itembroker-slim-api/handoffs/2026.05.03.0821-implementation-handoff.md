# IMPLEMENTATION HANDOFF

## Goal
Refactor `Amium.ItemBroker` so consumers use one slim public broker API instead of constructing transport-oriented message records for normal operations.

## Scope
Included:
- redesign the public broker surface around minimal operation-based methods
- remove or repurpose the current message-centric `IItemBroker` shape if it is no longer needed
- keep clear semantic separation between publish operations and write operations
- update current in-repo callers, demos, and tests to the new API

Excluded:
- unrelated broker features
- behavior changes in retention, routing, or coercion unless needed to support the API refactor
- new transports or protocol work

## Starting Point
- Current public broker usage is centered around `Item...Message` records in `src/Amium.ItemBroker/Messages.cs`.
- Current broker contract is defined in `src/Amium.ItemBroker/Contracts.cs`.
- Current demo callers construct messages directly, for example in `src/Amium.ItemBroker.MqttDemoWinForms/Controllers/WritableDemoController.cs`.
- The design goal agreed in chat is: as much as necessary, as little as possible.
- There should be one preferred public API path, not parallel equal entry points.

## Tasks
1. Review current public broker methods and identify which transport-specific parameters can be removed from the standard public path.
2. Define the new main contract around operation-style methods:
   `SubscribeAsync`, `PublishSnapshotAsync`, `PublishValueAsync`, `PublishParameterAsync`, `WriteValueAsync`, `WriteParameterAsync`, and `RemoveAsync`.
3. Decide whether the `IItemBroker` name is kept for the new main contract or whether the existing interface is removed and replaced.
4. Refactor broker implementation and any supporting abstractions so timestamps and other technical metadata are assigned internally on the standard path.
5. Update current repository callers to the new API and remove direct message construction from normal usage.
6. Keep or reduce message records as internal/infrastructure contracts only where still required.
7. Update documentation and tests to reflect the single public way of using the broker.

## File-Level Changes
- `src/Amium.ItemBroker/Contracts.cs` -> redefine the public broker contract around the slim operation-based API.
- `src/Amium.ItemBroker/InMemoryItemBroker.cs` -> implement the new public contract and internalize metadata handling where appropriate.
- `src/Amium.ItemBroker/Messages.cs` -> reduce public prominence, adapt visibility/usage if feasible, and keep only what is needed internally.
- `src/Amium.ItemBroker.MqttDemoWinForms/Controllers/WritableDemoController.cs` -> switch demo usage to the new publish/write API.
- `src/Amium.ItemBroker.MqttDemoWinForms/Controllers/DemoPublisherController.cs` -> align current publish usage with the new API if it still constructs transport messages directly.
- `src/Amium.ItemBroker` tests and related callers -> update to the new contract and add coverage for publish vs write semantics.
- Markdown docs under the broker area -> update examples and usage guidance.

## Implementation Order
1. Finalize the new contract shape in `Contracts.cs`.
2. Adapt `InMemoryItemBroker` and supporting internals to satisfy the new contract.
3. Migrate demos and in-repo callers off direct message construction.
4. Clean up message exposure and remove obsolete broker API members.
5. Update tests and documentation.

## Technical Constraints
- Follow existing project conventions and keep code, comments, and identifiers in English.
- Keep the public API minimal: one obvious way per operation.
- Preserve the semantic difference between publish and write operations.
- Do not broaden scope into unrelated broker redesign.
- Prefer path-based public operations as the dominant style.

## Acceptance Criteria
- Consumers can publish values and parameters without constructing `ItemValueChangedMessage` or `ItemParameterChangedMessage`.
- Consumers can request writes through `WriteValueAsync` and `WriteParameterAsync`.
- The public API no longer requires standard callers to pass `Timestamp`, `CorrelationId`, or repeated `SourceClientId`.
- There is one clear public broker surface rather than two equal public entry paths.
- Current repository callers compile and use the simplified API.

## Verification
- Build the relevant solution or broker projects successfully.
- Run affected broker tests and any demo-related tests if available.
- Manually review updated demo/controller code to confirm direct message construction is removed from the standard path.

## Out of Scope
- Introducing new publish modes, retention features, or transport adapters.
- Redesigning item path normalization behavior.
- Reworking `Amium.Items` model semantics beyond what the new broker API strictly requires.

## Risks / Watchouts
- `IItemBroker` may already be referenced broadly; removing or reshaping it can create wider ripple effects.
- Message records might still be needed by tests or transport adapters even if they are no longer the main public API.
- Over-simplifying the API must not blur the difference between a write request and a published state change.

# IMPLEMENTATION HANDOFF

## Goal
Refactor `Amium.Item.Server` to a slim public API based on snapshot registration plus unified updates, and enforce owner protection so existing item owners cannot be overwritten implicitly.

## Scope
Included:
- public API centered on:
  - `PublishSnapshotAsync(...)`
  - `UpdateValueAsync(...)`
  - `UpdateParameterAsync(...)`
  - `SubscribeAsync(...)`
  - `RemoveAsync(...)`
- owner registration through snapshot publication
- unified update handling after ownership is established
- explicit rejection or conflict handling when a different owner attempts to republish an already owned item
- updates to wrappers, demos, docs, and tests

Excluded:
- unrelated broker feature additions
- transport expansion
- implementing a full ownership transfer feature unless required by the refactor

## Starting Point
- Existing workitem: `docs/workitems/2026.05.03.0821-itembroker-slim-api/`
- Current direction in chat:
  - snapshot registers the item and owner
  - later value and parameter changes use unified `Update...` methods
  - callers should not need to choose between public publish and write methods
- Newly added constraint:
  - once an owner exists for an item path, another owner must not be able to overwrite it implicitly through snapshot publication

## Tasks
1. Audit the current broker, client-session, wrapper, and demo surfaces for publish/write responsibilities and ownership assumptions.
2. Define the target public contract around `PublishSnapshotAsync`, `UpdateValueAsync`, `UpdateParameterAsync`, `SubscribeAsync`, and `RemoveAsync`.
3. Decide the exact owner-registration signature for `PublishSnapshotAsync`.
4. Implement broker-side ownership tracking so:
   - first owner registration succeeds
   - same owner refresh succeeds
   - different owner on the same item path is rejected clearly
5. Refactor update handling so sender identity is always available internally and ownership-aware routing still works behind the public API.
6. Decide how ownership release works, especially whether `RemoveAsync` by the current owner can free the path.
7. Update demos, wrappers, tests, and documentation to the new ownership-safe public model.

## File-Level Changes
- `src/Amium.Item.Server/Contracts.cs` -> redefine the public contract around snapshot registration, updates, and ownership-safe behavior.
- `src/Amium.Item.Server/InMemoryitem_broker.cs` -> implement owner registration, conflict rejection, and sender-aware unified updates.
- `src/Amium.Item.Server/Messages.cs` -> keep or adapt low-level routing messages only as internal implementation details if still required.
- `src/Amium.Item.Client/IItemClientSession.cs` -> align the client session contract with the new snapshot-plus-update model.
- `src/Amium.Item.Client/ItemClientSession.cs` -> remove standard public publish/write split usage and align to ownership-safe snapshots and updates.
- `src/Amium.Item.Client.Mqtt/*` -> adapt wrappers or remote client surfaces that currently expose the old split or weak ownership assumptions.
- `src/Amium.Item.Server.MqttDemoWinForms/Controllers/*` -> migrate demos to the new API and validate ownership behavior.
- `docs/amium-itembroker-usage.md` -> rewrite usage and ownership explanation.
- affected tests across broker, host, editor, and demo areas -> add coverage for ownership registration and ownership conflict rejection.

## Implementation Order
1. Finalize the public contract and ownership semantics.
2. Implement ownership tracking and snapshot conflict handling in the broker.
3. Implement or adapt unified update handling on top of sender-aware internals.
4. Update wrappers, demos, and client/session usage.
5. Update tests and documentation.

## Technical Constraints
- Keep the public API minimal and easy to explain.
- Do not require standard callers to distinguish between public publish and write methods for later item changes.
- Existing owners must not be overwritten implicitly.
- Ownership conflict behavior must be explicit and deterministic.
- Keep code, docs, identifiers, comments, and XML docs in English.
- Avoid unrelated refactors.

## Acceptance Criteria
- `PublishSnapshotAsync` registers owner information for new item paths.
- Republishing a snapshot from the same owner succeeds.
- Republishing a snapshot from a different owner on the same item path is rejected clearly.
- Standard callers use `UpdateValueAsync` and `UpdateParameterAsync` for later changes.
- Sender identity is captured internally for update handling.
- Updated repository callers, demos, and docs consistently reflect the ownership-safe model.

## Verification
- Build the affected broker, client, and demo projects.
- Run the affected tests.
- Add or update tests for:
  - first owner registration
  - same-owner refresh
  - different-owner conflict
  - owner and non-owner update flows
- Review documentation and demos to confirm ownership is explained clearly.

## Out of Scope
- Full ownership transfer workflows unless they are required to complete the refactor safely.
- New transport features.
- Broader redesign outside the broker API simplification.

## Risks / Watchouts
- If ownership conflict reporting is vague, the simplified API may still be hard to diagnose in real systems.
- Some existing flows may have relied on silent ownership replacement without making that explicit.
- The design should avoid backing into an implicit transfer mechanism through `RemoveAsync` or reconnect scenarios unless intentionally defined.

# IMPLEMENTATION HANDOFF

## Goal
Refactor `Amium.Item.Server` so the standard public API uses explicit snapshot registration plus unified update operations, removing the need for callers to choose between public publish and write methods for normal item changes.

## Scope
Included:
- redesign the public broker API around:
  - `PublishSnapshotAsync(...)`
  - `UpdateValueAsync(...)`
  - `UpdateParameterAsync(...)`
  - `SubscribeAsync(...)`
  - `RemoveAsync(...)`
- establish item ownership when publishing snapshots
- pass sender identity internally on update operations
- let the broker internally decide whether an update is an owner-authoritative state update or a non-owner change request
- update callers, demos, tests, and docs to the new API model

Excluded:
- unrelated broker feature work
- transport expansion
- broad architecture changes outside the broker API simplification

## Starting Point
- Existing workitem: `docs/workitems/2026.05.03.0821-itembroker-slim-api/`
- Earlier plans in this workitem explored path-based and item-based publish/write APIs.
- Current chat direction has shifted again: the main public API should be easier than explicit `Publish` versus `Write` choice for normal changes.
- The desired user model is:
  - publish a snapshot to register the item and owner
  - use update methods afterward regardless of caller role
  - let the broker resolve owner versus non-owner behavior internally

## Tasks
1. Audit current public broker, client-session, and wrapper APIs that expose separate publish/write methods for value and parameter changes.
2. Define the target public contract around snapshot registration and unified updates.
3. Decide the final public signature for snapshot registration, including how owner identity is supplied.
4. Update broker internals so update operations always carry sender identity and ownership can be checked centrally.
5. Preserve or introduce internal routing that distinguishes:
   - owner update -> direct applied update
   - non-owner update -> routed request or rejection according to broker rules
6. Remove public ambiguity by replacing standard `PublishValue...`, `PublishParameter...`, `WriteValue...`, and `WriteParameter...` usage with `UpdateValue...` and `UpdateParameter...`.
7. Update demos, wrappers, tests, and docs to the new public model.

## File-Level Changes
- `src/Amium.Item.Server/Contracts.cs` -> redefine the public API around snapshot and update operations.
- `src/Amium.Item.Server/InMemoryitem_broker.cs` -> implement ownership-aware unified update handling.
- `src/Amium.Item.Server/Messages.cs` -> keep or adapt low-level publish/write message contracts only as internal implementation details if still needed.
- `src/Amium.Item.Client/IItemClientSession.cs` -> align the session contract with the new public model.
- `src/Amium.Item.Client/ItemClientSession.cs` -> remove normal public publish/write split usage in favor of snapshot plus update operations.
- `src/Amium.Item.Client.Mqtt/*` -> adapt wrappers or remote client surfaces that currently expose publish/write split APIs.
- `src/Amium.Item.Server.MqttDemoWinForms/Controllers/*` -> migrate demos to snapshot registration plus update usage.
- `docs/amium-itembroker-usage.md` -> rewrite usage examples and explanations around the new model.
- affected tests across broker, host, editor, and demo areas -> update call sites and assertions.

## Implementation Order
1. Finalize the new public contract and naming.
2. Implement owner registration and sender-aware update handling in broker internals.
3. Remove or de-emphasize the public publish/write split for value and parameter changes.
4. Update client/session wrappers and demos.
5. Update tests and documentation.

## Technical Constraints
- Keep the public API minimal and easy to explain.
- Do not require standard callers to decide between owner-publish and non-owner-write for normal value changes.
- Preserve clear ownership handling internally.
- Keep code, docs, identifiers, comments, and XML docs in English.
- Avoid unrelated refactors.

## Acceptance Criteria
- Standard public usage no longer requires choosing between public `PublishValue...` and `WriteValue...`.
- `PublishSnapshotAsync` is the explicit full-state and ownership-registration operation.
- `UpdateValueAsync` and `UpdateParameterAsync` are the preferred public methods for later changes.
- Sender identity is captured internally for update calls.
- The broker can distinguish owner updates from non-owner requests without exposing that complexity as the normal public API.
- Updated repository callers, demos, and docs consistently use the new model.

## Verification
- Build the affected broker, client, and demo projects.
- Run the affected tests.
- Review updated docs and demos to confirm the public model is explained as snapshot registration plus unified updates.
- Manually verify one owner flow and one non-owner update flow after migration.

## Out of Scope
- New transport features.
- Unrelated retention-policy redesign.
- Broader `Amium.Items` redesign beyond what this broker API change requires.

## Risks / Watchouts
- `Update...` is easier to use but semantically less explicit than separate publish/write methods, so ownership handling must be documented well.
- Existing routing behavior may rely heavily on explicit write messages and could need careful internal adaptation.
- Snapshot registration must clearly define whether it creates ownership, refreshes ownership, or merely refreshes retained state.

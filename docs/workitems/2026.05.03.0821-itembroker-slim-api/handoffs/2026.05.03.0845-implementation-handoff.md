# IMPLEMENTATION HANDOFF

## Goal
Refactor `Amium.Item.Server` to expose one slim public broker API and remove overlapping public publish concepts, especially `PublishItemAsync` versus explicit snapshot publishing.

## Scope
Included:
- define one public main API for broker operations
- remove `PublishItemAsync` from the public programming model
- standardize snapshot publishing on `PublishSnapshotAsync(item, retained: true)`
- standardize command-style writes on `WriteValueAsync` and `WriteParameterAsync`
- migrate current repository callers, docs, and tests

Excluded:
- unrelated broker feature additions
- protocol expansion
- deep retention-policy redesign beyond what the new API requires

## Starting Point
- Existing workitem: `docs/workitems/2026.05.03.0821-itembroker-slim-api/`
- Current broker contract: `src/Amium.Item.Server/Contracts.cs`
- Current broker implementation: `src/Amium.Item.Server/InMemoryitem_broker.cs`
- Current session convenience layer: `src/Amium.Item.Client/ItemClientSession.cs`
- Current docs include both `PublishSnapshotAsync` and `PublishItemAsync`, creating ambiguity.
- Agreed design rule: one main public API, as much as necessary, as little as possible.

## Tasks
1. Audit the current public broker and client-session surfaces for all publish and write entry points.
2. Define the target public API around these operations:
   `SubscribeAsync`, `PublishSnapshotAsync`, `PublishValueAsync`, `PublishParameterAsync`, `WriteValueAsync`, `WriteParameterAsync`, `RemoveAsync`.
3. Remove `PublishItemAsync` as a public main API method and replace its usage with explicit snapshot or delta calls.
4. Decide whether `IItemServer` is removed or repurposed as the single slim public interface.
5. Refactor implementation internals so the standard public path avoids message construction by callers.
6. Update demos, tests, remote/session wrappers, and docs to the new explicit API.
7. Reduce public exposure of low-level message-shaped usage where feasible.

## File-Level Changes
- `src/Amium.Item.Server/Contracts.cs` -> redefine the public broker contract around the slim API.
- `src/Amium.Item.Server/InMemoryitem_broker.cs` -> implement the new contract and keep snapshot/value/parameter/write semantics explicit.
- `src/Amium.Item.Client/IItemClientSession.cs` -> align the client session contract with the new public API.
- `src/Amium.Item.Client/ItemClientSession.cs` -> remove `PublishItemAsync` usage and migrate to explicit snapshot/value/parameter publishing.
- `src/Amium.Item.Client.Mqtt/*` -> update any wrappers or remote clients that still expose `PublishItemAsync`.
- `src/Amium.Item.Server.MqttDemoWinForms/Controllers/*` -> migrate demos to the explicit API.
- `docs/amium-itembroker-usage.md` -> document only the preferred API path.
- broker tests and host/editor tests -> update affected call sites and assertions.

## Implementation Order
1. Finalize the public contract and naming.
2. Update the broker implementation to support the final public contract.
3. Update client session and wrappers to remove overlapping publish methods.
4. Migrate demos and tests.
5. Update documentation and remove obsolete examples.

## Technical Constraints
- Keep chat-established semantics intact:
  - `Publish...` = already happened
  - `Write...` = requested change
- Keep the public API minimal and avoid parallel equal entry points.
- Prefer path-based operations, with explicit item snapshot publishing where needed.
- Keep code, docs, identifiers, comments, and XML docs in English.
- Do not add unrelated abstractions or features.

## Acceptance Criteria
- `PublishItemAsync` is removed from the public main API.
- `PublishSnapshotAsync(item, retained: true)` is the explicit snapshot mechanism.
- `WriteValueAsync` and `WriteParameterAsync` exist as the public write operations.
- Normal callers no longer construct low-level broker messages for standard usage.
- Repository call sites, demos, and docs use the simplified API consistently.

## Verification
- Build the affected solution/projects.
- Run broker and affected client/session tests.
- Review updated docs and demos to confirm only the preferred public path remains.

## Out of Scope
- New retention modes or publish modes.
- Item model redesign not required by the API refactor.
- Transport-layer redesign unrelated to the API simplification.

## Risks / Watchouts
- `PublishItemAsync` may be referenced in multiple wrappers and tests, so removal could have wider fallout.
- The client session currently contains snapshot-vs-delta convenience logic; removing the public method may require relocating or simplifying that behavior.
- The API should stay explicit enough that users still understand when they are publishing a full snapshot versus a delta.

# IMPLEMENTATION HANDOFF

## Goal
Bring HornetStudio and the BrokerWidget onto the slim ItemBroker API model: snapshots register ownership, later value and parameter changes use updates, and writable behavior continues to work through the existing MQTT path.

## Scope
Included:
- BrokerWidget local publish flow.
- HornetStudio host-side broker client facade.
- MQTT remote item client facade.
- Focused tests or verification for publish/update behavior.

Not included:
- New MQTT transport formats unless absolutely required.
- Broad UI changes.
- Ownership transfer workflows.
- Release/version updates.

## Starting Point
- Workitem: `docs/workitems/2026.05.03.0821-itembroker-slim-api/`
- Current API direction:
  - `PublishSnapshotAsync(...)` registers retained item state and ownership.
  - `UpdateValueAsync(...)` publishes owner updates or routes non-owner writes.
  - `UpdateParameterAsync(...)` publishes owner parameter updates or routes non-owner writes.
- Current observed gap:
  - `BrokerClientControl.HostItemBrokerPublisher` still calls `PublishSnapshotAsync` for every initial, interval, and change-triggered publish.
  - `IHostItemBrokerClient`, `HostItemBrokerClient`, and `MqttRemoteItemClient` do not expose update methods.
  - Items published from the widget can stop reacting because owner registration and later updates are not consistently represented through the new API.

## Tasks
1. Extend the HornetStudio broker client facade.
2. Extend the MQTT remote client facade.
3. Refactor BrokerWidget publishing to distinguish first snapshot registration from later updates.
4. Preserve snapshot publication for newly discovered paths and structural refreshes.
5. Review writable write-back behavior for echo and ownership side effects.
6. Add or adjust focused tests when practical.
7. Build and manually verify the BrokerWidget flow.

## File-Level Changes
- `src/HornetStudio.Host/HostItemBrokerClient.cs` -> add `UpdateValueAsync` and `UpdateParameterAsync` to `IHostItemBrokerClient`; forward from `HostItemBrokerClient` to `MqttRemoteItemClient`.
- `src/Amium.ItemBroker.Mqtt.Client/MqttRemoteItemClient.cs` -> add update methods that call the connected `MqttItemBrokerClientSession`.
- `src/HornetStudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs` -> update `HostItemBrokerPublisher` to track registered broker paths and use snapshots only for initial or structural registration, updates for later value/parameter changes.
- `src/HornetStudio.Editor/Widgets/Broker/HostItemBrokerWriteBackClient.cs` -> review and adjust only if update routing causes echo or writable regressions.
- `src/HornetStudio.Editor.Tests/Program.cs` or related existing test file -> add focused coverage if there is already a suitable test harness.
- `src/Amium.ItemBroker.Tests/Program.cs` -> add broker/facade coverage only if needed to lock the ownership/update interaction.

## Implementation Order
1. Read the current interfaces and implementations:
   - `IHostItemBrokerClient`
   - `HostItemBrokerClient`
   - `MqttRemoteItemClient`
   - `MqttItemBrokerClientSession`
   - `BrokerClientControl.HostItemBrokerPublisher`
2. Add `UpdateValueAsync(Item item, CancellationToken cancellationToken = default)` to `IHostItemBrokerClient`.
3. Add `UpdateParameterAsync(Item item, string parameterName, CancellationToken cancellationToken = default)` to `IHostItemBrokerClient`.
4. Implement both methods in `HostItemBrokerClient` by forwarding to `_remoteClient`.
5. Implement both methods in `MqttRemoteItemClient` by forwarding to `GetRequiredSession().UpdateValueAsync(...)` and `UpdateParameterAsync(...)`.
6. In `HostItemBrokerPublisher`, introduce a path state set keyed by normalized broker path.
7. During initial publication, call `PublishSnapshotAsync` and mark every published path as registered.
8. During later `OnDataItemChanged` handling:
   - resolve the configured local item
   - map it to the configured broker path
   - if the broker path was not registered yet, publish a snapshot
   - if the changed item corresponds to the published root value, call `UpdateValueAsync`
   - if the changed item indicates a parameter-only change and the parameter name is known, call `UpdateParameterAsync`
   - if the change is structural or ambiguous, publish a snapshot to keep ownership and retained structure consistent
9. During interval publishing:
   - prefer update for already registered leaf values
   - use snapshot if structure needs refreshing or the path has not been registered
10. Keep the existing exception logging style with `HostLogger.Log.Warning`.
11. Review `HostItemBrokerWriteBackClient` for echo behavior after updates; adjust only if necessary.
12. Build and run relevant tests.

## Acceptance Criteria
- `IHostItemBrokerClient` exposes snapshot, value update, parameter update, and subscribe methods.
- `HostItemBrokerClient` and `MqttRemoteItemClient` implement the new update methods.
- BrokerWidget initial publication still sends retained snapshots.
- BrokerWidget later value changes use `UpdateValueAsync` when the path is already registered.
- BrokerWidget later parameter changes use `UpdateParameterAsync` when parameter identity is known.
- Ambiguous structural changes still publish snapshots and do not break ownership registration.
- Existing demo MQTT clients still publish and react.
- Writable BrokerWidget entries still apply remote changes back to local registry items.

## Risks / Watchouts
- `DataChangedEventArgs` may not expose parameter identity clearly enough for all parameter update cases; fall back to snapshot for ambiguous changes.
- Full snapshot fallback may hide missing update coverage, so keep it limited to registration or structural ambiguity.
- Update calls through MQTT currently return local accepted acknowledgements from the client session, not broker-side ownership acknowledgements.
- Echo suppression in `HostItemBrokerWriteBackClient` must still prevent local update loops.
- The standalone `Amium.ItemBroker.Mqtt.Client` build previously exited with code 1 during restore without printed errors, while it built successfully as part of the editor project; verify through the broader project build if this repeats.

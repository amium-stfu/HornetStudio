# DEBUG REPORT

## Problem
Beim Projektwechsel oder beim Laden eines neuen Projekts wirft `BrokerClientControl` eine `Avalonia.AvaloniaInternalException` an `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs:839` beim Aufbau von `AttachedItems`.

## Expected Behavior
Beim Projektwechsel soll das Broker-Widget sauber herunterfahren und keine UI-Elemente oder Bindings mehr aktualisieren, sobald es aus dem Visual Tree entfernt wird.

## Actual Behavior
Während das Control bereits aus dem Visual Tree entfernt wird, werden `AttachedItems` und weitere UI-gebundene Properties erneut aktualisiert. Dadurch versucht Avalonia Item-Container bzw. Bindings für ein bereits abgehängtes Control aufzubauen und meldet:

`Visual is atached to visual tree but root could not be found.`

## Error Messages / Logs
- `Avalonia.AvaloniaInternalException`
- Message: `Visual is atached to visual tree but root could not be found.`
- Beobachtete Stelle: `BrokerClientControl.axaml.cs:839`

## Relevant Code
- `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs:260`
  `OnDetachedFromVisualTree()` ruft `DisconnectInternal()` auf.
- `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs:495`
  `DisconnectInternal()` ist `async void` und ruft am Ende `RefreshPresentation()` auf.
- `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs:756`
  `RefreshPresentation()` ruft `RebuildAttachedItemRows()` auf.
- `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs:839`
  `RebuildAttachedItemRows()` fügt neue `BrokerAttachedItemRow`-Einträge zu `AttachedItems` hinzu.
- `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml:43`
  `ItemsControl` ist direkt an `AttachedItems` gebunden.

## Attempted Fixes
- Codevergleich mit `UdlClientControl` -> dort wird beim Detach aggressiver aufgeräumt und nachlaufende Refreshes werden gestoppt.
- Lebenszyklusprüfung von `BrokerClientControl` -> bestätigt mehrere Pfade, die nach Detach noch `RefreshPresentation()` / `RebuildAttachedItemRows()` auslösen können.
- Prüfung des Projektwechselpfads -> `LoadBook_Click()` stoppt Host-Runtimes, verhindert aber nicht nachlaufende UI-Updates bereits bestehender Broker-Controls.

## Current Hypothesis (optional)
- Hauptursache ist ein Lifecycle-Race:
  `OnDetachedFromVisualTree()` startet asynchrones Disconnect,
  danach laufen noch UI-Refreshes weiter,
  und `RebuildAttachedItemRows()` mutiert die an das `ItemsControl` gebundene Collection, obwohl das Control keinen stabilen Visual Root mehr hat.
- Verstärkend kommt hinzu, dass `OnDataContextChanged()`, `OnObservedItemPropertyChanged()`, Timer-Ticks und Client-Callbacks ebenfalls `RefreshPresentation()` bzw. `RebuildAttachedItemRows()` triggern, ohne einen Guard für detachte Controls.

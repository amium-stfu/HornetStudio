# DEBUG REPORT

## Problem
Values can be edited in the HornetStudio UI, but the edited value does not persist and is replaced again by the previous value.

## Expected Behavior
When a user edits a writable value that represents a broker item, the write should reach the owning broker/client path. The UI should then remain on the accepted value or show a clear rejection.

## Actual Behavior
The UI value changes briefly, then is overwritten by the value coming from the broker/owner refresh.

## Attempted Fixes
- Checked whether HornetStudio already exposes the new API:
  - `IHostItemBrokerClient` now has `UpdateValueAsync` and `UpdateParameterAsync`.
  - `HostItemBrokerClient` forwards both methods.
  - `MqttRemoteItemClient` forwards both methods to `MqttItemClientSession`.
- Checked current BrokerWidget publisher behavior:
  - it now uses update methods for already registered configured publish paths.
  - it still falls back to snapshots for unregistered or structural/ambiguous changes.
- Checked UI write path:
  - UI edits currently update the local `HostRegistries.Data` item directly.
  - for broker-received runtime items, this is only the local mirror, not the broker owner.

## Current Hypothesis (optional)
- The rollback is very likely caused by editing a broker-received mirror item in the UI.
- The UI write updates only `HostRegistries.Data`, so the displayed item changes temporarily.
- Because the local mirror is not the broker owner and no broker write request is sent from this UI path, the actual owner still has the old value.
- The next BrokerWidget/MQTT refresh calls `UpsertSnapshot(...)` for the remote item and overwrites the local mirror with the owner value.
- This is related to the new owner rule: non-owner writes must be routed through broker update/write-request semantics, but the generic UI write path still performs a local registry mutation.

## Current Solution Direction
- Add a broker-aware UI write route for targets below `runtime.item_broker...` or for broker-received metadata.
- That route should translate the visible runtime path back to the flat broker path and call `IHostItemBrokerClient.UpdateValueAsync(...)` / `UpdateParameterAsync(...)` instead of only updating `HostRegistries.Data`.
- Keep local immediate refresh only as optimistic UI state if the broker write was accepted or intentionally pending.
- If no connected BrokerWidget/client can route the write, return a clear user-visible error instead of applying a local-only value that will be overwritten.

# IMPLEMENTATION HANDOFF

## Goal
Extend `Item.Server.Monitor` so the monitor starts with a local item server runtime but without automatically started transports. The UI should allow selectively starting and stopping available adapters. MQTT is the first available adapter, but the structure must be ready for additional adapters later.

## Scope
- Reuse the existing `docs/workitems/2026.05.07.1122-item-server-monitor/` workitem.
- Add adapter lifecycle support to `src/Item.Server.Monitor`.
- Start the monitor with an in-process `IItemServer` runtime.
- Do not start any adapter automatically.
- Expose available adapters in the UI.
- Support starting and stopping the MQTT adapter from the UI.
- Track adapter status, endpoint/options, and last error.
- Keep the implementation prepared for multiple adapters.
- Do not implement `config.yaml` yet.
- Record future `config.yaml` startup configuration as follow-up only.

## Starting Point
- `src/Item.Server.Monitor` already exists as an Avalonia monitor project.
- `MainWindowViewModel` currently reads `HostRegistries.Data` through `MonitorItemSource`.
- `Amium.Item.Server` already defines `IItemServerTransport`.
- `MqttItemServerAdapter` already implements `IItemServerTransport`.
- `MqttItemServerHost` exists, but it is too MQTT-specific for the monitor UI lifecycle.
- `ItemServerServiceHost` starts MQTT directly and should not be reused as the monitor runtime structure.
- The desired startup behavior is: monitor opens with server runtime ready, but no transport/adapter running.

## Tasks
1. Inspect the current monitor implementation and identify the minimal project references needed for `IItemServer`, `InMemoryItemServer`, and `MqttItemServerAdapter`.
2. Add a monitor hosting layer that owns the local `IItemServer` instance.
3. Add an adapter catalog with MQTT as the first adapter definition.
4. Add adapter runtime state for start/stop/status/error tracking.
5. Add an MQTT adapter factory that creates an `IItemServerTransport` from monitor adapter options.
6. Wire adapter start/stop commands through `MonitorHost`, not directly through `MainWindowViewModel`.
7. Extend the UI with an adapter management area.
8. Ensure the monitor does not start MQTT automatically.
9. Add focused tests for adapter runtime state transitions where practical.
10. Add a TODO or handoff note for future `config.yaml` adapter startup configuration.
11. Verify with build/tests without starting the UI application.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, identifiers, and user-visible messages must be English.
- Keep the monitor transport-neutral.
- Do not hardwire MQTT lifecycle into `MainWindowViewModel`.
- Use existing `IItemServerTransport` instead of creating a competing transport abstraction.
- Avoid a plugin system for now.
- Do not implement automatic startup from `config.yaml` in this step.
- Do not start demo, UI, service, or worker projects during verification.
- Use async/await for adapter start/stop.
- Dispose or stop running adapters when the monitor closes.
- Do not introduce new external dependencies.

## Acceptance Criteria
- The monitor starts without running MQTT or any other adapter.
- The monitor has a local item server runtime available for adapters.
- The UI lists MQTT as an available adapter.
- The user can start MQTT from the UI.
- The user can stop MQTT from the UI.
- Adapter status is visible in the UI.
- Last adapter error is visible when startup or shutdown fails.
- Adapter lifecycle code is structured so additional adapter factories can be added without rewriting the UI.
- `config.yaml` auto-start is not implemented.
- The complete solution builds without starting applications.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Run relevant tests if test coverage is added or an existing monitor/server test command is available.
- Manual UI verification is optional and should only be done if explicitly requested.

## Out of Scope
- Automatic adapter startup from `config.yaml`.
- Persisting adapter options.
- Multiple simultaneous MQTT instances unless the existing structure naturally supports it with no extra complexity.
- Remote monitor mode.
- Write/publish UI.
- History, charts, or long-term recording.
- Reworking `ItemServerServiceHost`.
- Reworking `MqttItemServerHost` unless a small extraction is strictly necessary.

## Risks / Watchouts
- If `MonitorItemSource` continues to read `HostRegistries.Data`, confirm that data published through the local `IItemServer` becomes visible there. If not, isolate the mismatch and prefer adjusting the source boundary rather than coupling UI code to server internals.
- `MqttItemServerHost` includes health publishing and host lifecycle behavior. The monitor may need only `MqttItemServerAdapter` to avoid MQTT-specific host coupling.
- Start/stop commands must guard against concurrent clicks and repeated starts.
- Failed starts should leave the adapter in a clear `Failed` or `Stopped` state and keep the UI usable.
- Running adapters must be stopped/disposed on application shutdown.
- Port `1883` may already be in use; the UI should expose the error clearly.

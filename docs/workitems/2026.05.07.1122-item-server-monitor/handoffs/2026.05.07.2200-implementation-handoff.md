# IMPLEMENTATION HANDOFF

## Goal
Fix the MQTT inbound handling so external MQTT publishes are imported as observed item state and become visible in the Item.Server.Monitor tree, while preserving the existing flat MQTT topic structure and existing echo-prevention behavior.

## Scope
Included:
- Keep the current MQTT topic mapping unchanged.
- Treat external MQTT publishes as observed broker state when they are not writable/control writes.
- Preserve writable item handling for intentional write/control paths.
- Preserve existing echo-prevention mechanisms based on MQTT client id, source client id, origin tracking, and injected mirror topic filtering.
- Ensure the monitor registry bridge can see imported external MQTT item paths.
- Add or update focused tests for the new inbound observed-state behavior.

Not included:
- No `command` topic implementation.
- No `config.yaml`.
- No external broker client/bridge configuration.
- No routing, ownership, discovery, or federation implementation.
- No topic layout redesign.

## Starting Point
- `MqttItemServerAdapter.HandleIncomingPublishAsync` maps incoming MQTT topics, tracks visible topics, tracks writable state, and currently rejects the publish when the target item is not known as writable.
- The monitor starts a local `InMemoryItemServer`, starts `MonitorRegistryBridge`, and starts the MQTT adapter.
- `MonitorRegistryBridge` currently subscribes only to `runtime` and `ItemServerHealthPaths.Root`.
- The demo MQTT client publishes paths such as `Edm1.Temperature` through the existing flat shared MQTT topic mapper.
- Existing BrokerWidget echo prevention relies on client ids, source client ids, published paths, value comparisons, origin skip, and injected topic filtering rather than sender payload fields.

## Tasks
1. Review the existing MQTT inbound tests and identify which behavior must remain write/control-specific.
2. Extend MQTT server adapter options with a minimal inbound publish policy if needed, defaulting to the current safe behavior for non-monitor use unless the project direction clearly supports observed inbound data by default.
3. Update `MqttItemServerAdapter.HandleIncomingPublishAsync` so external non-system MQTT publishes can be imported into the local broker as observed state instead of being rejected solely because the item is not writable.
4. Keep writable path behavior intact: known writable items still use the existing write target resolution and source-client origin tracking.
5. Use the MQTT publishing client id as the broker `sourceClientId` for imported observed state where available.
6. Ensure server-owned output topics, `$SYS` topics, injected mirror topics, and adapter self-publishes are not imported as normal observed data.
7. Update `MonitorRegistryBridge` subscription behavior so imported external MQTT item paths are mirrored into `HostRegistries.Data`.
8. Add or update tests for observed inbound MQTT publish import, non-writable behavior, writable write behavior, and echo/mirror guard behavior where practical.
9. Run the relevant test project and complete solution build without starting applications.

## Implementation Order
1. Inspect current tests around MQTT incoming write mapping and non-writable write blocking.
2. Decide the smallest inbound policy shape needed to avoid breaking existing non-monitor behavior.
3. Change `MqttItemServerAdapter` inbound handling so observed external MQTT state is published into `IItemServer`.
4. Adjust monitor adapter options if the observed import is not globally enabled.
5. Expand `MonitorRegistryBridge` subscriptions to include imported external item paths.
6. Update tests to document the new distinction between observed data and controlled writes.
7. Run tests and build.

- Do not introduce new dependencies.
- Prefer existing project patterns and existing item server abstractions.

## Acceptance Criteria
- External MQTT publishes from the demo client appear in the monitor tree.
- Existing flat MQTT topics continue to map as before.
- Writable item writes still work.
- Non-system external MQTT publishes are not rejected solely because the item is not writable when observed import is enabled.
- Adapter self-publishes, injected mirror topics, and system/server-owned topics are not imported as normal observed items.
- Existing BrokerWidget-style echo assumptions remain compatible.

## Out of Scope
- MQTT command/control topic design.
- YAML configuration file loading.
- External broker bridge/client connection management.
- Ownership enforcement.
- Sender payload envelope design.
- Message id, TTL, hop count, routing rules, discovery, federation, binary payload, protobuf, or MessagePack transport.

## Risks / Watchouts
- A global change from blocked non-writable inbound publishes to observed imports may affect security expectations in non-monitor scenarios. Prefer an explicit option if existing tests indicate the old behavior is relied on.
- A broad monitor subscription may accidentally mirror internal system topics unless reserved/system paths are filtered.
- Importing MQTT publishes into the broker may cause publish-back loops if origin tracking or injected mirror filtering is bypassed.
- The topic mapper treats JSON object payloads as meta snapshots; preserve that behavior.

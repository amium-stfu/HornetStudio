# IMPLEMENTATION HANDOFF

## Goal
Create a new read-only Avalonia desktop monitor application for the Item Server under `src/Item.Server.Monitor`. The monitor should host or observe the server runtime in-process, display all known item paths in a performant tree-like UI, show live values directly in the tree, and provide a detail area for the selected item.

## Scope
- Create a new Avalonia desktop project named `Item.Server.Monitor`.
- Use Avalonia `11.2.6`, matching the existing `HornetStudio` project.
- Target Windows and Linux where feasible.
- Use `net9.0` unless an existing required project reference forces a narrower target.
- Implement read-only monitoring only.
- Use `HostRegistries` as the initial runtime item source because no other runtime source is currently known.
- Display the item structure raw from the server/item paths.
- Display current values directly in the tree rows.
- Add search as a filter.
- Add a right-side detail area for the selected item.
- Add Pause/Freeze for UI updates.
- Add an adjustable UI update throttle interval for performance testing.
- Keep History out of the first implementation and record it as future work only.

## Starting Point
- `src/HornetStudio/ItemTreeWindow.axaml` already contains a simple Avalonia `TreeView` with Path and Value columns.
- `src/HornetStudio/ViewModels/ItemTreeWindowViewModel.cs` already derives a tree from `HostRegistries.Data`, caches nodes, and formats item values.
- The existing `HornetStudio` project uses Avalonia `11.2.6`.
- `HostRegistries` should remain server-oriented; the monitor should consume it through a small monitor source boundary instead of spreading direct registry calls throughout UI code.
- Current expected item count is below 1,000, but the UI should still be designed around visible-row updates and throttling.

## Tasks
1. Add the new `src/Item.Server.Monitor` Avalonia desktop project and include it in the solution.
2. Reference only the existing projects needed to access the server runtime and `HostRegistries`.
3. Build a minimal application shell with a main window, left tree area, top controls, and right detail area.
4. Implement a monitor item source that reads item paths and values from `HostRegistries`.
5. Implement a snapshot store that keeps the latest item state by path.
6. Implement a path-based tree store that preserves raw path structure.
7. Expose a flat visible-row list for the UI instead of binding a recursive `TreeView` directly as the performance hot path.
8. Implement expand/collapse state, row selection, and filter behavior.
9. Implement throttled UI update scheduling with a selectable interval.
10. Implement Pause/Freeze so UI updates stop while source snapshots can continue to advance.
11. Implement the selected-item detail view.
12. Add focused tests for path/tree/filter/snapshot logic where practical.
13. Add or update relevant Markdown documentation and create a TODO entry for History if the project uses `docs/todos/`.
14. Verify with a full solution build without running the application.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, identifiers, and user-visible messages must be English.
- Use Avalonia `11.2.6`.
- Prefer existing project patterns and minimal abstractions.
- Do not introduce new dependencies unless technically justified.
- Keep `HostRegistries` usage behind a small source boundary.
- Do not implement write/publish functionality in version 1.
- Do not implement History in version 1.
- Do not use a classic recursive `TreeView` as the primary live-update hot path unless measurements or implementation constraints justify it.
- Prefer a path tree plus flat visible-row projection for performance.
- UI updates must be throttled and must not update every row for every source event.
- Do not update non-visible rows through UI bindings; update only source snapshots and project visible rows as needed.

## Acceptance Criteria
- A new `Item.Server.Monitor` Avalonia project exists under `src/Item.Server.Monitor`.
- The project is included in the solution.
- The monitor can display all known item paths from `HostRegistries`.
- The monitor shows current values directly in the tree-like list.
- Search filters visible rows.
- Selecting a row updates the detail area.
- Pause/Freeze stops UI refreshes without requiring the source state to stop.
- The UI update interval can be changed for performance testing.
- History is not implemented and is documented as future work.
- The complete solution builds without starting any application.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- If tests exist or are added for the monitor logic, run the relevant test command without starting UI applications.
- Manually inspect the monitor UI only if explicitly requested; normal build verification must not use `dotnet run`.

## Out of Scope
- Remote client communication.
- MQTT communication.
- Write/publish operations.
- History capture, charts, or long-term recording.
- Persisted connections, window layout, or tree expansion state.
- Full custom drawn tree renderer.
- Reworking the existing `HornetStudio` `ItemTreeWindow`.
- Changing `HostRegistries` behavior.

## Risks / Watchouts
- Referencing existing server/runtime projects may force target framework or platform constraints; prefer keeping the monitor cross-platform, but do not fight required project constraints without reviewing the cause.
- `HostRegistries` is the only known runtime source, but direct access should stay isolated to avoid coupling the UI to registry internals.
- Avalonia recursive `TreeView` may be acceptable for small counts but should not be the planned hot path for live values.
- Filtering tree paths must preserve enough ancestor context to keep matched rows understandable.
- Pause/Freeze must avoid losing the latest state; resuming should consolidate to the current snapshot.
- Value formatting should avoid expensive formatting on the UI thread for large or complex values.

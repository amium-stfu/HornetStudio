# DEBUG REPORT

## Problem
The `Item.Server.Monitor` MQTT adapter can be started and an MQTT client can connect, but the client item data does not appear in the monitor tree.

## Expected Behavior
When `Amium.Item.Client.MqttDemoClient` publishes retained item data such as `Edm1.Temperature` and `Edm1.Pressure`, the monitor should show those paths and live values in the tree.

## Actual Behavior
The MQTT adapter starts and accepts client connections, but the monitor tree does not show the client data.

## Error Messages / Logs
No concrete runtime exception has been reported. The observed symptom is missing item data despite a successful MQTT client connection.

## Relevant Code
- `src/Item.Server.Monitor/Hosting/MonitorHost.cs`
  - Creates a local `InMemoryItemServer`.
  - Starts `MonitorRegistryBridge`.
  - Starts the MQTT adapter against the local broker.
- `src/Item.Server.Monitor/Hosting/MonitorRegistryBridge.cs`
  - Subscribes only to `runtime` and `ItemServerHealthPaths.Root`.
  - Mirrors received broker messages into `HostRegistries.Data`.
- `src/Item.Server.Monitor/Monitoring/MonitorItemSource.cs`
  - Reads visible monitor data from `HostRegistries.Data`.
- `src/Item.Server.Monitor/Hosting/MqttMonitorAdapterFactory.cs`
  - Creates `MqttItemServerAdapter` with `SubscriptionRootPath = "runtime"`.
- `src/Amium.Item.Server/Mqtt/MqttItemServerAdapter.cs`
  - Maps incoming MQTT publishes.
  - Updates adapter-internal visible topic tracking.
  - Rejects incoming publishes for paths that are not known as writable.
- `src/Amium.Item.Client.MqttDemoClient/Program.cs`
  - Publishes demo paths below `Edm1`, not below `runtime`.

## Attempted Fixes
- No fix has been applied yet.

## Current Hypothesis
- The monitor reads from `HostRegistries.Data`, but the MQTT client data is not mirrored into `HostRegistries.Data`.
- `MonitorRegistryBridge` only subscribes to `runtime` and server health paths. The demo client publishes paths such as `edm1.temperature`, so those paths are outside the bridge subscription.
- More importantly, `MqttItemServerAdapter.HandleIncomingPublishAsync` currently treats incoming MQTT publishes as write attempts. For unknown/non-writable paths it updates internal visible topic tracking and then rejects the broker update. That means no `IItemServer.PublishSnapshotAsync`, `UpdateValueAsync`, or retained broker state is produced for normal external MQTT item publishers.
- Existing `MqttItemServerHost` runtime counts can still increase because they use `MqttItemServerAdapter.VisibleItemCount`, which is adapter-internal state and not the broker/registry state shown by the monitor.

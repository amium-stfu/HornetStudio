# PLAN

## Topic
MQTT ItemBroker target structure for selfhosted service, remote client, and hybrid usage.

## Problem
The current MQTT-related projects already cover large parts of the needed behavior, but the intended product shape is still unclear. The system must support external solutions via NuGet or DLL, device-side deployments, and selfhosted broker scenarios similar to the existing BrokerWidget behavior. Without an explicit package and responsibility model, the client package can become overloaded and the host-side behavior can become too tightly coupled to HornetStudio.

## Goals
- Define a clear target structure for `Amium.ItemBroker`, `Amium.ItemBroker.Client`, `Amium.ItemBroker.Mqtt`, and `Amium.ItemBroker.Mqtt.Client`.
- Support three operating modes with minimal ambiguity:
  - Selfhosted service host
  - Remote MQTT client
  - Hybrid host plus local participant
- Keep external consumption simple enough for device and service scenarios.
- Preserve the existing broker semantics: resolve, writable reaction, and publishing behavior.

## Scope
Included:
- Package boundaries
- Role separation and responsibilities
- Suggested public entry points and facades
- Migration direction from current structure to target structure
- Verification strategy

Excluded:
- Concrete implementation
- Final API naming freeze
- Packaging and release mechanics
- Unrelated HornetStudio UI changes

## Current State
- `Amium.ItemBroker` contains the transport-neutral broker contracts and runtime behavior.
- `Amium.ItemBroker.Client` provides a generic in-process client session model.
- `Amium.ItemBroker.Mqtt` currently represents the MQTT broker bridge and embedded server behavior.
- `Amium.ItemBroker.Mqtt.Client` already supports MQTT connection, publishing, subscriptions, writes, and remote item reconstruction.
- `HornetStudio.Host/HostItemBrokerClient` adds host-specific convenience behavior on top of the MQTT client package.

## Target Structure
- `Amium.ItemBroker`
  - Transport-neutral core
  - Broker contracts, messages, policies, path handling, store abstractions

- `Amium.ItemBroker.Client`
  - Transport-neutral client session abstractions
  - Common publish/write/subscribe semantics
  - Shared high-level client contracts when they are not MQTT-specific

- `Amium.ItemBroker.Mqtt`
  - Selfhosted and bridge-side MQTT behavior
  - Embedded MQTT server hosting
  - Broker-to-MQTT bridge
  - Writable routing, retained state publishing, and broker-owned topic handling
  - The primary package for device-side or service-side selfhosted deployments

- `Amium.ItemBroker.Mqtt.Client`
  - Remote participant package
  - Connect to an existing MQTT-backed ItemBroker
  - Publish items and deltas
  - Subscribe to broker updates
  - Write to writable items through broker topics
  - Reconstruct remote item state and expose change notifications

- Optional future package: `Amium.ItemBroker.Mqtt.Hosting`
  - Only if startup ergonomics become large enough to justify separation
  - DI registration, options binding, background lifecycle helpers

## Operating Modes
1. Selfhosted service host
   - A process hosts the MQTT bridge locally and exposes the broker as a service interface.
   - Best fit for device installations and edge scenarios.

2. Remote client
   - A process connects to an already running MQTT-backed broker.
   - Best fit for external integrations, monitoring apps, and secondary services.

3. Hybrid
   - A process hosts a broker and also participates as a local client.
   - Best fit for HornetStudio-like scenarios where local publication and external visibility must coexist.

## Design Decisions
1. Selfhosted behavior belongs to the MQTT host side, not to the MQTT client package.
2. The MQTT client package should remain consumer-oriented and easy to adopt from other solutions.
3. Hybrid mode should be composed from host and client building blocks, not from a single mixed-responsibility class.
4. HornetStudio-specific conveniences should stay outside the transport packages unless they clearly generalize.
5. Shared semantics should be lifted into transport-neutral contracts only when they are truly reusable.

## Recommended Public Shape
- Low-level access remains available for advanced consumers.
- High-level facades should exist for common scenarios:
  - connect/disconnect
  - publish item or value
  - subscribe by path
  - write to path
  - observe remote item tree changes
- Selfhosted entry points should favor explicit startup objects over hidden static behavior.

## Migration Direction
1. Keep the existing core package boundaries.
2. Stabilize `Amium.ItemBroker.Mqtt.Client` as the external-consumer package.
3. Move reusable host-side convenience behavior out of HornetStudio only if it is broadly applicable.
4. Introduce explicit host-facade types on the MQTT host side if the current adapter is too low-level for device deployments.
5. Add a hybrid composition layer only after host and client roles are clean on their own.

## Risks
- The client package may become too broad if selfhosted concerns are added there.
- The host package may stay too low-level for device scenarios if no convenience facade is provided.
- Hybrid implementations may duplicate state reconstruction or publish filtering if composition boundaries remain unclear.
- Premature API generalization can lock in HornetStudio-specific assumptions.

## Acceptance Direction
- A new contributor can clearly answer which package to use for selfhosted service hosting versus remote participation.
- External consumers get a small, understandable MQTT client package.
- Selfhosted deployments do not require HornetStudio-specific classes.
- Hybrid usage is achievable through composition without collapsing package boundaries.

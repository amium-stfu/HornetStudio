# IMPLEMENTATION HANDOFF

## Goal
Restructure the MQTT ItemBroker solution surface so that selfhosted service hosting, remote MQTT participation, and hybrid usage are all supported with clear package responsibilities and simple entry points for external consumers.

## Scope
Included:
- Clarify and implement package responsibilities for:
  - `Amium.Item.Server`
  - `Amium.Item.Client`
  - `Amium.Item.Server.Mqtt`
  - `Amium.Item.Client.Mqtt`
- Add or refine public facades needed for selfhosted and remote usage.
- Reduce HornetStudio-specific coupling where generic behavior should live in reusable packages.

Not included:
- HornetStudio widget redesign
- Release packaging and publish versioning
- New transport implementations beyond MQTT
- Broad API redesign outside the MQTT ItemBroker area

## Starting Point
- Current behavior:
  - The broker side already resolves, reacts to writable items, and publishes MQTT state.
  - The MQTT client side already connects, publishes, subscribes, writes, and reconstructs remote items.
  - HornetStudio adds host-specific convenience behavior in `HostItemBrokerClient`.
- Relevant assumptions:
  - External consumers should be able to consume the MQTT client package via NuGet or DLL with minimal setup.
  - Device deployments may need to selfhost the service interface without Hornetstudio.
  - Hybrid scenarios are real and should remain supported.
- Required prerequisites:
  - Preserve existing broker semantics and topic mapping behavior.
  - Avoid introducing package roles that overlap ambiguously.

## Tasks
1. Audit current MQTT-related public types and classify each as core, host-side, client-side, or HornetStudio-specific.
2. Define the minimal public API surface for a reusable selfhosted MQTT host package.
3. Define the minimal public API surface for a reusable remote MQTT client package.
4. Decide which host-side conveniences in HornetStudio should stay local and which should move into reusable packages.
5. Implement a composition approach for hybrid scenarios without merging host and client responsibilities into one type.
6. Add or update tests that prove selfhosted, remote, and hybrid scenarios behave correctly.
7. Update relevant package or usage documentation once the structure is implemented.

## File-Level Changes
- `src/Amium.Item.Server/*` -> only adjust if a transport-neutral contract is genuinely needed by both MQTT host and MQTT client sides.
- `src/Amium.Item.Client/*` -> hold transport-neutral client abstractions and shared client helpers only.
- `src/Amium.Item.Server.Mqtt/*` -> host-side MQTT server, broker bridge, retained publishing, writable routing, and selfhosted entry points.
- `src/Amium.Item.Client.Mqtt/*` -> remote MQTT participant session, connection lifecycle, subscriptions, writes, diagnostics, and remote item registry.
- `src/Hornetstudio.Host/HostItemBrokerClient.cs` -> keep only HornetStudio-specific composition behavior or slim it down if reusable concerns move out.
- `src/Amium.Item.Server.Tests/*` -> add or update tests for new structure and behavior boundaries.
- `CHANGELOG.md` -> record the structural change when implementation is complete.

## Implementation Order
1. Inventory current public types and decide their target package responsibility.
2. Refine `Amium.Item.Server.Mqtt` so selfhosted hosting is consumable without Hornetstudio.
3. Refine `Amium.Item.Client.Mqtt` so remote participation is easy and self-contained for external consumers.
4. Rework any shared abstractions only where duplication or ambiguity requires it.
5. Revisit `HostItemBrokerClient` and keep it as a composition layer rather than a transport owner.
6. Add tests covering selfhosted, remote client, and hybrid flows.
7. Update docs and usage guidance.

## Technical Constraints
- Use existing project conventions, patterns, and package boundaries wherever possible.
- Keep code, comments, and public identifiers in English.
- Do not move selfhosted server responsibilities into the MQTT client package.
- Do not introduce HornetStudio-specific assumptions into reusable packages.
- Prefer small, explicit facades over large mixed-responsibility classes.

## Acceptance Criteria
- There is a clearly consumable selfhosted MQTT host path for device or service deployments.
- There is a clearly consumable remote MQTT client path for external integrations.
- Hybrid behavior can be built by composition of host and client roles.
- Package responsibilities are understandable from the public API and project structure.
- Existing broker semantics for publish, resolve, and writable handling are preserved.

## Verification
- Build the affected solution or projects after the structural changes.
- Run or extend automated tests for broker publish/subscribe/write flows.
- Manually verify:
  - selfhosted host startup works
  - remote client can connect and receive reconstructed remote items
  - writable item updates still route correctly
  - hybrid setup does not mirror its own published items incorrectly

## Out of Scope
- UI work in BrokerWidget or UdlClient
- New non-MQTT transports
- Release automation
- Unrelated cleanup or refactoring outside the MQTT ItemBroker area

## Risks / Watchouts
- Moving convenience logic too early can break stable consumers.
- Hybrid loops and self-mirroring remain a sensitive area and need explicit tests.
- Public API changes may affect future NuGet consumers, so naming and layering should be reviewed carefully before release.

## Notes
- Treat the current `HostItemBrokerClient` as evidence of needed ergonomics, not automatically as the reusable API to expose.
- If a dedicated hosting package is not yet justified, keep hosting in `Amium.Item.Server.Mqtt` and revisit only when startup concerns become too large.
- Favor a migration path that keeps current behavior working while improving package clarity incrementally.

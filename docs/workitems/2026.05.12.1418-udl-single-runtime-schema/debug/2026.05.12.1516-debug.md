# DEBUG REPORT

## Problem
UDL received items disappear from the `Received Items` section after clicking `Attach`. Attached rows show a red status indicator, and the attached-row `Edit` dialog does not offer bitmask channel selections.

## Expected Behavior
- A live runtime module remains visible under `Received Items` after it is attached.
- The received-row indicator changes from gray to green.
- The same path appears under `Attached Items` with a green indicator when the runtime item is still live.
- The attached-row `Edit` dialog shows runtime channels such as `read`, `set`, `out`, `state`, and `alert` so bit helpers can be configured.

## Actual Behavior
- After `Attach`, the entry disappears from `Received Items`.
- Existing attached rows such as `m430`, `m431`, `m432`, and `m433` show red indicators.
- The attached-row `Edit` dialog has no bitmask choices.
- The current screenshot shows live received modules `m434` and `m435`, while saved attachments are `m430` through `m433`.

## Relevant Code
- `UdlClientControl.RebuildAttachSectionRows()` builds attached-row status from `TryResolveRuntimeItem(attachedPath, out _)`.
- `UdlClientControl.TryResolveRuntimeItem()` resolves only against `EnumerateClientItems()`, not registry mirror paths.
- `UdlClientControl.GetRuntimeChannelDescriptors()` also uses `EnumerateClientItems()` and only returns items whose relative path has exactly two segments.
- `UdlClientControl.OnEditAttachedItemClicked()` passes `GetRuntimeChannelDescriptors()` to `UdlModuleExposureDialogWindow.ShowAsync(...)` and filters by `row.ModuleName`.
- `BuildReceivedAttachSectionRows(...)` has a focused test, but that test bypasses the full attach path and does not prove that the real runtime item still resolves after `SynchronizeAttachedItems()`.

## Attempted Fixes
- Existing code change separates received-row construction into `BuildReceivedAttachSectionRows(...)`.
  Result: The helper-level test can pass, but the observed UI behavior remains unchanged.
- Existing code change adds descendant signal source enumeration through `MainWindowViewModel.EnumerateSignalSourceOptions()`.
  Result: It may explain why a once-created helper could be selected by a signal widget, but it does not fix empty UDL `Edit` choices when the attached item is red/unresolved.
- Added regression `UiFolderInitialPublishPreservesSourceRuntimePaths`.
  Result: Failed before the fix because the initial attached mirror publish changed `source.Path` from `runtime.ui_folder_mirror.m002` to `studio.mirror_test.m002`.
- Updated `UiFolderContext.AttachedItemLink.OnTargetChanged(...)` to ignore the initial direct `SnapshotUpserted` event for the attached target path.
  Result: The initial mirror publish no longer writes the attached `studio...` path back into the runtime source item.
- Updated UDL bit helper publishing from `Bits.BitN` to `bits.bitN` and changed helper metadata keys to lowercase snake_case.
  Result: Bit helper item creation matches the enforced path segment policy and the existing signal source discovery expectation.
- Added regression `UdlRuntimeExposureBitsUseSnakeCasePaths`.
  Result: Verifies `bits.bit0` creation, bit values, labels, and lowercase metadata.

## Current Hypothesis
Confirmed root cause:

- `UiFolderContext.Attach(...)` creates an attached clone under `studio.<folder>.<client>.<module>`.
- `UdlClientControl.SynchronizeAttachedItems()` publishes that attached clone with `HostRegistries.Data.UpsertSnapshot(...)`.
- The `AttachedItemLink` already listens to target changes at that point, so the initial direct target `SnapshotUpserted` was treated as a user-side target edit.
- `ApplySnapshotToSource(...)` then applied the attached `studio...` snapshot back to the runtime source item.
- The runtime module still existed in the client dictionary, but its `Path` had been rewritten from `runtime.udl_client...m002` to `studio...m002`.
- `GetReceivedRuntimeRootPaths(...)` only treats `runtime.udl_client...` paths as runtime roots, so the received root list became empty immediately after attach.

Additional confirmed root cause:

- `UpsertRuntimeExposureBits(...)` still created helper items as `Bits.Bit0`, `Bits.Bit1`, and so on.
- The current item model requires lowercase snake_case path segments, so `new ItemModel("Bits", path: runtimeChannel.Path)` throws before helper bits can be published.
- The writeback metadata reader already expected lowercase keys such as `module_name`, `channel_name`, and `bit_index`, while the publisher wrote PascalCase metadata.

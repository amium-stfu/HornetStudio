# IMPLEMENTATION HANDOFF

## Goal
Make UDL demo mode and real UDL mode use one identical runtime item schema so a project can be developed with demo modules and later run with `Demo = false` without changing attachments, exposure definitions, or widget bindings.

## Scope
- Align demo and real UDL runtime item structure.
- Remove UDL legacy assumptions for `Value`, `Request`, PascalCase channel names, and mixed write properties.
- Ensure demo module names are real-compatible: lowercase `m` followed by exactly three digits, for example `m310` and `m411`.
- Keep the existing widget workflow: configure demo modules, attach received items, expose bits, then switch demo off for real modules.
- Update focused documentation only if an existing matching widget documentation file is present.

Not included:
- Release/versioning changes.
- New dependencies.
- New global architecture layers unless an existing project reference forces a small shared helper.
- Backward-compatible migration for old persisted demo definitions or old `Request` paths.

## Starting Point
- `src/Amium.UdlClient/Module.cs` already creates lowercase channels: `read`, `set`, `out`, `state`, `alert`.
- `src/Amium.UdlClient/UdlClient.cs` mostly writes channel read values through `Properties["read"]`, but still uses `Item.Value`, `Request`, `set`, `Set`, `Write`, and current-value checks against `Value`.
- `src/HornetStudio.Host/UdlModule.cs` mirrors the lowercase channel model.
- `src/HornetStudio.Host/HostUdlClient.cs` has similar legacy write and value fallback logic.
- `src/HornetStudio.Editor/Widgets/UdlClient/SimulatedHostUdlClient.cs` still creates PascalCase channels: `Read`, `Set`, `Out`, `State`, `Alert`.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` contains UDL attach, exposure, bit writeback, and runtime-channel resolution logic that should operate against one schema.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlDemoModulesDialogWindow.axaml.cs` has already been adjusted in the current working tree to create `m001`-style names and validate `m000` format.

## Tasks
1. Define and enforce the canonical runtime schema locally in UDL code:
   - Module child names: `read`, `set`, `out`, `state`, `alert`.
   - Runtime read value: `channel.Properties["read"].Value`.
   - Runtime write target: `channel.Properties["write"].Value`.
   - No `Request` child items.
   - No PascalCase channel names.
   - No UDL business logic based on `Item.Value`.
2. Align the simulated UDL client with the real client schema.
3. Remove legacy write/read fallback logic from both UDL client implementations.
4. Simplify widget runtime channel resolution and exposure writeback to use the canonical schema only.
5. Verify the demo-to-real workflow manually or with focused tests where existing test infrastructure supports it.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/UdlClient/SimulatedHostUdlClient.cs` -> create lowercase channels, write snapshots to `read`, read SetDriven targets from `set.write`, stop writing UDL channel values through `Item.Value`.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` -> normalize internal channel expectations to lowercase only, simplify write target resolution to channel `write`, ensure Received Items derive from runtime items rather than demo definitions.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlModuleExposureDialogWindow.axaml.cs` -> verify persisted channel names are canonical lowercase; if UI labels remain friendly, keep display separate from stored value.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlDemoModulesDialogWindow.axaml.cs` -> keep/enforce `m000` demo module name validation.
- `src/Amium.UdlClient/UdlClient.cs` -> remove `Request`, `Set`, `Write`, and `Item.Value` write/read fallbacks from UDL behavior.
- `src/Amium.UdlClient/Module.cs` -> keep as the canonical module shape; adjust only if necessary to remove legacy cleanup behavior.
- `src/HornetStudio.Host/HostUdlClient.cs` -> mirror the `Amium.UdlClient.UdlClient` cleanup for host-integrated real UDL runtime.
- `src/HornetStudio.Host/UdlModule.cs` -> keep aligned with `Amium.UdlClient.Module`; adjust only if necessary to remove legacy cleanup behavior.
- Matching docs under `HornetStudio/docs/widgets/`, `HornetStudio/docs/widgets/descriptions/`, or `HornetStudio/docs/widgets/help/` -> update only if those paths exist in the active checkout.

## Implementation Order
1. Start with `SimulatedHostUdlClient.cs`:
   - Replace `Read`/`Set`/`Out`/`State`/`Alert` with `read`/`set`/`out`/`state`/`alert`.
   - Ensure `Items[definition.Name]` receives a module whose path matches real module paths.
   - Ensure SetDriven reads `GetChannel(module, "set").Properties["write"].Value`.
2. Clean `UdlClient.cs`:
   - Make desired write value lookup read only `write`.
   - Make current value lookup read only `read`.
   - Remove `Request`, `Set`, `Write`, and `set` fallback handling.
   - Remove or stop using `module.Value = value` for runtime state.
3. Apply the same real-client cleanup to `HostUdlClient.cs`.
4. Review `Module.cs` and `UdlModule.cs`:
   - Keep channel construction canonical.
   - Remove legacy cleanup methods only if they no longer serve a current purpose.
5. Update `UdlClientControl.axaml.cs`:
   - Ensure runtime channel descriptors use lowercase channel names.
   - Ensure `TryResolveRuntimeChannel` resolves canonical lowercase definitions.
   - Replace `ResolveWriteValueItem` behavior with canonical channel-as-write-owner behavior.
   - Ensure bit exposure reads `read` and writes `write`.
6. Review `UdlModuleExposureDialogWindow.axaml.cs`:
   - Keep display labels user-friendly if desired.
   - Store canonical channel names only.
7. Run focused build and validation.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML docs, UI text, file names, and technical identifiers must be English.
- Do not introduce new dependencies.
- Prefer existing UDL classes and local helpers over new services or abstractions.
- Do not add migration/fallback behavior for old UDL demo schemas; this is still development.
- Public classes and public methods touched during implementation must keep or receive XML documentation according to repository rules.
- Avoid changing unrelated widget behavior.

## Acceptance Criteria
- Demo modules appear under `Received Items` exactly like real modules.
- A demo module named `m310` creates the same relative root path as a real module named `m310`.
- Attaching a demo module and then switching `Demo = false` keeps the same attachment path valid when the real module exists.
- Runtime channels are always lowercase: `read`, `set`, `out`, `state`, `alert`.
- UDL current values are represented through `Properties["read"]`.
- UDL write requests are represented through `Properties["write"]`.
- No UDL-specific code path requires `Request` child items for the intended workflow.
- No UDL-specific code path requires `Read`, `Set`, `Out`, `State`, or `Alert` child names.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore is available.
- At minimum, run `dotnet build src/HornetStudio.Editor/HornetStudio.Editor.csproj --no-restore` for focused verification.
- Manual demo workflow:
  1. Enable UDL demo mode.
  2. Add module `m310`.
  3. Confirm `m310` appears under `Received Items`.
  4. Attach `m310`.
  5. Confirm attached item resolves live.
  6. Disable demo mode and connect to real UDL source that publishes `m310`.
  7. Confirm the same attached path resolves live.
- Inspect runtime tree/logs to confirm lowercase channel paths and `read`/`write` properties.

## Out of Scope
- Persisted legacy layout migration.
- Compatibility with old `DemoModule01` names.
- Compatibility with old `Request` paths.
- Release publish, version bump, changelog release entry, or NuGet packaging.
- Broad Item/Broker architecture redesign.

## Risks / Watchouts
- `Item.Value` may still be used by generic registry or UI mirror behavior. For UDL runtime channels, use `read`; if generic infrastructure requires `Value` for display, isolate that decision explicitly and do not treat it as the UDL source of truth.
- Existing exposure definitions may currently persist display names like `Read`. If so, adjust the exposure editor to persist lowercase values while optionally displaying friendly labels.
- `HostUdlClient.cs` and `Amium.UdlClient/UdlClient.cs` are duplicate-ish implementations. Keep changes behaviorally aligned to avoid demo/editor and library divergence.
- Build can update tracked plugin binaries under `src/HornetStudio.Host/plugins/Amium.UdlClient/`; restore those artifacts if they are only build output and not part of the intended change.

# IMPLEMENTATION HANDOFF

## Goal
Fix the UDL client attach workflow after the single runtime schema change so received runtime modules stay visible after attach, attached modules expose bit helper items correctly, and UDL attached items or helper bits can be selected by signal widgets.

## Scope
Included:
- Keep `Received Items` as a stable view of live UDL runtime root modules.
- Keep `Attached Items` as a view of saved `UdlAttachedItemPaths`.
- Make the received-row status indicator switch from gray to green when the same runtime item is attached.
- Restore or complete bitmask helper publishing for attached UDL module channels.
- Ensure signal source pickers can discover the relevant attached UDL items or UDL bit helper items.
- Update focused UDL widget documentation if widget code changes.

Not included:
- Release/version changes.
- Broad registry redesign.
- New dependencies.
- Compatibility migration for old PascalCase UDL channels or old `Request` child paths.
- Unrelated widget refactoring.

## Starting Point
- Demo modules now appear initially under `Received Items`, for example `m310`.
- After clicking `Attach`, the module disappears from `Received Items`; expected behavior is that it remains visible and only the status dot changes from gray to green.
- `Attached Items` already shows saved attachments such as `m001`, but those entries are not available as signal widget sources.
- Bitmask exposure logic exists in `UdlClientControl.axaml.cs` through `UdlModuleExposureDefinitions`, `PublishExposureItems()`, `UpsertRuntimeExposureBits(...)`, and bit writeback helpers.
- Runtime schema should remain canonical:
  - channels: `read`, `set`, `out`, `state`, `alert`
  - current value: channel `Properties["read"].Value`
  - write request: channel `Properties["write"].Value`

## Tasks
1. Analyze why `Received Items` loses a module after attaching it.
2. Separate runtime discovery from attached mirror publishing.
3. Keep received rows visible and change only their attached presentation state.
4. Verify that `SynchronizeAttachedItems()` does not remove or mutate runtime items used by received discovery.
5. Restore or complete UDL bitmask helper publishing for configured module exposures.
6. Make UDL attached items or helper bits discoverable by signal source pickers.
7. Add focused tests for attach-row stability and signal-source discovery where existing test infrastructure supports it.
8. Update matching UDL widget documentation if widget behavior changes.

- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml` -> adjust bindings only if the existing row state model cannot express the expected status/action behavior.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlModuleExposureDialogWindow.axaml.cs` -> ensure exposure rows store canonical lowercase channel names and still provide useful bitmask defaults.

- `src/HornetStudio.Host/HostRegistries.cs` -> inspect capability behavior for descendants; change only if signal-source discovery requires registry-level support and no narrower widget fix is sufficient.
- Signal source picker code, likely under `src/HornetStudio.Editor/Widgets/CustomSignals/CustomSignalsControl.axaml.cs` or signal widget editor code -> include display-capable UDL attached/helper paths if they are currently omitted.

## Implementation Order
1. Reproduce the attach transition in code by tracing `OnAttachReceivedItemClicked`, `UdlAttachedItemPaths`, `RebuildAttachSectionRows()`, `GetAttachOptions()`, `TryResolveRuntimeItem()`, and `SynchronizeAttachedItems()`.
2. Make `Received Items` derive from live runtime roots only and use `UdlAttachedItemPaths` only for row presentation.
3. Ensure `Attached Items` is still built from saved paths and resolves live status independently.
4. Check `UiFolderContext.Attach(...)`, `ReleaseUiFolderContext()`, and `HostRegistries.Data.UpsertSnapshot(...)` interactions so attached mirror cleanup cannot affect live runtime discovery.
5. Verify bit helper creation under the canonical runtime channel tree and ensure helper items carry enough metadata for display and writeback.
6. Ensure signal source pickers can list the intended UDL attached/helper paths without exposing widget-internal operational status entries.
7. Add or update focused tests.
8. Update UDL widget docs.
9. Run focused build/tests, then solution build when restore state allows it.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Do not introduce new dependencies.
- Keep the canonical UDL runtime schema: lowercase channels and `read`/`write` properties.
- Prefer existing registry, widget, path helper, and test patterns.
- Do not add fallback behavior for old `Read`, `Set`, `Out`, `State`, `Alert`, or `Request` paths.
- Keep changes scoped to UDL attach, bit helper publishing, and signal-source discoverability.

## Acceptance Criteria
- A demo module such as `m310` remains visible in `Received Items` after clicking `Attach`.
- The received row status indicator changes from gray to green when the item is attached.
- The received row action is no longer executable for already attached items.
- The same path appears in `Attached Items` and shows green when its runtime source resolves.
- Detaching removes the attached row and returns the received row to the unattached presentation state.
- Configured bitmask helper items are created for selected UDL channels with the configured bit count.
- Bit helper values are derived from channel `read` values unless the existing exposure definition intentionally routes read input to `set.write`.
- UDL attached items or generated helper bits appear as selectable signal widget sources.
- No UDL workflow depends on PascalCase channel names or `Request` child items.

## Out of Scope
- Release publishing, version bump, tags, or NuGet packaging.
- Runtime dependency changes.
- Global registry architecture redesign.
- Persisted migration for obsolete UDL paths.
- Reworking non-UDL widget behavior beyond necessary source picker discovery.

## Risks / Watchouts
- `UiFolderContext.Dispose()` removes attached mirror roots; ensure this does not delete runtime roots or confuse discovery.
- `HostRegistries.Data.GetAllKeys()` returns root keys, while `TryResolve()` can resolve descendants; signal source discovery may need descendant-aware handling.
- `UpsertSnapshot(..., pruneMissingMembers: true)` can remove descendants from an existing root if source snapshots do not contain them.
- Bit helper items under runtime channels may be cloned into attached mirrors; decide whether signal widgets should bind to runtime paths, attached mirror paths, or both, and keep that consistent.
- Avoid exposing widget status paths as signal sources when expanding discovery.

## Notes
- The desired model is intentionally simple: runtime discovery finds live modules; attach state only annotates those modules; attached mirrors are separate UI-facing copies.
- Do not solve the disappearing received row by hiding already attached modules. The expected behavior is persistent visibility with a changed status indicator.
- Keep the demo-to-real workflow intact: a demo module path attached today must still resolve when demo mode is disabled and the real module with the same name exists.

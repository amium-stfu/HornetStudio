# IMPLEMENTATION HANDOFF

## Goal

Refine the WinForms MQTT demo TreeView so it presents observed item state as an MQTT Explorer-like topic tree.

Item nodes must show structure only. Values and properties must be shown as child leaf nodes.

Expected shape:

```text
edm1
  pressure
    raw
      read = 0.0
      epoch = 177844...
    epoch = 177844...
    format = 0.0
    meta = {}
    unit = hPa
    read = 1014.3
  temperature
    raw
      read = 0.0
      epoch = 177844...
    epoch = 177844...
    format = 0.0
    meta = {}
    unit = degC
    read = 20.8
```

Do not show item nodes like:

```text
edm1.pressure = 1014.3
```

## Scope

Included:

- Update the WinForms sample TreeView rendering.
- Show item nodes by item name only, not full path and not value.
- Show item value as a child leaf named `read`.
- Show item properties as child leaf nodes without `@` prefix.
- Show useful item properties including `meta`, `epoch`, `format`, and `unit`.
- Hide only technical model metadata such as `name` and `path`.
- Make `raw` consistent in the demo publisher as a child item, not both a property and a child item.
- Update sample documentation if it describes the old display behavior.

Not included:

- No further client registry redesign.
- No MQTT protocol changes.
- No retained broker cleanup feature.
- No new dependencies.
- No changes to `amium_item/src/Amium.Item/Item.cs`.

## Starting Point

- `MqttRemoteItemRegistry` now exposes a cleaner item root forest.
- `MainForm.BuildItemNode(...)` currently shows item text as `path = value`.
- `MainForm.BuildItemNode(...)` currently renders properties as `@property = value`.
- `MainForm.IsHiddenTechnicalProperty(...)` currently hides `meta`, `epoch`, and `read`, which conflicts with the desired MQTT Explorer-like view.
- `DemoItemPublisher` currently creates `raw` as a child item with `_temperature["raw"].Value`, but also publishes `raw` via `UpdatePropertyAsync(...)`, mixing child item and property semantics.

## Tasks

1. Change TreeView item node labels to show only the item name.
2. Add a `read = <value>` leaf below an item when the item value is present.
3. Render item properties as leaf nodes named `<propertyName> = <value>` without the `@` prefix.
4. Hide only technical model metadata:
   - `name`
   - `path`
5. Do not hide:
   - `meta`
   - `epoch`
   - `format`
   - `unit`
   - `read`
6. Render child items as child nodes below the item's leaf-property nodes or in a stable readable order.
7. Make `raw` consistent in `DemoItemPublisher`:
   - keep it as child item if that is the chosen structure,
   - publish/update it as a child item value,
   - do not also publish it as a property on `pressure` or `temperature`.
8. Ensure `meta` is either published intentionally or not shown as misleading empty data. Prefer publishing simple metadata if the demo already has helper code for it.
9. Build the sample.

## Implementation Order

1. Update `MainForm.cs` TreeView rendering:
   - item node text becomes `item.Name`,
   - value becomes `read = ...`,
   - properties become plain leaf nodes,
   - hidden-property filter is reduced to model metadata only.
2. Update `DemoItemPublisher.cs` to make `raw` a proper child item only.
3. Decide and implement metadata behavior:
   - either publish simple `meta` values and show them,
   - or remove `meta` from expected demo output/documentation.
4. Review the resulting visual tree against MQTT Explorer.
5. Update docs if needed.
6. Build the WinForms sample.

## Technical Constraints

- Chat responses are German; code, comments, XML documentation, UI strings, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Do not add new dependencies.
- Keep the change scoped to the sample display/publisher behavior.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Avoid new abstractions; this is sample UI rendering, not a reusable framework.

## Acceptance Criteria

- Item TreeView nodes show names only, e.g. `pressure`, not `edm1.pressure = 1014.3`.
- Item values appear as `read = <value>` child leaves.
- Properties appear as plain leaves such as `unit = hPa`, `format = 0.0`, `epoch = ...`, `meta = ...`.
- No property leaf uses an `@` prefix.
- `meta`, `epoch`, `format`, `unit`, and `read` are visible when present.
- `name` and `path` are hidden.
- `raw` is not shown both as a property and as a child item.
- The demo TreeView is visually close to MQTT Explorer for the same retained topics.
- The WinForms sample builds.

## Verification

- Run focused build:
  `dotnet build amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms/Amium.Item.Client.MqttDemoWinForms.csproj --no-restore`
- If needed after broader client changes, run:
  `dotnet build amium_item/Amium.Item.sln --no-restore`
- Manual verification when desktop UI is available:
  - start the sample from freshly built output,
  - create server,
  - create both clients,
  - start publishing,
  - compare the TreeView with MQTT Explorer,
  - confirm item nodes have no inline values and `read`/properties appear as leaves.

## Out of Scope

- Client registry architecture changes.
- MQTT topic cleanup.
- Writable demo behavior.
- Broker ownership or per-client topic grouping.
- Authentication/TLS/configuration UI.

## Risks / Watchouts

- Existing retained broker topics can still contain old `raw` property topics. Manual verification should use a clean broker state or distinguish old retained topics from new publisher behavior.
- `Item` default properties may create `meta` even when empty. Decide whether showing `meta = {}` is useful for the sample.
- Do not solve this by hiding too much; the goal is a clear MQTT Explorer-like view, not a minimal summary.

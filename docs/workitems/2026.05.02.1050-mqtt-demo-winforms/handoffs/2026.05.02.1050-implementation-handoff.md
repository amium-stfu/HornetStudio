# IMPLEMENTATION HANDOFF

## Goal
Create a new WinForms demo application that demonstrates a selfhosted MQTT ItemBroker service, two periodically published demo items, and one writable item that reacts to external writes.

## Scope
Included:
- New WinForms project for the MQTT ItemBroker demo
- Buttons to start and stop a local selfhosted service
- Two demo items equivalent in spirit to the current console `MqttDemoClient`
- One writable item with visible reaction to external writes
- Basic status and log output in the UI

Not included:
- Replacing or deleting the existing console demo projects
- Reworking the core MQTT package architecture
- Advanced WinForms theming or complex visualization
- Installer, publish, or release packaging work

## Starting Point
- Current behavior:
  - `src/Amium.ItemBroker.Service/Program.cs` shows how to start a selfhosted MQTT ItemBroker host.
  - `src/Amium.ItemBroker.MqttDemoClient/Program.cs` shows how to publish two simple demo items over MQTT.
  - `src/Amium.ItemBroker.Mqtt/MqttItemBrokerHost.cs` already provides the reusable host-side runtime.
  - `src/Amium.ItemBroker.Mqtt.Client/MqttItemBrokerClientSession.cs` already supports connect, publish, subscribe, and write behavior.
- Relevant assumptions:
  - The WinForms app runs on Windows and is intended primarily as an internal demo and teaching tool.
  - The demo should make the host, publisher, and writable-item roles visible without hiding too much of the actual system behavior.
  - Existing package behavior should be reused instead of duplicated in the UI project.
- Required prerequisites:
  - Choose a WinForms-compatible target framework and project setup that fits the current solution.
  - Keep UI code responsive and avoid blocking calls on the UI thread.

## Tasks
1. Add a new WinForms project `src/Amium.ItemBroker.MqttDemoWinForms`.
2. Reference the required MQTT host, MQTT client, broker, and item packages.
3. Create a main form with sections for:
   - service start/stop
   - published item values
   - writable item status
   - log output
4. Implement a small service host controller around `MqttItemBrokerHost`.
5. Implement a publisher controller based on the existing `MqttDemoClient` logic for `Temperature` and `Pressure`.
6. Implement a writable demo controller that:
   - publishes an initial writable item snapshot
   - marks the item writable
   - reacts to external write requests
   - updates the UI-visible state
7. Wire controller events into the form with safe UI-thread marshaling.
8. Add basic validation and lifecycle guards for repeated start/stop button usage.
9. Verify the demo manually and with build validation.

## Implementation Order
1. Create the WinForms project and add references.
2. Build the static form layout and wire application startup.
3. Implement the service host controller and validate start/stop behavior.
4. Implement the two-item publisher controller using the current demo logic as source material.
5. Implement the writable demo controller with actual broker write reaction.
6. Connect controller events to the UI and logging surface.
7. Test the full end-to-end flow with a second participant writing to the writable item.
8. Add any small polish needed for usability and lifecycle stability.

## Technical Constraints
- Keep code, comments, UI text, and public identifiers in English.
- Keep chat responses in German, but all user-visible application text must be English.
- Reuse existing packages and conventions instead of introducing demo-only duplicates.
- Avoid blocking calls such as `.Result` or `.Wait()` on the UI thread.
- Keep the form code thin; place background behavior in focused helper classes.
- Do not expand scope into unrelated package restructuring.

## Acceptance Criteria
- A WinForms demo project exists and builds successfully.
- The UI can start and stop a local selfhosted MQTT ItemBroker service.
- Two demo items are published and visibly updated in the UI.
- One writable item is published and reacts visibly to external write requests.
- Logging and status output make the demo understandable for desktop-oriented users.
- The UI remains responsive during runtime operations.

## Verification
- Build the new WinForms project and the affected solution.
- Launch the demo and verify:
  - service start works
  - service stop works
  - temperature and pressure values update continuously after publishing starts
  - the writable item is visible and initialized
  - an external MQTT participant can write to the writable item
  - the UI updates to reflect the external write
- If feasible, validate that repeated start/stop actions do not leave the app in a broken state.

## Out of Scope
- Replacing existing console demos
- Formal automated UI testing
- Production-grade visual design
- Broader refactoring of `Amium.ItemBroker.Mqtt` or `Amium.ItemBroker.Mqtt.Client`

## Risks / Watchouts
- External write behavior depends on correct item ownership and path consistency.
- Using the same client identity for multiple internal roles could create confusing behavior.
- Background timers and subscriptions must be disposed cleanly when stopping the demo.
- WinForms cross-thread access errors are likely if controller callbacks update controls directly.

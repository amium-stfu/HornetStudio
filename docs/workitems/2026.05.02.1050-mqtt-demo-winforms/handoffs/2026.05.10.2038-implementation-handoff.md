# IMPLEMENTATION HANDOFF

## Goal

Rebuild `Amium.Item.Client.MqttDemoWinForms` as a very flat, human-readable WinForms sample that can also serve as a starter code template for MQTT-backed Amium Item clients.

The sample should demonstrate:

- Starting a local MQTT item server.
- Creating two MQTT clients connected to that server.
- Showing visible item state for both clients in two `TreeView` controls.
- Publishing simple demo values, based on the existing console sample behavior.

## Scope

Included:

- Replace the current WinForms sample structure with a simpler sample-focused implementation.
- Keep the project under `amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms`.
- Use existing `Amium.Item`, `Amium.Item.Client`, and `Amium.Item.Server` APIs.
- Keep code readable enough to be copied as a template.
- Add concise XML documentation for public classes and public methods if any are introduced.
- Keep user-visible UI strings and validation/error text in English.
- Update relevant documentation if sample usage or behavior changes.

Not included:

- No new runtime dependencies.
- No advanced settings UI for host, port, base topic, credentials, or TLS.
- No retained-topic cleanup tooling unless an existing API makes it trivial.
- No editable TreeView, write-back editor, or writable item demo in the first rebuild.
- No broad refactoring outside this sample and directly relevant documentation.

## Starting Point

- Current WinForms sample is located at `amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms`.
- Current code uses separate controller classes under `Controllers/`, but the new sample should be flatter and easier to follow.
- Current project file has `RootNamespace` set to `Amium.Item.Client.MqttDemoWinForms`, while existing code namespaces use `Amium.Item.Server.MqttDemoWinForms`; this mismatch should be fixed.
- Existing console publisher sample is at `amium_item/apps/samples/Amium.Item.Client.MqttDemoClient/Program.cs`.
- Current usage documentation is at `amium_item/docs/amium-itembroker-usage.md`.
- Prefer `BaseTopic = string.Empty` for consistency with current docs and the console sample.
- Prefer demo paths that visually map well into a tree, such as `demo_client.sensors.temperature` and `demo_client.sensors.pressure`. If reusing the console sample verbatim conflicts with that, align the WinForms sample and documentation explicitly.

## Tasks

1. Review the current WinForms sample and remove the old controller-oriented shape where it makes the sample harder to read.
2. Rework the UI into a simple template-style form with lifecycle buttons, two client `TreeView` controls, and a log/status area.
3. Implement local server creation/start logic directly enough that sample readers can follow the lifecycle.
4. Implement two client creation flows, both connected to the local server.
5. Implement a small demo publisher based on the console sample, publishing temperature and pressure values periodically.
6. Populate both TreeViews with visible item values and properties from the clients' observed remote state.
7. Add clear start/stop/dispose handling so the form shuts down cleanly.
8. Update documentation only where behavior or usage instructions changed.
9. Build the affected project and, if practical, the relevant `amium_item` solution.

## Implementation Order

1. Normalize namespace and project identity for `Amium.Item.Client.MqttDemoWinForms`.
2. Replace the UI with the simplified layout:
   - top command row,
   - two side-by-side TreeViews,
   - status/log area.
3. Add local server lifecycle:
   - create/start on `Create Server`,
   - disable conflicting buttons while running,
   - dispose on form close.
4. Add client lifecycle:
   - `Create Client 1`,
   - `Create Client 2`,
   - both connect to the local server with distinct client IDs.
5. Add TreeView refresh logic for each client.
6. Add `DemoItemPublisher`:
   - initial retained snapshots,
   - periodic value updates,
   - optional simple metadata property updates if consistent with the console sample.
7. Wire `Start Publish` and optional `Stop Publish`.
8. Remove obsolete controller files and unused UI fields.
9. Update documentation if final paths, topics, commands, or behavior differ from current docs.
10. Run focused build verification.

## Technical Constraints

- Answer/chat language remains German; code, comments, XML documentation, UI strings, file names, and technical identifiers must be English.
- Do not add new dependencies.
- Use existing project APIs and patterns from `Amium.Item.Client` and `Amium.Item.Server`.
- Keep the sample deliberately flat; avoid new services, wrappers, or abstractions unless needed to keep the code readable.
- Prefer named arguments where calls have more than two parameters or ambiguous values.
- Use async/await correctly; avoid `.Result` and `.Wait()`.
- Dispose async resources during form closing.
- User-visible errors and validation messages must be English.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.

## Acceptance Criteria

- WinForms sample builds.
- Starting the app shows a simple UI with server/client/publish controls, two TreeViews, and a log/status area.
- `Create Server` starts a local MQTT item server on `127.0.0.1:1883`.
- `Create Client 1` and `Create Client 2` connect to the local server with distinct client IDs.
- `Start Publish` publishes temperature and pressure demo items periodically.
- Both TreeViews show the same observed demo item values/properties after publishing starts.
- Closing the form stops publishing and disposes clients/server without unhandled exceptions.
- Obsolete controller classes are removed if they are no longer part of the sample.
- Documentation is consistent with the final demo paths/topics.

## Verification

- Run:
  `dotnet build amium_item/Amium.Item.sln --no-restore`
- If the solution build is too broad or already known to need restore, run the focused project build:
  `dotnet build amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms/Amium.Item.Client.MqttDemoWinForms.csproj --no-restore`
- Manually launch the WinForms sample if the environment supports desktop UI and verify:
  - server can be created,
  - both clients can be created,
  - publishing starts,
  - both TreeViews update,
  - closing the form is clean.

## Out of Scope

- Repairing the existing controller design without simplifying it.
- Adding MQTT authentication, TLS, remote broker configuration, or persistent settings.
- Adding editable item values or write-request demonstrations.
- Changing core client/server library behavior.
- Adding new unit-test infrastructure for WinForms UI.

## Risks / Watchouts

- Port `1883` may already be in use; report this clearly in the UI/log instead of failing silently.
- The sample should avoid reflecting a client's own published items in confusing ways; choose the client API and visible state behavior intentionally.
- TreeView updates must marshal back to the UI thread.
- High-frequency updates can make the TreeView flicker; refresh at a human-friendly cadence if direct updates are too noisy.
- Documentation currently mentions paths/topics that may not exactly match the console sample. Align the rebuilt WinForms sample and docs.

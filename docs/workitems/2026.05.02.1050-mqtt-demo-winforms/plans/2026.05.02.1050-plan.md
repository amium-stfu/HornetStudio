# PLAN

## Topic
WinForms demo application for the MQTT ItemBroker with selfhosted service control, two published demo items, and one writable item reacting to external changes.

## Problem
The current demo setup is split across console applications and is technically useful, but not ideal for teams that are still strongly oriented around WinForms. A Windows desktop demo would make the selfhosted service role, local publishing behavior, and writable item routing much easier to understand and demonstrate. Without a dedicated structure, there is a risk that demo UI code, broker hosting, and MQTT client behavior become tightly mixed.

## Goals
- Add a WinForms-based demo application tailored for colleagues who prefer desktop-style tooling.
- Demonstrate three concrete capabilities in one app:
  - start and stop a local selfhosted MQTT ItemBroker service
  - publish two simple demo items similar to the existing `MqttDemoClient`
  - publish one writable item that visibly reacts to external writes
- Keep internal responsibilities separated so the demo remains understandable and maintainable.
- Preserve the existing console demos as small technical examples.

## Scope
Included:
- New WinForms demo project
- UI structure and interaction model
- Demo composition for host, publisher, and writable item roles
- Logging and status presentation needed for demonstration

Excluded:
- Replacing the existing console demos
- Broad refactoring of the MQTT packages
- Production-grade styling or advanced desktop UX
- Packaging or release work

## Current State
- `Amium.Item.Server` already demonstrates starting a local selfhosted MQTT ItemBroker host.
- `Amium.Item.Server.MqttDemoClient` already demonstrates publishing two simple items over MQTT.
- `MqttItemServerHost` already provides the selfhosted service-side runtime.
- `MqttItemClientSession` already provides the MQTT client behavior required for publishing, subscribing, and reacting to writes.
- There is currently no WinForms demo that combines these concepts into one visible desktop workflow.

## Target Structure
- New project: `src/Amium.Item.Server.MqttDemoWinForms`
- Internal structure should separate:
  - `Service Host Controller`
  - `Demo Publisher Controller`
  - `Writable Demo Controller`
  - `MainForm` UI composition and event wiring

## UI Direction
- One main form is sufficient.
- Recommended visible sections:
  - `Service`
    - Start button
    - Stop button
    - Host/port/base-topic display
    - running/stopped state
  - `Published Items`
    - current temperature value
    - current pressure value
    - start publishing button
    - stop publishing button
  - `Writable Item`
    - current writable item value
    - last external write value or timestamp
    - indication that the item is writable
  - `Log`
    - chronological status and diagnostic output

## Design Decisions
1. Keep the WinForms demo as a separate project rather than replacing the existing console demos.
2. Use the current selfhosted host and MQTT client packages instead of introducing demo-only broker logic.
3. Model the writable item as a real owner-published item so that external writes exercise the actual broker routing semantics.
4. Keep the form thin by moving service, publisher, and writable responsibilities into dedicated demo classes.
5. Prefer clarity and observability over feature richness.

## Demo Behavior
1. The user can start a local selfhosted MQTT service from the UI.
2. The user can start local publishing of two demo items:
   - `Temperature`
   - `Pressure`
3. The app also exposes one writable item:
   - initially published by the local demo client
   - marked writable for external participants
   - updates locally when a remote write request is received
4. The app logs lifecycle actions and important item changes.

## Suggested Internal Responsibilities
- `Service Host Controller`
  - own and lifecycle-manage `MqttItemServerHost`
  - expose start/stop state changes
  - surface errors and diagnostics

- `Demo Publisher Controller`
  - own a `MqttItemClientSession`
  - publish retained snapshots for the two demo items
  - run the periodic update loop
  - expose current values to the UI

- `Writable Demo Controller`
  - own or share a session, depending on final composition choice
  - publish the writable item snapshot
  - react to incoming write requests
  - surface last written value and status to the UI

- `MainForm`
  - wire buttons and status labels
  - marshal background updates onto the UI thread
  - append logs without embedding transport logic

## Sequencing
1. Create the new WinForms project and add the required project references.
2. Build the main form layout for service, published items, writable item, and log sections.
3. Implement the service host controller around `MqttItemServerHost`.
4. Implement the demo publisher controller based on the current `MqttDemoClient` behavior.
5. Implement the writable demo controller with real external write reaction.
6. Wire all background updates safely into the WinForms UI.
7. Verify lifecycle handling, publish behavior, and external write behavior.

## Risks
- Async lifecycle handling may freeze or confuse the UI if marshaling is not done carefully.
- Combining host and client roles in one app can create accidental self-mirroring if session identities are not chosen carefully.
- Writable item behavior may appear broken if ownership, path choice, or initial publishing is inconsistent.
- A too-thin logging surface can make it hard to explain what happened during writes or reconnects.

## Acceptance Direction
- The app starts and stops a local MQTT ItemBroker service from buttons.
- The app publishes two demo items and updates them visibly.
- The app exposes one writable item and visibly updates when another participant writes to it.
- The UI remains responsive during start, stop, publish, and external write activity.
- The code remains easy to follow for future demo maintenance.

# IMPLEMENTATION HANDOFF

## Goal
Modularize the repository agent instructions by moving the large root `AGENTS.md` rule set into focused Markdown files under `/agents`, while keeping `AGENTS.md` as the authoritative entry point for mode priority, loading order, and references.

Also extend the `DEBUG` mode rules with the validation and stop-condition rules provided by the user from `t:\debug.txt`. The file `t:\debug.txt` is only source material for the new rules and must not become part of the runtime workflow or be referenced as a required input file.

## Scope
Included:
- Create an `/agents` folder at the repository root.
- Split existing general and mode-specific instructions into focused files.
- Keep root `AGENTS.md` as the main entry point with priority rules, mode handling, and references to `/agents/*.md`.
- Move solution-specific rules into `/agents/solution.md`.
- Add the debug validation and autonomous-fix stop rules from the user-provided `debug.txt` content into the appropriate modular files.
- Preserve existing behavior and intent of the current agent instructions.
- Update Markdown only.

Explicitly NOT included:
- No source code changes.
- No build, test, or release logic changes.
- No change to actual project structure outside documentation/rules files.
- No implementation of new tooling that reads `/agents/*.md`.
- No reference to `t:\debug.txt` as a future required file.

## Starting Point
- Root `AGENTS.md` currently contains priority rules, mode aliases, mode definitions, workflow rules, workitem rules, coding style, documentation, testing, git/security, release, publish, and FOSS/license rules in one large file.
- The user wants this content split into single-purpose files.
- The desired target files are:
  - `/agents/AGENTS.md`
  - `/agents/solution.md`
  - `/agents/modes.md`
  - `/agents/planning.md`
  - `/agents/implementation.md`
  - `/agents/debugging.md`
  - `/agents/architecture.md`
  - `/agents/naming.md`
  - `/agents/documentation.md`
  - `/agents/testing.md`
  - `/agents/release.md`
- Because path-relevant new names must use `snake_case`, all newly created files/folders should use lowercase snake_case where applicable. `AGENTS.md` is an existing convention/name and may remain uppercase by intent.
- The root `AGENTS.md` should remain present because external tooling commonly discovers that file name at the repository root.

2. Create `/agents` and the target Markdown files.
3. Move or rewrite the rules into focused files without changing their meaning.
4. Keep root `AGENTS.md` concise and authoritative:
   - priority rules
   - language rules

   - loading/reference order
   - links to `/agents/*.md`
5. Decide whether `/agents/AGENTS.md` is needed as a mirrored entry point or whether root `AGENTS.md` should be the only entry point. If created, it must not conflict with root `AGENTS.md`.
6. Put mode aliases and full mode definitions in `/agents/modes.md`.
7. Put planning/workitem/handoff rules in `/agents/planning.md`.

12. Remove duplicated or conflicting text across files.

14. Verify Markdown files are readable and do not include machine-specific references except examples already present in the established rules.

## Technical Constraints
- Chat responses must remain in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must remain in English.
- Newly created path-relevant names must use `snake_case`, except intentional convention names like `AGENTS.md`.
- Do not introduce dependencies.
- Do not alter source code.
- Do not alter public APIs.
- Do not create release versions.
- Do not reference `t:\debug.txt` as a required future workflow artifact.
- Preserve the existing priority that explicit `MODE` instructions override all other request rules.

## Acceptance Criteria
- Root `AGENTS.md` is concise and points to the modular rule files.
- `/agents` contains the agreed focused Markdown files.
- Mode aliases and mode rules are still unambiguous.
- `DEBUG` mode includes the new reproducible-validation and autonomous-stop rules.
- `solution.md` contains only solution-specific rules and does not duplicate general policy.
- General rules are stored in the appropriate thematic files.
- No conflicting duplicate rule sets remain.
- The resulting instruction structure is easier to maintain than the current single large file.

## Verification
- Manually review `AGENTS.md` and all `/agents/*.md` files for priority conflicts.
- Run:
  - `Get-ChildItem -LiteralPath 'agents'`
  - `Select-String -Path 'AGENTS.md','agents/*.md' -Pattern 'debug.txt','t:\\debug.txt'`
- Expected result: no future workflow reference to `debug.txt`; the concepts from that file are present in `debugging.md` and/or `testing.md`.
- No solution build is required because only instruction Markdown files are changed.

## Out of Scope
- Implementing code changes in `amium_item` or other projects.
- Adding tooling to automatically merge/read agent rule files.
- Reworking existing workitem contents.
- Creating or changing release artifacts.
- Changing actual debug report storage outside the established `docs/workitems/.../debug/` pattern.

## Risks / Watchouts
- Splitting rules can make precedence unclear. Root `AGENTS.md` must define a clear priority/loading order.
- Duplicating the same rule in multiple files can create future contradictions. Prefer one canonical location plus references.
- `/agents/AGENTS.md` may confuse tooling if it appears to compete with root `AGENTS.md`; if it is created, clearly state that root `AGENTS.md` is authoritative.
- Moving mode rules out of root `AGENTS.md` must not weaken the absolute priority of explicit mode instructions.
- Debug stop-condition rules should prevent speculative fix loops without blocking normal focused debugging.

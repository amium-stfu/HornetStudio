# IMPLEMENTATION HANDOFF

## Goal
Create a dedicated `Amium.Item.sln` area with clean production libraries, tests, samples, and tools so the Amium.Item domain can become a reusable foundation for HornetStudio and other solutions.

The intended production assemblies are:

- `Amium.Item.dll`
- `Amium.Item.Server.dll`
- `Amium.Item.Client.dll`
- `Amium.Item.Mesh.dll`

`Amium.Item.dll` is the central core/domain/contracts assembly. It must not become an all-in-one runtime assembly. Server, client, and mesh functionality remain separate assemblies.

## Scope
Included:

- Create a dedicated `Amium.Item.sln` in the repository.
- Reorganize Amium.Item-related projects into a clean solution-oriented layout.
- Keep production libraries separate from tests, samples, demos, hosts, monitors, and stress tools.
- Clean up legacy Amium.Item project leftovers and merge folders during the extraction.
- Preserve the current HornetStudio solution while preparing HornetStudio to consume the Amium.Item libraries through explicit project references or later package references.
- Keep MQTT as an internal transport area inside `Amium.Item.Server` and `Amium.Item.Client`.
- Move or prepare multimaster mesh logic under `Amium.Item.Mesh`.
- Update project references so the Amium.Item project graph is clear and reusable.
- Update relevant documentation for the new solution structure.

Explicitly not included:

- Publishing NuGet packages.
- Creating release versions or Git tags.
- Moving to a separate repository.
- Making MQTT transport-neutral.
- Combining all functionality into a single `Amium.Item.dll`.
- Implementing ESP32-specific runtime code.
- Redesigning the MQTT topic protocol or mesh conflict model beyond what is necessary for clean structure.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- Newly created path-relevant names must use `snake_case`, unless preserving existing project names or existing .NET naming conventions.
- Use existing libraries and project patterns.
- Do not introduce new runtime dependencies unless technically required and justified.
- Keep MQTT as the built-in transport inside server/client.
- Do not create separate `Amium.Item.Mqtt.*` assemblies.
- Do not make `Amium.Item.dll` depend on server, client, mesh, MQTTnet, UI, or tool projects.
- Do not move demo UI code into production libraries.
- Keep refactoring limited to what is necessary for clean structure and references.
- Public classes and public methods changed or added during the implementation must have XML documentation.
- Update tests for moved or changed logic.

## Acceptance Criteria
- `Amium.Item.sln` exists and builds.
- Production assemblies are clearly separated:
  - `Amium.Item.dll`
  - `Amium.Item.Server.dll`
  - `Amium.Item.Client.dll`
  - `Amium.Item.Mesh.dll`
- `Amium.Item.dll` has no dependency on server, client, mesh, MQTTnet, UI, samples, tools, or tests.
- `Amium.Item.Server.dll` may depend on `Amium.Item.dll` and MQTT server dependencies.
- `Amium.Item.Client.dll` may depend on `Amium.Item.dll` and MQTT client dependencies, but not on `Amium.Item.Server.dll`.
- `Amium.Item.Mesh.dll` may depend on `Amium.Item`, `Amium.Item.Server`, and `Amium.Item.Client` only if needed for orchestration.
- Tests are under `tests/`.
- Samples/demos are under `samples/`.
- Operational utilities are under `tools/`.
- Obsolete `.__merge` folders are removed or reduced to no relevant content.
- Standalone `Amium.Item.Client.Mqtt` is removed or explicitly documented as a temporary compatibility exception.
- `HornetStudio.sln` still builds after reference updates.
- HornetStudio references only the Amium.Item assemblies it actually needs.

## Verification

- Verify `Amium.Item.csproj` has no MQTTnet, server, client, mesh, UI, sample, tool, or test references.
- Verify no source in `Amium.Item.Client` uses `Amium.Item.Server` unless explicitly documented as a temporary compatibility exception.
- Verify `HornetStudio.sln` does not reference removed project paths.

## Out of Scope
- Release publishing.
- Version number generation.
- Git tag creation.
- NuGet package publishing.
- Separate repository migration.
- ESP32-specific implementation.
- Transport-neutral architecture redesign.
- MQTT topic model redesign.
- Mesh conflict-resolution redesign.

## Risks / Watchouts
- Moving projects can break solution entries, relative project references, generated paths, and test discovery.
- `Amium.Item.Client.Mqtt` currently references `Amium.Item.Server`; breaking that dependency may require carefully moving shared mapping/value logic into `Amium.Item`.
- The `.__merge` folders may contain obsolete or duplicated code; do not blindly copy them into the new structure.
- Demo code may contain useful production mesh logic mixed with UI assumptions. Move only reusable non-UI logic.
- A single central `Amium.Item.dll` should not become a hidden monolith. Keep it dependency-light.
- HornetStudio may still need server/client references if it hosts or connects to runtime Item services. Do not force a single-reference model if the actual behavior requires more assemblies.
- Path moves are broad. Keep commits/review chunks small if this is implemented in stages.

# IMPLEMENTATION HANDOFF

## Goal
Improve the MQTT stress test metric semantics and final assessment so the tool clearly distinguishes configured signal count, total update rate, live pending backlog, and final delivery quality.

## Scope
Included:
- Rename ambiguous UI labels.
- Add a readonly average updates-per-value display.
- Replace live `Lost` with `Pending`.
- Replace live `Loss rate` with `Max pending`.
- Track `MaxPending` inside the metrics model.
- Replace the existing end-of-run assessment log with a clearer PASS/WARN/FAIL assessment block.
- Keep the existing random signal selection and total-rate publish model.
- Build the complete solution.

Explicitly NOT included:
- Per-signal update-rate mode.
- SLA configuration UI.
- Persistent report output.
- Charts.
- Changes to MQTT topic mapping or ItemBroker runtime behavior.

## Starting Point
- Current UI has `Signals` and `Messages/s`.
- `Messages/s` currently means total updates per second across all generated signal values.
- Current live `Lost` is calculated as `Published - Received`, so it behaves as live backlog and can decrease back to zero.
- Current final assessment logs delivery, latency, and throughput, but still uses the ambiguous `lost` terminology.
- Relevant project is `src/amium_itembroker_mqtt_stress_test/`.
- Current solution builds successfully after the initial stress test implementation.

## Tasks
1. Update load UI labels:
   - `Signals` -> `Signal values`
   - `Messages/s` -> `Total updates/s`
2. Add a readonly load UI value:
   - `Avg updates/s per value`
   - Formula: `Total updates/s / Signal values`
   - Update when either signal count or total updates/s changes.
3. Update live metrics labels:
   - `Lost` -> `Pending`
   - Replace `Loss rate` with `Max pending`
4. Extend `StressTestMetrics`:
   - Rename snapshot concept from live `Lost` to `Pending`.
   - Add `MaxPending`.
   - Update max pending whenever published or received counters change.
5. Update `StressTestMetricsSnapshot`:
   - Replace `Lost` with `Pending`.
   - Replace `LossRatePercent` with `MaxPending`.
   - Keep existing throughput and latency fields.
6. Update `MainForm` rendering:
   - Render pending and max pending.
   - Render average updates per value.
   - Keep UI-thread updates through existing timer/event pattern.
7. Update `StressTestController` final assessment:
   - Compute `finalLost` from the final snapshot pending value after the settle delay.
   - Log `Assessment overall`.
   - Log `Delivery`, `Throughput`, `Latency`, `Backlog`, and `Load profile`.
8. Apply assessment thresholds:
   - Delivery `FAIL`: final lost, duplicates, or out-of-order above zero.
   - Throughput `PASS`: achieved >= 99%; `WARN`: achieved >= 95%; `FAIL`: achieved < 95%.
   - Latency classes: `EXCELLENT <= 10 ms`, `GOOD <= 50 ms`, `ELEVATED <= 200 ms`, `HIGH > 200 ms`.
   - Backlog `FAIL`: final pending above zero; `WARN`: high max pending with final pending zero; `GOOD`: otherwise.
9. Run build verification.

## Acceptance Criteria
- The UI no longer shows ambiguous `Messages/s`.
- The UI makes clear that update rate is total update rate.
- The UI shows average updates per value as a derived readonly value.
- Live backlog is shown as `Pending`, not `Lost`.
- The UI shows `Max pending`.
- End-of-run assessment logs overall, delivery, throughput, latency, backlog, and load profile.
- Final delivery loss is assessed only after the settle delay.
- The solution builds successfully.

## Out of Scope
- Per-signal update-rate mode.
- User-configurable assessment thresholds.
- Historical result export.
- Automated WinForms UI tests.
- MQTT broker changes.

## Risks / Watchouts
- Renaming snapshot record fields will require updating every UI/controller reference.
- `MaxPending` should be tracked inside metrics rather than inferred from UI polling, otherwise short backlog spikes can be missed.
- A short settle delay reduces false final loss, but cannot prove broker-level delivery under every QoS/session condition.
- With random signal selection, average updates per value is informational and not a guarantee for each individual signal.

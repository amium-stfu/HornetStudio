# IMPLEMENTATION HANDOFF

## Goal
Create a standalone WinForms MQTT ItemBroker stress test application that publishes configurable random signal updates and measures delivery quality through a direct raw MQTT subscriber.

## Scope
Included:
- New standalone stress test GUI application.
- Configurable broker connection settings.
- Configurable random signal count.
- Configurable steady message rate and test duration.
- Direct MQTT subscription for measuring received messages.
- Live metrics for published, received, lost, duplicate, out-of-order, throughput, and latency.
- Basic logging and run lifecycle controls.

Explicitly NOT included:
- Editor integration.
- Replacing existing demo applications.
- New external runtime dependencies beyond the MQTTnet version already used by the MQTT client project.
- Persistent database reporting.
- Complex charts.
- Distributed multi-machine measurement.

## Starting Point
- Current behavior:
  - `src/Amium.Item.Server.MqttDemoClient/Program.cs` publishes two demo values through `MqttItemClientSession`.
  - `src/Amium.Item.Server.MqttDemoWinForms/` provides a small WinForms reference style.
  - `src/Amium.Item.Client.Mqtt/MqttItemClientSession.cs` can publish ItemBroker value updates and can subscribe through an item abstraction.
  - `src/Amium.Item.Server.Mqtt/MqttTopicMapping.cs` defines the MQTT topic mapping.
- Relevant assumptions:
  - Loss measurement must observe every MQTT message, not just reconstructed item state.
  - Publisher and subscriber can run inside the same stress test process for version 1.
  - Local timestamp latency is valid for version 1 because send and receive timestamps come from the same machine and process.
  - Stress payloads can use a test-specific textual envelope in the `read` value.
- Required prerequisites:
  - A running MQTT ItemBroker endpoint, usually `127.0.0.1:1883`.
  - The implementation should build without running any application as part of automated verification.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Newly created path-relevant names should use `snake_case`.
- Public classes and public methods need XML documentation.
- Use existing libraries, frameworks, and project conventions.
- Do not introduce new external dependencies without justification.
- Use `async` and `await`; avoid `.Result`, `.Wait()`, and UI-thread blocking.
- Keep the WinForms UI responsive by polling metric snapshots or throttling UI updates.
- Do not update UI controls directly from MQTT callbacks or background tasks.
- Avoid retained messages by default for stress runs.
- Keep stress data under a dedicated root such as `stress`.

## Acceptance Criteria
- A new standalone WinForms stress test project exists.
- The solution includes the new project.
- The user can configure host, port, base topic, signal count, message rate, duration, and retained flag.
- The user can start, stop, and reset a test run.
- The publisher emits deterministic stress signal paths with per-signal sequence numbers.
- The subscriber receives raw MQTT messages through a direct MQTT subscription.
- The metrics view shows published total, received total, lost total, loss rate, duplicates, out-of-order count, publish rate, receive rate, average latency, max latency, p95 latency, and p99 latency.
- The UI remains responsive while a test is running.
- Build verification succeeds.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore`.
- If solution build is too broad for the current environment, run `dotnet build src/amium_itembroker_mqtt_stress_test/amium_itembroker_mqtt_stress_test.csproj --no-restore`.
- Manual verification:
  - Start or connect to an MQTT ItemBroker at `127.0.0.1:1883`.
  - Launch the stress test application.
  - Start a small steady test, for example 10 signals at 100 messages/sec.
  - Confirm published and received counts increase.
  - Confirm loss stays at zero under low local load.
  - Increase signal count or message rate and observe latency and loss behavior.
  - Stop the run and confirm counters stop changing.
  - Reset the run and confirm old messages do not pollute a new run.

## Out of Scope
- Automated UI testing.
- Persistent historical reports.
- Advanced chart rendering.
- Broker implementation changes.
- Release packaging.
- Multi-process or multi-machine clock synchronization.

## Risks / Watchouts
- JSON payloads on `read` topics can interact badly with current topic mapping heuristics; use a compact non-JSON textual payload for version 1.
- Retained messages can create false duplicate or stale receive events at run start; keep retained disabled by default and clear measurement state per run.
- Updating WinForms controls for every received message can overload the UI; throttle UI updates to a fixed interval.
- Percentile latency calculations should use bounded samples or a rolling window for long-running soak tests.
- Very high rates may expose broker limits and client-side limits at the same time; label metrics clearly so conclusions are not overstated.

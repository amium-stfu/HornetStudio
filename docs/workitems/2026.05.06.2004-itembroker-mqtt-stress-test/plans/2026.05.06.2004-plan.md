# PLAN

## Topic
Small standalone MQTT ItemBroker stress test application with configurable random signals and measurable broker transport quality.

## Problem
The current MQTT demo clients are useful for functional checks, but they do not measure whether the broker behaves reliably under higher load. A stress test that only publishes random values would show publishing throughput, but it would not prove delivery quality. To evaluate loss, latency, jitter, duplicates, and message ordering, the tool needs an independent receive path that observes the raw MQTT traffic produced by the test.

## Goals
- Add a standalone desktop application for MQTT ItemBroker stress testing.
- Allow the user to configure the number of random signals and the generated update load.
- Measure delivery quality with direct MQTT subscription rather than relying on a remote item registry snapshot.
- Keep the first version small, understandable, and useful for local broker validation.
- Reuse existing MQTT, item, and broker conventions without adding new external runtime dependencies.

## Scope
Included:
- New standalone GUI application for stress testing.
- Configurable broker connection settings.
- Configurable random signal count and load profile.
- Publisher role for generated test signals.
- Independent direct MQTT subscriber role for raw receive measurement.
- Live metrics for throughput, loss, duplicates, ordering, and latency.
- Basic run lifecycle, logging, and reset behavior.

Excluded:
- Integration into the HornetStudio editor UI.
- Replacing existing MQTT demo applications.
- Production-grade reporting, database persistence, or long-term monitoring.
- Advanced charts in the first version.
- Distributed multi-machine latency measurement.
- Broker core refactoring unless a blocking limitation is discovered.

## Current State
- `src/Amium.Item.Server.MqttDemoClient/Program.cs` publishes simple demo items over MQTT.
- `src/Amium.Item.Server.MqttDemoWinForms/` already demonstrates a small WinForms application around the MQTT broker.
- `src/Amium.Item.Client.Mqtt/MqttItemClientSession.cs` publishes ItemBroker item updates and internally maintains a remote item registry.
- `MqttItemClientSession.RemoteItems` reconstructs item state and is useful for UI state views, but it is not the preferred measurement source for loss because state reconstruction can hide overwritten intermediate updates.
- `src/Amium.Item.Server.Mqtt/MqttTopicMapping.cs` defines the current item-to-topic mapping and should be reused for consistent topic handling.

## Target Structure
Create a new standalone stress test project. Because project-specific rules require newly created path-relevant names to use `snake_case`, the planned project folder should use a snake_case path while keeping .NET identifiers, assembly name, and namespace in normal .NET casing.

Recommended target:

- `src/amium_itembroker_mqtt_stress_test/`

Internal responsibilities:

- `MainForm`
  - GUI composition, button wiring, settings display, metric display, and log output.
- `StressTestController`
  - Owns a test run lifecycle and coordinates publisher, subscriber, metrics, and cancellation.
- `StressTestRunSettings`
  - Captures host, port, base topic, signal count, load mode, message rate, duration, and retained flag.
- `RandomSignalCatalog`
  - Builds deterministic signal paths and per-signal state.
- `StressTestPublisher`
  - Publishes generated signal updates.
- `RawMqttStressSubscriber`
  - Subscribes directly to the MQTT item topics and forwards every matching raw message into the metrics layer.
- `StressTestMetrics`
  - Tracks expected sequences, received sequences, latency samples, throughput windows, duplicates, out-of-order events, and errors.
- `StressTestPayload`
  - Defines the compact textual payload envelope used for measurable messages.

## Key Architecture Decision
Use a direct MQTT subscriber for measurement.

The publisher may use the existing `MqttItemClientSession` to publish through the same client abstraction as normal ItemBroker clients. The measurement subscriber should use MQTTnet directly and subscribe to the relevant raw item topics. This avoids hiding message loss or order changes behind item-state reconstruction.

## Payload Strategy
For the first version, publish a compact textual value payload on each signal's `read` topic rather than JSON.

Recommended payload fields:

- sequence number
- UTC send timestamp or timestamp ticks
- random value

Reason:
- The current MQTT topic mapper uses payload shape to distinguish some item values from metadata.
- A non-JSON textual payload avoids confusing value messages with metadata messages.
- The subscriber can parse the exact sequence and timestamp from the same message whose delivery is being measured.

Example conceptual payload shape:

- `seq=42;sent=2026-05-06T18:04:12.1234567Z;value=0.7342`

This is intentionally not a public protocol change. It is a stress-test payload format scoped to the new tool.

## Load Modes
First version should support at least:

1. Steady load
   - Constant signal count and message rate.
   - Best for a repeatable baseline.
2. Ramp up
   - Starts with low throughput and increases periodically.
   - Helps find the point where loss or latency becomes unacceptable.
3. Burst
   - Alternates idle or low-load periods with short high-load bursts.
   - Useful for queue and backpressure behavior.
4. Soak
   - Long-running steady test.
   - Useful for memory, timer, reconnect, and stability issues.

If implementation time must be reduced, start with steady load and leave the other modes as clearly isolated follow-up options.

## Metrics Model
Track metrics globally and per signal.

Global metrics:

- published message count
- received message count
- expected message count
- lost message count
- loss rate
- duplicate message count
- out-of-order message count
- publish messages per second
- receive messages per second
- latency min, max, average, p95, p99
- jitter
- current error count

Per-signal metrics:

- last published sequence
- last received sequence
- missing sequence count
- duplicate sequence count
- out-of-order sequence count
- last latency
- max latency

Loss detection:

- Each signal has monotonically increasing sequence numbers.
- If received sequence jumps from `N` to `N + K`, count `K - 1` missing messages for that signal.
- If a sequence lower than the last accepted sequence arrives, count out-of-order.
- If the same sequence arrives more than once, count duplicate.

Latency detection:

- Compare local receive timestamp with embedded send timestamp.
- Since publisher and subscriber run in the same application process for version 1, no clock synchronization is required.

## Design Decisions
1. Build a separate application, not an editor widget.
2. Use WinForms to match the existing MQTT demo style and keep implementation small.
3. Use direct MQTT subscription for measurement.
4. Keep the first UI metric-focused, not chart-focused.
5. Use deterministic signal names so repeated runs are comparable.
6. Keep generated signal paths under a dedicated root, for example `stress.signal_000001`.
7. Avoid retained messages by default during active stress testing; allow retained publishing only as an explicit option.
8. Reset measurement state at the start of every run so retained or previous messages do not pollute results.

## Suggested Signal Paths
Use a dedicated test root to avoid mixing stress data with normal broker data:

- `stress.signal_000001.read`
- `stress.signal_000002.read`
- `stress.signal_000003.read`

The visible item path should be normalized through the same item path/topic mapper rules used by the existing MQTT implementation.

## Risks
- If the publisher uses JSON payloads on `read`, current mapping heuristics can interpret messages differently than expected.
- If retained messages are enabled by default, old messages can distort received counts at the start of a run.
- Very high message rates can overload the WinForms UI if every message updates controls directly.
- Latency percentiles require bounded sample storage or rolling windows to avoid unbounded memory growth during soak tests.
- Direct MQTT subscription is better for measurement, but it means the measurement path intentionally bypasses `MqttRemoteItemRegistry`.

## Acceptance Direction
- The tool can start a stress run against a configured broker.
- The user can configure at least signal count and message rate.
- The tool publishes random signal updates under a dedicated stress root.
- The tool receives matching raw MQTT messages through an independent direct subscriber.
- The UI reports loss, latency, duplicates, ordering issues, and throughput live.
- The UI remains responsive during a running test.
- The solution builds without introducing new external runtime dependencies.

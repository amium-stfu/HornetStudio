# MQTT STRESS TEST METRICS AND ASSESSMENT PLAN

## Goal
Clarify the stress test load semantics and replace the current ambiguous live loss interpretation with interface-test metrics that can produce a useful end-of-run assessment.

## Problem Framing
The current UI uses `Signals` and `Messages/s`, but `Messages/s` means total publish rate, not per-signal rate. This can be read as `signals * messages/s`, which is not how the publisher works.

The current live `Lost` metric is also misleading. During a run it is mostly the current receive backlog:

```text
Pending = Published - Received
```

It can move from `1` back to `0`, which means the message was delayed or still in flight, not actually lost. Final loss should only be assessed after the run stops and the subscriber has had a short settle window.

## Proposed UI Semantics
Rename the load inputs to:

- `Signal values`
- `Total updates/s`
- `Duration s`

Add a readonly derived field:

- `Avg updates/s per value`

Formula:

```text
Avg updates/s per value = Total updates/s / Signal values
```

This should be presented as a descriptive average only because the current signal selection is random and does not guarantee exact per-signal distribution.

## Proposed Live Metrics
Keep live metrics focused on observable state:

- `Published`
- `Received`
- `Pending`
- `Max pending`
- `Duplicates`
- `Out of order`
- `Publish rate`
- `Receive rate`
- `Avg latency`
- `Max latency`
- `P95 latency`
- `P99 latency`
- `Elapsed`

Remove or avoid live `Loss rate` unless it is renamed to a pending/backlog concept. Prefer not showing a pending rate in the first iteration because it is less intuitive than absolute pending and max pending.

## Proposed Final Assessment
At the end of a run, after a short settle delay, log one compact assessment block:

- Overall status: `PASS`, `WARN`, or `FAIL`
- Delivery status
- Throughput status
- Latency status
- Backlog status
- Load profile

Example:

```text
Assessment overall: PASS
Delivery: PASS - 30,000 published, 30,000 received, 0 final lost, 0 duplicates, 0 out-of-order.
Throughput: PASS - target 500.0/s, actual 499.8/s, achieved 99.96%.
Latency: EXCELLENT - avg 0.44 ms, p95 0.76 ms, p99 1.11 ms, max 7.63 ms.
Backlog: GOOD - max pending 2, final pending 0.
Load profile: 500 signal values, 500 total updates/s, avg 1.0 updates/s per value.
```

## Assessment Rules
Use explicit thresholds rather than a numeric score.

Delivery:

- `FAIL` when `final lost > 0`
- `FAIL` when `duplicates > 0`
- `FAIL` when `out-of-order > 0`
- `PASS` otherwise

Throughput:

- `PASS` when target achieved is `>= 99%`
- `WARN` when target achieved is `>= 95%` and `< 99%`
- `FAIL` when target achieved is `< 95%`

Latency:

- `EXCELLENT` when `p99 <= 10 ms`
- `GOOD` when `p99 <= 50 ms`
- `ELEVATED` when `p99 <= 200 ms`
- `HIGH` when `p99 > 200 ms`

Latency should not force an overall `FAIL` without a user-defined SLA. `HIGH` should produce at most an overall `WARN` in this version.

Backlog:

- `GOOD` when `final pending = 0` and max pending is small relative to total updates/s
- `WARN` when `final pending = 0` but max pending is high
- `FAIL` when `final pending > 0`

Overall:

- `FAIL` if Delivery fails, Throughput fails, or final pending remains above zero.
- `WARN` if Throughput warns, Latency is `HIGH`, or Backlog warns.
- `PASS` otherwise.

## Architecture
Keep the change local to the stress test project.

The existing `StressTestMetrics` should remain the source of live counters and snapshots. Extend it with backlog-oriented fields rather than creating a second metric system.

The existing `StressTestController` should remain responsible for final assessment because it owns lifecycle, settings, and the final snapshot timing.

The UI should only render metric values and not contain assessment rules.

## Scope
Included:

- Rename ambiguous UI labels.
- Add readonly average updates-per-value display.
- Rename live `Lost` to `Pending`.
- Replace live `Loss rate` with `Max pending`.
- Track `MaxPending` in metrics.
- Update final assessment log output.
- Keep current random update distribution.
- Run a full solution build.

Not included:

- Per-signal rate mode.
- SLA configuration UI.
- Charts.
- Persistent reports.
- Exact per-signal distribution guarantees.

## Risks
- Changing labels without changing the underlying semantics could still confuse users if the final log does not repeat the load profile.
- `Max pending` depends on snapshot timing unless tracked directly in `StressTestMetrics`; it should be updated whenever published or received counters change.
- A fixed final settle delay can still report pending under high backlog if the subscriber is very delayed. This should be documented by the assessment text as final-after-settle, not absolute network truth.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Manual run examples:
  - `Signal values = 1`, `Total updates/s = 2000`
  - `Signal values = 500`, `Total updates/s = 500`
  - `Signal values = 500`, `Total updates/s = 2000`
- Confirm the publish rate matches `Total updates/s`, not `Signal values * Total updates/s`.
- Confirm live pending can move up and down while final lost is only assessed after completion.

# DEBUG REPORT

## Problem
The MQTT stress test stops immediately after start with an item value type mismatch.

## Expected Behavior
The stress publisher should write compact textual payloads to generated stress signal `read` values.

## Actual Behavior
The first update for a generated signal can fail because the item was initialized with a `double` and later receives a `string` payload.

## Error Messages / Logs
`Cannot assign value of type 'System.String' to property 'stress.signal_000072.read' of type 'System.Double'.`

## Relevant Code
`RandomSignalCatalog.CreateSignal` initialized each generated item with a numeric `InitialValue`.

`StressTestPublisher.PublishNextAsync` writes `StressTestPayload.Format()` into `signal.Item.Value`.

## Attempted Fixes
- Initialize generated item values with `null` while retaining the deterministic numeric `InitialValue` separately -> avoids the `double` to `string` type change and keeps the textual stress payload format.

## Current Hypothesis (optional)
- The type guard in `ItemProperty.Value` is working as designed; the stress test item initialization was inconsistent with its intended textual payload.

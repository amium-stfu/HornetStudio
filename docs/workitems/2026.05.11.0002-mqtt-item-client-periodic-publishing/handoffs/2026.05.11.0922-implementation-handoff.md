# IMPLEMENTATION HANDOFF

## Goal
Extend the current `MqttItemClient` periodic publishing helper from value-only publishing to a shared periodic publish scheduler that supports values, properties, and snapshots. Then simplify the MQTT demo applications so their loops mutate item state only while `MqttItemClient` owns cyclic publishing.

## Scope
Included:
- Keep the existing public `AddPeriodicValue(ItemModel item, int periodMilliseconds)` API.
- Add `AddPeriodicProperty(ItemModel item, string propertyName, int periodMilliseconds)`.
- Add `AddPeriodicSnapshot(ItemModel item, int periodMilliseconds)`.
- Refactor the current internal `PeriodicValueGroup` / `PeriodicValueRegistration` model into a shared periodic publish model.
- Group all periodic registrations by `periodMilliseconds`, regardless of publish kind, so one worker handles all registrations for the same period.
- Update the CLI demo in `amium_item/apps/samples/Amium.Item.Client.MqttDemoClient`.
- Update the WinForms demo publisher in `amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms`.
- Preserve demo-visible behavior where practical: temperature, pressure, meta, epoch, and raw values should still be updated and published if the demo currently depends on them.
- Add or update focused tests for periodic property and snapshot publishing.

Not included:
- Do not add a generic delegate-based periodic API.
- Do not add new dependencies.
- Do not change MQTT topic structure or retained semantics.
- Do not alter broker/server behavior.
- Do not redesign item modeling.
- Do not migrate unrelated samples.

## Starting Point
- `MqttItemClient` already has `AddPeriodicValue(ItemModel item, int periodMilliseconds)`.
- Internally it currently uses:
  - `_periodicValueGroups`
  - `PeriodicValueGroup`
  - `PeriodicValueRegistration`
  - `RunPeriodicValueGroupAsync`
  - `RemovePeriodicValueRegistrationAsync`
- The current worker calls `UpdateValueAsync`.
- `UpdatePropertyAsync` and `PublishSnapshotAsync` already exist as manual public client methods.
- The CLI demo still owns a `PeriodicTimer` loop and manually calls `UpdateValueAsync` and `UpdatePropertyAsync`.
- The WinForms demo publisher still uses `MqttItemClientSession` directly and manually publishes values/properties inside its loop.

## Tasks
1. Replace value-specific periodic internals with shared publish-kind internals.
2. Add a periodic publish kind representation for:
   - value
   - property
   - snapshot
3. Keep `AddPeriodicValue` as the value registration entry point.
4. Add `AddPeriodicProperty`.
5. Add `AddPeriodicSnapshot`.
6. Validate all public API inputs.
7. Make all registration methods return disposable handles.
8. Ensure one worker is used per period even if value, property, and snapshot registrations share the period.
9. Ensure worker loops copy registrations under lock and publish outside the lock.
10. Route each registration kind to the existing public manual publish method:
    - value -> `UpdateValueAsync`
    - property -> `UpdatePropertyAsync`
    - snapshot -> `PublishSnapshotAsync`
11. Update diagnostics so messages identify publish kind, item path, and property name when relevant.
12. Update tests for value/property/snapshot periodic publishing.
13. Refactor CLI demo so periodic publishing is registered once and the loop only mutates values/properties and writes console state.
14. Refactor WinForms demo publisher to use `MqttItemClient` instead of `MqttItemClientSession` for publishing.
15. Refactor WinForms demo loop so it only mutates item values/properties and raises UI state.

   - `RunPeriodicValueGroupAsync` -> `RunPeriodicPublishGroupAsync`

4. Add optional `PropertyName` storage to registrations.

7. Add `AddPeriodicSnapshot(ItemModel item, int periodMilliseconds)`.
8. Update worker dispatch to call the matching existing public method.
9. Update group start/stop/remove logic to use the shared model.
10. Build or run focused tests to catch API breakage before touching demos.

    - keep initial snapshot publication
    - register periodic value and meta property publishing after connect

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public methods require XML documentation with parameter descriptions.
- Use `periodMilliseconds` as the parameter name.
- Use named parameters for ambiguous method calls, especially item/property/period calls.
- Do not hold `_sync` across awaits.
- Do not introduce new public types unless the existing `IAsyncDisposable` return type becomes insufficient.
- Keep new scheduler types private inside `MqttItemClient` unless there is a clear compile-time need.
- Preserve existing retained publish behavior by reusing current public methods.
- Handle expected cancellation explicitly; do not add silent catch blocks.
- Avoid parallel fan-out inside one period worker for now.

## Acceptance Criteria
- `AddPeriodicValue`, `AddPeriodicProperty`, and `AddPeriodicSnapshot` are available on `MqttItemClient`.
- All three methods validate null/empty/out-of-range inputs with clear exceptions.
- Registrations sharing the same `periodMilliseconds` share one worker group.
- Disposing a registration stops publishing that registration.
- Empty period groups are removed and their worker is stopped.
- Disconnect stops workers.
- Reconnect restarts workers for registrations that still exist.
- Periodic property publishing updates the expected MQTT retained property.
- Periodic snapshot publishing updates the expected MQTT retained item snapshot.
- CLI demo no longer manually calls `UpdateValueAsync` or `UpdatePropertyAsync` from its cyclic loop.
- WinForms demo publisher no longer uses `MqttItemClientSession` for its cyclic publish loop.
- Demo loops are simpler and mainly mutate item state.

## Out of Scope
- Immediate first-tick publishing behavior changes.
- Custom publish delegates.
- Backpressure queues or skipped-tick accounting.
- Parallelizing item publishing within a period.
- New sample UI features.
- Reworking unrelated direct `MqttItemClientSession` usages outside the two named demo applications.
- Release versioning, tags, publish artifacts, or notice-file updates.

## Risks / Watchouts
- `AddPeriodicSnapshot` can publish many MQTT messages per tick. Do not use it in 100 ms demo paths unless that behavior is explicitly wanted.
- The WinForms demo currently publishes `epoch`, `meta`, and raw values. Preserve these with `AddPeriodicProperty` and `AddPeriodicValue` where they are still mutated.
- A disposed registration may publish one final time if a worker already copied the registration list. This is acceptable unless existing tests require stricter behavior.
- Calling public publish methods from the periodic worker preserves behavior but also triggers `RemoteItemsChanged`. Avoid adding separate duplicate notifications.
- When converting WinForms from `MqttItemClientSession` to `MqttItemClient`, remember that the facade's `UpdatePropertyAsync` is retained by design, matching current demo intent.
- Keep cancellation handling explicit when stopping registration handles before disconnecting.

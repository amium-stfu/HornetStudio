# IMPLEMENTATION HANDOFF

## Goal
Make periodic publishing easier for application code by letting `MqttItemClient` own periodic registrations by default. Callers should be able to call `AddPeriodicValue`, `AddPeriodicProperty`, or `AddPeriodicSnapshot` without storing returned handles. Returned handles should remain useful for explicitly removing individual periodic registrations.

## Scope
Included:
- Adjust `MqttItemClient` periodic registration lifetime semantics so the client owns registrations internally.
- Keep the existing public API names and signatures if possible.
- Preserve the existing return value as an optional removal handle.
- Ensure client disposal/periodic cleanup removes all registrations and stops workers.
- Simplify the CLI MQTT demo by removing its local `List<IAsyncDisposable>` registration storage.
- Update tests or add focused tests for ignored handles and explicit handle disposal.

Not included:
- Do not redesign the periodic worker model.
- Do not add new dependencies.
- Do not change MQTT publish semantics.
- Do not remove the returned handle unless absolutely necessary.
- Do not update WinForms demo in this first step unless compilation requires it.

## Starting Point
- `MqttItemClient` currently stores periodic registrations in `_periodicPublishGroups`.
- Each registration is represented by private `PeriodicPublishRegistration : IAsyncDisposable`.
- `AddPeriodicValue`, `AddPeriodicProperty`, and `AddPeriodicSnapshot` already add the registration to the internal group and return the registration object.
- The CLI demo currently stores returned handles in a local field:
  - `private readonly List<IAsyncDisposable> _periodicRegistrations = [];`
- The demo disposes these handles in `DisposePeriodicRegistrationsAsync`.
- This makes the example look more complex than desired for normal usage.

## Tasks
1. Review whether current `MqttItemClient` already keeps registrations alive without the caller storing handles.
2. If current behavior already works with ignored handles, document and test that behavior.
3. Add client-level cleanup that can remove all remaining periodic registrations when the client is disposed.
4. Decide whether `DisconnectAsync` should only stop workers or also clear registrations:
   - Preferred: `DisconnectAsync` stops workers but keeps registrations for reconnect.
   - `DisposeAsync` clears registrations permanently.
5. Keep explicit handle disposal behavior: disposing a returned handle removes only that registration.
6. Simplify CLI demo:
   - remove `_periodicRegistrations`
   - remove `DisposePeriodicRegistrationsAsync`
   - call `AddPeriodicValue` and `AddPeriodicProperty` directly in `RegisterPeriodicPublishing`
   - keep the value source loop unchanged
7. Add or update tests to prove:
   - ignored registration handles still publish periodically
   - explicit handle disposal stops that registration
   - client dispose stops/clears all periodic registrations

3. Update `DisposeAsync` to disconnect and clear registrations.
4. Keep `DisconnectAsync` behavior as worker-stop only, preserving registrations for later reconnect.
5. Simplify CLI demo registration code so the user-facing example calls `AddPeriodic...` directly.
6. Remove dead fields/methods/imports from the CLI demo.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public XML documentation should clearly explain that returned handles are optional removal handles.
- Use existing private scheduler structure; avoid new public abstractions.
- Do not hold locks across awaits.
- Keep reconnect behavior: periodic registrations should survive `DisconnectAsync` and restart after `ConnectAsync`.
- Avoid finalizers or GC-dependent behavior.
- Do not add new dependencies.

## Acceptance Criteria
- Application code can call `AddPeriodicValue`, `AddPeriodicProperty`, or `AddPeriodicSnapshot` and ignore the returned handle.
- The periodic registration still remains active because the client owns it.
- Explicitly disposing the returned handle removes that individual registration.
- `DisconnectAsync` stops workers but does not erase configured periodic registrations.
- `ConnectAsync` after disconnect restarts configured periodic registrations.
- `DisposeAsync` stops workers and clears periodic registrations.
- CLI demo no longer contains a periodic registration list.
- CLI demo still separates periodic publish configuration from value source logic.
- Build succeeds.

## Risks / Watchouts
- Because the returned handle is optional, its XML documentation must not imply that the caller must keep it alive for publishing to continue.
- Clearing registrations on `DisconnectAsync` would make reconnect behavior surprising and should be avoided.
- If `DisposeAsync` clears registrations, ensure in-flight worker tasks are stopped before clearing group state.
- Tests should avoid relying on garbage collection behavior. The ownership guarantee should come from client-held references.
- Do not simplify the demo by discarding the value source loop; it is still needed to update item values.

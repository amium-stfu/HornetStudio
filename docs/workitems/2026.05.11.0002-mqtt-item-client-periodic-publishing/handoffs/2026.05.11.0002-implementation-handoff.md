# IMPLEMENTATION HANDOFF

## Goal
Add a small periodic publishing helper to `MqttItemClient` so application code can register mutable items for cyclic value publishing with different periods. Items with the same period should share one internal worker.

## Scope
Included:
- Add a public `MqttItemClient` helper for cyclic value updates.
- Use an `int periodMilliseconds` API because the project preference for this feature is millisecond-based periods.
- Group registrations by `periodMilliseconds`.
- Run one worker task per period group.
- Start workers when the client is connected.
- Stop workers during `DisconnectAsync` and `DisposeAsync`.
- Return a registration handle so callers can remove an item from cyclic publishing.
- Add or update focused tests for registration grouping, lifecycle, and removal.
- Update relevant XML documentation.

Not included:
- Do not implement cyclic snapshot publishing in the first step.
- Do not redesign `MqttItemClientSession`.
- Do not change MQTT topic semantics, retained behavior, remote item filtering, or subscription behavior.
- Do not introduce new dependencies.
- Do not migrate sample publishers unless explicitly requested in a later implementation request.

## Starting Point
- `amium_item/src/Amium.Item.Client/MqttItemClient.cs` currently exposes manual publish methods:
  - `PublishSnapshotAsync(ItemModel item, CancellationToken cancellationToken = default)`
  - `UpdateValueAsync(ItemModel item, CancellationToken cancellationToken = default)`
  - `UpdatePropertyAsync(ItemModel item, string parameterName, CancellationToken cancellationToken = default)`
- `MqttItemClient` owns the connected `MqttItemClientSession` and hides self-published remote paths via `_publishedServerPaths`.
- Current demos implement their own `PeriodicTimer` loops.
- `MqttItemClientSession` already contains reconnect handling for individual MQTT publishes.
- The requested helper should live at the client facade level, not inside the low-level session.

## Tasks
1. Add a public cyclic value registration method to `MqttItemClient`.
2. Validate `item` and `periodMilliseconds`.
3. Store registrations grouped by period in milliseconds.
4. Create one internal worker per unknown period.
5. Have each worker publish `UpdateValueAsync` for all currently registered items in its period group.
6. Ensure worker loops do not hold locks across awaits.
7. Start workers after successful `ConnectAsync`.
8. Stop workers during `DisconnectAsync` and `DisposeAsync`.
9. Return a disposable registration handle that removes the item from its period group.
10. Remove an empty period group and stop its worker when the last registration is disposed.
11. Add diagnostics for publish loop failures without permanently killing the loop.
12. Add tests for period grouping, add/remove behavior, and disconnect cleanup.

## File-Level Changes
- `amium_item/src/Amium.Item.Client/MqttItemClient.cs` -> add the public registration API, internal period groups, worker lifecycle, and registration handle.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> add focused tests using the existing test style and MQTT test infrastructure where possible.
- Optional only if required by compilation or style: add a small public interface for the returned registration handle in the existing client project. Prefer `IAsyncDisposable` if sufficient.

## Implementation Order
1. Inspect current tests around `MqttItemClient` and decide whether existing MQTT broker test setup can verify cyclic publishing without adding infrastructure.
2. Add private fields to `MqttItemClient` for period groups and lifecycle cancellation.
3. Add the public method, preferably named `AddPeriodicValue(ItemModel item, int periodMilliseconds)`.
4. Implement validation:
   - `item` must not be null.
   - `periodMilliseconds` must be greater than zero.
5. Implement an internal period group model with:
   - period in milliseconds
   - registrations list or set
   - cancellation source
   - worker task
6. Implement the worker loop with `PeriodicTimer(TimeSpan.FromMilliseconds(periodMilliseconds))`.
7. On each tick, copy the registered items under lock, then publish outside the lock.
8. Use existing `UpdateValueAsync` to preserve retained behavior and self-published path filtering.
9. Add worker start calls after successful `ConnectAsync`.
10. Add worker stop calls before or during `DisconnectAsync`.
11. Ensure registration disposal is idempotent.
12. Add tests.
13. Run targeted tests/build.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public methods and classes need XML documentation.
- Use named parameters where calls have multiple or ambiguous parameters.
- Keep the implementation inside the existing client facade unless a separate type has a clear technical benefit.
- Do not add dependencies.
- Do not hold `_sync` while awaiting asynchronous operations.
- Avoid silent catch blocks. Worker exceptions should either be diagnostics or explicitly handled cancellation.
- Use `async`/`await` correctly; avoid `.Result` and `.Wait()`.
- Keep changes limited to the periodic value publishing feature.

## Acceptance Criteria
- Application code can register an item for cyclic value publishing with `periodMilliseconds`.
- Multiple items with the same period are processed by one period worker.
- Different periods use different workers.
- Invalid periods are rejected with a clear exception.
- Disposing the registration removes the item from cyclic publishing.
- Disconnecting the client stops active workers.
- Reconnecting starts workers for existing registrations again.
- Existing manual publish APIs keep their behavior.
- Existing tests still pass.

## Out of Scope
- Cyclic snapshot publishing.
- Cyclic property publishing.
- Per-item custom publish delegates.
- Backpressure queues.
- Parallel publish fan-out inside one period group.
- Sample application migration to the new helper.
- Public API breaking changes.
- Release versioning or changelog updates unless the later implementation request explicitly treats this as release work.

## Risks / Watchouts
- `ItemModel` is mutable. The worker will publish the current item state at tick time. Do not clone by default unless tests or behavior show inconsistent reads matter for value updates.
- A long publish cycle can overrun its period. The first implementation should publish sequentially and skip complex catch-up semantics.
- If `UpdateValueAsync` throws because the client is disconnected during shutdown, treat expected cancellation/disconnect paths explicitly and avoid noisy diagnostics.
- `PeriodicTimer` starts after the first period elapses. If immediate first publish is required, that is a separate behavior decision and should not be assumed silently.
- Do not use `int period` as the parameter name. Use `periodMilliseconds` so call sites remain unambiguous.
- Be careful that registration disposal while a tick is in progress may allow one final publish from a copied item list. That is acceptable if documented or covered by tests.

## Notes
- Preferred public method name: `AddPeriodicValue(ItemModel item, int periodMilliseconds)`.
- Preferred return type: `IAsyncDisposable` if no richer public contract is needed.
- Snapshot publishing can later be added as `AddPeriodicSnapshot(ItemModel item, int periodMilliseconds)` with separate period groups or a publish-kind-aware group, but it should not be included in this first implementation.

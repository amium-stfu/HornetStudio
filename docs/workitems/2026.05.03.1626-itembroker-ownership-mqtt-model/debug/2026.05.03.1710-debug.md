# DEBUG REPORT

## Problem
External MQTT control no longer worked when the published BrokerWidget item was also attached to a SignalWidget.

## Expected Behavior
An external MQTT write on a writable shared topic should follow the same effective write target as a SignalWidget user interaction.

## Actual Behavior
BrokerWidget write-back wrote value updates directly to the configured `LocalPath`.

## Error Messages / Logs
No explicit exception. The visible symptom was that external writes reached the broker path but did not trigger the same local control behavior as SignalWidget input.

## Relevant Code
- `src/Hornetstudio.Editor/Widgets/Broker/HostItemBrokerWriteBackClient.cs`
- `src/Hornetstudio.Editor/Models/PageItemModel.cs`
- `src/Hornetstudio.Host/UdlModule.cs`

## Attempted Fixes
- Matched BrokerWidget value write-back to SignalWidget write target resolution:
  - honor `WritePath`
  - honor `WriteMode=Request`
  - write to the resolved `Request` child when present
- Added a regression test for Broker write-back using a request write target.

## Current Hypothesis (optional)
- UDL and SignalWidget-facing items expose readable channels with `Writable=true`, `WritePath=<channel path>`, and `WriteMode=Request`. Writing the read channel value directly only changes local display state; writing `<channel>.Request` triggers the UDL write path.

# DEBUG REPORT

## Problem
The terminal was flooded with repeated `MQTT write rejected for non-writable item path ...` messages while BrokerWidget-published items were active.

## Expected Behavior
Normal non-writable item publishes should not spam the terminal. Rejected write attempts should remain diagnosable when explicitly enabled.

## Actual Behavior
Every non-writable shared-topic publish handled by the MQTT adapter wrote a rejection line to `Console.Error`.

## Error Messages / Logs
`MQTT write rejected for non-writable item path 'studio.Folder1.UdlClient1.m300.Read' parameter 'Value'.`

## Relevant Code
`src/Amium.Item.Server.Mqtt/MqttItemServerAdapter.cs`

## Attempted Fixes
- Changed rejection logging from unconditional `Console.Error.WriteLine(...)` to opt-in diagnostics behind `Amium.Item.Server.Mqtt.WriteDiagnostics` with a suppression limit -> terminal flood is prevented while diagnostics remain available.

## Current Hypothesis (optional)
- BrokerWidget publishes normal MQTT shared-topic state through the same MQTT server that also accepts external write attempts. The adapter cannot treat every non-writable shared-topic publish as an operator-facing error without producing noise.

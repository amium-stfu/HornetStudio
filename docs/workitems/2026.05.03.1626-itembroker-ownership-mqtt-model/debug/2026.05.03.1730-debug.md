# DEBUG REPORT

## Problem
External writes to `hornet/Studio/Folder1/UdlClient1/m300/Set/Request` still reverted to the previous value after the BrokerWidget publisher snapshot fallback was fixed.

## Expected Behavior
The MQTT value topic should keep the new request value as the current retained MQTT state so existing and newly subscribing clients see the same value.

## Actual Behavior
`MqttRemoteItemClient.PublishSnapshotAsync` wrote retained MQTT messages, but `UpdateValueAsync` and `UpdateParameterAsync` delegated to `MqttItemBrokerClientSession` without `retained: true`. The old retained snapshot value could remain in the broker and reappear after refresh or resubscription.

## Error Messages / Logs
No exception is required. The visible symptom is an MQTT Explorer write changing briefly and then returning to the previous value.

## Relevant Code
- `src/Amium.ItemBroker.Mqtt.Client/MqttRemoteItemClient.cs`
  - `UpdateValueAsync`
  - `UpdateParameterAsync`
- `src/Amium.ItemBroker.Tests/Program.cs`
  - `MqttRemoteClientValueUpdateRefreshesRetainedState`

## Attempted Fixes
- BrokerWidget change-update snapshot fallback was fixed first -> reduced one stale-publish path but did not solve the retained MQTT state.
- `MqttRemoteItemClient.UpdateValueAsync` now calls the MQTT session with `retained: true`.
- `MqttRemoteItemClient.UpdateParameterAsync` now calls the MQTT session with `retained: true`.
- Added a regression test that publishes a retained snapshot value `1`, updates the value to `42`, then reads the retained MQTT topic with a fresh MQTT client and expects `42`.

## Current Hypothesis (optional)
- The remaining overwrite was caused by stale retained MQTT state, not by the local BrokerWidget write-back path. For Host/BrokerWidget publishing, value and parameter updates are authoritative MQTT state changes and must refresh retained topics.

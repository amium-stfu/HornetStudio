# DEBUG REPORT

## Problem
SignalWidget writes changed only the SignalWidget display value. The InApp TreeView and MQTT publish path did not react.

## Expected Behavior
Writing through a SignalWidget should update `HostRegistries.Data`, raise a `ValueUpdated` event, refresh other in-app views, and publish the value through the BrokerWidget when the path is configured for publishing.

## Actual Behavior
The SignalWidget log reported success:
- `[SignalWrite] result=ok ... registryUpdated=True`

But no `[DataRegistryValue]`, `[SignalWidgetRegistry]`, `[BrokerWidgetPublish]`, or `[HostItemBrokerClientPublish]` followed. The signal display changed locally, while TreeView and MQTT stayed unchanged.

## Error Messages / Logs
SignalWidget write:
- `[SignalWrite] item=Folder1.Demo1 target=Studio.Folder1.CustomSignals1.Demo1 targetParam=Value targetValue=80 (Double) writeTarget=Studio.Folder1.CustomSignals1.Demo1 writeParam=Value raw=10 (Double) converted=10 (Double)`
- `[SignalWrite] result=ok item=Folder1.Demo1 writeTarget=Studio.Folder1.CustomSignals1.Demo1 writeParam=Value value=10 (Double) registryUpdated=True`

CustomSignals list edit:
- `[DataRegistryValue] result=updated key=Studio.Folder1.CustomSignals1.Demo1 ... old=60 incoming=90 stored=90`
- `[SignalWidgetRegistry] applying ... kind="ValueUpdated" ... value=90`
- `[BrokerWidgetPublish] kind=value ... value=90`
- `[HostItemBrokerClientPublish] kind=value ... value=90`

## Relevant Code
- `src/HornetStudio.Editor/Models/PageItemModel.cs`
- `src/HornetStudio.Editor.Tests/Program.cs`

## Attempted Fixes
- Moved SignalWidget and interaction writes so they call `HostRegistries.Data.UpdateValue(...)` / `TryUpdateUserParameter(...)` before mutating the local `Item` instance.
- Kept the local mutation only for the snapshot fallback path when the registry update cannot resolve the target path.
- Added a regression test proving a SignalWidget write emits a `DataChangeKind.ValueUpdated` event.

## Current Hypothesis (optional)
- The previous no-op suppression in `DataRegistry.UpdateValue` is correct, but it exposed a caller-order bug: `PageItemModel.TryUpdateTargetParameterValue` mutated the same registry item instance before calling `UpdateValue`. The registry then saw `old == incoming`, returned success, and suppressed the change event. Moving the registry update first restores TreeView and MQTT propagation.

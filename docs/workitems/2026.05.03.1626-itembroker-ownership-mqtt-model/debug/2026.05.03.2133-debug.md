# DEBUG REPORT

## Problem
After startup, MQTT writes initially updated the app, UI writes also worked, but later MQTT writes stopped updating the app. After some time, MQTT writes could start working again.

## Expected Behavior
External MQTT writes should always flow through BrokerWidget write-back into `HostRegistries.Data`, unless they are proven self-echoes from the current HornetStudio client.

## Actual Behavior
The BrokerWidget write-back received the MQTT message but classified it as an echo:
- `[BrokerWidgetWriteBack] received type=ItemValueChangedMessage brokerPath=studio.Folder1.CustomSignals1.Demo1 parameter=Value value=11 source=shared`
- `[BrokerWidgetWriteBack] result=echo brokerPath=studio.Folder1.CustomSignals1.Demo1 localPath=studio.Folder1.CustomSignals1.Demo1 writeTarget=studio.Folder1.CustomSignals1.Demo1 parameter=Value value=11`

Because the message was discarded before `HostRegistries.Data.UpdateValue(...)`, the app did not react.

## Error Messages / Logs
No exception was reported. The relevant symptom is `result=echo` for a message with `source=shared`, which is not the HornetStudio client id.

## Relevant Code
- `src/Hornetstudio.Editor/Widgets/Broker/HostItemBrokerWriteBackClient.cs`

## Attempted Fixes
- Narrowed the echo filter so local value equality alone no longer discards MQTT messages.
- Echo detection now ignores:
  - values already applied by this write-back client, and
  - same-value messages from the current HornetStudio client id.
- Messages from other sources, including `shared`, are allowed to reach the registry. If the value is truly unchanged, the registry no-op guard suppresses downstream events there instead of the BrokerWidget discarding the message early.

## Current Hypothesis (optional)
The previous echo filter was too broad. It treated any incoming MQTT value equal to the local value as an echo, independent of `SourceClientId`. During startup or after UI writes, local and broker state can briefly match before a user expects the external MQTT write to drive the app. Classifying `source=shared` as echo prevented the registry update path from running.

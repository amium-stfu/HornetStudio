# DEBUG REPORT

## Problem
An external MQTT write to `hornet/Studio/Folder1/UdlClient1/m300/set/request` is accepted, but when a SignalWidget is attached to the same `.Request` item the value is immediately overwritten again.

## Expected Behavior
The external request write should be applied as a value update and republished as an ItemBroker value update, without sending a stale full snapshot back to MQTT.

## Actual Behavior
The BrokerWidget publisher could handle a local `ValueUpdated` change as a snapshot fallback when the changed broker path was not yet registered. That snapshot could contain the previous registry/widget value and overwrite the external MQTT Explorer write.

## Error Messages / Logs
No new exception was required to reproduce this path. The observable symptom is that the MQTT Explorer value changes and then immediately reverts.

## Relevant Code
- `src/Hornetstudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs`
  - `HostItemBrokerPublisher.PublishDefinitionIfAvailable`
  - `PublishValueUpdateAsync`
  - `PublishSnapshotAsync`

## Attempted Fixes
- Checked write-back routing to `.Request` -> useful for parent items with `WriteMode=Request`, but not the direct root cause when the MQTT topic already points at `.Request`.
- Added a publisher regression test for an unregistered request value change -> reproduced the stale snapshot fallback.
- Moved `ChangeUpdate` handling before snapshot registration fallback and publish `ValueUpdated`/parameter changes as updates whenever a changed broker path can be resolved -> editor tests pass.

## Current Hypothesis (optional)
- The SignalWidget makes the request path participate in the local registry update loop. The external MQTT write updates the registry correctly, but the BrokerWidget publisher reacts to that local change and must preserve update semantics. Falling back to a full snapshot on `ValueUpdated` is what causes the immediate overwrite.

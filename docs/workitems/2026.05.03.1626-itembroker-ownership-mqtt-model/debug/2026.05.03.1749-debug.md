# DEBUG REPORT

## Problem
The log fills with internal broker write-back echoes and the request value can fall back to the previous value even without an external MQTT Explorer write. After the first fix, the application crashed during a heavy trace run, most likely because unchanged runtime values still generated registry events, broker publishes, and UI log entries.

## Expected Behavior
When a source child such as `Read` changes, only the matching Studio child path should be updated. Existing writable request children such as `set.request` must keep their current value. Repeated writes of the same value should not be treated as new value changes.

## Actual Behavior
`UiFolderContext.AttachedItemLink.OnSourceChanged` mirrored every source child change by cloning the complete source tree and upserting it at the Studio target root. A read/source update could therefore carry an older `set.request` value back into `studio.Folder1.UdlClient1.m300.set.request`.

After that was fixed, `DataRegistry.UpdateValue` still accepted identical values as `ValueUpdated`, logged them, and raised `ItemChanged`. The trace shows `old=0 incoming=0 stored=0` followed by `SignalWidgetRegistry`, `BrokerWidgetPublish`, and `HostItemBrokerClientPublish`, so one unchanged cyclic runtime poll can still fan out through the signal and broker pipeline.

The visible log widget had a second overflow risk: `ProcessLog` keeps only the newest 1000 buffered rows, but `EditorLogControl.LogEntries` kept appending live entries while the page was active.

## Attempted Fixes
- Changed source child mirroring from full root `UpsertSnapshot` to targeted `UpdateValue` / `UpdateParameter` on the corresponding target child path.
- Kept full snapshot fallback only when the target child path does not exist.
- Applied the same fix to the legacy `UiPageContext` copy.
- Added a regression test where `studio.MirrorTest.m300.set.request` is externally set to `42`, then `source.Read` changes; the request child remains `42`.
- Changed registry value and parameter updates so unchanged converted values return success but do not raise change events or emit update debug logs.
- Limited the live log display collection to the newest 1000 entries, matching the existing `ProcessLog` buffer limit.
- Added regression tests for unchanged value and parameter updates.

## Current Hypothesis (optional)
- The original fallback was caused by full source-tree snapshots from the UI folder mirror, not by MQTT retained state.
- The later crash was caused by unchanged values being forwarded as real changes through registry, signal, broker publish, and the visible log widget. Suppressing no-op registry events should stop the self-amplifying log/publish loop while preserving real updates.

# PLAN

## Topic
MQTT shared-topic write ownership for BrokerWidget-published items.

## Goal
Define and implement a corrected ownership/write model where BrokerWidget-published items use one MQTT topic per value or parameter, while external MQTT writes can update writable local sources without being silently overwritten by stale local state.

## Problem
The current strict broker ownership rule treats `PublishSnapshotAsync` as owner registration and routes non-owner updates as write requests. That model works for an in-process broker, but the MQTT mapping intentionally exposes shared topics without owner identity. As a result, the same MQTT topic is currently used both as retained state output and as an external write input. This makes strict topic ownership ambiguous and can cause external writes to be overwritten by later local snapshots or refresh-triggered publishes.

## Decided Model
- MQTT remains shared-topic based: every item value or parameter has one public topic.
- BrokerWidget/Studio is the local authority for BrokerWidget-published local items.
- External MQTT publishes to the same topic are treated as write requests when the published item is writable.
- `Writable=true` allows the external value to be applied to the configured local source or write target.
- `Writable=false` blocks external writes and should produce diagnostics instead of silently pretending success.
- Local source changes are the confirmation path: after the local item changes, BrokerWidget publishes `UpdateValueAsync` or `UpdateParameterAsync` back to the same MQTT topic.
- Ownership must not require distinct MQTT owner topics for this workflow.

## Scope
Included:
- BrokerWidget publish/write-back behavior.
- MQTT adapter/client behavior for shared item topics.
- In-process owner routing only where it affects MQTT shared-topic writes.
- Documentation of the effective MQTT write model.
- Focused verification for writable and non-writable published items.

Excluded:
- Release publishing.
- New MQTT topic hierarchy with explicit `/request` topics.
- New non-MQTT transports.
- Large BrokerWidget UI redesign.
- Broad refactoring of SignalWidget or UDL runtime code.

## Architecture Notes
The corrected flow should be:

1. BrokerWidget publishes initial retained state for a local item or parameter.
2. MQTT clients observe and edit the same shared topic.
3. The MQTT bridge/client receives external publishes and maps them to broker updates.
4. If the item is writable, Studio applies the requested value to the local source/write target.
5. Local `HostRegistries.Data.ItemChanged` or equivalent item change events trigger an update publish.
6. The same MQTT topic receives the confirmed local value.

This means snapshots register/publish available state, while update messages represent confirmed local changes. External writes are requests until the local source accepts and emits the new value.

## Major Decisions
- Do not introduce separate MQTT request topics for now.
- Do not encode owner identity into the public MQTT item topic.
- Keep the user-facing contract simple: same topic for reading and writing.
- Treat retained MQTT state as observable state, not proof of ownership.
- Prefer changing the MQTT/shared-topic path over weakening all in-process ownership semantics.

## Sequencing
1. Audit current write flow from external MQTT publish to local `HostRegistries.Data` update.
2. Adjust the MQTT/broker path so external writes on shared topics reach writable local definitions.
3. Ensure local item changes publish update messages after initial snapshot registration.
4. Prevent non-writable external writes from mutating local state.
5. Add or update focused tests around shared-topic writable behavior.
6. Update architecture/usage documentation.
7. Build affected projects.

## Open Details
- The implementation may either keep `HostItemBrokerWriteBackClient` as the local write-request consumer or replace it with a more direct write-target handler if that is smaller and cleaner.
- Diagnostics for blocked writes should use existing logging/diagnostic channels.
- Parameter writes should remain constrained by existing user-write parameter policy.

## Acceptance Criteria
- External MQTT writes on writable BrokerWidget-published topics update the intended local item or write target.
- The confirmed local value is republished to the same MQTT topic.
- External writes on non-writable topics do not mutate local state.
- SignalWidget visualization alone does not cause an old local value to overwrite an accepted external write.
- Snapshot publishing does not repeatedly override writable external updates after the item is registered.
- The chosen MQTT write model is documented.

## Verification
- Build `src/Hornetstudio.Editor/Hornetstudio.Editor.csproj`.
- Build or run the relevant ItemBroker/MQTT tests.
- Manually verify with an MQTT client:
  1. Publish a local item through BrokerWidget.
  2. Change the same MQTT value topic externally.
  3. Confirm the local source receives the value when writable.
  4. Confirm Studio republishes the accepted value on the same topic.
  5. Repeat with `Writable=false` and confirm the local value is not changed.

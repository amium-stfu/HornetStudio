# PLAN

## Topic
Correct the ItemBroker ownership model for MQTT, BrokerWidget, external writes, and HornetStudio UI mirrors.

## Problem
The newly introduced ItemBroker owner rule improves internal broker consistency, but it currently makes real HornetStudio/MQTT workflows worse because MQTT and the BrokerWidget do not clearly distinguish owner-published state from non-owner write requests.

Observed issues:
- BrokerWidget-published UdlClient modules do not expose or carry owner information over MQTT.
- External MQTT writes can be overwritten immediately by the local HornetStudio source value.
- Visualizing the same item through a SignalWidget can trigger local registry activity that makes the old local value win again.
- UI edits can update only the local broker mirror, then get overwritten by the next broker refresh.

## Scope
Included:
- Re-evaluate ownership semantics across `InMemoryItemServer`, MQTT transport, BrokerWidget, UdlClient-published items, external MQTT clients, and UI/SignalWidget mirrors.
- Decide whether the strict owner rule should be relaxed, delayed, or represented explicitly over MQTT.
- Define how external writes should reach the true local owner/write target.
- Produce an implementation-ready direction after the model is clarified.

Excluded:
- Release publishing.
- Broad unrelated UI changes.
- New transports beyond the current MQTT path.
- Large refactors before the ownership/write model is agreed.

## Current Understanding
- `InMemoryItemServer` now treats `PublishSnapshotAsync` as owner registration.
- `UpdateValueAsync` and `UpdateParameterAsync` are owner updates or non-owner write requests.
- MQTT topics currently represent retained value/parameter state but do not encode owner registration.
- BrokerWidget acts as an owner proxy when publishing local UdlClient/HostRegistry items to MQTT.
- Broker-received runtime items in HornetStudio are local mirrors, not authoritative owners.

## Key Questions
1. Should MQTT explicitly represent owner registration, or should ownership remain in-process only?
2. Should external MQTT publishes to an owner-published item be interpreted as retained state or as write requests?
3. Should BrokerWidget-published UdlClient items be considered owned by the BrokerWidget proxy or by the UdlClient source?
4. How should SignalWidget/UI writes detect broker mirrors and route writes back to the broker owner?
5. Should the strict owner conflict rule be relaxed until MQTT has a clear registration/write protocol?

## Recommended Direction
Prefer a clear three-lane model:
- Owner state publish: authoritative state from the owner/proxy.
- Write request: non-owner request to change an owner-controlled item.
- UI mirror: local visualization of remote broker state, never authoritative by itself.

## Acceptance Criteria
- A new model explains why external MQTT writes are or are not allowed to override BrokerWidget-published values.
- BrokerWidget-published UdlClient modules have a clearly defined owner/proxy role.
- UI/SignalWidget interactions have a clear broker-aware write path.
- The next implementation handoff can be executed without guessing ownership semantics.

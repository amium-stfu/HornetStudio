# IMPLEMENTATION HANDOFF

## Goal
Implement the MQTT shared-topic write model for BrokerWidget-published items. External MQTT clients must be able to write to the same topic they read from when the published item is writable, and Studio must apply the write to the local source before republishing the confirmed value.

## Scope
Included:
- BrokerWidget-published local items and parameters.
- External MQTT publishes on shared item topics.
- Writable routing back to local `HostRegistries.Data` items or configured write targets.
- Local change confirmation through `UpdateValueAsync` / `UpdateParameterAsync`.
- Diagnostics for blocked or failed writes.
- Focused tests and documentation updates.

Explicitly NOT included:
- Release/versioning work.
- New MQTT `/request` topic structure.
- Explicit MQTT owner metadata topics.
- Non-MQTT transport redesign.
- Unrelated UI redesign or broad cleanup.

## Starting Point
- The current MQTT topic mapper maps every item value/parameter to one shared topic.
- `MqttItemBrokerAdapter` tracks `Writable` and `WritePath` metadata and maps incoming MQTT publishes to broker updates.
- `InMemoryItemBroker` currently registers snapshot owners and routes non-owner updates as `ItemWriteRequestMessage`.
- `BrokerClientControl.HostItemBrokerPublisher` already subscribes to `HostRegistries.Data.ItemChanged` and can publish snapshots or updates.
- `HostItemBrokerWriteBackClient` subscribes to writable broker paths and writes broker changes back into local `HostRegistries.Data`.
- External writes can still be overwritten by old local snapshots or by refresh-triggered publishes.

## Tasks
1. Trace the current external MQTT write path:
   - `MqttItemBrokerAdapter.HandleIncomingPublishAsync`
   - `InMemoryItemBroker.UpdateValueAsync` / `UpdateParameterAsync`
   - `HostItemBrokerWriteBackClient.HandleMessageAsync`
   - `HostRegistries.Data.UpdateValue` / parameter write
   - `HostItemBrokerPublisher.OnDataItemChanged`
2. Make MQTT shared-topic writes route as write requests for writable items without requiring distinct MQTT owner topics.
3. Ensure initial BrokerWidget snapshots register/publish the item once, then local changes use update messages where possible.
4. Ensure external writes are only applied when the published definition is writable.
5. Ensure blocked writes are diagnosable.
6. Add focused tests for writable shared-topic behavior and blocked non-writable behavior.
7. Update relevant docs to describe the shared-topic write contract.
8. Build affected projects.

## Implementation Order
1. Add or adjust tests around the broker/MQTT write path before making broad behavior changes where feasible.
2. Fix routing so external MQTT publishes on writable shared topics become owner/local write requests, not competing owner state.
3. Prevent immediate MQTT echo from being treated as a confirmed state update when the local source has not accepted the value yet.
4. Make `HostItemBrokerWriteBackClient` or equivalent local write handler apply only writable definitions and respect protected parameter policy.
5. Make `HostItemBrokerPublisher` publish confirmed local `ValueUpdated` and `ParameterUpdated` changes as update messages after initial registration.
6. Reduce snapshot republishing where it can overwrite external writes after registration.
7. Add diagnostics for rejected non-writable or unresolved write targets.
8. Update docs.
9. Run builds/tests.

## Acceptance Criteria
- A BrokerWidget-published writable item can be changed externally through the same MQTT topic used for reading.
- The external value reaches the configured local source or write target.
- Studio republishes the accepted local value on the same MQTT topic.
- A non-writable published item ignores or rejects external writes without changing local state.
- Blocked writes produce useful diagnostics.
- SignalWidget visualization does not trigger stale snapshot overwrites.
- The in-process broker still has a coherent ownership model after the MQTT-specific correction.

## Verification
- Run `dotnet build src/HornetStudio.Editor/HornetStudio.Editor.csproj --no-restore -v:minimal`.
- Run relevant ItemBroker/MQTT tests, for example `dotnet run --project src/Amium.ItemBroker.Tests/Amium.ItemBroker.Tests.csproj` if that is the established test command.
- Manual MQTT scenario:
  1. Start the broker/MQTT runtime.
  2. Connect BrokerWidget.
  3. Publish a local UDL item or parameter with `Writable=true`.
  4. Write a new value externally to the exact same MQTT topic.
  5. Confirm `HostRegistries.Data` receives the value.
  6. Confirm the same topic receives the confirmed Studio value.
  7. Repeat with `Writable=false` and confirm local state is unchanged.

## Out of Scope
- Release publish.
- New MQTT topic contract with request/response topics.
- MQTT authorization/security rules.
- Large SignalWidget or UDL refactors.
- General UI redesign.

## Risks / Watchouts
- Retained MQTT messages can look authoritative even when they are only mirror output.
- A snapshot after an external write can overwrite the external value if the local source did not first accept it.
- Echo suppression must not hide the confirmed local publish.
- Parameter writes must not bypass protected parameter policy.
- Interval publishing can still overwrite external writes if the local source rejects or cannot apply the value.
- WritePath resolution must be normalized consistently with existing target path helpers.

## Notes
- The preferred behavior is not last-writer-wins between MQTT and Studio. The external MQTT write is a request; the local source confirms by changing and republishing.
- Do not require users to publish to a separate write topic.
- If implementation reveals the current `HostItemBrokerWriteBackClient` cannot reliably distinguish request from confirmed state, prefer a small explicit routing change over broad ownership removal.

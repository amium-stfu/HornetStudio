# IMPLEMENTATION HANDOFF

## Goal
Re-evaluate and correct the ItemBroker ownership model across MQTT, BrokerWidget, external MQTT writes, and HornetStudio UI mirrors. The immediate objective is not to continue the previous implementation blindly, but to decide the right ownership/write architecture and then implement the smallest corrective change.

## Scope
Included:
- `Amium.Item.Server` owner rule behavior.
- MQTT topic and adapter semantics.
- BrokerWidget publishing of local `HostRegistries.Data` / UdlClient items.
- External MQTT value changes.
- SignalWidget/UI visualization and editing of broker items.
- Write-back behavior for writable broker-published items.

Explicitly NOT included:
- Release/versioning work.
- New non-MQTT transports.
- Unrelated BrokerWidget UI redesign.
- Large cleanup unrelated to ownership/write behavior.

Current behavior / observations:
- The strict owner rule was introduced in `InMemoryItemServer`.
- `PublishSnapshotAsync` registers ownership.
- `UpdateValueAsync` / `UpdateParameterAsync` publish owner updates or route non-owner writes to the registered owner.
- MQTT topics currently do not encode owner identity or snapshot-registration intent.
- BrokerWidget-published UdlClient modules do not expose owner information in MQTT.
- External MQTT Explorer writes can be immediately overwritten by the old local value.
- The overwrite becomes visible/reproducible when the same item is visualized through a SignalWidget.
- UI edits of broker-received runtime items can update only the local mirror and then get overwritten by broker refresh.

## Tasks
1. Audit the exact current data flow for a BrokerWidget-published UdlClient module.
2. Audit external MQTT write flow into `MqttItemServerAdapter`.
3. Audit SignalWidget/PageItemModel write and visualization flow for broker-received runtime items.
4. Decide the correct ownership model for MQTT:
   - in-process only ownership
   - explicit MQTT ownership metadata
   - relaxed/no strict owner enforcement over MQTT
   - dedicated write-request topics or conventions
5. Decide what BrokerWidget owns:
   - actual owner
   - owner proxy for local UdlClient/HostRegistry source
   - pure bridge with no ownership
6. Decide how external writes should behave:
   - retained state overwrite
   - write request to owner
   - rejected unless an explicit writable contract exists
7. Update or replace the previous implementation handoff with the chosen model.
8. Implement only the minimal corrective change after the model is chosen.

## Implementation Order
1. Reproduce or reason through the failing scenario:
   - BrokerWidget publishes a local UdlClient module.
   - External MQTT Explorer changes the same value topic.
   - SignalWidget visualizes the same item.
   - MQTT value is overwritten by the old local value.
2. Draw the actual event path:
   - local source -> HostRegistry -> BrokerWidget publish -> MQTT topic
   - external MQTT topic write -> adapter/client/session -> local mirror or broker update
   - SignalWidget target refresh/write -> HostRegistry update -> BrokerWidget publish
3. Identify which actor is currently treated as authoritative at each step.
4. Make one explicit architecture decision:
   - Option A: Relax owner enforcement for MQTT-shared topics until protocol support exists.
   - Option B: Add explicit MQTT write-request semantics and keep strict owner enforcement.
   - Option C: Add explicit MQTT owner/snapshot metadata and keep strict owner enforcement.
5. Update the plan/handoff if the chosen option materially changes scope.
6. Implement the smallest code change matching the chosen option.
7. Add a focused regression test or manual verification script/checklist.
8. Build affected projects.

## Acceptance Criteria
- The model defines whether BrokerWidget, UdlClient, or another actor owns BrokerWidget-published UdlClient items.
- External MQTT writes no longer get silently overwritten without a clear reason.
- If external writes are allowed, they reach the intended local owner/write target.
- If external writes are rejected, the behavior is explicit and diagnosable.
- SignalWidget visualization alone does not make old local values overwrite external writes unexpectedly.
- UI edits of broker-received mirrors do not pretend to persist locally when the broker owner will overwrite them.
- The chosen model is documented in usage/architecture docs.

## Risks / Watchouts
- The strict owner rule may be correct for in-process broker clients but too strong for the current MQTT shared-topic protocol.
- MQTT retained state makes old values look authoritative even when they are only bridge output.
- A SignalWidget can trigger registry refreshes that indirectly cause BrokerWidget publishes.
- Broker-received runtime items are local mirrors; treating them as writable authoritative items causes temporary UI changes followed by rollback.
- Adding owner metadata to MQTT may be a protocol change and should be documented clearly.
- Relaxing owner enforcement may reintroduce silent owner replacement unless constrained to MQTT/shared-topic flows.

## Notes
- Do not treat the previous owner-rule implementation as automatically correct.
- The next chat should first decide the model, then implement.
- The safest conceptual model is:
  - Owner Publish: authoritative state from the owner or owner proxy.
  - External Write Request: non-owner request routed to the owner/write target.
  - UI Mirror: local visualization, not authoritative state.

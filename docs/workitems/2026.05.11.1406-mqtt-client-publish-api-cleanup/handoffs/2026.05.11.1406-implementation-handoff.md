# IMPLEMENTATION HANDOFF

## Goal

Clean up the `Amium.Item.Client` MQTT publish API so user code describes publishing explicitly and no longer exposes locally generated `ItemServerAckMessage` results.

Target API:

```csharp
await client.PublishReadAsync(item);
await client.PublishPropertyAsync(item, parameterName: "write");
await client.PublishSnapshotAsync(item);
```

This is a deliberate breaking change. Do not keep compatibility wrappers for the removed old client methods.

## Scope

- Replace client-side `UpdateValueAsync(...)` with `PublishReadAsync(...)`.
- Replace client-side `UpdatePropertyAsync(...)` with `PublishPropertyAsync(...)`.
- Remove client-side `RequestWriteAsync(...)`.
- Remove `ItemServerAckMessage` from the `Amium.Item.Client` publish method return path.
- New publish methods return `Task`, not `Task<ItemServerAckMessage>`.
- Update periodic publishing, samples, tests, host integration, and docs to the new API.
- Keep server-side APIs and message contracts only where still required outside the client publish path.

Explicitly not included:

- Do not add compatibility wrappers for `UpdateValueAsync(...)`, `UpdatePropertyAsync(...)`, or `RequestWriteAsync(...)`.
- Do not add a new ack/request-response protocol.
- Do not expose MQTTnet publish result types in the public `Amium.Item.Client` API.
- Do not change `amium_item/src/Amium.Item/Item.cs`.
- Do not introduce new dependencies.

## Starting Point

- `MqttItemClient.UpdateValueAsync(...)`, `UpdatePropertyAsync(...)`, and `RequestWriteAsync(...)` currently return `Task<ItemServerAckMessage>`.
- `MqttItemClientSession.UpdateValueAsync(...)`, `UpdatePropertyAsync(...)`, and `RequestWriteAsync(...)` currently create local `ItemServerAckMessage` instances.
- The returned ack is not a real broker/server/owner acknowledgement; it only indicates local publish completion.
- `RequestWriteAsync(...)` hides a special non-retained publish to the `write` property.
- The desired public API is explicit publish naming:
  - `PublishReadAsync(...)`
  - `PublishPropertyAsync(...)`
  - `PublishSnapshotAsync(...)`
- A repository search shows callers in tests, samples, mesh controllers, Host integration, Broker widget code, stress test tooling, and docs.

## Tasks

1. Add/rename public client publish methods:
   - `PublishReadAsync(ItemModel item, bool publishEpoch = true, bool retained = false, CancellationToken cancellationToken = default)`
   - `PublishPropertyAsync(ItemModel item, string parameterName, bool retained = false, CancellationToken cancellationToken = default)`
   - ensure `PublishSnapshotAsync(ItemModel item, bool retained = true, CancellationToken cancellationToken = default)` remains available
2. Remove public client methods:
   - `UpdateValueAsync(...)`
   - `UpdatePropertyAsync(...)`
   - `RequestWriteAsync(...)`
3. Update `MqttItemClientSession` publish methods to return `Task`.
4. Remove local `ItemServerAckMessage` creation from the client/session publish path.
5. Make `PublishReadAsync(... publishEpoch: true)` publish both `read` and `epoch` using the same `retained` value.
6. Make `PublishPropertyAsync(item, "write")` the explicit way to publish a write property.
7. Update periodic publishing internals to call the new methods.
8. Update all compile-time callers.
9. Update tests and documentation.
10. Build and run focused self-tests.

## Technical Constraints

- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Public classes and public methods need XML documentation.
- Prefer existing project patterns and libraries.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Do not introduce new dependencies.
- This is an intentional breaking change: do not keep compatibility wrappers for the removed client methods.
- Use named parameters where they improve readability, especially with multiple or ambiguous arguments.

## Acceptance Criteria

- `MqttItemClient` exposes:
  - `Task PublishReadAsync(ItemModel item, bool publishEpoch = true, bool retained = false, CancellationToken cancellationToken = default)`
  - `Task PublishPropertyAsync(ItemModel item, string parameterName, bool retained = false, CancellationToken cancellationToken = default)`
  - `Task PublishSnapshotAsync(ItemModel item, bool retained = true, CancellationToken cancellationToken = default)`
- `MqttItemClient` no longer exposes:
  - `UpdateValueAsync(...)`
  - `UpdatePropertyAsync(...)`
  - `RequestWriteAsync(...)`
- `MqttItemClientSession` publish methods return `Task`, not `Task<ItemServerAckMessage>`.
- `ItemServerAckMessage` is not returned from the client publish path.
- `PublishReadAsync(... publishEpoch: true)` publishes both `read` and `epoch` with the same retained flag.
- `PublishPropertyAsync(item, "write")` replaces `RequestWriteAsync(...)` for write-property publishing.
- Single read/property publishes default to `retained: false`.
- Snapshot publishes default to `retained: true`.
- All repository callers compile against the new API.
- Client README and usage docs describe the new API and no longer recommend removed methods.

## Out of Scope

- Adding real broker/server acknowledgement support.
- Adding MQTTnet publish-result exposure to public APIs.
- Changing item path validation.
- Changing `Item` read/write property behavior.
- Changing received-item registration semantics.
- Adding compatibility wrappers for old client publish methods.

## Risks / Watchouts

- This is a broad breaking API change and will touch multiple callers.
- Server-side APIs also contain `UpdateValueAsync` and `ItemServerAckMessage`; avoid renaming/removing unrelated server concepts unless remaining usages prove they are dead or client-only.
- `PublishReadAsync(... publishEpoch: true)` adds an extra publish. Tests and docs should make that visible.
- Changing default retained behavior from previous client update methods may alter monitoring/demo startup behavior. Snapshot remains the retained state publication mechanism.
- `write` is now just an explicit property publish. Do not reintroduce hidden write-specific convenience behavior.

## Notes

- The new API uses `Publish...` names because the client publishes MQTT item state; it does not synchronously update a remote owner.
- Failures should be represented by thrown exceptions from the publish operation.
- A future real acknowledgement feature should be designed separately instead of reusing the old local `ItemServerAckMessage` return.

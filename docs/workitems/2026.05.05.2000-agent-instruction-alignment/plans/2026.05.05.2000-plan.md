# PLAN

## Goal
Die Arbeitsregeln für Visual Studio, GitHub Copilot, VS Code und das Codex-Plugin so konsolidieren, dass `AGENTS.md` die führende gemeinsame Regelquelle bleibt und `.github/copilot-instructions.md` nur ergänzende oder abgeleitete Regeln enthält.

## Problem Framing
Aktuell überschneiden sich `AGENTS.md` und `.github/copilot-instructions.md` in mehreren allgemeinen Regeln. Gleichzeitig ist `AGENTS.md` für den VS-Code-/Codex-Workflow relevant, weshalb eine Verlagerung aller Regeln in `.github/copilot-instructions.md` die Tool-Kompatibilität verschlechtern würde. Es braucht daher eine klare Rollenverteilung ohne widersprüchliche oder doppelte Vorgaben.

## Scope
In Scope:
- Rollenklärung zwischen `AGENTS.md` und `.github/copilot-instructions.md`
- Bereinigung redundanter oder konkurrierender Regeln
- Beibehaltung der Mode-/Workflow-Logik in `AGENTS.md`
- Definition, welche allgemeinen Engineering-Regeln in beiden Dateien stehen dürfen und welche nur einmal gepflegt werden sollen
- Konsistenz für Visual Studio, GitHub Copilot und VS Code + Codex

Out of Scope:
- Änderung fachlicher Produktlogik
- Projekt- oder Solution-Umstrukturierung außerhalb der Regeldateien
- Einführung neuer Tools oder externer Abhängigkeiten

## Current State
- `AGENTS.md` enthält allgemeine Regeln, Workflow-Regeln, Modes, Workitem-Vorgaben und Qualitätsregeln.
- `.github/copilot-instructions.md` enthält ebenfalls allgemeine Verhaltens-, Architektur- und Dokumentationsregeln.
- Für VS Code + Codex ist `AGENTS.md` relevant und darf daher nicht auf einen bloßen Restbestand reduziert werden.

## Target State
- `AGENTS.md` ist die führende, vollständige Regelquelle für den agentenübergreifenden Workflow.
- `.github/copilot-instructions.md` ist auf Copilot-spezifische Ergänzungen und eine schlanke, kompatible Spiegelung reduziert.
- Doppelte Regeln sind minimiert.
- Konflikte zwischen beiden Dateien sind ausgeschlossen oder klar priorisiert.

## Decisions
1. `AGENTS.md` bleibt die führende Datei für alle toolübergreifenden Regeln.
2. Mode-, Workflow-, Workitem-, Plan-, Todo- und Debug-Regeln bleiben ausschließlich oder primär in `AGENTS.md`.
3. `.github/copilot-instructions.md` soll nur Regeln enthalten, die Copilot-spezifisch sind oder als kompatible Kurzfassung ohne konkurrierende Logik benötigt werden.
4. Allgemeine Regeln dürfen nicht unterschiedlich formuliert werden, wenn sie denselben Sachverhalt regeln.

## Steps
1. Bestehende Regeln in beiden Dateien kategorisieren: allgemeine Regeln, Workflow/Modes, Codequalität, Dokumentationspflichten, bereichsspezifische Spezialregeln.
2. Überschneidungen identifizieren und entscheiden, welche Datei je Kategorie führend ist.
3. `AGENTS.md` so bereinigen, dass sie die vollständige zentrale Regelbasis für den Cross-Tool-Workflow bleibt.
4. `.github/copilot-instructions.md` so kürzen oder umformulieren, dass sie `AGENTS.md` ergänzt statt konkurriert.
5. Beide Dateien auf Widersprüche, doppelte Prioritätsaussagen und mehrdeutige Regeln prüfen.
6. Abschließend die neue Rollenverteilung in beiden Dateien klar und knapp sichtbar machen.

## Affected Files
- `AGENTS.md`
- `.github/copilot-instructions.md`
- optional `CHANGELOG.md`, falls die Regelpflege dort dokumentiert werden soll

## Risks
- Zu aggressive Kürzung von `.github/copilot-instructions.md` könnte Copilot-spezifische Hilfestellung verlieren.
- Zu viele Doppelungen würden das ursprüngliche Problem nicht lösen.
- Unterschiedliche Prioritätsformulierungen könnten weiterhin uneindeutig bleiben.

## Verification Strategy
- Manuelle Gegenprüfung beider Dateien auf widersprüchliche Regeln
- Prüfen, ob `AGENTS.md` allein den notwendigen Cross-Tool-Workflow vollständig beschreibt
- Prüfen, ob `.github/copilot-instructions.md` ohne `AGENTS.md` keine konkurrierenden Entscheidungen vorgibt

## Expected Result
Eine stabile, verständliche und redundanzarme Regelstruktur, die in Visual Studio und VS Code mit Codex gleichermaßen funktioniert.

# IMPLEMENTATION HANDOFF

## Goal
Remove the public `ItemProperty.Epoch` metadata API and use only the normal item property `Properties["epoch"]` as the item value update timestamp, so the timestamp is published automatically through the existing item/property publishing flow.

## Scope
- Remove `ItemProperty.Epoch` from `amium_item/src/Amium.Item/Item.cs`.
- Keep `epoch` as a normal item property on every `Item`.
- Initialize `Properties["epoch"]` with a Unix timestamp in milliseconds.
- Update `Properties["epoch"]` whenever `Item.Value` writes to `read` or `write`.
- Migrate current code that reads or writes `ItemProperty.Epoch` to use the item-level `epoch` property where behavior still requires a timestamp.
- Update serialization and cloning so no special `"Epoch"` metadata field is emitted for individual item properties.
- Update or add focused tests if an existing test project covers `Amium.Item`.

This supersedes the older handoff `docs/workitems/2026.05.08.2156-itemproperty-epoch-api/handoffs/2026.05.08.2156-implementation-handoff.md`, which planned to keep `ItemProperty.Epoch`.

## Starting Point
- `ItemProperty` currently exposes `public ulong Epoch { get; set; }`.
- `ItemProperty.Value` currently updates `Epoch` on every property value assignment.
- `Item.Value` currently updates `Properties["epoch"].Value` after writing `read` or `write`.
- The constructor currently contains a bug-like initialization: `Properties["epoch"].Value = _path;`.
- Existing host, mesh, demo, clone, and JSON code still references `ItemProperty.Epoch`.
- The intended timestamp granularity is now item-level, not property-level.
- Assumption: Direct assignments to arbitrary properties, such as `Properties["meta"].Value`, do not update `Properties["epoch"]` unless they happen through `Item.Value`.

## Tasks
1. Remove `Epoch` from `ItemProperty`.
2. Remove all timestamp writes from `ItemProperty.Value` and the `ItemProperty` constructor.
3. Ensure every `Item` creates an `epoch` property with the current Unix milliseconds timestamp.
4. Keep `Item.Value` updating `Properties["epoch"].Value` after a successful write to `read` or `write`.
5. Fix the current `_path` assignment in the main `Item` constructor so `epoch` is a numeric timestamp.
6. Update `ItemExtension` serialization, deserialization, and cloning to stop treating `"Epoch"` as special per-property metadata.
7. Migrate all `ItemProperty.Epoch` usages under `amium_item/` and `src/`.
8. Update documentation references that describe per-property `Epoch` behavior.
9. Run targeted search and build verification.

## File-Level Changes
- `amium_item/src/Amium.Item/Item.cs` -> remove `ItemProperty.Epoch`; remove automatic per-property timestamping; initialize and update `Properties["epoch"]` as a normal item property.
- `amium_item/src/Amium.Item/ItemExtension.cs` -> remove `"Epoch"` from `ItemPropertyToJsonString()` payload; remove deserialization of `"Epoch"` metadata; remove clone copying of `ItemProperty.Epoch`.
- `amium_item/src/Amium.Item.Mesh/Controllers/mesh_demo_node_controller.cs` -> replace timestamp reads from `read` or `value` property metadata with item-level `Properties["epoch"]`.
- `amium_item/samples/amium_item_server_multimaster_demo/Controllers/DemoNodeController.cs` -> replace timestamp reads from `read` or `value` property metadata with item-level `Properties["epoch"]`.
- `src/HornetStudio.Host/HostRegistries.cs` -> update APIs and data copy paths that currently accept or assign per-property timestamps.
- `src/HornetStudio.Host/UiFolderContext.cs` -> migrate propagated timestamps from property metadata to item-level `epoch`.
- `src/HornetStudio.Host/Legacy/UiPageContext.cs` -> migrate propagated timestamps from property metadata to item-level `epoch`.
- `src/HornetStudio/docs/data-flow-and-signals.md` -> update wording if it describes `Epoch` as property metadata.
- Existing tests under `amium_item/` or `tests/` -> update expected JSON/clone behavior and add coverage for item-level `epoch` if applicable.

## Implementation Order
1. Update `Item.cs` first so the core model has only the new timestamp source.
2. Update `ItemExtension.cs` so clone and JSON no longer depend on removed metadata.
3. Search for `.Epoch` usages and classify each one:
   - UI formatting identifiers such as `EpochToDatetime` are not part of `ItemProperty.Epoch` and should stay.
   - Runtime accesses to `ItemProperty.Epoch` must be migrated.
4. Migrate host registry methods that currently take or write per-property timestamps. Prefer deriving the timestamp from the owning item's `epoch` property.
5. Migrate UI folder/page propagation to pass or copy the item-level `epoch` property through the normal property set.
6. Migrate mesh/demo timestamp display code to read `Properties["epoch"]`.
7. Update tests and documentation.
8. Run searches for removed API usage and build.

## Technical Constraints
- No new dependencies.
- Preserve existing project structure and naming.
- Keep code, comments, XML documentation, UI text, and technical identifiers in English.
- Chat responses remain German.
- Keep the change focused on timestamp API migration only.
- Do not refactor unrelated host, mesh, or editor code.
- Keep `epoch` as lowercase because it is a normal item property key.
- Use Unix timestamp in milliseconds as the stored value.
- Preserve the existing `Item.Value` write-channel behavior: write to `write` when present, otherwise to `read`.

## Acceptance Criteria
- `ItemProperty` no longer exposes `Epoch`.
- No runtime code under `amium_item/` or `src/` references `ItemProperty.Epoch`.
- `Item` instances contain a normal `epoch` property.
- New item instances initialize `epoch` to a numeric Unix milliseconds timestamp.
- Assigning `Item.Value` updates `Properties["epoch"].Value`.
- `ItemPropertyToJsonString()` no longer emits a special `"Epoch"` metadata field.
- Clone and JSON roundtrip preserve `epoch` as a normal property because it is part of the property dictionary.
- Existing UI `EpochToDatetime` formatting support remains untouched.
- Build succeeds without `CS1061` or timestamp API errors.

## Verification
- `rg -n 'ItemProperty\\.Epoch|\\.Epoch\\b' amium_item src tests`
- Manually review remaining `.Epoch` matches and confirm they are unrelated UI formatting identifiers or non-`ItemProperty` domain model fields.
- `dotnet build HornetStudio.sln --no-restore`
- If the Amium.Item solution is built separately in this repo state, also run the relevant `dotnet build` command for that solution or project.
- Run existing tests if a test project is available and restore/build state allows it.

## Out of Scope
- Changing item publishing transport behavior beyond making `epoch` available as a normal property.
- Adding new timestamp formats or date/time display features.
- Renaming `EpochToDatetime` UI formatting options.
- Introducing per-property replacement metadata under another name.
- Adding new external dependencies.
- Performing release versioning or publish output.

## Risks / Watchouts
- This is a breaking API change for code that consumed `ItemProperty.Epoch`.
- Existing saved JSON that contains per-property `"Epoch"` metadata will no longer restore that metadata. The value should instead be stored as the normal `epoch` property.
- Some host APIs may currently model timestamps per property. Migrating them naively can weaken conflict resolution if the surrounding code truly needs per-property timestamps.
- `epoch` updates triggered by `Item.Value` can raise a second property change event after `read` or `write`; verify that consumers tolerate this.
- Avoid updating `epoch` recursively when setting `Properties["epoch"].Value` directly.

## Notes
- Prefer helper methods only if the migration otherwise duplicates timestamp parsing in several places.
- A minimal safe helper inside existing classes may be acceptable, but avoid introducing a new timestamp service or abstraction.
- When a timestamp value is missing or invalid, preserve the current behavior as closely as possible and document the fallback in code only if the surrounding code is not self-explanatory.

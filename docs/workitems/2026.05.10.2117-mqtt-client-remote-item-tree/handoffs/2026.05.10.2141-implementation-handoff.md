# IMPLEMENTATION HANDOFF

## Goal

Finish the MQTT client remote-state cleanup so the client represents retained MQTT topics as a pure item/topic forest with no artificial remote-client grouping and no synthetic root status metadata.

The client view must match the shape a user sees in MQTT Explorer for normal item topics:

```text
edm1
  pressure
    raw
    epoch
    format
    unit
    read
  temperature
    raw
    epoch
    format
    unit
    read
```

The client must not create this shape for normal retained topics:

```text
shared
  shared
    edm1
```

## Scope

Included:

- Correct the previous partial implementation of `2026.05.10.2117-mqtt-client-remote-item-tree`.
- Make `MqttRemoteItemRegistry` a pure retained item forest.
- Remove synthetic root concepts for normal retained topics:
  - no artificial `shared` root,
  - no `remote_client_id`,
  - no `connection_status`,
  - no `last_seen_utc`,
  - no `stale`.
- Keep topic-to-item reconstruction and value/property updates.
- Keep `MqttRemoteItemClient.GetRemoteItemSnapshots()` returning roots keyed by item root name.
- Keep local published-path hiding behavior, adapted to the item forest.
- Update WinForms sample rendering only as needed to show the cleaned roots directly.
- Add or adjust focused tests so this regression is caught.
- Update documentation if it still describes remote client roots or synthetic shared grouping.

Not included:

- No new dependencies.
- No MQTT broker cleanup feature.
- No authentication/TLS/configuration UI.
- No writable-item sample.
- No per-publisher ownership model.
- No change to `amium_item/src/Amium.Item/Item.cs`.

## Starting Point

The first implementation pass appears to have partially changed the model:

- `MqttItemTopicMapper.TryMapTopic(...)` now returns `ClientId: null` for normal topics.
- `MqttRemoteItemRegistry` now has `_itemRoots`, but it still treats roots like remote clients by creating synthetic metadata such as `connection_status`, `last_seen_utc`, and `stale`.
- The UI can still show paths like `shared.shared.edm1`, which means either:
  - retained MQTT topics really exist as `shared/shared/edm1/...`, or
  - an old build/old process is still running, or
  - remaining registry/UI logic is still preserving old artificial grouping.

This handoff is intentionally stricter than the previous one: normal retained item roots must be plain item roots, not status-bearing client roots.

## Tasks

1. Inspect current implementation and identify all remaining remote-client-root assumptions in client MQTT code.
2. Simplify `MqttRemoteItemRegistry` so it only stores item roots and descendants from mapped item paths.
3. Remove root status metadata creation for normal item roots.
4. Remove or isolate methods that imply root/client status, such as `MarkRootOffline`, unless a real caller needs them for a separate non-item concept.
5. Ensure path reconstruction for `edm1/pressure/read` creates:
   - root item `edm1`,
   - child item `pressure`,
   - value/property data on `pressure`.
6. Ensure `shared/shared/edm1/...` appears only if the broker topic literally contains `shared/shared/edm1/...`.
7. Add tests for both positive and negative behavior.
8. Update WinForms sample to display direct item roots and avoid showing technical default properties if they are not useful for the sample.
9. Update docs and XML documentation to remove stale remote-client-root wording.
10. Build and run focused verification.

## Implementation Order

1. Add regression tests first:
   - Map/apply `edm1/pressure/read` with payload `1019.9`.
   - Assert snapshots contain root key `edm1`.
   - Assert root `edm1` contains child `pressure`.
   - Assert there is no root key `shared`.
   - Assert no root has synthetic properties `remote_client_id`, `connection_status`, `last_seen_utc`, or `stale`.
2. Add a negative/edge test:
   - If the input topic is literally `shared/shared/edm1/pressure/read`, the resulting item path may be `shared.shared.edm1.pressure`.
   - This proves the client is not inventing or hiding real path segments.
3. Refactor `MqttRemoteItemRegistry`:
   - `GetItemRoots()` returns clones of plain item roots.
   - `Apply(...)` parses payload and updates the item resolved from `mapping.Path`.
   - `GetOrCreateItem(...)` creates root from the first path segment and descendants from remaining segments.
   - No root status touch/update is performed.
4. Clean event semantics:
   - Keep `MqttRemoteItemChangedEventArgs` compatible if needed, but do not require `RemoteClientId` for normal item changes.
   - For normal mapped topics, use `string.Empty` or `null`-equivalent behavior consistently.
5. Verify `MqttRemoteItemClient.RemoveDescendant(...)` against paths like `edm1.pressure`.
   - Removing `edm1.pressure` from root `edm1` must remove child `pressure`.
   - Removing root `edm1` should either remove/empty the root consistently or be explicitly documented.
6. Update WinForms TreeView rendering:
   - Display root item nodes returned from snapshots directly.
   - Do not display old client wrapper nodes.
   - Filter technical default properties only if they are not part of the sample's teaching goal.
7. Update docs/XML comments.
8. Run build/tests.

## Technical Constraints

- Chat responses are German; code, comments, XML documentation, UI strings, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Do not add new dependencies.
- Prefer existing project APIs and patterns.
- Avoid overengineering and wrapper services.
- Public classes and public methods must have XML documentation.
- Use named arguments where method calls have more than two parameters or ambiguous arguments.
- Do not use blocking async calls such as `.Result` or `.Wait()`.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.

## Acceptance Criteria

- `edm1/pressure/read` produces a snapshot root `edm1`, child `pressure`, and value `1019.9`.
- `edm1/pressure/unit` produces property `unit` on `edm1.pressure`.
- `edm1/temperature/read` appears under the same root `edm1`.
- `GetRemoteItemSnapshots()` returns item roots keyed by item root name.
- Normal retained topics do not create a `shared` root unless the actual MQTT topic path starts with `shared`.
- No normal item root contains synthetic properties:
  - `remote_client_id`,
  - `connection_status`,
  - `last_seen_utc`,
  - `stale`.
- The WinForms sample shows `edm1 -> pressure/temperature` without a wrapper root.
- Existing publish, update, subscribe, and retained reconstruction behavior still works.
- Tests fail if artificial `shared` grouping is reintroduced.

## Out of Scope

- Retained MQTT topic deletion/cleanup.
- Broker migration tooling.
- Editable TreeView values.
- Writable item demo.
- Per-publisher ownership or client-group topic schema.
- Full redesign of server health topics.

## Risks / Watchouts

- Existing retained broker data can still contain old topics. The client should not hide real `shared/shared/...` topics; manual verification must distinguish real retained topics from client-invented grouping.
- A stale running executable can make it look like the implementation did not change. Manual verification should use freshly built binaries.
- `Item` default properties may include `path`, `meta`, or `read`. Do not confuse these with synthetic root status metadata.
- If compatibility with old `RemoteClientId` event semantics is required, keep the record shape but stop using it as the storage key for normal retained item state.
- Do not solve this with a WinForms-only display hack. The client API must expose the clean tree.

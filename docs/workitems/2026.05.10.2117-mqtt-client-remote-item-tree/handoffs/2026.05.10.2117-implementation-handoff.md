# IMPLEMENTATION HANDOFF

## Goal

Clean up the MQTT client remote-state model so retained MQTT item topics are represented as a natural item/topic tree, similar to MQTT Explorer, instead of being grouped under the artificial `shared` client root.

Expected visible structure:

```text
edm1
  pressure
    raw
    epoch
    format
    unit
    read
  temperature
    raw
    epoch
    format
    unit
    read
edm2
  ...
```

Not expected:

```text
shared
  edm1
    pressure
```

And especially not:

```text
shared
  shared
    edm1
```

## Scope

Included:

- Refactor the client-side remote item registry from remote-client-root grouping to a normal item root forest.
- Remove the visible `shared` grouping from normal retained item topics.
- Keep retained MQTT topic reconstruction behavior.
- Keep local published-path hiding behavior in `MqttRemoteItemClient`.
- Update the WinForms MQTT demo to render the cleaned item roots without special `shared` handling.
- Update focused tests and documentation affected by the changed client snapshot semantics.

Not included:

- No new dependencies.
- No MQTT protocol redesign.
- No authentication, TLS, broker configuration UI, or retained cleanup feature.
- No broad refactoring of server runtime behavior unless required by the mapper contract.
- No changes to `amium_item/src/Amium.Item/Item.cs`.

## Starting Point

- `MqttItemTopicMapper.TryMapTopic(...)` currently maps normal item topics to `new MqttTopicMapping(..., ClientId: "shared")`.
- `MqttRemoteItemRegistry` stores remote state in `_clientRoots`, keyed by `RemoteClientId`.
- `MqttRemoteItemClient.GetRemoteItemSnapshots()` returns snapshots keyed by remote client id.
- Normal shared retained topics therefore become visible under a synthetic `shared` root.
- The WinForms sample currently displays the remote snapshots as returned by `GetRemoteItemSnapshots()`, so it exposes this synthetic grouping.
- The desired behavior is closer to MQTT Explorer: visible roots should be item/topic roots such as `edm1`, `edm2`, and possibly `$SYS`/system roots, not `shared`.

## Tasks

1. Redesign the client remote registry to store retained MQTT state as an item forest keyed by the first item path segment.
2. Adjust topic mapping so normal shared item topics do not require a synthetic `ClientId`.
3. Preserve source/client metadata only where it is real and useful; do not create visible placeholder roots.
4. Update `MqttRemoteItemClient.GetRemoteItemSnapshots()` semantics and documentation from "keyed by remote client id" to "keyed by item root path/name".
5. Keep hiding of locally published paths working against the new item forest.
6. Update affected code paths in `MqttItemClientSession` that assume `mapping.ClientId` is required for registry updates.
7. Update WinForms sample TreeView rendering to show item roots directly.
8. Update tests or console self-tests that assert old `shared` grouping.
9. Update documentation to describe the natural retained MQTT item tree.

## File-Level Changes

- `amium_item/src/Amium.Item/Server/Mqtt/MqttTopicMapping.cs` -> adjust `MqttTopicMapping` semantics and normal topic mapping; remove or neutralize artificial `shared` for normal item topics.
- `amium_item/src/Amium.Item.Client/Mqtt/MqttRemoteItemRegistry.cs` -> replace `_clientRoots` model with item root forest; rename APIs if appropriate, or keep API names only if compatibility is more important than clarity.
- `amium_item/src/Amium.Item.Client/Mqtt/MqttRemoteItemClient.cs` -> update snapshot semantics, local published-path hiding, event forwarding, and XML documentation.
- `amium_item/src/Amium.Item.Client/Mqtt/MqttItemClientSession.cs` -> ensure received retained topics are applied even without a synthetic `ClientId`; keep message source fields meaningful.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> update or add focused assertions for retained topic reconstruction without `shared`.
- `amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms/MainForm.cs` -> render item roots directly.
- `amium_item/docs/amium-itembroker-usage.md` -> update remote client snapshot/tree explanation and examples.

## Implementation Order

1. Add or adjust focused tests that describe the desired item-tree result:
   - topic `edm1/pressure/read` maps to item path `edm1.pressure`,
   - remote snapshots expose root `edm1`,
   - no `shared` root exists for normal retained topics.
2. Refactor `MqttRemoteItemRegistry`:
   - store item roots by first path segment,
   - create descendants from remaining segments,
   - apply value/property updates at the correct item node,
   - raise change events with path and item clone.
3. Adjust `MqttTopicMapping` and `MqttItemTopicMapper.TryMapTopic(...)`:
   - normal topics should not get artificial `ClientId = "shared"`,
   - server health mapping may keep a source marker only if needed, but should not force visible `shared`.
4. Update `MqttItemClientSession` receive handling:
   - apply mappings without requiring `ClientId`,
   - keep `ItemValueChangedMessage` and `ItemPropertyChangedMessage` source fields coherent.
5. Update `MqttRemoteItemClient.GetRemoteItemSnapshots()`:
   - return item roots keyed by item root path/name,
   - update XML docs,
   - ensure locally published paths are removed correctly from the returned forest.
6. Update WinForms sample TreeView rendering if it currently assumes an outer client grouping node.
7. Update documentation and any affected tests.
8. Run focused build/test validation.

## Technical Constraints

- Chat responses are German; code, comments, XML documentation, UI strings, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Do not add new dependencies.
- Prefer existing APIs and project patterns.
- Keep the solution simple; do not introduce extra services or abstraction layers for hypothetical future grouping.
- Public classes and public methods must have XML documentation.
- Use named arguments where method calls have more than two parameters or ambiguous arguments.
- Avoid blocking async calls such as `.Result` and `.Wait()`.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.

## Acceptance Criteria

- Normal retained item topics are exposed as item roots, not under `shared`.
- `edm1/pressure/read` and related property topics produce a visible root `edm1` with child `pressure`.
- `edm1/temperature/read` appears under the same `edm1` root.
- No `shared.shared...` path can appear from normal retained item topics.
- The WinForms sample shows a tree similar to MQTT Explorer.
- Local published-path hiding in `MqttRemoteItemClient` still hides paths published by that same remote client facade.
- Existing publish/update/subscribe behavior remains functional.
- Documentation no longer describes normal remote snapshots as remote-client-root snapshots if that is no longer true.

## Out of Scope

- MQTT retained cleanup.
- Editable client tree.
- Writable item sample.
- Broker-side ownership model.
- Per-publisher topic namespace design.
- Authentication/TLS/configuration UX.

## Risks / Watchouts

- `GetRemoteItemSnapshots()` currently documents snapshots keyed by remote client id. Changing this is a semantic API change and should be documented clearly.
- Some existing tests or consumers may rely on `shared`; update only intentional call sites.
- Source client id is not encoded in current normal item topics, so the client cannot honestly group retained state by publisher.
- Do not replace a fake `shared` root with another fake grouping concept.
- System topics may need a clear root strategy; prefer a natural root such as `sys` or the existing mapped health path rather than `shared`.

## Notes

- The clean model should follow the retained MQTT topic hierarchy because the broker topic tree is the source of truth.
- `ClientId` should remain the local MQTT connection identity; it should not become a visible grouping root unless the topic protocol actually carries ownership.
- If compatibility concerns are high, consider adding a clearly named new method such as `GetRemoteItemRootSnapshots()` and deprecating the old client-root wording later. Prefer the simpler direct semantic update if this library is still allowed to evolve.

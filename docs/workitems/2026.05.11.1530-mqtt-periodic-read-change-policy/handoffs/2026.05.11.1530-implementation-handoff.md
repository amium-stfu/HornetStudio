# IMPLEMENTATION HANDOFF

## Goal

Extend periodic read publishing so `MqttItemClient.PublishReadPeriodically(...)` can publish only when the read value changes, with an optional forced refresh after a configured number of cycles.

Target API:

```csharp
client.PublishReadPeriodically(
    item,
    periodMilliseconds: 100,
    onlyOnChanged: true,
    forceEveryCycles: 50);
```

## Scope

- Extend `PublishReadPeriodically(...)` with change-detection parameters.
- Apply this policy only to periodic read publishing.
- Keep periodic property and snapshot publishing unchanged.
- Update tests and user-facing documentation.

Explicitly not included:

- Do not add change detection to `PublishPropertyPeriodically(...)`.
- Do not add snapshot diffing to `PublishSnapshotPeriodically(...)`.
- Do not change one-shot `PublishReadAsync(...)`.
- Do not introduce new dependencies.
- Do not change `amium_item/src/Amium.Item/Item.cs`.

## Starting Point

- The publish API cleanup has already been implemented.
- Current public periodic methods include:
  - `PublishReadPeriodically(ItemModel item, int periodMilliseconds)`
  - `PublishPropertyPeriodically(ItemModel item, string parameterName, int periodMilliseconds)`
  - `PublishSnapshotPeriodically(ItemModel item, int periodMilliseconds)`
- Current read periodic behavior publishes on every timer tick.
- `PublishReadAsync(...)` can publish `read` and, by default, `epoch`.

## Tasks

1. Extend `PublishReadPeriodically(...)` signature:
   - `bool onlyOnChanged = false`
   - `int? forceEveryCycles = null`
2. Preserve current behavior when `onlyOnChanged` is `false`.
3. When `onlyOnChanged` is `true`:
   - first tick publishes
   - subsequent ticks publish only when the `read` value changed
   - if `forceEveryCycles` is set, publish at least every configured number of cycles
4. Compare only the read value, not the epoch or whole item.
5. Update periodic registration state to track:
   - last published read value
   - whether an initial publish has occurred
   - skipped/elapsed cycle count for forced refresh
6. Validate `forceEveryCycles` when provided.
7. Update tests.
8. Update README and usage docs.
9. Build and run focused self-tests.

## Acceptance Criteria

- `PublishReadPeriodically(...)` accepts `onlyOnChanged` and `forceEveryCycles`.
- Existing callers continue to work because new parameters have defaults.
- With `onlyOnChanged: false`, periodic read publishing behaves as before.
- With `onlyOnChanged: true`, first tick publishes.
- With `onlyOnChanged: true`, unchanged read values are skipped.
- With `onlyOnChanged: true`, changed read values are published.
- With `forceEveryCycles` set, unchanged read values are still published at the configured cycle interval.
- Read comparison ignores `epoch`.
- Periodic property and snapshot publishing behavior is unchanged.
- Documentation explains the new options clearly.

## Out of Scope

- Property change detection.
- Snapshot diffing.
- Deep comparison for complex read values.
- Configuring change detection globally through `MqttItemClientOptions`.
- Real publish acknowledgements or broker feedback.

## Risks / Watchouts

- `Item.Value` may write to `write` when a write channel exists. Compare the actual `read` property value, not blindly `Item.Value`, if that distinction matters in current `Item` behavior.
- `object.Equals(...)` is acceptable for scalar values, but it is not a deep comparer for arbitrary complex objects.
- If `publishEpoch: true` changes epoch during publish, do not treat that epoch change as a read change on the next cycle.
- Forced refresh counting should be based on cycles, not elapsed wall-clock time.
- Keep this as an opt-in optimization; default behavior should remain simple and unchanged.

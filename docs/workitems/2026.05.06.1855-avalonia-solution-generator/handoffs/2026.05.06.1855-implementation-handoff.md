# IMPLEMENTATION HANDOFF

## Goal

Build an Avalonia desktop tool that creates a new .NET solution folder from UI inputs and writes reusable project instruction files derived from this repository's `AGENTS.md` without solution-specific references.

## Scope

Included:

- Avalonia desktop UI for collecting generator inputs.
- Creation of a new solution directory.
- Creation of a `src/` based .NET solution structure.
- Creation of one initial project.
- Optional creation of one test project.
- Generation of `AGENTS.md`.
- Generation of `.github/copilot-instructions.md`.
- Generation of `.gitignore`.
- Optional `git init`.
- Optional opening of the created folder in VS Code.
- Validation and user-visible error reporting in English.
- Unit tests for validation and template generation.

Not included:

- Full template marketplace.
- Dynamic parsing of arbitrary existing `AGENTS.md` files.
- Complex multi-project architecture presets.
- Release publishing.
- Git commits or Git tags.
- Installing missing SDKs, Git, VS Code, or Avalonia tools automatically.

## Starting Point

- Current repository has a detailed `AGENTS.md` with reusable rules and solution-specific references.
- The new tool should use a curated generic template, not raw copying.
- Chat responses are German, but code, comments, XML documentation, UI text, file names, and technical identifiers are English.
- Required prerequisites on the implementation PC:
  - .NET SDK installed.
  - Avalonia-compatible development setup.
  - Git installed if Git initialization will be tested.
  - VS Code installed and `code` available on PATH if VS Code opening will be tested.

## Tasks

1. Create or choose the target repository for the generator.
2. Add an Avalonia desktop app project.
3. Add a test project if the target repository has test conventions or create one beside the app.
4. Build a wizard-like UI with fields for target directory, solution name, first project name, project template, create tests, initialize Git, and open VS Code.
5. Add validation for names, paths, conflicting output folders, and optional external tools.
6. Add a generation service that creates folders, runs `dotnet` commands, writes templates, and reports progress.
7. Add static templates for `AGENTS.md`, `.github/copilot-instructions.md`, and `.gitignore`.
8. Add template replacement for placeholders such as solution name and current year if needed.
9. Add optional post-generation steps for `git init` and `code <solution-root>`.
10. Add unit tests for validation, template output, and command planning.
11. Manually verify a generated sample solution builds.

## Implementation Order

1. Decide whether this is implemented inside `HornetStudio` or as a standalone repository.
2. Create the Avalonia app and verify it builds before adding generator logic.
3. Create model types for generation options and validation results.
4. Implement validation without file creation.
5. Add static templates and template rendering.
6. Implement a dry-run command plan so tests can verify expected `dotnet` commands.
7. Implement actual directory creation and `dotnet new` execution.
8. Add optional Git init and VS Code open steps.
9. Connect the UI to validation and generation progress.
10. Add tests for validation and template output.
11. Generate a sample solution in a temporary folder and build it.
12. Review generated files to ensure no `HornetStudio`, `Amium`, or local paths remain in templates.

## Technical Constraints

- Use Avalonia for the desktop UI.
- Prefer existing .NET libraries and framework APIs.
- Do not introduce new runtime dependencies unless necessary and documented.
- Use async process execution and avoid blocking `.Result`, `.Wait()`, or long work on the UI thread.
- User-visible UI text and validation messages must be English.
- Public classes and public methods must have XML documentation.
- Handle failures explicitly with contextual error messages.
- Do not silently ignore failed external commands.
- Keep the first version focused and avoid plugin/template marketplaces.
- Generated path-relevant artifact names should be predictable and safe.

## Acceptance Criteria

- The app lets the user select a target directory and enter a solution name.
- The app can generate a new solution folder with `src/`, `.sln`, and at least one project.
- The generated root contains `AGENTS.md`, `.github/copilot-instructions.md`, and `.gitignore`.
- The generated `AGENTS.md` contains reusable rules but no `HornetStudio`, `Amium`, or local machine paths.
- The generated `.gitignore` excludes common .NET, Visual Studio, VS Code, build, log, and test output folders.
- Optional Git init works when Git is available.
- Optional VS Code open works when `code` is available.
- Failures are shown clearly in the UI and do not leave the user guessing.
- Unit tests cover validation and template rendering.

## Verification

- Run `dotnet build <GeneratorSolution>.sln --no-restore` after restore has completed.
- Run the generator tests with the repository's normal test command.
- Manually generate a sample solution into a temporary directory.
- Run `dotnet build <GeneratedSolution>.sln --no-restore` for the generated solution after restore has completed.
- Search generated files for forbidden solution-specific terms:
  - `HornetStudio`
  - `Amium`
  - `b:\`
  - current local user profile paths
- Test failure handling by selecting an existing non-empty output folder.

## Out of Scope

- Creating GitHub repositories.
- Publishing NuGet packages.
- Creating release versions or tags.
- Supporting non-.NET projects.
- Automatically installing external tools.
- Creating a full graphical project designer.

## Risks / Watchouts

- Raw-copying this repository's `AGENTS.md` would leak solution-specific rules and paths into new projects.
- Running external tools from a UI can freeze the app if process execution is not async.
- Folder overwrite behavior can destroy user work if not guarded carefully.
- `code` may not be on PATH even when VS Code is installed.
- `dotnet new` template names vary by installed SDK and workloads.
- Avalonia project templates may require restore or package access on a fresh PC.

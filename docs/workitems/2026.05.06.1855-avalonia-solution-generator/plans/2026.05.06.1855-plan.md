# Avalonia Solution Generator Plan

## Goal

Create a small Avalonia desktop tool that generates new .NET solution folders from a few UI inputs. The generated output should include a clean solution layout, a generic `AGENTS.md` derived from this repository's rules without solution-specific details, GitHub Copilot instructions, a practical `.gitignore`, optional Git initialization, and optional opening in VS Code.

## Problem Framing

The current repository contains useful project rules in `AGENTS.md`, but those rules include solution-specific details such as the repository name, local path, project names, build command, publish behavior, and existing workflow history. A reusable generator should not parse the current `AGENTS.md` as executable configuration. It should use a curated generic template that preserves the reusable rules and fills only safe placeholders from UI input.

## Proposed Product Shape

- Desktop app built with Avalonia.
- First version should be a focused wizard, not a broad project scaffolding framework.
- The UI should collect:
  - target directory
  - solution name
  - optional first project name
  - project template choice for the first project
  - create test project yes/no
  - initialize Git yes/no
  - open folder in VS Code yes/no
- The generator should produce:
  - solution root
  - `src/`
  - `.sln`
  - one initial project under `src/<ProjectName>/`
  - optional test project under `src/<ProjectName>.Tests/`
  - `AGENTS.md`
  - `.github/copilot-instructions.md`
  - `.gitignore`

## Architecture

Keep the implementation intentionally small:

- UI layer: Avalonia views and view models for input, validation, preview, and execution status.
- Generation layer: validates names and paths, creates directories, runs `dotnet` commands, writes template files, and reports progress.
- Template layer: embedded or content-file templates for `AGENTS.md`, Copilot instructions, and `.gitignore`.
- Process layer: thin service for external commands such as `dotnet`, `git`, and `code`.

Avoid introducing a plugin system, dynamic template engine, or complex profile model in the first version. Static templates with placeholder replacement are sufficient.

## Template Strategy

The generic `AGENTS.md` template should keep reusable rules from this repository:

- mode control and aliases
- German chat responses
- English code, comments, documentation, UI texts, file names, and identifiers
- minimal, scoped changes
- existing conventions first
- XML documentation for public APIs
- tests for changed logic
- no secrets
- release versioning format
- FOSS and license documentation expectations

The template must remove or generalize:

- `b:\HornetStudio`
- `HornetStudio.sln`
- `HornetStudio`, `Amium.*`, and other project-specific names
- existing workitem history
- build commands that reference this solution by name
- publish output assumptions that may not apply to every generated project unless explicitly selected

Copilot instructions should be shorter than `AGENTS.md` and focused on coding conventions, response language, tests, documentation, and safety.

## Execution Flow

1. Validate UI inputs.
2. Resolve the final solution root.
3. Refuse generation when the target directory already contains conflicting files unless the user explicitly selected an empty/new folder.
4. Create the folder structure.
5. Run `dotnet new sln`.
6. Run `dotnet new <template>` for the initial project.
7. Add project to the solution.
8. Optionally create and add a test project.
9. Write `AGENTS.md`, `.github/copilot-instructions.md`, and `.gitignore`.
10. Optionally run `git init`.
11. Optionally run `code <solution-root>`.
12. Show a concise success or failure report.

## Validation Rules

- Solution name must be a valid .NET identifier-like project name with no path separators.
- Target directory must exist or be creatable.
- Generated folder path must not overwrite an existing non-empty project unless explicitly supported later.
- External tool availability should be checked before generation:
  - `dotnet --info`
  - `git --version` only when Git init is selected
  - `code --version` only when VS Code open is selected

## Testing Strategy

- Unit tests for name/path validation.
- Unit tests for template placeholder replacement.
- Unit tests for command planning without executing external tools.
- Integration-style test with a temporary directory can be added if the repository already supports it comfortably.
- Manual verification on Windows:
  - generate a sample solution
  - build the generated solution
  - confirm generated instructions do not contain `HornetStudio`, `Amium`, or local paths
  - confirm VS Code open is optional and failure is reported cleanly

## Suggested Implementation Location

If implemented inside this repository, use existing `src/` conventions:

- `src/HornetStudio.SolutionGenerator/` for the Avalonia app
- `src/HornetStudio.SolutionGenerator.Tests/` for tests

If implemented as a standalone project on another PC, use:

- `src/SolutionGenerator/`
- `src/SolutionGenerator.Tests/`

The second option is cleaner if the tool is intended to become independent from this repository.

## Codex Desktop Assessment

Codex Desktop is a good fit for implementation because this task benefits from local file creation, `dotnet new`, build checks, and UI iteration. This chat is enough for planning, but the implementation should run on a machine with the .NET SDK, Avalonia templates or packages available, Git, and VS Code installed if those optional features should be tested.

For a different PC, use the implementation handoff in this workitem. It is intentionally self-contained and should not require reading this chat.

## Decisions

- Use Avalonia for the UI.
- Use static curated templates instead of parsing `AGENTS.md` dynamically.
- Generate both `AGENTS.md` and `.github/copilot-instructions.md`.
- Keep the first version focused on .NET solution scaffolding.
- Treat Git init and VS Code open as optional post-generation steps.

## Open Questions

- Should the generator live inside this repository or be created as a separate repository?
- Which first project templates should be supported initially: `classlib`, `console`, `worker`, `webapi`, Avalonia app, or only a minimal subset?
- Should generated paths follow strict `snake_case`, or should .NET project folder names follow project names as usual?
- Should the generated `AGENTS.md` include the full mode system, or a shorter default subset?

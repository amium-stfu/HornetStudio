# IMPLEMENTATION HANDOFF

## Goal

Migrate UDL module item structure from nested `Request` child items and `Command` to flat item properties using `read` and optional `write` channels.

## Scope

Included:

- UDL module shape changes.
- UDL write routing changes from `Request` child item updates to `write` property updates.
- Minimal item-model support needed for `Alert` without a read channel.
- Broker/editor write-back updates directly required by the new flat UDL shape.
- Focused tests and documentation updates.

Not included:

- Unrelated broker ownership redesign.
- Widget visual redesign.
- Release/publish/version changes.
- Broad cleanup outside UDL/item write-channel behavior.

## Starting Point

- `Item.Value` currently reads `Properties["read"]` and writes to `Properties["write"]` when the property exists.
- `Item` currently creates `read` by default.
- `Module`, `UdlModule`, `ExtendedSignalModule`, and the simulator still create nested `Request` items.
- UDL write processing currently observes `ReadRequest`, `SetRequest`, `OutRequest`, and `CommandRequest`.
- `HostItemBrokerWriteBackClient` still falls back to a `Request` child and honors `write_path` / `write_mode`.
- `Properties["kind"]` is not needed for UDL routing but may be used as category metadata by consumers.

## Tasks

1. Extend `Item` construction only as needed so an item can intentionally omit the `read` property.
2. Update UDL channel creation to create channels with exact capabilities:
   - `Read`: read + write
   - `Set`: read + write
   - `Out`: read + write
   - `State`: read + write
   - `Alert`: no read channel
3. Remove `Command` from new UDL module structures.
4. Replace request-child write subscriptions in UDL clients with `write` property change handling.
5. Remove UDL dependence on `writable`, `write_path`, and `write_mode` where `write` provides the same information.
6. Update editor UDL exposure logic to detect write capability through `Properties.Has("write")`.
7. Update broker write-back logic so flat `write` channels are preferred and `Request` is only a legacy fallback if still needed.
8. Update simulator/demo module generation to match the real UDL shape.
9. Update tests that assert old `Request` / `Command` structure.
10. Update documentation that describes UDL module channels and write paths.

## File-Level Changes

- `src/Amium.Item/Item.cs` -> add minimal constructor support for items without `read`; keep existing behavior as default.
- `src/Amium.UdlClient/Module.cs` -> replace request-channel creation with flat read/write channel creation; remove `Command`; decide whether to keep or omit `kind`.
- `src/Amium.UdlClient/UdlClient.cs` -> subscribe to and process `write` property changes for UDL write channels; remove command request handling.
- `src/HornetStudio.Host/UdlModule.cs` -> mirror the flat UDL module structure.
- `src/HornetStudio.Host/HostUdlClient.cs` -> mirror UDL write routing changes.
- `src/HornetStudio.Host/ExtendedSignalModule.cs` -> align only if it represents the same UDL channel model; otherwise keep separate with explicit reasoning.
- `src/HornetStudio.Editor/Widgets/UdlClient/SimulatedHostUdlClient.cs` -> generate flat channels and no `Command`.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` -> update writeability and value-source resolution away from `Request`.
- `src/HornetStudio.Editor/Widgets/Broker/HostItemBrokerWriteBackClient.cs` -> prefer `write` property for write-back routing; limit `Request` fallback to legacy only.
- `src/Amium.Item.Server.Tests/Program.cs` -> update/add item write-channel tests where relevant.
- `src/HornetStudio.Host.Tests/Program.cs` -> update UDL/host registry tests that currently expect `Request`.
- `docs/amium-item-communication-structure.md` -> update UDL structure and write path documentation.
- `docs/amium-itembroker-usage.md` and `docs/amium-itembroker-architecture.md` -> update only if they still document UDL `Request` write paths.

## Implementation Order

1. Inspect the current `Item` constructors and all UDL module builders.
2. Add the smallest `Item` constructor option needed to omit `read`, preserving existing default constructor behavior.
3. Update `src/Amium.UdlClient/Module.cs` first and keep names/properties minimal.
4. Update `src/Amium.UdlClient/UdlClient.cs` to react to channel `write` property changes.
5. Mirror the same changes in `HornetStudio.Host` UDL classes.
6. Update simulator/demo generation.
7. Update editor UDL exposure logic.
8. Update broker write-back routing.
9. Update tests.
10. Update documentation.
11. Run build and relevant tests.

## Technical Constraints

- Chat and planning text may be German, but code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Keep properties flat and minimal.
- Prefer existing project patterns.
- Do not introduce new dependencies.
- Use named parameters when constructor calls become ambiguous.
- Keep public API changes as small as possible and call out breaking changes.
- Public classes and public methods touched during implementation need XML documentation maintained.
- Do not implement unrelated refactorings.

## Acceptance Criteria

- `Request` child items are no longer created for new UDL modules.
- `Command` is no longer created for new UDL modules.
- `Read`, `Set`, `Out`, and `State` have a `write` property.
- `Alert` has no `read` property, unless implementation explicitly documents a temporary compatibility compromise.
- Writing a UDL channel updates the channel `write` property and triggers the correct UDL write behavior.
- UDL write routing no longer depends on `write_path` or `write_mode`.
- Broker/editor write-back works with flat write channels.
- Tests reflect the new channel model.
- Documentation reflects the new channel model.

## Verification

- `dotnet build HornetStudio.sln --no-restore`
- Run item tests covering `read`, `write`, and no-read item construction.
- Run host tests covering UDL module registration and write-back behavior.
- Run item broker tests covering writable item handling.
- Manually verify one UDL module write from UI or broker path reaches the UDL write processor.

## Out of Scope

- Keeping `Request` as the primary write mechanism.
- Adding a replacement `Command` concept.
- Broad broker ownership changes.
- UI redesign.
- Release packaging.

## Risks / Watchouts

- Existing saved layouts or external MQTT clients may still target old `.Request` paths.
- Removing `Command` may expose stale configuration references.
- `Alert` without `read` can affect generic code that assumes every item has a readable value.
- Removing `kind` from UDL modules can affect category display in signal descriptors.
- `ExtendedSignalModule` may intentionally differ from UDL modules; do not force alignment without confirming its role.

## Notes

- Treat `write` presence as channel capability, not as a generic mutability flag.
- Prefer `Properties.Has("write")` over `writable`, `write_path`, or `write_mode` for new UDL logic.
- Keep `kind` out of routing decisions. If it remains, it should be metadata only.
- If legacy compatibility is required, isolate it in read/write-back adapters rather than reintroducing `Request` into the new module structure.

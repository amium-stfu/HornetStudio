# UDL Module Flat Channels Plan

## Goal

Flatten the UDL module item structure so channel direction is represented by item properties instead of nested `Request` child items.

## Problem

The current UDL module structure still models write requests with child items such as `Read.Request`, `Set.Request`, `Out.Request`, and `Command.Request`. The generic `Item` model now has a `read` property and an optional `write` property, so the nested request structure duplicates the same concept with more hierarchy and extra metadata.

The updated domain information also changes the expected UDL channel shape:

- There is no `Command` channel.
- `State` has a write channel.
- `Alert` has no read channel.
- Properties should stay flat, minimal, and only contain what is needed.

## Scope

- Define the target UDL module item shape.
- Plan migration from `Request` child items to `write` properties.
- Identify affected host, editor, broker, simulator, and test areas.
- Clarify the expected role of `Properties["kind"]`.

## Out of Scope

- Implementing the migration.
- Redesigning the generic broker ownership model.
- Adding new runtime dependencies.
- Changing unrelated widget layout or styling.
- Release/versioning work.

## Target Structure

The desired module structure is:

- `Read` with `read` and `write`
- `Set` with `read` and `write`
- `Out` with `read` and `write`
- `State` with `read` and `write`
- `Alert` without a read channel
- no `Command`
- no nested `Request` items

This keeps channel capabilities on the item itself instead of spreading write intent across child nodes and metadata.

## Property Model

Prefer the smallest useful property set:

- Keep standard item properties required by `Item`, currently `name`, `path`, `read`, optional `write`, and `meta`.
- Keep UDL-specific properties only when they carry real external or runtime meaning, for example `module_id`, `text`, and `unit`.
- Remove or stop depending on `writable`, `write_path`, and `write_mode` for UDL channels once `write` is the canonical write channel.
- Do not use `Request` as an implicit write path fallback for the new UDL structure.

## `kind` Decision

`Properties["kind"]` is not required to perform UDL read/write routing. It is currently used elsewhere as a category-like metadata property, especially by signal descriptors and some runtime snapshots.

Recommended decision:

- Do not require `kind` for UDL module logic.
- Keep compatibility with existing consumers that may read `kind`.
- If removing it from UDL modules, first make consumers tolerate missing `kind` and verify UI/broker classification still behaves acceptably.
- Treat removal of `kind` as a separate, small cleanup decision if it is not needed after the channel migration.

## Architecture Approach

1. Make the generic `Item` model capable of representing an item without a `read` channel before changing `Alert`.
2. Update UDL module construction to create child channels explicitly with the intended read/write capabilities.
3. Replace UDL write routing from `Request` item changes to `write` property changes.
4. Remove `Command` from UDL modules after write routing is moved to the correct real channel.
5. Update editor exposure and broker write-back logic so it detects write capability through `Properties.Has("write")`.
6. Keep any legacy fallback for old snapshots narrow and temporary, only where needed for migration.

## Sequencing

1. Add or confirm item-level support for missing read channels.
2. Introduce the new UDL module structure in one module implementation first.
3. Align duplicate host/editor UDL module builders to the same structure.
4. Move UDL write processing to `write` property changes.
5. Remove `Request`-based write metadata from UDL channels.
6. Remove `Command` from UDL module shape and simulator/demo generation.
7. Update broker and UI write-back detection.
8. Update tests and documentation.

## Risks

- Existing UI or broker code may still write to `Request` and silently stop controlling UDL channels.
- Removing `Command` can break saved configurations or demos that still reference it.
- Removing or ignoring `kind` can reduce category information in signal descriptors or UI displays.
- Alert without `read` needs generic item support; simply leaving `read = null` is easier but weaker semantically.
- Existing tests may encode the old request structure and need careful updates instead of blanket removal.

## Acceptance Criteria

- New UDL modules do not create `Request` child items.
- New UDL modules do not create a `Command` channel.
- `Read`, `Set`, `Out`, and `State` expose `write` as the write channel.
- `Alert` no longer exposes a read channel, or the chosen temporary compromise is explicitly documented.
- UDL writes are triggered from `write` changes, not `Request.Value` changes.
- Broker/UI write-back uses the flat write channel model.
- Tests cover item read/write behavior and UDL module structure.
- Documentation no longer describes `Request` as the normal UDL write path.

## Verification

- Build the full solution with `dotnet build HornetStudio.sln --no-restore`.
- Run relevant item, item broker, host, and UDL-related tests.
- Manually verify a UDL channel write through the editor/broker path reaches the UDL client write processing.
- Verify old `kind` assumptions do not cause null or classification failures if `kind` is removed or omitted.

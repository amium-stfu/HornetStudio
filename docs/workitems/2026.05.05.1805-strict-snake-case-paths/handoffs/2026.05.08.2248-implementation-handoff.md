# IMPLEMENTATION HANDOFF

## Goal
Consolidate Amium item path handling so there is exactly one canonical item path format: dot-separated `snake_case` segments. Remove unnecessary fragmentation around item path helpers and reject non-canonical paths instead of silently migrating or normalizing legacy formats.

## Scope
Included:

- Make `Amium.Items.ItemPath` the single source for general item path rules.
- Enforce dot-separated `snake_case` paths.
- Stop accepting slash, backslash, PascalCase, camelCase, mixed-case, empty-segment, or separator-normalized legacy input as valid item paths.
- Move or replace recursive item repathing so it is owned by `Item` behavior instead of `ItemPathExtensions`.
- Remove `ItemServerItemPath` if its remaining purpose is only `item.Path ?? item.Name` plus normalization.
- Keep `ItemServerPath` only for server-specific semantics such as the global subscription token `*`, matching, and descendant checks.
- Update direct usages and tests affected by the stricter behavior.
- Update relevant `Amium.Item` documentation.

Explicitly NOT included:

- Runtime migration of old item paths.
- Compatibility aliases for slash/backslash or PascalCase paths.
- Broad restructuring outside path handling.
- Release versioning or publish work.

## Starting Point
- `amium_item/src/Amium.Item/ItemPath.cs` currently normalizes many legacy forms by replacing `/` and `\` with `.`, splitting on `.`, converting segments to `snake_case`, and removing empty segments.
- `amium_item/src/Amium.Item/ItemPathExtensions.cs` exposes `Repath(this Item item, string absolutePath)` and mutates `Item._path`, `name`, `path`, property paths, and child paths recursively.
- `amium_item/src/Amium.Item/ItemServerItemPath.cs` only resolves `item.Path ?? item.Name` and passes it to `ItemServerPath.Normalize(...)`.
- `amium_item/src/Amium.Item/ItemServerPath.cs` wraps `ItemPath.Normalize(...)` and adds server-specific `GlobalSubscriptionPath = "*"`, equality, matching, and descendant behavior.
- `amium_item/src/Amium.Item/Item.cs` constructs paths through `ItemPath.Normalize(...)` and `ItemPath.Combine(...)`.
- Current tests include legacy-compatible examples such as `runtime/device\Read` and `runtime.Device.Read`; these should be changed because migrations are intentionally out of scope.

## Tasks
1. Redesign `ItemPath` around strict validation and canonical path composition.
2. Decide the final API names inside `ItemPath`, favoring explicit methods such as `Validate`, `Normalize` only if it now means "validate and return unchanged canonical value", `Combine`, and segment validation.
3. Replace the segment converter behavior with strict segment validation for lowercase `snake_case`.
4. Ensure invalid input throws clear `ArgumentException` messages in English.
5. Move recursive `Repath` behavior into `Item` or replace usages with an equivalent `Item` method that preserves item/property path consistency.
6. Remove `ItemPathExtensions.cs` after usages are updated, unless the implementation intentionally keeps a minimal compatibility extension for one transition; compatibility is not required by the goal.
7. Remove `ItemServerItemPath.cs` and update call sites to use canonical `item.Path` or a small local helper only where a null-path guard is still technically required.
8. Keep `ItemServerPath` focused on server-specific path matching and `*`; it should delegate canonical item path validation to `ItemPath`.
9. Update all call sites that currently rely on `ItemPath.ToSnakeCaseSegment(...)` converting arbitrary property names.
10. Update mesh/demo defaults and tests so generated or supplied paths are already canonical.
11. Update `amium_item/src/Amium.Item/README.md` to document strict dot-separated `snake_case` item paths and the absence of legacy migration.
12. Run focused tests and the complete solution build.

## File-Level Changes
- `amium_item/src/Amium.Item/ItemPath.cs` -> own all canonical item path validation/composition rules; reject non-canonical paths instead of converting them.
- `amium_item/src/Amium.Item/Item.cs` -> own recursive path reassignment if `Repath` is moved here; ensure constructors, child creation, properties, and path metadata use canonical paths consistently.
- `amium_item/src/Amium.Item/ItemPathExtensions.cs` -> delete or reduce only if a deliberate minimal bridge is still needed; preferred end state is removal.
- `amium_item/src/Amium.Item/ItemServerItemPath.cs` -> delete and replace usages because it is a thin indirection.
- `amium_item/src/Amium.Item/ItemServerPath.cs` -> keep only server-specific behavior; ensure `*` remains accepted only where server subscriptions need it.
- `amium_item/src/Amium.Item/README.md` -> update usage examples and notes for strict paths.
- `amium_item/src/Amium.Item.Server/**` -> update server usages of `ItemServerItemPath.Resolve(...)`, `ItemPath.ToSnakeCaseSegment(...)`, and legacy-compatible test expectations.
- `amium_item/src/Amium.Item.Client/**` -> update client usages of `ItemServerItemPath.Resolve(...)`, parameter-name canonicalization, and registry path handling.
- `amium_item/src/Amium.Item.Mesh/**` -> ensure generated item paths and names are canonical before they reach `ItemPath`.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> replace legacy-path assertions with strict-valid and strict-invalid expectations.

## Implementation Order
1. Inspect all `ItemPath`, `ItemServerPath`, `ItemServerItemPath`, `Repath`, and `ToSnakeCaseSegment` usages.
2. Update `ItemPath` to strict validation/composition first.
3. Move recursive path reassignment into `Item` or introduce the final chosen `Item` API.
4. Update all `.Repath(...)` call sites to the final API.
5. Remove or collapse `ItemPathExtensions.cs`.
6. Replace all `ItemServerItemPath.Resolve(...)` usages with direct canonical `item.Path` handling or `ItemServerPath` where server subscription semantics are required.
7. Review all `ToSnakeCaseSegment(...)` usages and replace conversion behavior with validation of already-canonical property names.
8. Update tests to assert rejection of old formats and acceptance of canonical paths.
9. Update `Amium.Item` README.
10. Build and run relevant tests.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible errors and validation messages must be English.
- Do not add dependencies.
- Keep changes limited to `amium_item` path handling and directly affected tests/docs.
- Do not migrate old path values.
- Public classes and public methods that remain or are added need XML documentation.
- Preserve C# naming conventions; the `snake_case` rule applies to item paths and path-relevant string values, not namespaces, class names, methods, or framework identifiers.
- Prefer existing project patterns and avoid new helper layers unless they replace current fragmentation.

## Acceptance Criteria
- Valid item paths are dot-separated `snake_case` only, for example `runtime.device.read` and `nodes.node_1.dynamic_item`.
- Invalid paths such as `runtime/device/read`, `runtime\device\read`, `runtime.Device.Read`, `Runtime.device`, `runtime..device`, `runtime.device.`, and `runtime-device-read` are rejected.
- `ItemPath` is the central general path rule implementation.
- `ItemServerItemPath.cs` is removed or has no remaining functional need.
- `ItemPathExtensions.cs` is removed or no longer owns core item path mutation.
- Server-specific `*` subscription behavior still works through `ItemServerPath`.
- Item constructors, child creation, property paths, and recursive path reassignment keep metadata internally consistent.
- Tests no longer expect silent path migration.
- Documentation states the strict canonical path format.

## Out of Scope
- Automatic conversion of old saved data.
- Preserving slash/backslash/PascalCase compatibility.
- Renaming unrelated source files or namespaces.
- UI/editor path identity work outside directly affected `amium_item` consumers.
- Release packaging, version number generation, Git tags, or notice-file updates.

## Risks / Watchouts
- Current tests and demo code intentionally use legacy path shapes; many failures after strict validation are expected until those inputs are changed.
- `Item.Name` currently stores the original constructor name. Decide whether item names must also be canonical path segments when they define paths; avoid silently displaying non-canonical names as path identities.
- Property names passed into MQTT/server update APIs may currently rely on automatic conversion. Tightening this can expose callers that still send `Read`, `Write`, or mixed-case names.
- `ItemServerPath.Equals(...)` currently uses `OrdinalIgnoreCase`; once canonical paths are lowercase only, equality can likely be ordinal, but changing matching semantics should be done deliberately.
- Deleting public extension APIs can be a breaking change. This is acceptable only because the requested goal explicitly rejects migration/compatibility.
- Avoid broad case transformations; previous strict snake-case work accidentally changed C# namespaces and identifiers.

# IMPLEMENTATION HANDOFF

## Goal
Enforce canonical `snake_case` for all path-relevant names globally and remove remaining logic that still generates or expects new PascalCase path segments.

## Scope
Included:

- Audit path-relevant name generation and normalization across item, host, editor, broker, and widget layers.
- Ensure newly written runtime, studio, attach, broker, and generated artifact paths use canonical `snake_case`.
- Remove or replace remaining hard-coded PascalCase path literals where they influence canonical path creation.
- Keep read-side compatibility only where legacy persisted or runtime data still needs to be discovered.
- Update focused tests for canonical path generation and compatibility parsing.

Explicitly NOT included:

- Renaming C# types, namespaces, public APIs, or UI display text to `snake_case`.
- Broad refactoring unrelated to path generation or path parsing.
- Release/version/publish work.
- Changing non-path naming rules for normal .NET identifiers.

## Starting Point

- `AGENTS.md` already states that newly created path-relevant names must use `snake_case`.
- `Amium.Item.ItemPath` already provides canonical path normalization and `ToSnakeCaseSegment(...)`.
- Several higher-level helpers already normalize names through `TargetPathHelper` or `UdlPathHelper`.
- The codebase still contains mixed behavior: some path-sensitive logic uses canonical helpers, while some widgets still build or parse PascalCase path fragments directly.
- Earlier strict path migration already exposed the risk of applying casing rules too broadly to code identifiers instead of only to path-relevant names.

## Tasks

1. Define the enforcement boundary precisely:
   - path-relevant names must be canonical `snake_case`
   - code identifiers and display names keep existing conventions
2. Inventory all remaining places that create canonical paths from raw names without centralized normalization.
3. Inventory all remaining hard-coded PascalCase path literals that affect runtime behavior.
4. Route canonical path generation through existing central helpers wherever possible.
5. Replace direct PascalCase path construction with canonical helper-based construction.
6. Keep legacy mixed-case support only on read/discovery paths where existing data may still exist.
7. Add or update tests that prove:
   - new writes are canonical `snake_case`
   - legacy mixed-case paths are only tolerated where explicitly intended
8. Verify build and targeted tests.

## File-Level Changes

- `src/Amium.Item/ItemPath.cs` -> verify it remains the single canonical segment normalization source.
- `src/HornetStudio.Editor/Helpers/TargetPathHelper.cs` -> ensure path-building entry points consistently normalize path-relevant segments.
- `src/HornetStudio.Editor/Helpers/UdlPathHelper.cs` -> keep canonical UDL runtime/status path generation centralized.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` -> remove remaining direct PascalCase path fragments that affect runtime/status/attach paths.
- `src/HornetStudio.Editor/Widgets/Dialogs/AttachItemsEditorDialogWindow.axaml.cs` -> align attach discovery with canonical helpers plus explicit compatibility reads where needed.
- `src/HornetStudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs` -> inspect and fix remaining mixed-case status/attach path literals if they still generate canonical paths incorrectly.
- Relevant test project(s) -> add or update focused tests for path generation and compatibility behavior.
- Documentation only where current markdown still documents path-producing logic with obsolete PascalCase examples.

## Implementation Order

1. Confirm the canonical rule and boundary in existing helpers and docs.
2. Search the codebase for remaining path-producing PascalCase literals such as `Runtime`, `Status`, `AttachOptions`, `UdlClientruntime`, and similar path fragments.
3. Classify each hit:
   - code/display only
   - canonical path write
   - legacy compatibility read
4. Replace canonical path writes with helper-based `snake_case` generation.
5. Isolate any legacy compatibility handling so it is explicit and read-only.
6. Update tests around canonical path generation and path discovery.
7. Run build and targeted tests.

## Technical Constraints

- Chat answers must remain German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Keep changes minimal and directly related to path-relevant naming.
- Prefer existing helpers and patterns over new abstraction layers.
- Do not introduce new dependencies.
- Do not mechanically rename .NET identifiers to `snake_case`.
- Preserve existing behavior except where canonical path enforcement requires path output changes.

## Acceptance Criteria

- New canonical runtime/studio/attach/broker paths are always written in `snake_case`.
- Remaining PascalCase path strings do not influence canonical path generation anymore.
- Legacy mixed-case paths are only supported where explicitly required for discovery or backward compatibility.
- No code identifiers, namespaces, or public APIs are renamed solely because of the path rule.
- Focused tests cover canonical generation and intended compatibility cases.
- The solution builds successfully.

## Verification

- `dotnet build HornetStudio.sln --no-restore`
- Run targeted tests for item path normalization, editor path helpers, UDL path handling, and broker path generation where available.
- Manually inspect representative generated paths for:
  - UDL runtime/status/attach branches
  - broker status/attach branches
  - host/editor generated target paths

## Out of Scope

- Full migration tooling for legacy persisted documents.
- Renaming historical persisted data already stored with mixed casing unless required for read compatibility.
- General naming cleanup unrelated to filesystem or runtime path behavior.

## Risks / Watchouts

- A broad search-and-replace can break case-sensitive .NET identifiers, XAML type resolution, or namespaces.
- Some PascalCase strings are valid display text and must not be normalized blindly.
- Compatibility reads can create duplicate discovery entries if canonical and legacy branches are both present.
- Some helpers may already normalize input, so additional local transforms could over-normalize or alter intended names.

## Notes

- The architectural rule should be phrased as: canonical path identity uses `snake_case`, while code-level naming follows existing project and .NET conventions.
- The main technical risk is not missing normalization support in the lower layers, but higher-level code bypassing those helpers with hard-coded path strings.

# IMPLEMENTATION HANDOFF

## Goal

Implement strict snake_case paths and strict snake_case path-relevant item property keys across the item model, broker, MQTT, host registry, editor helpers, defaults, tests, and documentation.

## Scope

Included:
- Canonical snake_case path generation and normalization.
- Default item property keys changed to `name`, `path`, `value`, and `meta`.
- Path-relevant metadata property keys changed to snake_case.
- Hard-coded generated paths changed to snake_case.
- Tests and docs updated for the new canonical format.

Not included:
- Migration of existing persisted layouts.
- Compatibility aliases for PascalCase property keys.
- Backward compatibility for old MQTT topics or old registry paths.
- Unrelated refactoring.

## Starting Point

- `src/Amium.Item/Item.cs` currently creates paths from raw names and uses default property keys `Name`, `Path`, `Value`, and `Meta`.
- `src/Amium.Item/ItemPathExtensions.cs` normalizes separators but does not convert segments to snake_case.
- `src/Amium.Item.Server/ItemServerPath.cs` normalizes separators and trims segments but preserves case and segment spelling.
- Host/editor helpers contain root path names like `Studio`, `Runtime`, `Logs`, `Commands`, `ItemBroker`, and widget path fragments in PascalCase.
- Tests and YAML files contain many hard-coded PascalCase paths.
- There is no migration requirement.

## Tasks

1. Add or reuse a central snake_case segment conversion helper in the lowest practical layer.
2. Update `Amium.Item` to generate strict snake_case paths and snake_case default property keys.
3. Update `ItemPathExtensions.Repath` to normalize absolute paths and descendant paths to snake_case.
4. Update all direct accesses to default property keys from PascalCase to snake_case.
5. Update path-relevant metadata property keys to snake_case.
6. Update broker path normalization and broker item path resolution.
7. Update MQTT topic mapping and broker adapter property references.
8. Update host registry, runtime path helpers, and signal path helpers.
9. Update editor target path helpers and widget-generated path strings.
10. Update YAML defaults, templates, docs, and tests.
11. Run focused tests and builds, then fix remaining compile/test failures in scope.

## Implementation Order

1. Implement the canonical path/key normalization in `Amium.Item`.
2. Update all `Amium.Item` tests or add focused tests if missing.
3. Compile to expose direct PascalCase property key usage.
4. Update broker and MQTT layers.
5. Compile again and update host registry/runtime layers.
6. Update editor helpers and widget-generated paths.
7. Update YAML/template/doc examples.
8. Run relevant tests and builds.

## Technical Constraints

- Chat response must remain German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Do not add new dependencies.
- No compatibility aliases.
- No migration logic.
- Keep changes scoped to path and path-relevant property key behavior.
- Preserve existing architecture and project patterns.
- Public changed classes and methods must keep XML documentation accurate.

## Acceptance Criteria

- New `Item("default_layout")` has path `default_layout`.
- Child path generation converts every segment to snake_case.
- Default properties are stored under `name`, `path`, `value`, and `meta`.
- `Properties["Value"]` and similar PascalCase accesses are not supported.
- `Repath("studio.default_layout.UdlClient1.set.request")` stores `studio.default_layout.udl_client1.set.request`.
- Broker paths normalize to snake_case.
- MQTT topics generated from broker paths contain snake_case topic segments.
- Editor target paths and generated widget paths are snake_case.
- Tests pass with snake_case expectations.

## Verification

- Run `dotnet build AutomationExplorer.sln --no-restore`.
- Run `dotnet build Editor\AutomationExplorer.Editor.csproj --no-restore` if this project path remains valid in the current workspace.
- Run available test projects under `src/*Tests`.
- Manually inspect YAML defaults for remaining PascalCase path examples.
- Use `rg` to check remaining path-sensitive strings such as `Properties["Value"]`, `Properties["Path"]`, `studio.`, `runtime.`, `ItemBroker`, `default_layout`, and `set.request`.

## Out of Scope

- Persisted layout migration.
- Legacy path compatibility.
- Runtime conversion of old MQTT retained topics.
- UI redesign or unrelated widget behavior changes.
- New external libraries.

## Risks / Watchouts

- Some PascalCase strings are display text or class names and should not be mechanically changed.
- Some protocol acronyms may need deterministic conversion, for example `MQTT` -> `mqtt`, `UdlClient` -> `udl_client`, and `PublishIntervalMs` -> `publish_interval_ms`.
- Case-insensitive dictionaries may hide missed strictness; tests should verify exact stored keys and exact string values.
- Protected property policies must use the new snake_case keys or write protection can be bypassed.

# IMPLEMENTATION HANDOFF

## Goal
Extend the path identity name lock to folder/page identities and update new project creation so the first folder name is requested and validated instead of silently creating `Folder1`.

## Scope
Included:

- Treat `FolderModel.Name` as a path-relevant identity.
- Validate add-folder / add-page names with the same strict `snake_case` identity rule.
- Generate folder/page defaults as `snake_case`.
- Prompt for the first folder name during new project creation.
- Use the validated first folder identity for directory creation and `Folder.yaml`.
- Preserve legacy project loading.

Explicitly NOT included:

- Automatic migration of existing legacy folders such as `Folder1` or `Page1`.
- Renaming existing project directories during load.
- Forcing display-only captions into `snake_case`.
- Release/version/publish work.

## Starting Point
- `src/HornetStudio/ViewModels/MainWindowViewModel.cs` currently contains `FallbackFolderTemplate` with `Folder1`.
- `CreateNewBook(...)` currently creates a first page directory using `Folder1`.
- `TryCreateNewPage(...)` appears to be the MainWindow add-folder / add-page path.
- `src/HornetStudio/MainWindow.axaml.cs` calls `TryCreateNewPage(pageName, ...)`.
- Folder/page names are used as path roots by editor and runtime path helpers.
- Existing plans already cover widget, item, signal, CustomSignals, and EnhancedSignals path identity validation.

## Tasks
1. Include `FolderModel.Name` in the central strict path identity validation rule.
2. Inspect current new-project UI flow in `MainWindow.axaml.cs`.
3. Add a first-folder-name prompt before calling the new project creation logic.
4. Use a valid default suggestion such as `main` or `page_1`.
5. Validate the prompt result with the central identity validator.
6. Cancel new project creation cleanly when the user cancels the prompt.
7. Update `CreateNewBook(...)` so it receives or otherwise uses the validated first folder name instead of hard-coded `Folder1`.
8. Update `TryCreateNewPage(...)` to reject invalid folder/page names.
9. Update fallback/default folder templates to use canonical `snake_case` values.
10. Add or update focused tests if the ViewModel exposes testable seams.
11. Update documentation examples only where they describe newly generated defaults.
12. Run build and relevant tests.

## File-Level Changes
- `src/HornetStudio/MainWindow.axaml.cs` -> prompt for the first folder name before creating a new project and pass the validated value into the creation flow.
- `src/HornetStudio/ViewModels/MainWindowViewModel.cs` -> remove hard-coded `Folder1` from new project creation, validate `TryCreateNewPage(...)`, and use canonical default/fallback folder identities.
- `src/HornetStudio.Editor/Helpers/TargetPathHelper.cs` -> ensure strict identity validation helper is reusable by folder validation if this helper owns the rule.
- Existing relevant test project(s) -> add tests for folder identity validation and new project creation behavior if practical.
- Documentation examples -> update only where examples describe generated folder/page defaults.

## Implementation Order
1. Implement or reuse the central identity validator from the broader path identity lock work.
2. Locate the exact `MainWindow.axaml.cs` new project command path around `CreateNewBook(...)`.
3. Add the first-folder prompt in the UI layer.
4. Change `CreateNewBook(...)` signature or add an overload to receive the validated first folder name.
5. Replace `Folder1` in new project creation and fallback template with a canonical value.
6. Tighten `TryCreateNewPage(...)` validation for AddFolder/AddPage.
7. Add focused tests where available.
8. Run `dotnet build HornetStudio.sln --no-restore`.

## Technical Constraints
- Chat answers must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation messages must be English.
- Keep changes minimal and directly related to path-relevant folder identity names.
- Prefer existing dialog and validation patterns.
- Do not introduce new dependencies.
- Do not migrate existing project folders automatically.

## Acceptance Criteria
- New project creation asks for the first folder name.
- New project creation does not create `Folder1` by default.
- Invalid first folder names such as `Folder1`, `Page1`, `MainPage`, and `main-page` are rejected.
- Valid first folder names such as `main`, `page_1`, and `production_line_1` are accepted.
- AddFolder/AddPage rejects non-`snake_case` names.
- Created folder directories and `Folder.yaml` use the validated identity name.
- Existing legacy projects still load.
- The solution builds successfully.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Manual verification:
  - Create a new project and confirm a first-folder-name prompt appears.
  - Try `Folder1`; confirm validation blocks it.
  - Try `main`; confirm project creation succeeds.
  - Confirm the created folder directory and `Folder.yaml` use `main`.
  - Add another folder with `Page1`; confirm validation blocks it.
  - Add another folder with `page_2`; confirm creation succeeds.
  - Load an existing legacy project with `Folder1`; confirm it still opens.

## Out of Scope
- Renaming old folder directories.
- Changing display captions beyond what is required by existing model fields.
- Release publishing.
- Broad project format migration.

## Risks / Watchouts
- New project creation may currently create files before the first folder name is known; avoid leaving partial directories if the prompt is cancelled.
- If `FolderModel.Name` doubles as display text, new folders will visibly use `snake_case` unless a separate caption is already available.
- Some docs and YAML definitions still use `Page1`; update only relevant default-generation documentation unless the implementation changes those templates.
- Keep validation messages concise and user-visible text in English.

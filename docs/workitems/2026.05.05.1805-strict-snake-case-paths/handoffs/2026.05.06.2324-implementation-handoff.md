# IMPLEMENTATION HANDOFF

## Goal
Add an editor-side validation lock for all newly created or edited path-relevant identity names so widgets, items, signals, custom signals, and comparable path-producing definitions cannot be saved with PascalCase or other non-`snake_case` names.

## Scope
Included:

- Add or expose a central validator for path-relevant identity name segments.
- Update new widget/item default identities to `snake_case`.
- Block invalid `FolderItemModel.Name` edits for path-relevant controls.
- Block invalid custom signal definition names.
- Ensure CustomSignals registry and trigger path builders emit canonical `snake_case` path segments.
- Review EnhancedSignals for equivalent path-relevant definition names and include the same validation if applicable.
- Add or update focused tests where existing test structure allows it.

Explicitly NOT included:

- Renaming C# types, namespaces, properties, public APIs, or XAML classes.
- Forcing display-only text into `snake_case`.
- Full migration tooling for existing persisted layouts.
- Release/version/publish work.

## Starting Point

- Path normalization helpers already convert many generated paths to `snake_case`.
- The missing layer is input validation: path-relevant `Name` values can still be stored as mixed-case text.
- `MainWindowViewModel.GetSuggestedControlName(...)` still produces names such as `CustomSignals1` and `Signal1`.
- `MainWindowViewModel.TryValidateControlName(...)` currently permits uppercase letters.
- `CustomSignalEditorDialogViewModel.TryBuildDefinition(...)` only checks that `Name` is non-empty.
- `CustomSignalsControl.BuildRegistryPath(...)` uses local `SanitizeSegment(...)`, which trims but does not enforce canonical `snake_case`.
- `CustomSignalsControl.BuildManualTriggerPath(...)` appends `.Trigger`.

## Tasks
1. Create or expose a central helper method for strict path identity names.
2. Define strict identity validation:
   - lowercase `a-z`
   - digits `0-9`
   - underscore `_`
   - no uppercase letters
   - no whitespace
   - no dots, slashes, backslashes, or hyphens
   - no empty value
3. Add a helper for generating canonical numbered defaults from control kinds.
4. Update `GetSuggestedControlName(...)` and related default identity generation to return values such as `signal_1`, `custom_signals_1`, `broker_widget_1`, and `udl_client_control_1`.
5. Update control name validation so invalid `Name` values cannot be saved from the editor dialog.
6. Decide how unchanged legacy names are handled:
   - allow load/display of existing mixed-case names
   - reject saving a changed or newly created invalid name
7. Update CustomSignals signal editor validation to reject invalid path-relevant names.
8. Update CustomSignals path builders to use canonical helper methods and `.trigger`.
9. Inspect EnhancedSignals definition names for path relevance and apply the same rule if they generate registry/runtime paths.
10. Add focused tests for validator behavior and CustomSignals path generation if accessible.
11. Run build and relevant tests.

## File-Level Changes
- `src/HornetStudio.Editor/Helpers/TargetPathHelper.cs` -> add strict identity-name validation and canonical identity-name helpers if no better existing helper exists.
- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs` -> update default name generation and `TryValidateControlName(...)` / `IsValidControlNameFormat(...)`.
- `src/HornetStudio.Editor/Widgets/CustomSignals/CustomSignalEditorDialogWindow.axaml.cs` -> validate custom signal `Name` before building definitions.
- `src/HornetStudio.Editor/Widgets/CustomSignals/CustomSignalsControl.axaml.cs` -> replace `SanitizeSegment(...)` path building with canonical helpers and use lowercase `trigger`.
- `src/HornetStudio.Editor/Widgets/EnhancedSignals/*` -> inspect and update only if definition names are path-relevant and currently unvalidated.
- Existing relevant test project(s) -> add focused tests if helper logic is testable without UI startup.
- Documentation -> update only if docs describe PascalCase identity examples for path-producing names.

## Implementation Order
1. Add central helper methods:
   - `NormalizeIdentitySegment(...)` if generation needs canonical conversion
   - `IsValidIdentitySegment(...)`
   - `TryValidateIdentitySegment(...)` or equivalent
2. Update default name mapping for `ControlKind` to canonical base names.
3. Update uniqueness checks to work with the new generated names.
4. Tighten editor dialog validation for `FolderItemModel.Name`.
5. Tighten CustomSignal definition validation.
6. Normalize CustomSignals generated registry and trigger paths.
7. Review EnhancedSignals and apply the same pattern where needed.
8. Add tests.
9. Build and report remaining warnings/errors.

## Technical Constraints
- Chat answers must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Keep changes minimal and directly related to path-relevant identity names.
- Prefer existing helpers and patterns.
- Do not introduce new dependencies.
- Do not mechanically rename existing C# identifiers or UI display text.
- Do not silently migrate all existing persisted mixed-case layout names unless explicitly required.

## Acceptance Criteria
- New widgets/items/signals receive `snake_case` identity names by default.
- Manual widget/item `Name` edits reject uppercase and other non-`snake_case` characters.
- Custom signal names reject `CustomSignal1` and accept `custom_signal_1`.
- Generated CustomSignals registry paths contain only canonical `snake_case` segments.
- Manual trigger path suffix is `trigger`.
- Display labels remain human-readable where separate display fields exist.
- Existing mixed-case persisted layouts can still load.
- The solution builds successfully.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Run targeted tests for editor path helpers and CustomSignals if available.
- Manual verification:
  - Add a Signal and confirm the technical `Name` is `signal_1` or equivalent `snake_case`.
  - Add a CustomSignals widget and confirm the technical `Name` is `custom_signals_1`.
  - Try to save a widget name `CustomSignals1`; confirm validation blocks it.
  - Try to add a custom signal named `CustomSignal1`; confirm validation blocks it.
  - Add `custom_signal_1`; confirm registry path is `studio.<page>.<widget>.custom_signal_1`.
  - Confirm display text/captions still look intentional.

## Out of Scope
- Automatic migration of all old layout files.
- Rewriting user-facing labels to `snake_case`.
- Release publishing.
- Global API or namespace renames.

## Risks / Watchouts
- Existing mixed-case names may be used as display labels. Ensure defaults set display fields separately where needed.
- Legacy layouts with unchanged mixed-case names need a deliberate load/edit behavior so users are not blocked by merely opening a file.
- Some names, especially formula variables, are not path segments and should not be validated with this rule.
- CustomSignals duplicate checks should compare canonical identity names consistently.

- Treat `Name` as path identity for path-producing models.
- Treat `Title`, `Text`, captions, and labels as display text.
- Prefer blocking invalid new names over silently saving normalized variants, because silent conversion can hide collisions.

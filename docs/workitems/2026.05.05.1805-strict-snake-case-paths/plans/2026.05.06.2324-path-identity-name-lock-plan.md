# Path Identity Name Lock Plan

## Problem

The strict `snake_case` path migration normalized many generated paths, but it does not yet block invalid path-relevant names at creation or edit time. A user can still create or save names like `CustomSignal1`, and those names are path-relevant because widgets, items, signals, and custom signal definitions participate in generated runtime, studio, registry, attach, or target paths.

## Scope

This follow-up closes the missing validation layer for path-relevant identity names.

Included:

- Path-relevant widget and item names.
- Page/folder names where they become path roots.
- Custom signal definition names.
- Enhanced signal definition names if they generate registry/runtime paths.
- Default names created by add flows.
- Editor validation for manual edits.

Not included:

- Renaming C# identifiers, namespaces, class names, or APIs.
- Changing display-only text such as captions, button text, body text, labels, or titles unless they are directly reused as identity names.
- Migration tooling for existing persisted mixed-case layouts.
- Release or publish work.

## Architectural Decision

`Name` must be treated as a technical identity whenever the model participates in generated paths. Human-readable labels should use existing display fields such as `Title`, `Text`, `ControlCaption`, `BodyCaption`, or widget-specific captions.

Canonical path-relevant identity names must use strict `snake_case`:

- lowercase `a-z`
- digits `0-9`
- underscore `_`
- no uppercase letters
- no whitespace
- no dots, slashes, backslashes, or hyphens
- no empty value

Numbered defaults should be generated as `signal_1`, `custom_signals_1`, `broker_widget_1`, and similar names.

## Plan

1. Add or expose a central path identity name validator near existing path helpers.
2. Define one validation API for path-relevant identity segments.
3. Update widget/item default name generation to produce `snake_case` names.
4. Update widget/item edit validation so invalid `Name` values cannot be saved.
5. Update CustomSignals signal editor validation so `CustomSignal1` and similar mixed-case names are rejected.
6. Update CustomSignals path builders to use canonical path helper behavior and lowercase reserved path suffixes such as `trigger`.
7. Review EnhancedSignals for the same issue and apply the same identity-name rule if its definitions generate paths.
8. Keep legacy persisted values readable where possible, but block newly saved invalid identity names.
9. Add focused tests for the validator and path-producing flows where test seams already exist.
10. Run build and targeted tests.

## Acceptance Criteria

- Adding a new widget creates a `snake_case` identity name.
- Adding a new Signal creates a `snake_case` identity name.
- Adding a CustomSignals widget creates a `snake_case` identity name.
- Adding a custom signal rejects `CustomSignal1` and accepts `custom_signal_1`.
- Editing a path-relevant widget or item name rejects uppercase names.
- Generated CustomSignals registry paths use `snake_case` segments only.
- Manual trigger paths use `.trigger`, not `.Trigger`.
- Existing display captions remain human-readable and are not forced into `snake_case`.
- The solution builds successfully.

## Risks

- Existing persisted layouts may contain mixed-case `Name` values. The implementation must avoid breaking load unnecessarily.
- Changing default identity names can affect tests or snapshots that assert persisted layout names.
- Some current UI labels may rely on `Name` because no separate display field was configured. Defaults should set display fields explicitly where needed.
- Validation must not be applied to formula variable names if those variables are only formula tokens and not path segments.

# Folder Identity Name Lock Plan

## Problem

The path identity lock plan must also include folder/page identities. A folder name becomes a path root in generated paths such as `studio.<folder_name>...`, persisted target paths, runtime bindings, registry paths, and layout directories. The current new project flow still creates `Folder1`, and the add-folder flow can create or accept names that are not strict `snake_case`.

This means the path model is still open at the root level even if widget, item, and signal names are fixed.

## Scope

Included:

- New project first-folder creation.
- MainWindow add-folder / add-page flow.
- Folder/page rename validation where supported by the UI.
- Folder/page directory names when they are created from folder identity names.
- Documentation and examples that describe new default folder/page names.

Not included:

- Automatic migration of existing `Folder1`, `Page1`, or other legacy folder directories.
- Renaming existing user projects during load.
- Changing display-only folder captions if a separate caption field exists.
- Release or publish work.

## Architectural Decision

`FolderModel.Name` is path-relevant and must follow the same strict path identity rule as widget, item, signal, custom signal, and enhanced signal identities.

New project creation should not silently create an invalid technical identity such as `Folder1`. It should ask for the first folder name before creating the project structure. The proposed default can be valid, for example `main` or `page_1`.

The project creation flow must only continue when the first folder identity is valid `snake_case`.

## Plan

1. Extend the central path identity validation scope to include `FolderModel.Name`.
2. Update add-folder / add-page validation to require strict `snake_case`.
3. Update folder/page default name generation to use values such as `page_1` or `folder_1`.
4. Update new project creation so it prompts for the initial folder name instead of hard-coding `Folder1`.
5. Pre-fill the prompt with a valid suggestion such as `main` or `page_1`.
6. Block project creation until a valid folder identity has been provided or let cancellation abort the new project flow cleanly.
7. Ensure created folder directories and `Folder.yaml` use the validated identity name.
8. Preserve legacy project loading behavior for existing mixed-case folder names.
9. Update tests or add focused tests around folder identity validation if existing test seams allow it.
10. Update documentation examples where they describe newly generated folder/page names.

## Acceptance Criteria

- Creating a new project no longer silently creates `Folder1`.
- New project creation asks for the first folder name.
- Invalid names such as `Folder1`, `Page1`, `MainPage`, or `main-page` are rejected for the first folder.
- Valid names such as `main`, `page_1`, or `production_line_1` are accepted.
- AddFolder / AddPage rejects non-`snake_case` names.
- Newly created folder directories use the validated `snake_case` folder identity.
- Existing projects with legacy folder names still load.
- The solution builds successfully.

## Risks

- New project creation likely runs through `MainWindow.axaml.cs` plus `MainWindowViewModel.CreateNewBook(...)`; the prompt belongs in the UI layer, not deep file creation logic, unless there is already a reusable dialog pattern.
- Existing tests or docs may assume `Folder1`.
- If folder name and display caption are currently the same field, the UI may show `snake_case`; a separate display caption should be used if a prettier label is required.
- Cancelling the first-folder prompt must leave no partial project structure behind.

# IMPLEMENTATION HANDOFF

## Goal
Restore the `Amium.Items.Item` read-channel model so every item always owns a `read` property, no item uses a `value` property as fallback storage, and optional write capability is controlled only by the existing `hasWriteChannel` constructor parameter.

## Scope
Included:

- Update `Amium.Items.Item` so `read` is always created and used as the item value property.
- Remove `value` fallback behavior from `Item`.
- Remove the `hasReadChannel` constructor parameter from `Item`.
- Keep `hasWriteChannel` behavior: when `true`, create `write`; when `false`, do not create `write`.
- Update UDL module/channel creation so `Alert` has `read` but no `write`.
- Update tests that currently expect `Alert` to have no `read`.
- Update directly related Amium.Item documentation if it mentions `hasReadChannel`, `value` fallback behavior, or `Alert` without `read`.

Explicitly not included:

- New item architecture.
- New transport behavior.
- New MQTT topic mapping rules.
- New dependencies.
- Release versioning or publishing.
- Broad refactoring outside the directly affected files.

## Starting Point
- `amium_item/src/Amium.Item/Item.cs` currently contains `DefaultValuePropertyName = "read"` and `FallbackValuePropertyName = "value"`.
- The main `Item` constructor currently accepts `hasReadChannel`.
- When `hasReadChannel` is `false`, `Item` can avoid creating `read` and can store an initial value under `value`.
- The `Value` property currently reads `read`, then falls back to `value`.
- UDL module creation currently creates `Alert` with `hasReadChannel: false`.
- UDL metadata cleanup currently removes `read` from `Alert` and may copy it to `value`.
- Host tests currently assert that `Alert` does not have `read`.

## Tasks
1. Update `Item` so all constructors create `Properties["read"]`.
2. Remove `FallbackValuePropertyName` and all `value` fallback logic from `Item`.
3. Remove the `hasReadChannel` parameter from the public `Item` constructor.
4. Update `Item.Value` so it reads from `read` and writes to `write` when present, otherwise to `read`.
5. Update XML documentation in `Item.cs` to describe the invariant `read` property and optional `write` property.
6. Update UDL channel creation calls to stop passing `hasReadChannel`.
7. Remove UDL cleanup logic that removes `read` or creates `value`.
8. Ensure `Alert` channels are created without `write` but still have `read`.
9. Update tests to assert `Alert.Properties.Has("read")` and `!Alert.Properties.Has("write")`.
10. Search for remaining references to `hasReadChannel`, `FallbackValuePropertyName`, `Properties["value"]`, and `Remove("read")`.
11. Update relevant documentation if it describes the removed behavior.
12. Build the relevant solutions and run focused tests when feasible.

## File-Level Changes
- `amium_item/src/Amium.Item/Item.cs` -> enforce `read` invariant, remove `value` fallback, remove `hasReadChannel`, keep `hasWriteChannel`.
- `src/Amium.UdlClient/Module.cs` -> create `Alert` without `write`, stop removing `read`, remove `value` fallback handling.
- `src/HornetStudio.Host/UdlModule.cs` -> same UDL channel model correction for host module creation.
- `src/HornetStudio.Editor/Widgets/UdlClient/SimulatedHostUdlClient.cs` -> stop passing `hasReadChannel: false` for simulated `Alert`.
- `src/HornetStudio.Host.Tests/Program.cs` -> update UDL assertions for `Alert`.
- `amium_item/src/Amium.Item/README.md` -> update only if it references optional `read` or `value` fallback behavior.
- `amium_item/docs/*.md` -> update only directly incorrect statements about `Alert` having no `read` or `value` fallback behavior.

## Implementation Order
1. Modify `Item.cs` first to make `read` the only value property.
2. Update all call sites that pass `hasReadChannel`.
3. Remove UDL `EnsureNoReadChannel` behavior and replace it with logic that only ensures no `write` channel exists for `Alert` if needed.
4. Update tests to match the corrected `Alert` model.
5. Update directly related documentation.
6. Run searches for stale symbols and string patterns.
7. Build Amium.Item and HornetStudio.
8. Run focused tests for UDL module shape and item behavior.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- Newly created path-relevant names must use snake_case.
- Do not introduce new dependencies.
- Keep changes minimal and limited to the read/write item model correction.
- Preserve `hasWriteChannel`.
- Do not preserve `hasReadChannel`; it represents an invalid model because `read` is mandatory.
- Public API change is intentional and should be noted in the final summary as a breaking change.

## Acceptance Criteria
- Every newly constructed `Item` has `Properties.Has("read") == true`.
- No newly constructed `Item` uses or creates a `value` property as item value fallback.
- `Item.Value` reads from `read`.
- `Item.Value` writes to `write` when `write` exists.
- `Item.Value` writes to `read` when `write` does not exist.
- `Alert` items have `read`.
- `Alert` items do not have `write`.
- No code references `hasReadChannel`.
- No code references `FallbackValuePropertyName`.
- No UDL code removes `read` from `Alert`.

## Verification
- Run `rg "hasReadChannel|FallbackValuePropertyName|Properties\\[\"value\"\\]|Remove\\(\"read\"\\)" amium_item src`.
- Run `dotnet build amium_item/Amium.Item.sln --no-restore`.
- Run `dotnet build HornetStudio.sln --no-restore`.
- Run relevant host/editor tests that cover UDL module shape and item property behavior.

## Out of Scope
- MQTT protocol redesign.
- Item server behavior redesign.
- NuGet packaging.
- Release version generation.
- Git tags.
- Moving files or changing repository split.
- Cleanup unrelated to `Item` read/write channel behavior.

## Risks / Watchouts
- Removing `hasReadChannel` is a public API breaking change for any callers still passing it.
- Documentation currently contains older statements that `Alert` has no separate `read` parameter; update only directly affected statements to avoid broad documentation churn.
- Existing serialized or brokered data may still contain legacy `value` properties; this task removes item fallback behavior but does not define a migration for persisted legacy data.
- Raw artifact references mean HornetStudio may need rebuilt Amium.Item artifacts before HornetStudio build verification.

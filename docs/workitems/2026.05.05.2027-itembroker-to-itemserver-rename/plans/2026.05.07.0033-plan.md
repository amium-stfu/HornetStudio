# PLAN

## Title
Rename the ItemBroker area to the Item Server structure and remove `Broker` from the active naming model.

## Goal
Replace the current `Amium.Item.Server`-centered structure with a clearer target structure based on:

- `Amium.Item`
- `Amium.Item.Server`
- `Amium.Item.Client`
- `Amium.Item.Client.Mqtt`

The rename includes projects, namespaces, type names, tests, demos, host integration, and relevant documentation.

## Scope
- Rename the current broker runtime project to `Amium.Item.Server`.
- Merge the current service entry-point responsibility into `Amium.Item.Server`.
- Keep server-side transport adapters inside `Amium.Item.Server` rather than creating separate server adapter projects.
- Rename the generic client package to `Amium.Item.Client`.
- Rename the MQTT remote client package to `Amium.Item.Client.Mqtt`.
- Remove active `Broker` naming from affected public and internal types where it refers to this runtime/client model.
- Update solution/project references, tests, demos, and relevant Markdown documentation.

## Out of Scope
- No functional redesign of broker/server behavior.
- No new transport architecture beyond the rename and structural consolidation.
- No additional adapter extraction into separate server projects.
- No unrelated cleanup outside the rename scope.

## Target Structure
- `Amium.Item`
  - Item domain model only
- `Amium.Item.Server`
  - authoritative server/runtime
  - in-memory store, ownership, update/write rules, path logic, policies
  - server-side adapters such as MQTT kept as internal project structure
  - composition root previously living in `Amium.Item.Server`
- `Amium.Item.Client`
  - generic client-facing runtime/session API
- `Amium.Item.Client.Mqtt`
  - MQTT-based remote client access

## Main Decisions
1. `Broker` is removed as the active product/runtime term.
2. The server remains one project even if multiple adapters are added later.
3. The former service project is folded into the server project instead of surviving as `Amium.Item.Server.Service`.
4. Tests, demos, and documentation are part of the same migration so the repository ends in one consistent naming state.

## Execution Strategy
1. Establish the final naming map for projects, assemblies, namespaces, folders, and key public types.
2. Rename and consolidate the core runtime/service projects first so the server target shape becomes stable.
3. Rename client and MQTT client packages against the stabilized server naming.
4. Propagate the rename through dependents such as tests, demos, host/editor integration, and docs.
5. Run repo-wide verification for stale `Broker` references, build breaks, and broken documentation links.

## Risk Areas
- Public API renames may ripple into HornetStudio host/editor integration and tests.
- Merging the service project into the server project can blur composition-root and runtime code if not kept structurally separated.
- Demo and test projects may still embed old project names, namespaces, or strings.
- Existing docs/workitems may contain historical `ItemBroker` references that should remain as history, so documentation updates need judgment rather than blind replacement.

## Verification
- Build the complete solution.
- Run relevant tests for renamed runtime/client areas.
- Search for remaining active `ItemBroker`/`Broker` references in source, project files, and current docs.
- Manually verify that the former service startup path still builds from its new location.

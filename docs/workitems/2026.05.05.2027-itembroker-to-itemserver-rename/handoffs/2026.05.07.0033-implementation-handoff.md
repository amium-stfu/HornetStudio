# IMPLEMENTATION HANDOFF

## Goal
Rename the current `ItemBroker`-based runtime/client package family to an `Item Server` structure, remove `Broker` from the active naming model, and leave the repository in one consistent state across code, tests, demos, and relevant documentation.

## Scope
Included:
- Rename `Amium.Item.Server` to `Amium.Item.Server`
- Merge `Amium.Item.Server` into `Amium.Item.Server`
- Rename `Amium.Item.Client` to `Amium.Item.Client`
- Rename `Amium.Item.Client.Mqtt` to `Amium.Item.Client.Mqtt`
- Keep server-side MQTT adapter code inside `Amium.Item.Server`
- Rename affected public/internal type names that still use `Broker` for this runtime model
- Update references in tests, demos, host/editor integration, solution/project files, and relevant docs

Excluded:
- Functional redesign of runtime behavior
- Splitting server adapters into separate server projects
- Unrelated cleanup or architecture changes outside the rename scope

## Starting Point
- Current projects include:
  - `src/Amium.Item`
  - `src/Amium.Item.Server`
  - `src/Amium.Item.Client`
  - `src/Amium.Item.Server.Mqtt`
  - `src/Amium.Item.Client.Mqtt`
  - `src/Amium.Item.Server`
  - `src/Amium.Item.Server.Tests`
  - demo and host/editor dependents that still reference broker naming
- The current server-side MQTT hosting logic lives outside the core runtime project.
- The chosen target model is one server project with internal adapter structure, not multiple server-side packages.

## Tasks
1. Define the exact rename map for projects, folders, assemblies, namespaces, and key types.
2. Move the runtime project to `Amium.Item.Server` and fold in the current service entry-point responsibilities.
3. Reorganize server internals so adapter/hosting code remains clearly separated inside the single server project.
4. Rename the client project to `Amium.Item.Client`.
5. Rename the MQTT remote client project to `Amium.Item.Client.Mqtt`.
6. Rename `Broker`-based runtime/client type names to `Server`, `ItemClient`, or another neutral target that matches the new structure.
7. Update all project references and dependent code across tests, demos, host, and editor integration.
8. Update relevant Markdown docs so active guidance reflects the new naming.
9. Run build/test/search verification and remove remaining active rename leftovers.

## File-Level Changes
- `src/Amium.Item.Server/*` -> move/rename to `src/Amium.Item.Server/*`; update assembly/root namespace; absorb former service startup code
- `src/Amium.Item.Server/*` -> merge applicable composition-root/startup files into `src/Amium.Item.Server/*`; remove obsolete standalone service project
- `src/Amium.Item.Client/*` -> move/rename to `src/Amium.Item.Client/*`; update client-facing type names and namespaces
- `src/Amium.Item.Client.Mqtt/*` -> move/rename to `src/Amium.Item.Client.Mqtt/*`; update namespaces and MQTT client naming
- `src/Amium.Item.Server.Mqtt/*` -> evaluate whether code belongs inside `Amium.Item.Server`; move or rename accordingly so server-side adapter code no longer presents as its own product package
- `src/Amium.Item.Server.Tests/*` -> rename/update to match the new server naming
- `src/Amium.Item.Server.MqttDemoClient/*` -> update references and visible names
- `src/Amium.Item.Server.MqttDemoWinForms/*` -> update references and visible names
- `src/HornetStudio*/*` and `src/Amium.UdlClient/*` where applicable -> update renamed references/usings/types
- `HornetStudio.sln` -> update project entries and names
- Relevant `docs/*.md` and active workitem docs -> update current guidance where the old name is no longer the intended target

## Implementation Order
1. Freeze the rename matrix before editing:
   - project names
   - folder names
   - assembly names
   - root namespaces
   - renamed public types
2. Rename and stabilize the server side first:
   - `Amium.Item.Server` -> `Amium.Item.Server`
   - fold `Amium.Item.Server` into that project
   - relocate/rename server-side MQTT adapter code into the server project structure if needed
3. Rename client packages after the server naming is stable:
   - `Amium.Item.Client` -> `Amium.Item.Client`
   - `Amium.Item.Client.Mqtt` -> `Amium.Item.Client.Mqtt`
4. Apply the type-level rename pass:
   - contracts
   - implementations
   - options
   - path helpers
   - adapters
   - host/editor-facing abstractions
5. Update tests, demos, and solution/project references.
6. Update relevant documentation and verify no active guidance still points to the obsolete structure.
7. Run build/tests/searches and fix remaining rename fallout.

## Technical Constraints
- Keep the existing runtime behavior intact; this is a structural/naming migration, not a behavior redesign.
- Keep server adapters inside `Amium.Item.Server`; do not create `Amium.Item.Server.Mqtt`.
- Keep the codebase aligned with existing project conventions and minimal-change principles where possible.
- Use English for code, identifiers, comments, XML docs, and UI text.
- Avoid introducing new dependencies.

## Acceptance Criteria
- The repository uses the target package structure:
  - `Amium.Item`
  - `Amium.Item.Server`
  - `Amium.Item.Client`
  - `Amium.Item.Client.Mqtt`
- The former service responsibility is hosted from `Amium.Item.Server` and no standalone `Amium.Item.Server` remains.
- Active runtime/client code no longer exposes `Broker` as the primary naming model.
- Tests, demos, and dependent projects compile against the renamed structure.
- Relevant documentation reflects the new naming model.

## Verification
- Build the complete solution with the standard solution build command.
- Run relevant tests for server/client/runtime areas.
- Search the repository for active `ItemBroker` and `Broker` references and review remaining hits for intentional historical/doc context.
- Verify the former service startup path still exists and builds from its new integrated location.

## Out of Scope
- New transport adapters
- Behavioral redesign of publish/update/write semantics
- Breaking API cleanup beyond what is required by the rename
- Historical documentation cleanup unrelated to active guidance

## Risks / Watchouts
- Blind renaming can produce weak names; review each public type rename for semantic quality.
- The service merge must not leak startup/composition concerns across the whole server runtime.
- Some historical docs/workitems should remain historical artifacts; do not rewrite history indiscriminately.
- Namespace and folder renames may conflict with the repository’s strict path rules; verify final path names carefully.

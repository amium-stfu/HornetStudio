# IMPLEMENTATION HANDOFF

## Goal
Implement Phase 2 for the multimaster mesh demo: read-side mirroring of peer Items into each node's local MQTT Item Server/Broker. After this phase, external tools such as MQTT Explorer connected to any one of the three local brokers should see the complete mesh Item tree, not only that node's locally owned Items.

## Scope
Included:

- Extend the implemented mesh observer demo with optional read mirroring.
- Mirror remote peer Items from observer snapshots into the local node's own broker.
- Keep each node's locally owned Items published by its local publisher session.
- Preserve the three-node mesh topology:
  - `server_a`: local broker on `127.0.0.1:18831`, base topic `hornet_multimaster_mesh_server_a`
  - `server_b`: local broker on `127.0.0.1:18832`, base topic `hornet_multimaster_mesh_server_b`
  - `server_c`: local broker on `127.0.0.1:18833`, base topic `hornet_multimaster_mesh_server_c`
- After startup, every local broker should expose:
  - `nodes.server_a.dyn_1`
  - `nodes.server_a.stat_2`
  - `nodes.server_b.dyn_3`
  - `nodes.server_b.stat_4`
  - `nodes.server_c.dyn_5`
  - `nodes.server_c.stat_6`
- Extend the mesh self-test so it verifies broker-level mirrored visibility, not only observer-level visibility.
- Keep conflict handling simple: last write wins.

Explicitly not included:

- External write-routing for writes made directly against mirrored remote Items through MQTT Explorer.
- Full bidirectional replication semantics.
- Peer discovery or configurable topology.
- Security, TLS, authentication, or multi-machine deployment.
- Complex conflict resolution.

## Starting Point
- Phase 1 mesh observer demo is implemented and appears green.
- Current observed external behavior: MQTT Explorer connected to broker A only sees `server_a` Items, broker B only sees `server_b` Items, and broker C only sees `server_c` Items.
- The desired Phase 2 behavior is that MQTT Explorer connected to any broker sees all three `nodes.server_*` branches.
- The current observer clients must remain read-only for observation. Mirrored writes into the local broker should use normal `MqttItemClientSession` methods, not `MqttRemoteItemClient` write methods.

## Tasks
1. Inspect the implemented mesh controller files and identify the current peer observation lifecycle.
2. Add a read-mirroring mechanism per mesh node:
   - Subscribe to peer observer change events or poll peer snapshots at controlled points.
   - For each remote Item snapshot, publish it into the local broker through a local mirror publisher session.
   - Use retained publishes so MQTT Explorer can inspect the mirrored tree.
3. Add metadata or local tracking to distinguish local-owned Items from mirrored Items.
4. Prevent echo loops:
   - Do not mirror Items that originate from the local node back into the local broker through peer snapshots.
   - Do not mirror already mirrored copies as if they were newly owned peer Items.
   - Prefer source-node/path tracking over string heuristics when available.
5. Implement last-write-wins behavior for mirrored updates:
   - Later value/property observations overwrite earlier mirrored values.
   - Avoid stale snapshot overwrites if the current APIs expose timestamps that can be compared.
   - If reliable timestamps are not available, document that arrival order is the first implementation rule.
6. Extend or add mesh self-test phases:
   - broker_mirror_baseline: connect/read each local broker and verify all six initial Items are present.
   - broker_mirror_cross_write: after cross-write, verify each broker exposes the final expected value.
   - broker_mirror_runtime_create: after runtime creation, verify every broker exposes all runtime-created Items.
7. Write self-test output to the existing mesh log directory or clearly mark Phase 2 checks inside the summary.
8. Add UI indication only if it stays small:
   - show whether read mirroring is enabled.
   - optionally show local-owned vs mirrored counts per node.
9. Keep the existing single-broker `--self-test` and Phase 1 mesh `--mesh-self-test` behavior available.

   - Prefer per-node mirroring if each node already owns local publisher/peer observer state.

4. On peer observer updates, enumerate visible remote snapshots and publish mirrored snapshots or value/property updates into the local broker.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Use existing MQTT client/server APIs and project patterns. Do not add dependencies.
- Observer clients remain read-only.
- Mirrored broker writes must go through `MqttItemClientSession` methods:
  - `PublishSnapshotAsync`
  - `UpdateValueAsync`
  - `UpdatePropertyAsync`
- Keep Phase 2 focused on read mirroring only.
- Do not implement external writes to mirrored Items in this phase.
- Last write wins for conflicts.
- Preserve the current working Phase 1 behavior and self-tests.
- Avoid broad refactoring of the existing demo UI.

## Acceptance Criteria
- Mesh demo starts three local brokers on separate ports.
- MQTT Explorer connected to any of the three brokers can see all three server branches under that broker's base topic.
- Each broker exposes the six initial Items after startup:
  - A's local Items
  - B's mirrored/local Items
  - C's mirrored/local Items
- Cross-write results are mirrored so every broker exposes the same final value for the tested path.
- Runtime-created Items are mirrored so every broker exposes every new runtime Item.
- Self-test validates broker-level mirrored visibility and passes.
- Existing observer-level mesh self-test behavior remains green.
- Existing single-broker self-test remains green.
- Observer clients are not used for writes.

## Out of Scope
- Writing with MQTT Explorer to a mirrored remote path and routing that write back to the owner broker.
- Multi-hop replication guarantees.
- Permanent persisted replication state.
- Peer discovery.
- Distributed locking, vector clocks, or consensus.
- Multi-machine network tests.

## Risks / Watchouts
- Echo loops are the main risk. A mirrored Item on broker A must not be observed by B and then treated as B-owned data.
- Retained messages can make stale mirrored Items visible after a restart. Use run-specific values in self-tests and consider clearing/removing retained mirror paths if supported.
- Publishing full snapshots on every observer change may create excessive duplicate messages. Add simple change detection if the current mesh logs show repeated growth.
- Dynamic Items update frequently; mirror logic must not cause unbounded publish storms.
- If `MqttRemoteItemClient.GetRemoteItemSnapshots()` hides locally published paths, make sure mirror publisher client ids do not mutate peer observer clients.
- External tools can write to mirrored paths after Phase 2, but routing those writes is intentionally not supported yet. Document this limitation if needed.

# IMPLEMENTATION HANDOFF

## Goal
Reorganize all `amium_item` `.cs` files into projects and folders that match their responsibility. Keep `Amium.Item` as the pure item model/core library, move shared server protocol contracts out of the core library, and keep concrete server/client/MQTT implementation code out of unrelated assemblies.

Target production project shape:

- `Amium.Item`
- `Amium.Item.Server.Abstractions`
- `Amium.Item.Mqtt`
- `Amium.Item.Server`
- `Amium.Item.Client`
- `Amium.Item.Mesh`

## Scope
Included:

- Move server protocol contracts and messages out of `Amium.Item`.
- Keep `Amium.Item` focused on item model, item properties, item serialization/cloning, and general `ItemPath` rules.
- Introduce `Amium.Item.Server.Abstractions` for contracts and message types used by both server and client.
- Introduce `Amium.Item.Mqtt` for MQTT topic mapping shared by server and client.
- Keep concrete in-memory server, retention policies, health publisher, service host, and MQTT server adapter in `Amium.Item.Server`.
- Keep concrete client sessions, remote client, and remote registry in `Amium.Item.Client`.
- Keep mesh/multimaster demo orchestration in `Amium.Item.Mesh`.
- Update project references and namespaces to avoid client depending on concrete server implementation.
- Update tests and documentation affected by the project split.

Explicitly NOT included:

- Runtime behavior changes beyond what is required by moves and dependency cleanup.
- Path validation redesign beyond existing strict path work.
- MQTT topic model redesign.
- Mesh protocol redesign.
- Release versioning, publishing, Git tags, or NuGet packaging.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- User-visible validation and error messages must be English.
- Do not add external runtime dependencies.
- Keep changes structural and behavior-preserving.
- Do not move concrete server implementation into abstractions.
- Do not make `Amium.Item` depend on server, client, mesh, MQTTnet, or abstractions.
- Avoid dependency cycles.
- Preserve public API names where practical, but moving assemblies/namespaces is acceptable for this structural cleanup.
- Public classes and public methods that are changed or added must keep XML documentation.

## Acceptance Criteria
- `Amium.Item` contains only core item model/path/serialization logic.
- `Amium.Item` has no `namespace Amium.Item.Server` files.
- `Contracts.cs`, `Messages.cs`, shared subscription options, shared server path helpers, and shared health path contracts are no longer compiled into `Amium.Item`.
- `Amium.Item.Client` does not reference `Amium.Item.Server`.
- `Amium.Item.Client` can use server messages/contracts through `Amium.Item.Server.Abstractions`.
- Shared MQTT topic mapping is not inside `Amium.Item`.
- Concrete server code remains in `Amium.Item.Server`.
- Concrete client code remains in `Amium.Item.Client`.
- Project graph has no cycles.
- Tests compile and relevant behavior remains unchanged.

## Out of Scope
- Strict path validation implementation details.
- Removing or redesigning `Repath`.
- Changing MQTT topic semantics.
- Changing message schemas.
- Changing mesh behavior.
- Creating a release.
- Updating version numbers.
- Publishing packages.

## Risks / Watchouts
- Moving `Messages.cs` directly into `Amium.Item.Server` would force clients to depend on the concrete server assembly; avoid that.
- `ItemServerValueCoercion` is currently used by both server store and client remote registry; either make it a neutral shared helper or duplicate only if responsibilities are truly different.
- `Health.cs` is referenced by MQTT topic mapping and tests, so it likely belongs in abstractions rather than concrete server.
- `SubscriptionPolicies.cs` may mix shared options and concrete retention decisions; split by responsibility instead of moving the whole file blindly.
- Namespace changes can create widespread compile errors; keep moves grouped by responsibility and build incrementally.
- Do not repeat previous broad casing mistakes: C# namespaces/types stay PascalCase even though item path values are `snake_case`.

# IMPLEMENTATION HANDOFF

## Goal
Reorganize the Amium.Item project structure into a flatter, clearer architecture while keeping MQTT as the built-in transport for the current server and client libraries.

The target structure is:

- `Amium.Item`
- `Amium.Item.Server`
- `Amium.Item.Client`
- `Amium.Item.Mesh`

`Amium.Item.Server` and `Amium.Item.Client` may contain internal `Mqtt/` areas because MQTT is the central transport for the foreseeable future. Mesh and multimaster mirroring must be isolated from the normal server and client libraries.

## Scope
Included:

- Keep all changes inside the existing `HornetStudio.sln`.
- Preserve the existing `Amium.Item` core/domain project.
- Keep MQTT server integration inside `Amium.Item.Server` under a clear `Mqtt/` namespace/folder.
- Move or merge MQTT client functionality into `Amium.Item.Client` under a clear `Mqtt/` namespace/folder.
- Introduce or prepare `Amium.Item.Mesh` as the place for multimaster mesh/mirroring logic.
- Remove the incorrect client-to-server project dependency where possible.
- Update project references, namespaces, tests, and demo references needed for the new structure.
- Keep behavior equivalent unless a dependency boundary forces a small API adjustment.

Explicitly not included:

- Creating a separate `Amium.Item.sln`.
- Creating `Amium.Item.Mqtt.Server` or `Amium.Item.Mqtt.Client` projects.
- Making MQTT transport-neutral.
- Adding new runtime dependencies.
- Adding ESP32-specific implementation code.
- Redesigning the protocol, topic model, or conflict-resolution semantics.

## Starting Point
- `src/Amium.Item/Amium.Item.csproj` exists as the core/domain project.
- `src/Amium.Item.Server/Amium.Item.Server.csproj` exists and currently references `MQTTnet.Server`.
- `src/Amium.Item.Server/Mqtt/` already contains MQTT server-related types such as topic mapping and server adapter logic.
- `src/Amium.Item.Client/Amium.Item.Client.csproj` exists.
- `src/Amium.Item.Client.Mqtt/Amium.Item.Client.Mqtt.csproj` exists and currently references both `Amium.Item.Client` and `Amium.Item.Server`.
- `src/amium_item_server_multimaster_demo/` contains demo-only multimaster and mesh controller code.
- Existing mesh/multimaster workitems are:
  - `docs/workitems/2026.05.08.0816-multimaster-mesh-observer-demo/`
  - `docs/workitems/2026.05.08.0854-multimaster-mesh-read-mirroring/`

Required assumptions:

- MQTT remains the primary transport and is intentionally built into the server/client libraries for now.
- `Amium.Item.Mesh` is intended for PC/gateway-class nodes, not constrained ESP32 devices.
- The first restructuring step remains inside the existing repository and solution.

## Tasks
1. Inspect all project references among `Amium.Item`, `Amium.Item.Server`, `Amium.Item.Client`, `Amium.Item.Client.Mqtt`, demos, host projects, monitor projects, and tests.
2. Define the concrete target project graph and verify there are no dependency cycles.
3. Move MQTT client source files from `Amium.Item.Client.Mqtt` into `Amium.Item.Client/Mqtt/`.
4. Adjust namespaces from `Amium.Item.Client.Mqtt` to the selected namespace pattern under `Amium.Item.Client.Mqtt` or `Amium.Item.Client`.
5. Remove the `Amium.Item.Client.Mqtt` project from the solution after all references are updated, unless a temporary compatibility step is required.
6. Remove the `Amium.Item.Client.Mqtt` project reference from all consuming projects and replace it with `Amium.Item.Client`.
7. Remove the `Amium.Item.Server` project reference from client code by moving shared non-server types into `Amium.Item` or by changing client code to avoid server-only types.
8. Keep existing MQTT server source inside `Amium.Item.Server/Mqtt/` and verify it remains cohesive.
9. Create or prepare `Amium.Item.Mesh` only for mesh/multimaster library code that should not remain in demo projects or the normal server/client.
10. Move reusable mesh/multimaster orchestration code out of demo projects only when it is clearly production library logic.
11. Leave UI/demo-only code in demo projects.
12. Update tests or add focused tests for moved client/server boundary logic.
13. Update documentation to describe the flat Amium.Item architecture and dependency rules.
14. Build the complete solution without starting applications.

## File-Level Changes
- `src/Amium.Item/Amium.Item.csproj` -> keep as core/domain library; add shared protocol-neutral item types only if needed to break client-to-server dependency.
- `src/Amium.Item/` -> receive shared non-server utilities only when they are required by both client and server and are not MQTT-specific.
- `src/Amium.Item.Server/Amium.Item.Server.csproj` -> keep `MQTTnet.Server`; keep dependency on `Amium.Item`; do not depend on `Amium.Item.Client` or `Amium.Item.Mesh`.
- `src/Amium.Item.Server/Mqtt/` -> keep MQTT server host/adapter/topic mapping functionality.
- `src/Amium.Item.Client/Amium.Item.Client.csproj` -> add required MQTT client package references and remove reliance on `Amium.Item.Client.Mqtt`.
- `src/Amium.Item.Client/Mqtt/` -> receive MQTT client session, options, remote client, and remote registry functionality.
- `src/Amium.Item.Client.Mqtt/` -> remove after migration, or leave temporarily only if the implementation explicitly documents why compatibility is needed.
- `src/Amium.Item.Mesh/Amium.Item.Mesh.csproj` -> create only if production mesh logic is moved out of the demo project in this step.
- `src/Amium.Item.Mesh/` -> contain multimaster mesh orchestration, peer mirroring, and read-replication logic that is not demo UI logic.
- `src/amium_item_server_multimaster_demo/` -> update references to use `Amium.Item.Client`, `Amium.Item.Server`, and optionally `Amium.Item.Mesh`; keep WinForms/demo presentation code here.
- `src/Amium.Item.Server.Tests/` -> update namespaces and references; add tests if moved shared logic changes behavior.
- `HornetStudio.sln` -> add/remove projects and references to match the target structure.
- `CHANGELOG.md` -> update only if the implementation changes public package/project structure or observable behavior.
- Relevant Markdown docs -> update architecture notes if such documentation exists or if new project boundaries need to be recorded.

## Implementation Order
1. Create an inventory of project references and source-level dependencies.
2. Decide the minimal shared types that must move to `Amium.Item` to remove the client dependency on `Amium.Item.Server`.
3. Move shared non-server types first and update server/client usages.
4. Move MQTT client files into `Amium.Item.Client/Mqtt/`.
5. Update `Amium.Item.Client.csproj` package/project references.
6. Update all consumers from `Amium.Item.Client.Mqtt` to `Amium.Item.Client`.
7. Remove `Amium.Item.Client.Mqtt` from the solution if no compatibility layer is required.
8. Keep MQTT server code in `Amium.Item.Server/Mqtt/` and verify no mesh logic remains there.
9. If enough reusable mesh logic exists, create `Amium.Item.Mesh` and move only non-UI production mesh code into it.
10. Update demos, tests, and host projects.
11. Update documentation.
12. Run solution build and relevant tests.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- Newly created path-relevant names must use `snake_case`, except existing project names and existing paths preserved by convention.
- Prefer existing project patterns and libraries.
- Do not add new dependencies unless technically required and justified.
- Keep MQTT as the built-in transport for server/client in this step.
- Do not introduce separate `Amium.Item.Mqtt.*` projects.
- Do not create a separate `Amium.Item.sln` in this step.
- Avoid broad refactoring unrelated to the dependency restructuring.
- Public classes and public methods changed or added during the implementation must have XML documentation.
- Update or create focused tests for moved or changed logic.

## Acceptance Criteria
- `Amium.Item.Client` no longer depends on `Amium.Item.Server`.
- MQTT client functionality is available from `Amium.Item.Client` without requiring `Amium.Item.Client.Mqtt`.
- MQTT server functionality remains available from `Amium.Item.Server`.
- Mesh/multimaster orchestration is not embedded in the normal server/client runtime path.
- Demo projects still compile against the new project structure.
- The solution has no project-reference cycles.
- No new `Amium.Item.Mqtt.Server` or `Amium.Item.Mqtt.Client` project is introduced.
- The full solution builds without starting any application.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore`.
- Run existing relevant tests, especially `Amium.Item.Server.Tests` if they are affected.
- Verify `HornetStudio.sln` no longer references removed projects after migration.
- Verify no source file in `Amium.Item.Client` has a `using Amium.Item.Server` directive unless a deliberate temporary compatibility exception is documented.
- Verify mesh-specific source is not part of the normal `Amium.Item.Server` server path.

## Out of Scope
- Publishing a release.
- Changing version numbers.
- Creating Git tags.
- Creating a separate repository.
- Creating a separate `Amium.Item.sln`.
- Implementing ESP32 firmware or embedded client code.
- Designing a transport-neutral plugin architecture.
- Reworking MQTT topic semantics beyond what is necessary for the move.

## Risks / Watchouts
- `Amium.Item.Client.Mqtt` currently depends on `Amium.Item.Server`; moving shared helpers incorrectly could pull server concerns into `Amium.Item`.
- Topic mapping may be shared by client and server. If it is not truly server-specific, consider moving only the minimal shared mapping/value logic to `Amium.Item` or a clearly named shared area inside `Amium.Item`.
- Removing `Amium.Item.Client.Mqtt` may break downstream project references; update all solution consumers in one pass.
- Moving mesh code too early may turn demo-specific assumptions into production APIs. Only move reusable non-UI orchestration.
- Avoid turning `Amium.Item.Mesh` into a copy of the server. Mesh should compose server/client behavior, not duplicate broker implementation.
- Keep the first step structural and behavior-preserving.

# IMPLEMENTATION HANDOFF

## Goal
Move core Item Server health generation out of the MQTT-specific host path and into server-owned runtime logic so every in-process server runtime can expose consistent health data. Keep MQTT-specific health separate from core server health.

## Scope
- Create or introduce a server-core health publisher owned by the Item Server runtime.
- Ensure core server health is available when the server is used by `Amium.Item.Server.Host`, `Item.Server.Monitor`, or future hosts.
- Keep MQTT adapter-specific values separate from core server values.
- Preserve existing public health paths where practical to avoid unnecessary breakage.
- Update tests for the new health ownership.
- Do not change MQTT inbound data/write classification in this workitem.
- Do not change monitor adapter UI behavior except where required to consume the corrected health source.

## Starting Point
- `src/Amium.Item.Server/Health.cs` defines `ItemServerHealthPaths` and `ItemServerHealthSnapshot`.
- `src/Amium.Item.Server/Mqtt/MqttItemServerHost.cs` currently publishes both core server health and MQTT-specific health.
- `src/Amium.Item.Server/Mqtt/MqttItemServerOptions.cs` currently contains `PublishHealth` and `HealthPublishInterval`, even though core health is not MQTT-specific.
- `src/Amium.Item.Server/InMemoryItemServer.cs` already exposes `RetainedItemCount`.
- `src/Item.Server.Monitor/Hosting/MonitorHost.cs` creates an `InMemoryItemServer` directly and does not use `MqttItemServerHost`.
- `src/Item.Server.Monitor/Hosting/MqttMonitorAdapterFactory.cs` currently sets `PublishHealth = false`.
- Existing tests assert self-hosted MQTT health behavior.

## Tasks
1. Inspect `MqttItemServerHost` health publishing and separate core server health values from MQTT-specific values.
2. Add a server-owned health publisher or runtime component in `Amium.Item.Server`.
3. Ensure the component publishes retained snapshots for core health paths before periodic updates start.
4. Publish periodic core health updates independently from MQTT adapter lifecycle.
5. Keep MQTT-specific health publishing in MQTT-owned code or a small adapter status publisher.
6. Update `MqttItemServerHost` to use the server-core health publisher instead of owning core health itself.
7. Update `Item.Server.Monitor` hosting so the local monitor server starts the same core health publisher.
8. Keep or deprecate `MqttItemServerOptions.PublishHealth` carefully; avoid breaking public APIs unless necessary.
9. Update tests to verify core health works without requiring the MQTT host facade.
10. Update existing MQTT health tests so they still verify MQTT-specific health where appropriate.
11. Update documentation or TODOs if any behavior or ownership changes are user-facing.
12. Run build and relevant tests.

## File-Level Changes
- `src/Amium.Item.Server/Health.cs` -> adjust health model if needed; keep XML documentation current.
- `src/Amium.Item.Server/ItemServerHealthPublisher.cs` -> new server-core component that publishes core health snapshots and periodic updates.
- `src/Amium.Item.Server/ItemServerHealthOptions.cs` -> optional options for enabled state and publish interval if needed.
- `src/Amium.Item.Server/InMemoryItemServer.cs` -> expose any required state for health metrics without coupling to MQTT.
- `src/Amium.Item.Server/Mqtt/MqttItemServerHost.cs` -> remove duplicated core health publishing and delegate to server-core health component.
- `src/Amium.Item.Server/Mqtt/MqttItemServerOptions.cs` -> preserve or clarify MQTT-specific option behavior.
- `src/Item.Server.Monitor/Hosting/MonitorHost.cs` -> start/dispose the server-core health publisher for the monitor-owned server.
- `src/Item.Server.Monitor/Hosting/MonitorRegistryBridge.cs` -> should continue mirroring `ItemServerHealthPaths.Root`.
- `src/Amium.Item.Server.Tests/Program.cs` -> add/update tests for core health and MQTT-specific health.
- Relevant Markdown docs or changelog -> update only if behavior or public usage changes.

## Implementation Order
1. Read the full `MqttItemServerHost` health implementation and list which paths are core server, process, and MQTT-specific.
2. Define the server-core health publisher API with minimal surface:
   - accepts an `IItemServer`
   - accepts a way to read retained item count when available
   - supports `StartAsync`, `StopAsync`, and async disposal
3. Move core paths to the new component:
   - `sys.status.state`
   - `sys.status.uptime_seconds`
   - `sys.status.started_at_utc`
   - `sys.status.last_updated_utc`
   - `sys.metrics.item_count`
   - process metrics if they remain intentionally server-runtime metrics
4. Leave MQTT-specific paths in MQTT code:
   - `sys.mqtt.status.state`
   - `sys.mqtt.status.client_count`
   - `sys.mqtt.status.endpoint`
   - `sys.mqtt.status.last_error`
5. Update `MqttItemServerHost` to compose the core health publisher and publish only MQTT-specific health itself.
6. Update `MonitorHost` to compose and start the core health publisher even when no adapter is running.
7. Add tests for health publishing on plain `InMemoryItemServer` plus core health publisher.
8. Update existing MQTT host tests to match the new ownership without reducing coverage.
9. Run verification.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, identifiers, and user-visible messages must be English.
- Use existing project patterns and avoid new external dependencies.
- Keep public API changes minimal.
- Do not implement MQTT inbound data behavior changes here.
- Do not introduce a broad hosting framework or dependency injection layer.
- Use async/await correctly and dispose timers/subscriptions cleanly.
- Health snapshots should be retained so late subscribers can see current health.
- Avoid silent catch blocks; health failures should surface with context or be represented in health state.

## Acceptance Criteria
- Core server health is generated without requiring `MqttItemServerHost`.
- `Item.Server.Monitor` shows server health even before any adapter is started.
- `Amium.Item.Server.Host` still exposes health through its MQTT path as before.
- MQTT-specific health remains available when MQTT is running.
- Core health and MQTT-specific health ownership are separated in code.
- Existing tests are updated or new tests are added for the new health ownership.
- The full solution builds successfully.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Run relevant `Amium.Item.Server.Tests` verification if available.
- Do not start UI, demo, service, or worker applications during normal build verification.

## Out of Scope
- MQTT inbound external data classification.
- Monitor tree/source redesign.
- Adapter catalog redesign.
- Remote monitor mode.
- `config.yaml` startup configuration.
- Release/publish changes.
- Renaming existing health paths unless required.

## Risks / Watchouts
- Existing MQTT health topics are mapped through `MqttItemTopicMapper`; preserving existing paths avoids breaking retained-topic clients.
- `MqttItemServerOptions.PublishHealth` currently controls mixed health. Changing its meaning can be a breaking behavior change; prefer preserving it or introducing a clearer server health option with compatibility.
- Process metrics may be host/process metrics rather than pure server metrics. Decide explicitly whether to keep them under server health for compatibility or move them later in a separate workitem.
- `MonitorRegistryBridge` currently mirrors `sys` separately; this should work once the monitor-owned server publishes core health.
- Avoid starting duplicate health publishers for the same broker instance.

# IMPLEMENTATION HANDOFF

## Goal
Fix the Item.Server.Monitor tree hover behavior so pointer hover no longer jumps or flickers on periodic refresh while selection remains stable.

## Scope
Included:
- Diagnose the remaining root cause of the hover jump in the monitor tree.
- Implement a minimal fix in Item.Server.Monitor only.
- Validate with a focused monitor project build and manual runtime verification.

Not included:
- Changes to Amium.Item.Server or the server health publisher work.
- General monitor UI redesign.
- Broad refactoring outside the monitor tree refresh and row rendering path.

## Starting Point
- Current behavior: the monitor updates successfully and selection can remain stable, but mouse hover on the tree jumps during refreshes.
- A first selection-preservation fix was already implemented in the main window view model.
- A later attempt changed visible row synchronization to reuse row view models by path instead of rebuilding the entire collection.
- Another attempt moved row hover styling into the row template and disabled list virtualization with a StackPanel items panel.
- Focused builds of Item.Server.Monitor succeeded after these changes, so the current problem is behavioral rather than a compile failure.
- The problem is most likely in the tree row rendering / pointer handling path in the monitor UI.

## Tasks
1. Reproduce and isolate the remaining hover jump in the monitor tree during periodic refresh.
2. Verify whether the jump is caused by row/container recreation, child control pointer handling, layout invalidation, or selection/scroll side effects.
3. Apply the smallest fix that stabilizes hover without regressing expand/collapse, selection, or refresh behavior.
4. Validate with a focused monitor build and manual UI verification while updates are active.

## File-Level Changes
- src/Item.Server.Monitor/ViewModels/MainWindowViewModel.cs -> inspect and adjust refresh synchronization logic only if still relevant to hover behavior.
- src/Item.Server.Monitor/ViewModels/MonitorTreeRowViewModel.cs -> keep or refine mutable row state only if required by the final fix.
- src/Item.Server.Monitor/Monitoring/MonitorTreeStore.cs -> inspect row ordering / expansion behavior if row identity or order still changes unexpectedly.
- src/Item.Server.Monitor/MainWindow.axaml -> likely primary location for the final fix in row template, pointer behavior, or list configuration.
- src/Item.Server.Monitor/App.axaml -> inspect hover-related styles for expand-toggle and other child controls.

## Implementation Order
1. Reproduce the issue with the current branch and confirm whether it still occurs after the latest row reuse and non-virtualized items panel changes.
2. Inspect pointer-related behavior in the tree row template and child controls, especially the expand button and row hit-test surface.
3. If needed, add a narrow runtime diagnostic or visual simplification to identify whether the hover jump follows row containers, child controls, or data updates.
4. Apply the minimal UI-side fix first; only revisit view-model refresh logic if the UI fix does not fully solve the issue.
5. Rebuild Item.Server.Monitor with a focused command and verify manually in the running app.

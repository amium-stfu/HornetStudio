# IMPLEMENTATION HANDOFF

## Goal
Rework the server direction from a central item state broker to a lean MQTT item host. MQTT retained topics are the primary last-known-state mechanism, and clients reconstruct item state from MQTT. The server should stay small, stable, and fast instead of maintaining a full duplicate item registry unless a concrete future requirement proves it is needed.

## Scope
Included:
- Treat MQTT as the fixed transport for the current product direction.
- Treat MQTT retained messages as the source of last-known item state.
- Keep item intelligence and remote item reconstruction in clients.
- Reduce `Amium.Item.Server` responsibility toward MQTT hosting, topic/payload forwarding, and minimal technical health.
- Remove or de-emphasize full server-side retained item state where it is only duplicating MQTT retained state.
- Remove server-side write policy, `writable`, `readonly`, `write_path`, Mesh logic, and domain intelligence from the server.
- Allow breaking changes in server APIs, tests, samples, and documentation when needed to simplify the server.
- Keep `Amium.Item.Mqtt` only if it remains useful as a neutral shared MQTT topic/payload protocol layer.

Not included:
- Do not add another transport.
- Do not add generic future transport abstractions.
- Do not design authorization, permissions, validation, conflict resolution, or Mesh policy.
- Do not implement a production Mesh protocol.
- Do not optimize for hypothetical non-MQTT deployments.

## Starting Point
- `MqttItemServerHost` currently owns or accepts an `IItemServer` and creates an `InMemoryItemServer` by default.
- `InMemoryItemServer` and `InMemoryItemServerStore` maintain retained item snapshots and subscription fanout.
- `MqttItemServerAdapter` bridges MQTT to `IItemServer` and subscribes back to retained server state.
- `ItemServerHealthPublisher` currently publishes through `IItemServer` and derives retained item count from `InMemoryItemServer`.
- Clients already subscribe to MQTT item topics and reconstruct remote item state through `MqttRemoteItemRegistry`.
- MQTT publish methods already use retained messages for item snapshots and value/property updates.

## Tasks
1. Reconfirm actual production usage:
   - Audit references to `IItemServer`, `InMemoryItemServer`, `InMemoryItemServerStore`, `MqttItemServerHost`, `MqttItemServerAdapter`, `ItemServerHealthPublisher`, `SubscribeAsync`, `PublishSnapshotAsync`, `UpdateValueAsync`, and `UpdatePropertyAsync`.
   - Separate production usage from tests, demos, tools, and old workitem docs.
2. Define the new server role:
   - Server hosts MQTT item communication.
   - Server may publish minimal technical health/status.
   - Server does not build a complete item tree/state registry for normal item traffic.
   - Server does not provide the authoritative current item snapshot; clients reconstruct that from MQTT retained topics.
3. Simplify `MqttItemServerHost`:
   - Prefer hosting the MQTT server directly.
   - Remove default dependency on `InMemoryItemServer` if it is only used to mirror retained MQTT state.
   - Keep lifecycle, endpoint options, connected-client tracking, and health publishing.
4. Simplify or replace `MqttItemServerAdapter`:
   - Remove the roundtrip model `MQTT publish -> IItemServer update -> server subscription -> MQTT publish` where it duplicates the broker's retained behavior.
   - Keep only needed MQTT interception/filtering if required by MQTTnet hosting.
   - Keep topic visibility/client metrics only if still used for health or monitoring.
   - Do not rebuild item state from every incoming MQTT item publish.
5. Rework health publishing:
   - Keep minimal technical health topics as retained MQTT messages.
   - Publish health directly to MQTT if the state server layer is removed from the host path.
   - Remove or replace retained item count if it depends on `InMemoryItemServer`; use visible MQTT topic count only if cheap and already tracked, otherwise drop the metric or set a clearly technical value.
6. Decide fate of `IItemServer` and in-memory state types:
   - If no production code needs them after host simplification, remove them or move them out of the primary server path.
   - If tests/tools still need an in-process item broker, keep it as a secondary compatibility/test utility and document that it is not the main MQTT server architecture.
   - Avoid keeping it wired into the default MQTT host unless required.
7. Decide fate of `Amium.Item.Mqtt`:
   - Keep it if both client and server still share topic/payload mapping.
   - If kept, make it neutral: namespace should match `Amium.Item.Mqtt`, and server-health-specific mapping should be reviewed.
   - Do not keep it as an accidental server namespace package.
8. Update tests:
   - Remove tests that assert server-retained item state as the primary last-known-state source.
   - Keep or add tests that assert MQTT retained topics contain the latest item payloads.
   - Keep tests for client reconstruction from retained MQTT messages.
   - Keep tests for MQTT topic mapping.
   - Adjust health tests to retained MQTT health topics rather than `IItemServer` retained snapshots if applicable.
9. Update docs:
   - State that MQTT retained is the current last-known-state source.
   - State that clients reconstruct item state.
   - State that the server is a lean MQTT item host, not a central policy/state engine.
   - Document that richer server-side state can be added later only if a concrete need appears.
10. Build and verify.

## Implementation Order
1. Run a fresh reference audit and classify each use as production, Mesh demo, test, tool, or docs.
2. Update architecture docs first or alongside code to make the target role unambiguous.
3. Simplify `MqttItemServerHost` around direct MQTT hosting and minimal health.
4. Simplify or remove `MqttItemServerAdapter` state-broker bridging.
5. Rework health publishing to retained MQTT output without requiring `InMemoryItemServer`.
6. Decide whether `IItemServer` / `InMemoryItemServer` stay as compatibility/test utilities or are removed.
7. Neutralize `Amium.Item.Mqtt` if it remains shared.
8. Update tests and samples.
9. Build `amium_item` and full `HornetStudio`.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and identifiers must be English.
- Prefer existing project patterns and dependencies.
- Do not add new dependencies.
- Do not introduce a transport abstraction for hypothetical future transports.
- Keep new path-relevant names in snake_case.
- Public classes and public methods must have XML documentation.
- Keep the MQTT hot path small and predictable.
- Avoid preserving server-side state infrastructure only because it already exists.

## Acceptance Criteria
- The default server path no longer maintains a full duplicate item state registry for normal MQTT item traffic.
- MQTT retained topics are documented and tested as the current last-known-state source.
- Clients remain responsible for reconstructing item state from MQTT.
- The server no longer contains write policy, Mesh policy, or domain intelligence.
- Minimal technical health remains available without requiring a full item state broker.
- `Amium.Item.Mqtt` is either removed deliberately or retained as a neutral shared MQTT protocol layer.
- Tests reflect MQTT-retained/client-state behavior rather than central server-retained state behavior.
- Relevant builds succeed.

## Out of Scope
- Non-MQTT transports.
- Authorization or validation systems.
- Central conflict resolution.
- Production Mesh design.
- Broad client API redesign outside compile-required adjustments.
- Premature batching/queue redesign.

## Risks / Watchouts
- Removing the server state broker may break tests or tools that use `InMemoryItemServer` directly; classify these before deleting APIs.
- Health metrics that depend on retained item count need a replacement or removal.
- Some current MQTT host behavior may rely on server subscription fanout to publish retained topics; replacing this must preserve externally visible retained MQTT payload behavior.
- Namespace cleanup for `Amium.Item.Mqtt` can cause widespread but mechanical compile changes.
- Avoid creating a new abstraction layer while removing the old one.

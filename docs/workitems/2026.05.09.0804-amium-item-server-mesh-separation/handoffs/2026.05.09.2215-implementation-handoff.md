# IMPLEMENTATION HANDOFF

## Goal
Complete the MQTT simplification by removing the legacy in-memory state-broker path from the default server architecture, reducing or deleting `Amium.Item.Server.Abstractions`, and replacing legacy-oriented selftests with tests that validate the retained-MQTT client reconstruction workflow.

## Scope
Included:
- Keep the target architecture explicit: lean MQTT host on the server, client-side topic mapping and item-state reconstruction, Mesh isolated.
- Audit and remove remaining production dependencies on `InMemoryItemServer`, `InMemoryItemServerStore`, and state-broker-only abstractions where possible.
- Reclassify selftests into "keep and adapt" versus "delete and rewrite" based on whether they validate the target architecture.
- Reduce or remove `Amium.Item.Server.Abstractions` based on actual remaining consumers.
- Preserve already completed own-broker migration to `MqttItemServerHost` and build on top of it.

Not included:
- No new transport abstraction.
- No attempt to preserve legacy state-broker semantics for compatibility alone.
- No unrelated HornetStudio UI or host refactors.

## Starting Point
- `MqttItemServerHost` already supports leaner hosting and the editor own-broker path was already migrated away from `InMemoryItemServer`.
- `Amium.Item.Server.Abstractions` still exists and currently provides `IItemServer`, `IItemServerTransport`, `IItemServerStore`, `IItemSubscription`, `IItemServerClient`, `IItemServerClock`, and `ItemSubscriptionOptions`.
- `InMemoryItemServer` and `InMemoryItemServerStore` still exist and are part of why the abstractions project still compiles.
- `HornetStudio.Host.Tests` still contains broad selftest debt, much of it caused by strict snake_case path validation and some tests still reflecting obsolete architecture assumptions.
- Builds currently succeed for `amium_item/Amium.Item.sln` and `HornetStudio.sln`, and `Amium.Item.Server.Tests` already passes.

## Tasks
1. Audit remaining source references to `Amium.Item.Server.Abstractions`, `InMemoryItemServer`, `InMemoryItemServerStore`, `IItemServerTransport`, and `IItemServerStore` across `amium_item/src` and `src`.
2. Classify each remaining abstraction as one of:
   - required by the retained-MQTT client API,
   - temporary migration residue,
   - obsolete server-state-broker contract.
3. Remove or migrate obsolete production usages first, before touching tests.
4. Reduce `Amium.Item.Server.Abstractions` to the minimum still required by real client-facing APIs; delete it entirely if no justified shared surface remains.
5. Remove `InMemoryItemServer` and `InMemoryItemServerStore` once no production path depends on them.
6. Rewrite selftests that encode obsolete architecture; only repair tests that still validate current intended behavior and merely fail because of path or naming drift.
7. Add or finalize an end-to-end retained-MQTT selftest that covers host start, retained publish, late subscriber, and client-side reconstruction.
8. Update docs and project references to match the reduced architecture.

## File-Level Changes
- `amium_item/src/Amium.Item.Server.Abstractions/Contracts.cs` -> remove or shrink obsolete server-state contracts.
- `amium_item/src/Amium.Item.Server.Abstractions/SubscriptionPolicies.cs` -> keep only options still exercised by real client APIs.
- `amium_item/src/Amium.Item.Server/InMemoryItemServer.cs` -> delete when no longer referenced.
- `amium_item/src/Amium.Item.Server/InMemoryItemServerStore.cs` -> delete when no longer referenced.
- `amium_item/src/Amium.Item.Server/Mqtt/MqttItemServerHost.cs` -> keep as lean host entry point; only adjust if abstraction removal requires constructor cleanup.
- `amium_item/src/Amium.Item.Server/Mqtt/MqttItemServerAdapter.cs` -> remove or reduce if it still exists only to bridge to removed contracts.
- `amium_item/src/Amium.Item.Client/**` -> absorb any remaining client-owned mapping or subscription responsibilities that should not live in `Server.Abstractions`.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> define retained-MQTT workflow assertions.
- `src/HornetStudio.Host.Tests/Program.cs` -> drop legacy-state expectations and keep only host/client behavior that still matters.
- `amium_item/Amium.Item.sln` and affected `.csproj` files -> remove obsolete project references if `Server.Abstractions` or other legacy projects disappear.

## Implementation Order
1. Run a focused reference audit and write down the remaining real consumers.
2. Remove production dependencies on legacy state-broker contracts.
3. Collapse or delete `Amium.Item.Server.Abstractions` based on the audit result.
4. Delete `InMemoryItemServer` and `InMemoryItemServerStore` when the build is clean without them.
5. Replace legacy selftests with retained-MQTT workflow tests.
6. Update docs and project wiring.
7. Run focused build and selftest validation.

## Technical Constraints
- Chat stays German; code, identifiers, comments, and docs stay English.
- Do not add compatibility abstractions for hypothetical future transports.
- Keep changes minimal and rooted in the target architecture.
- Prefer deleting obsolete behavior over preserving it behind new layers.
- Respect the existing two-stage build flow: build `amium_item/Amium.Item.sln` first, then `HornetStudio.sln`.
- Selftests in `Amium.Item.Server.Tests` and `HornetStudio.Host.Tests` are executable console suites and should be run via `dotnet <path-to-dll>`.

## Acceptance Criteria
- No production path requires `InMemoryItemServer` or `InMemoryItemServerStore`.
- `Amium.Item.Server.Abstractions` is either deleted or reduced to a clearly justified minimal surface.
- Server responsibilities are limited to MQTT hosting and technical health publication.
- Client-side retained-MQTT reconstruction is covered by a selftest.
- Legacy tests that only preserve the old architecture are removed or rewritten.
- `amium_item/Amium.Item.sln` and `HornetStudio.sln` still build successfully.

## Out of Scope
- Fixing all unrelated strict-path failures across the entire host test suite unless they block the migrated architecture tests.
- New feature work in Mesh, editor UI, or general host runtime.
- Broad renaming or cleanup with no architectural effect.

## Risks / Watchouts
- Some interfaces in `Server.Abstractions` may still be part of the public client API even if the project name is now architecturally misleading; verify before deleting.
- Tests failing due to snake_case enforcement should not be used as justification to keep old broker abstractions alive.
- Artifact-based HornetStudio references may hide stale compile-time dependencies until both solutions are rebuilt in the correct order.
- `MqttItemServerAdapter` may still be indirectly required by health publishing or test setup; confirm before removing it.

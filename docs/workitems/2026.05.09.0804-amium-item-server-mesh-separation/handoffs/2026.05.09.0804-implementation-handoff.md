# IMPLEMENTATION HANDOFF

## Goal
Simplify the `Amium.Item.Server` side by removing Mesh-specific and application-level writeability behavior from the generic server/MQTT server path, while keeping MQTT as the only planned transport and preserving a lean, high-performance Item Server core.

## Scope
Included:
- Keep `Amium.Item.Server` focused on receiving item messages, maintaining item state, publishing retained state, and notifying subscriptions.
- Keep MQTT as the only concrete server transport; do not introduce future transport abstractions.
- Remove or relocate server-side `writable` / `write_path` interpretation from the generic MQTT server adapter.
- Move Mesh-specific server composition and multi-node orchestration into `Amium.Item.Mesh`.
- Update tests that currently assert generic server-side writable blocking/routing behavior.
- Update server/client architecture documentation where it describes server responsibilities.

Not included:
- Do not redesign MQTT topic semantics beyond removing application-level writeability behavior from the generic server.
- Do not implement named pipes or another transport.
- Do not redesign the client API beyond compile-required adjustments caused by the server cleanup.
- Do not implement a full production Mesh protocol.
- Do not add runtime dependencies.

Breaking changes are allowed:
- Existing tests, samples, or demos may be adjusted if they relied on generic server-side `writable` / `write_path` behavior.
- Public message types may be removed or de-emphasized if they only exist to support the removed server-side writeability behavior and compile impact is acceptable.

## Starting Point
- `Amium.Item.Server.Mqtt.MqttItemServerAdapter` currently tracks `WritableItemState` in `_writableItems`.
- `MqttItemServerAdapter` updates writeability state from incoming item snapshots and parameter changes.
- Incoming MQTT `read` topic writes are currently converted to `ItemWriteRequestMessage` only when the target item is considered writable.
- `write_path` can redirect a generic server-side write request to another path.
- Tests in `amium_item/tests/Amium.Item.Server.Tests/Program.cs` assert writable routing and non-writable blocking behavior.
- `Amium.Item.Mesh` composes `MqttItemServerHost` through `mesh_demo_node_controller.cs`.
- `Amium.Item.Mesh` already has an internal MQTT orchestration type in `mesh_mqtt_node_client.cs`, but the local server host and Mesh behavior are still not fully separated from generic server assumptions.

## Tasks
1. Audit current server writeability and Mesh-related behavior:
   - Search for `Writable`, `write_path`, `ItemWriteRequestMessage`, `ItemServerAckMessage`, `MqttItemServerAdapter`, and `MqttItemServerHost`.
   - Separate generic server behavior from Mesh/demo/application behavior.
2. Simplify `MqttItemServerAdapter`:
   - Remove `_writableItems` and `WritableItemState`.
   - Remove `TrackWritableState`, `IsWritable`, and `GetWritableState`.
   - Stop blocking incoming MQTT writes because an item is not marked writable.
   - Stop redirecting writes through `write_path`.
   - Treat writeability metadata as ordinary item data when it arrives.
3. Define the lean generic behavior for incoming MQTT `read` updates:
   - Prefer mapping incoming write/read-topic payloads to direct item updates on the mapped path.
   - Avoid application-level permission decisions in the adapter.
   - Preserve existing retained/publish behavior where possible.
4. Re-evaluate `ItemWriteRequestMessage` and `ItemServerAckMessage`:
   - If still needed by `IItemServer.UpdateValueAsync` / `UpdatePropertyAsync`, keep them but document them as generic server API messages rather than MQTT writable-routing messages.
   - If only the removed MQTT writeability path uses a message, remove or reduce it only when compile impact remains small.
5. Move Mesh-specific server handling into `Amium.Item.Mesh`:
   - Add or extend an internal Mesh-owned server orchestration type if needed.
   - Keep local `MqttItemServerHost` creation in Mesh code, but ensure Mesh-specific mirroring, ownership, peer routing, and demo behavior stay in Mesh.
   - Do not push Mesh-specific state or decisions back into `Amium.Item.Server`.
6. Update tests:
   - Remove or replace tests that expect generic MQTT writes to be blocked by `writable=false`.
   - Remove or replace tests that expect generic MQTT writes to be redirected through `write_path`.
   - Add/update tests that assert incoming MQTT item updates are handled as generic item state updates.
   - Keep tests for retained publishing, server health, subscriptions, path normalization, and MQTT topic mapping.
7. Update documentation:
   - Clarify that `Amium.Item.Server` is a generic item-state server/broker, not a Mesh policy engine.
   - Clarify that `writable` / `write_path` are application-level metadata if still used by applications.
   - Clarify that Mesh behavior belongs to `Amium.Item.Mesh`.
8. Run builds/tests and fix compile issues.

## File-Level Changes

- `amium_item/src/Amium.Item.Server.Abstractions/Messages.cs` -> Keep, remove, or adjust write/ack message records depending on remaining generic API use.
- `amium_item/src/Amium.Item.Server.Abstractions/Contracts.cs` -> Adjust only if write/ack message contracts are intentionally simplified.
- `amium_item/src/Amium.Item.Server/InMemoryItemServer.cs` -> Adjust only if write/ack message contracts change; preserve normal item state behavior.
- `amium_item/src/Amium.Item.Server/Policies.cs` -> Remove write-request retention exceptions only if the message type is removed or no longer relevant.
- `amium_item/src/Amium.Item.Mesh/Controllers/mesh_demo_node_controller.cs` -> Keep Mesh-specific local server composition and mirroring decisions here or behind a Mesh-owned internal type.
- `amium_item/src/Amium.Item.Mesh/Controllers/mesh_mqtt_node_client.cs` -> Extend only if Mesh-specific MQTT/server orchestration needs a clearer home.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> Update tests for removed writable/write_path server behavior and add direct update expectations.
- `amium_item/docs/amium-itembroker-architecture.md` -> Update responsibility boundaries.
- `amium_item/docs/amium-itembroker-usage.md` -> Update usage guidance around server, MQTT, and writable metadata.
- `CHANGELOG.md` -> Add a relevant unreleased note if repository convention expects API/behavior changes here.

## Implementation Order
1. Run a fresh reference audit for writeability, write routing, and Mesh/server coupling.
2. Update `MqttItemServerAdapter` first, because it contains the main mixed responsibility.
3. Build the server solution to expose direct compile fallout.
4. Decide whether `ItemWriteRequestMessage` / `ItemServerAckMessage` remain useful generic messages or should be removed.
5. Adjust `InMemoryItemServer`, `Contracts.cs`, and `Policies.cs` only if the message/API decision requires it.
6. Update `Amium.Item.Server.Tests` to match the simplified server contract.
7. Check Mesh compile impact and move any Mesh-only behavior into `Amium.Item.Mesh`.
8. Update relevant docs and changelog.
9. Run final builds/tests.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and identifiers must be English.
- Keep new path-relevant names in snake_case.
- Prefer existing libraries, framework patterns, and project conventions.
- Do not add new dependencies.
- Do not add a generic transport abstraction for hypothetical future transports.
- Public classes and public methods must have XML documentation.
- Keep changes minimal and directly related to server/Mesh responsibility separation.
- Favor hot-path simplicity over extensibility that is not currently required.

## Acceptance Criteria
- `Amium.Item.Server` no longer treats `writable` or `write_path` as generic server-side routing/permission policy.
- `MqttItemServerAdapter` no longer maintains writable state for item paths.
- Incoming MQTT item updates are handled without application-level writeability checks in the generic server adapter.
- Mesh-specific mirroring, ownership, peer routing, and demo behavior remain isolated in `Amium.Item.Mesh`.
- No new transport abstraction is introduced.
- Tests no longer assert generic server-side writable blocking or write_path redirection.
- Documentation states the server/MQTT/Mesh responsibility split clearly.
- The complete relevant solutions build successfully.

## Out of Scope
- New transport support.
- MQTT batching redesign.
- Client API slimming work already covered by `docs/workitems/2026.05.09.0736-amium-item-client-api-slimming/`.
- Full production Mesh protocol design.
- Authorization, access control, or user permission systems.
- Broad namespace or project renames.

## Risks / Watchouts
- Existing demos or tests may rely on `writable` / `write_path` routing as behavior rather than metadata.
- Removing or changing write/ack messages can cascade into public API changes; keep them if they still support useful generic server calls.
- MQTT `read` topic naming may be semantically confusing after removing server-side writeability decisions; update docs/tests to make the intended behavior explicit.
- Mesh is conceptual and should not reintroduce generic server policy through shared server code.
- Avoid turning this cleanup into a larger transport or authorization redesign.

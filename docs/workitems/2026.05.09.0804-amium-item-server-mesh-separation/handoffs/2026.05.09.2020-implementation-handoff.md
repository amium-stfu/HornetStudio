# IMPLEMENTATION HANDOFF

## Goal
Simplify `Amium.Item.Server` into a lean, high-performance item state server that receives, stores, and publishes item state with minimal policy logic. Move application-level write semantics and Mesh intelligence out of the generic server path.

## Scope
Included:
- Simplify the generic server behavior around incoming MQTT item values.
- Remove `writable` / `write_path` handling from `Amium.Item.Server.Mqtt`.
- Treat incoming MQTT item payloads as direct item state updates.
- Keep server health as minimal technical health/status publishing.
- Keep MQTT as the only concrete transport; do not add future transport abstractions.
- Keep Mesh-specific intelligence in `Amium.Item.Mesh`.
- Allow breaking changes in server APIs, tests, samples, and documentation when they support the simplified server model.

Not included:
- Do not design a generic authorization, permission, or write-policy system.
- Do not implement a production Mesh protocol.
- Do not add named pipes or another transport.
- Do not optimize batching beyond what naturally falls out of removing mixed responsibilities.
- Do not redesign the client API except for compile-required adjustments.

## Starting Point
- `MqttItemServerAdapter` currently mixes MQTT mapping with application-level writeability state via `_writableItems`, `WritableItemState`, `TrackWritableState`, `IsWritable`, and `write_path` routing.
- Incoming MQTT value writes may be blocked when an item is not marked writable.
- `ItemWriteRequestMessage`, `ItemServerAckMessage`, `UpdateValueAsync`, and `UpdatePropertyAsync` exist as part of the current server contract.
- `ItemServerHealthPublisher` publishes technical server health and should remain, but only as minimal infrastructure status.
- `Amium.Item.Mesh` is conceptual and should own Mesh-specific node orchestration, mirroring, and conflict behavior.

## Tasks
1. Audit current server write path references:
   - `writable`
   - `write_path`
   - `WritableItemState`
   - `ItemWriteRequestMessage`
   - `ItemServerAckMessage`
   - `UpdateValueAsync`
   - `UpdatePropertyAsync`
2. Simplify `MqttItemServerAdapter`:
   - Remove writable state tracking.
   - Remove write_path redirection.
   - Remove server-side blocking for non-writable items.
   - Map incoming MQTT payloads directly to item state changes on the mapped path/property.
   - Keep topic and payload handling focused on MQTT-to-item translation.
3. Decide final fate of the server write API during implementation:
   - Remove `UpdateValueAsync`, `UpdatePropertyAsync`, `ItemWriteRequestMessage`, and `ItemServerAckMessage` if they only preserve old writable/write-request semantics.
   - Keep them only if they still represent a useful generic state-update API after simplification.
   - If kept, ensure they no longer imply server-owned permission or routing policy.
4. Keep `ItemServerHealthPublisher`:
   - Preserve minimal technical health values.
   - Do not add Mesh, client role, writeability, or application policy data to server health.
5. Keep `InMemoryItemServer` focused on state:
   - Retain normalized item storage.
   - Retain subscription fanout.
   - Retain retained-state retrieval.
   - Remove special behavior only tied to old write-request policy if the related messages are removed.
6. Keep Mesh intelligence in `Amium.Item.Mesh`:
   - Use or extend Mesh-owned internal orchestration for mirroring, peer routing, and demo behavior.
   - Do not push Mesh-specific assumptions into `Amium.Item.Server`.
7. Update tests:
   - Remove tests that assert server-side writable blocking.
   - Remove tests that assert write_path redirection by the generic MQTT server.
   - Add or adjust tests so incoming MQTT item values update server state directly.
   - Keep coverage for topic mapping, retained state, subscriptions, health publishing, and path normalization.
8. Update documentation:
   - State that `Amium.Item.Server` is a lean item state broker.
   - State that `writable`, `readonly`, and `write_path` are application/client/Mesh semantics, not generic server policy.
   - Clarify that MQTT is currently the only planned transport and no abstraction is being introduced for hypothetical future transports.
9. Build and verify.

## Implementation Order
1. Run reference audit for all writeability and write-request symbols.
2. Simplify `MqttItemServerAdapter` hot path first.
3. Build `amium_item/Amium.Item.sln --no-restore` to expose compile impact.
4. Decide whether write/ack contracts are still useful after adapter simplification.
5. Apply contract/message/server changes if needed.
6. Update tests to the lean server behavior.
7. Move any remaining Mesh-only assumptions into `Amium.Item.Mesh`.
8. Update docs and changelog.
9. Run final build/test verification.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and identifiers must be English.
- Use existing project patterns and dependencies.
- Do not add new dependencies.
- Do not introduce a generic future transport abstraction.
- Keep new path-relevant names in snake_case.
- Public classes and public methods must have XML documentation.
- Optimize for a small, predictable server hot path.
- Avoid broad refactoring outside server responsibility cleanup.

## Acceptance Criteria
- `Amium.Item.Server` primarily receives, stores, and publishes item state.
- `MqttItemServerAdapter` does not track writable state.
- `MqttItemServerAdapter` does not interpret `write_path`.
- Incoming MQTT item payloads update server state directly on the mapped path/property.
- Server health remains available as minimal technical status.
- Mesh-specific intelligence remains in `Amium.Item.Mesh`.
- No new transport abstraction is introduced.
- Tests reflect the simplified server behavior.
- Documentation clearly separates server, client/application, and Mesh responsibilities.
- Relevant builds succeed.

## Out of Scope
- Named pipe support.
- Additional transports.
- Authorization or permission system.
- Full Mesh production protocol.
- Client API cleanup already covered by `docs/workitems/2026.05.09.0736-amium-item-client-api-slimming/`.
- Broad folder/project restructuring.

## Risks / Watchouts
- Removing write/ack contracts may cause wider public API fallout; keep them if removal causes disproportionate churn.
- Existing samples may encode old `writable` / `write_path` behavior as expected behavior.
- MQTT `read` topic naming can be confusing once all incoming values are direct state updates; documentation should explicitly describe this.
- Server health should stay infrastructure-level and not become another policy surface.
- Mesh should not regain server policy through shared abstractions.

## Notes
- Recommended direction: server dumb and fast, clients/application/Mesh intelligent.
- `writable`, `readonly`, and `write_path` may still exist as ordinary item metadata if an application publishes them, but the generic server should not enforce or route based on them.
- Prefer direct state mutation semantics over request/ack semantics unless a concrete generic need remains.

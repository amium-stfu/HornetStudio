# IMPLEMENTATION HANDOFF

## Goal
Finish the Amium item MQTT simplification by reducing the server to a lean MQTT host, moving item topic mapping and item-state reconstruction to the client side, adding a server-client selftest for the retained MQTT workflow, and cleaning up obsolete state-broker abstractions/projects.

## Scope
Included:
- Make `Amium.Item.Server` a minimal MQTT host without item-state control, item-state reconstruction, write policy, or Mesh/domain intelligence.
- Treat MQTT retained messages as the current last-known-state source.
- Keep item-state reconstruction and MQTT item topic interpretation in clients.
- Remove or isolate legacy state-broker APIs and implementations that are no longer part of the default server path.
- Remove the server dependency on `Amium.Item.Mqtt` if the server no longer needs item topic mapping.
- Move MQTT item topic mapping into `Amium.Item.Client` unless a concrete shared core use remains.
- Add a server-client selftest that verifies retained MQTT publish and client reconstruction end to end.
- Update tests, docs, and project references to the simplified four-project target where feasible:
  - `Amium.Item`
  - `Amium.Item.Server`
  - `Amium.Item.Client`
  - `Amium.Item.Mesh`

Not included:
- Do not add another transport.
- Do not add generic transport abstractions.
- Do not implement authorization, write permissions, validation, conflict resolution, or Mesh policy.
- Do not build a new central state broker.
- Do not keep compatibility layers unless required by production code or a clearly documented transition.

## Starting Point
- `Amium.Item.Server` still references `Amium.Item.Mqtt`.
- `MqttItemServerAdapter` still uses `MqttItemTopicMapper` and `MqttTopicMapping`.
- `MqttItemServerAdapter` still implements `IItemServerTransport` / `IItemServerClient` and accepts `IItemServer`.
- `MqttItemServerHost` still exposes an optional `IItemServer? Broker`.
- `InMemoryItemServer`, `InMemoryItemServerStore`, `IItemServer`, `IItemServerStore`, and related subscription/retention APIs still exist.
- `Amium.Item.Client` uses `MqttItemTopicMapper`, `MqttTopicMapping`, `IItemServerClock`, `IItemServerClient`, `IItemSubscription`, and `ItemSubscriptionOptions`.
- `HornetStudio.Editor` still has an in-process `InMemoryItemServer` + `MqttItemServerAdapter` own-broker path.
- Tests still cover old state-broker behavior and MQTT topic mapping through the server namespace.

## Tasks
1. Audit and classify remaining legacy references:
   - Production: `src/HornetStudio.*`, `amium_item/src`.
   - Tests: `amium_item/tests`, `src/*Tests`.
   - Docs/workitems.
   - Decide which references must be migrated and which tests should be deleted/replaced.
2. Define final project ownership:
   - `Amium.Item` owns the item model and generic item/path helpers.
   - `Amium.Item.Client` owns MQTT item mapping, publish helpers, remote item registry, and client-side reconstruction.
   - `Amium.Item.Server` owns MQTT server hosting, lifecycle, options, connected client metrics, and minimal retained technical health publishing.
   - `Amium.Item.Mesh` owns Mesh/demo orchestration only.
3. Move or remove `Amium.Item.Mqtt`:
   - Move `MqttItemTopicMapper` and `MqttTopicMapping` into `Amium.Item.Client/Mqtt` if only clients need them after server cleanup.
   - Rename namespace away from `Amium.Item.Server.Mqtt`.
   - Remove `Amium.Item.Mqtt` project and project/artifact references if no longer needed.
   - If implementation proves a real shared non-server/non-client use remains, stop and document why keeping `Amium.Item.Mqtt` is justified.
4. Reduce `Amium.Item.Server`:
   - Remove server-side item topic mapping from the default server path.
   - Remove or heavily simplify `MqttItemServerAdapter` if it only exists to bridge MQTT to `IItemServer`.
   - Ensure `MqttItemServerHost` can start an MQTT server without an `IItemServer`.
   - Keep only minimal technical health output, preferably direct retained MQTT messages.
   - Do not parse normal item topics into item paths on the server hot path.
5. Clean state-broker abstractions:
   - Remove `IItemServerTransport`, `IItemServerStore`, and state-store contracts if unused after server reduction.
   - Remove `InMemoryItemServerStore` and `Policies.cs` if they only serve the old server state model.
   - Remove `InMemoryItemServer` if no production code needs it.
   - If some in-process test/client API still needs a local broker, move it behind a clearly named test/compatibility scope or keep temporarily with documentation and no default server dependency.
6. Clean client-facing abstractions carefully:
   - Keep `IItemSubscription`, `IItemServerClient`, `IItemServerClock`, and `ItemSubscriptionOptions` only if still required by the client public API.
   - Reduce `ItemSubscriptionOptions` to actually implemented options; remove speculative options such as batching/drop policy if not implemented.
   - Rename or relocate abstractions if `Server.Abstractions` no longer makes architectural sense.
7. Update HornetStudio own-broker mode:
   - Replace `InMemoryItemServer` + `MqttItemServerAdapter` usage with the lean `MqttItemServerHost`.
   - Keep external broker/client behavior stable.
   - Update docs that mention in-process `InMemoryItemServer`.
8. Add selftest server-client workflow:
   - Start a local `MqttItemServerHost` on a test port.
   - Connect publisher client.
   - Publish a retained item snapshot/value.
   - Connect observer client or subscribe observer after publishing.
   - Verify observer reconstructs the expected item/value from retained MQTT.
   - Verify no server-side state broker API is required for this flow.
9. Update tests:
   - Remove or replace tests that assert `InMemoryItemServer` retained state as the primary behavior.
   - Move topic mapping tests to client tests if mapping becomes client-owned.
   - Keep server tests focused on MQTT host lifecycle and technical health.
   - Keep client tests focused on publish, retained subscribe, and remote item reconstruction.
10. Update documentation and changelog:
   - Document the simplified server role.
   - Document that MQTT retained is the current last-known-state mechanism.
   - Document client-side item reconstruction.
   - Document project structure reduction.
   - Add a breaking cleanup note if convention requires it.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and identifiers must be English.
- Use existing dependencies; do not add new packages.
- Do not add transport abstractions for hypothetical future transports.
- Keep new path-relevant names in snake_case.
- Public classes and public methods must have XML documentation.
- Keep changes focused on server reduction, selftest, and cleanup.
- Preserve MQTT topic convention and item model behavior.
- Prefer localized changes over compatibility layers that keep obsolete architecture alive.

## Acceptance Criteria
- Default server path starts and hosts MQTT without `IItemServer`, `InMemoryItemServer`, or `InMemoryItemServerStore`.
- Server no longer interprets normal item MQTT topics into item state in the hot path.
- Server no longer depends on `Amium.Item.Mqtt` if mapping is client-owned.
- `Amium.Item.Mqtt` project is removed, or a concrete shared-use reason is documented.
- `Amium.Item.Server.Abstractions` is removed or reduced to only abstractions that are still genuinely required.
- Client publishes retained item data and reconstructs remote item state from MQTT retained messages.
- A selftest covers server start, client publish, client observe/reconstruct.
- Old state-broker tests are removed or rewritten to match MQTT-retained client behavior.
- Relevant builds succeed.

## Out of Scope
- New transports.
- Transport-neutral server framework.
- Central item-state broker redesign.
- Authorization/permissions/write policy.
- Mesh conflict handling.
- MQTT batching or queue redesign.
- Broad UI changes unrelated to own-broker host setup.

## Risks / Watchouts
- `HornetStudio.Editor` own-broker mode currently directly constructs `InMemoryItemServer` and `MqttItemServerAdapter`; migrate this before deleting those types.
- Some tests may encode old server-state behavior as expected behavior; replace with retained MQTT expectations rather than preserving obsolete APIs.
- `ItemSubscriptionOptions` and message interfaces may still be part of client-facing public APIs; shrink carefully to avoid unnecessary churn.
- Removing `Amium.Item.Mqtt` requires updating artifact references in `HornetStudio` project files and package/build output assumptions.
- Health publishing must still work after removing `IItemServer`; use retained MQTT messages directly.
- Do not move server-specific health topic behavior into the client mapping unless clients truly need to parse those topics.

# IMPLEMENTATION HANDOFF

## Goal

Replace the generic `Item` property `value` with explicit `read` and optional `write` properties. Preserve the `Item.Value` convenience wrapper, but make it read from `read` and write to `write` when a write channel exists. This is a hard breaking change with no migration or legacy `value` alias.

## Scope

Included:

- Core `Item` model changes.
- Broker and MQTT default property name changes from `value` to `read`.
- Editor default property selection updates.
- Tests and relevant documentation updates.

Explicitly NOT included:

- Legacy migration from `value` to `read`.
- Backward-compatible MQTT aliases.
- Large UDL request-structure refactoring.
- New dependencies.

## Starting Point

- `src/Amium.Item/Item.cs` initializes `Properties["value"]`.
- `Item.Value` currently gets and sets `Properties["value"].Value`.
- Broker and MQTT code special-case `"value"` as the primary item value parameter.
- Editor code inserts `"value"` as a preferred property option.
- Direct `Properties["value"]` usage appears concentrated in `Item.cs`, but string-based `"value"` handling exists in broker, MQTT, and editor code.

## Tasks

1. Change `Item` constructors to initialize `read` instead of `value`.
2. Add `bool hasWriteChannel = false` to the main public `Item` constructor.
3. Create `Properties["write"]` only when `hasWriteChannel` is `true`.
4. Update `Item.Value` so `get` returns `read`, and `set` writes to `write` if present, otherwise `read`.
5. Replace broker default parameter handling from `"value"` to `"read"`.
6. Replace MQTT default parameter handling from `"value"` to `"read"`.
7. Update editor defaults from `"value"` to `"read"`.
8. Update tests and documentation for the hard breaking change.
9. Build and run relevant tests.

## File-Level Changes

- `src/Amium.Item/Item.cs` -> core property model, constructor signature, `Value` wrapper, XML documentation.
- `src/Amium.Item/ItemExtension.cs` -> review serialization/deserialization behavior; do not add legacy migration.
- `src/Amium.Item.Server/InMemoryItemServer.cs` -> change primary value parameter name to `read`.
- `src/Amium.Item.Client.Mqtt/MqttItemClientSession.cs` -> change `ValueParameterName` and default parameter publishing to `read`.
- `src/Amium.Item.Client.Mqtt/MqttRemoteItemRegistry.cs` -> treat incoming `read` as the item value; do not treat `value` as an alias.
- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs` -> replace default selectable `"value"` property with `"read"`.
- `src/Amium.UdlClient/Module.cs` -> only update item construction if a clear write channel is required.
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` -> only update required call sites and naming assumptions; avoid larger UDL refactoring.
- `src/Amium.Item.Server.Tests/Program.cs` -> update broker/MQTT expectations.
- `src/HornetStudio.Editor.Tests/Program.cs` -> update editor/property expectations.
- `docs/amium-item-communication-structure.md` -> update item property terminology if it documents `value`.
- `docs/amium-itembroker-architecture.md` -> update broker terminology if it documents value parameters/topics.
- `docs/amium-itembroker-usage.md` -> update usage examples if they mention `value`.
- `CHANGELOG.md` -> add an entry if present or required by repository conventions for this breaking behavior change.

## Implementation Order

1. Update `src/Amium.Item/Item.cs` first.
2. Build once or run a targeted compile check to expose call-site errors.
3. Update broker/MQTT hard-coded `"value"` handling to `"read"`.
4. Update editor hard-coded `"value"` options to `"read"`.
5. Update tests and documentation.
6. Run the complete solution build.
7. Run relevant test projects or existing test commands.

## Technical Constraints

- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Do not add dependencies.
- Keep changes minimal and directly related to `read`/`write`.
- Do not add migration logic or compatibility aliases.
- Prefer named parameters when updating constructor calls with `hasWriteChannel`.
- Preserve existing project structure and naming conventions.
- Do not start applications during verification.

## Acceptance Criteria

- New `Item` instances contain `read` and no `value`.
- New `Item` instances contain `write` only when `hasWriteChannel` is `true`.
- `Item.Value` reads from `read`.
- `Item.Value` writes to `write` when present, otherwise `read`.
- Broker default item value parameter is `read`.
- MQTT default item value parameter/topic is `read`.
- No code path maps legacy `value` to `read`.
- Tests expecting `value` are updated to `read`.

## Verification

- `dotnet build HornetStudio.sln --no-restore`
- Run relevant existing tests for item broker behavior.
- Run relevant existing tests for editor item/property behavior.
- Search check: `rg -n '"value"|Properties\["value"\]' src docs` and manually verify remaining hits are unrelated local variable names or explicitly acceptable documentation of the breaking change.

## Out of Scope

- Data migration.
- MQTT compatibility bridge.
- UDL module architecture redesign.
- Renaming general local variables named `value`.
- Release publishing.

## Risks / Watchouts

- `new Item(...)` calls with positional arguments may become ambiguous if the constructor signature is changed carelessly. Add `hasWriteChannel` after existing optional parameters to preserve existing call sites.
- `ItemPropertyDictionary` lazily creates any requested property; avoid accidentally recreating `value` through old string constants.
- Editor property selection may still insert `value` through fallback logic.
- Tests may pass compile but still assert old MQTT topic names.
- UDL write behavior may need a later dedicated plan if `write` replaces existing `Request` items.

## Notes

- This change intentionally breaks old `value` consumers.
- `hasWriteChannel` models channel shape, not mutability.
- `write` should be treated as an explicit request channel, while `read` remains the observed state.

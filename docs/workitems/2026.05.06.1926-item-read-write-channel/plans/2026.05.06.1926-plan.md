# Item Read/Write Channel Plan

## Problem

`Item` currently uses a generic `value` property for both observed state and requested updates. This makes it unclear whether a value represents the latest read state or a write request. The requested change is a hard breaking change: replace `value` with `read`, add optional `write`, and do not support legacy migration.

## Goal

Introduce explicit `read` and optional `write` item properties:

- `read` stores the current observed item value.
- `write` stores a requested write value when the item has a dedicated write channel.
- `Item.Value` remains the convenience wrapper.
- No compatibility mapping from `value` to `read` is added.

## Scope

Included:

- Update the core `Item` model.
- Add `hasWriteChannel` to item construction.
- Replace broker and MQTT special handling of `value` with `read`.
- Update editor defaults that expose `value` as a selectable property.
- Update tests and documentation affected by the property name change.

Excluded:

- Large UDL structure refactoring.
- Migrating old serialized data.
- Supporting old MQTT topics or old JSON payloads that still use `value`.
- Adding new dependencies.

## Architecture Decision

Use `hasWriteChannel` instead of `isReadOnly`.

Reasoning:

- The important distinction is whether an item has a separate write channel, not whether it is read-only.
- Items without `write` can still be mutable; `Item.Value` can directly update `read`.
- Items with `write` separate observed state from requested state.

## Target Semantics

For every `Item`:

- `Properties["read"]` exists.
- `Properties["value"]` does not exist.
- `Value get` returns `Properties["read"].Value`.

For items without a write channel:

- `Properties["write"]` does not exist.
- `Value set` writes to `Properties["read"].Value`.

For items with a write channel:

- `Properties["write"]` exists.
- `Value set` writes to `Properties["write"].Value`.
- `Value get` still reads from `Properties["read"].Value`.

## Implementation Sequence

1. Update `src/Amium.Item/Item.cs`.
   - Replace all internal `value` property initialization with `read`.
   - Add `hasWriteChannel` constructor parameter.
   - Create `write` only when requested.
   - Update XML documentation.

2. Update broker core behavior.
   - Replace default parameter name `"value"` with `"read"` in broker publishing and update handling.
   - Keep `write` as a distinct item property. Do not silently route `write` through old `value` behavior.

3. Update MQTT client/session behavior.
   - Change the default MQTT parameter from `value` to `read`.
   - Publish item values through `read`.
   - Process received `read` topics as the main item value.
   - Do not process `value` as a legacy alias.

4. Update editor property defaults.
   - Replace default selectable property `"value"` with `"read"`.
   - Ensure `write` is only exposed when present in the item property list.

5. Update UDL-related call sites only where required.
   - Do not refactor the UDL request structure as part of this change.
   - Only opt into `hasWriteChannel: true` where the existing behavior clearly represents separate write requests.

6. Update tests.
   - Replace expected `"value"` names with `"read"`.
   - Add focused tests for `hasWriteChannel`.
   - Remove any implicit legacy compatibility expectations.

7. Update documentation.
   - Update relevant Markdown files that describe item properties, broker behavior, MQTT topics, or editor item binding.
   - Add a `CHANGELOG.md` entry if the repository already treats this as a relevant behavior change.

8. Verify.
   - Run the complete solution build.
   - Run available relevant tests if they are part of the repository workflow.

## Breaking Changes

- `Properties["value"]` is removed.
- Broker and MQTT default parameter name changes from `value` to `read`.
- Existing serialized item JSON containing only `value` is not migrated.
- Existing MQTT clients publishing or subscribing to `value` topics must be updated.

## Risks

- Broad compile failures where `"value"` is hard-coded.
- Runtime behavior changes in editor binding lists.
- Remote MQTT clients may silently stop exchanging values if they still use old topics.
- Existing saved project/runtime data may no longer resolve the intended property.

## Verification Targets

- `dotnet build HornetStudio.sln --no-restore`
- Existing item broker tests.
- Existing editor tests that resolve item properties.
- Manual review of MQTT topic names in broker/client code.

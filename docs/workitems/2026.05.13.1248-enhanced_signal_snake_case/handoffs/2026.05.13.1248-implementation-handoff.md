# IMPLEMENTATION HANDOFF

## Goal
Fix remaining EnhancedSignal / ExtendedSignalModule snake_case inconsistencies that can break runtime publishing when adding a new enhanced signal.

## Scope
Included:
- Make generated enhanced-signal registry paths use canonical snake_case segments.
- Align EnhancedSignal runtime path construction with the existing CustomSignals path pattern.
- Add focused tests for the generated enhanced-signal registry path.
- Update widget documentation only if the documented runtime path changes.

Not included:
- Renaming persisted widget type names such as `EnhancedSignals`.
- Broad property-name migration for all runtime metadata.
- Release/version changes.
- New dependencies.
- Refactoring the EnhancedSignal runtime architecture.

## Starting Point
- `EnhancedSignalRuntime.BuildRegistryPath(...)` currently returns `studio.<folder>.EnhancedSignals.<signal>`.
- Strict path rules expect snake_case path identities.
- `CustomSignalsControl.BuildRegistryPath(...)` already normalizes the widget and signal path segments to snake_case.
- The enhanced-signal editor already validates new signal names with `TargetPathHelper.IsValidPathIdentityName(...)`.
- Adding a signal likely crashes after the dialog saves the definition and the runtime rebuild publishes a path containing the PascalCase `EnhancedSignals` segment.

## Tasks
1. Change enhanced-signal runtime path generation so the static runtime segment is `enhanced_signals`.
2. Ensure folder and signal-name segments remain normalized with the existing path helpers.
3. Check whether `ExtendedSignalModule` needs only path-level changes or also a non-path metadata cleanup for `kind`.
4. Add focused tests that assert enhanced-signal runtime paths are fully snake_case.
5. Run focused editor/host tests and a no-restore build where available.
6. Update EnhancedSignals documentation if any documented runtime path currently shows `EnhancedSignals`.

## File-Level Changes
- `src/HornetStudio.Host/EnhancedSignalRuntime.cs` -> update `BuildRegistryPath(...)` to emit `enhanced_signals` instead of `EnhancedSignals`.
- `src/HornetStudio.Host/ExtendedSignalModule.cs` -> inspect only; change `kind` metadata only if it is validated as a technical path/identity and not just a display/runtime type marker.
- `src/HornetStudio.Editor.Tests/Program.cs` -> add or update focused test coverage for the enhanced-signal runtime path.
- `src/HornetStudio/docs/widgets/EnhancedSignals.md` -> update only if path examples are affected.
- `src/HornetStudio/docs/widgets/help/EnhancedSignals.help.md` -> update only if path examples are affected.
- `docs/data-flow-and-signals.md` and `src/HornetStudio/docs/data-flow-and-signals.md` -> update only if they still document `studio.<Folder>.EnhancedSignals.<SignalName>`.

## Implementation Order
1. Add or adjust a test that expects `studio.default_layout.enhanced_signals.enhanced_signal_1`.
2. Update `EnhancedSignalRuntime.BuildRegistryPath(...)`.
3. Run the focused test suite to confirm the path expectation.
4. Search for documented `EnhancedSignals` registry-path examples and update them to `enhanced_signals` only where they describe registry paths.
5. Run a focused build or solution build with `--no-restore` if restore state allows it.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Do not add dependencies.
- Keep changes local and avoid new abstractions.
- Do not rename the persisted widget type unless a failing test proves it is necessary.
- Keep compatibility behavior for resolving old paths if the existing registry normalization already supports it.

## Acceptance Criteria
- Adding an enhanced signal no longer publishes a registry key containing `.EnhancedSignals.`.
- New enhanced-signal registry paths use `.enhanced_signals.`.
- Existing snake_case signal names continue to be accepted.
- Uppercase enhanced-signal names continue to be rejected in the editor.
- Existing CustomSignals behavior remains unchanged.

## Verification
- Run `dotnet build src/HornetStudio.Editor/HornetStudio.Editor.csproj --no-restore` if dependencies are already restored.
- Run `dotnet build HornetStudio.sln --no-restore` if solution-level restore state allows it.
- Run the focused editor test executable or test command used for `src/HornetStudio.Editor.Tests`.
- Manual check: add a new EnhancedSignals widget signal and verify the published path uses `studio.<folder>.enhanced_signals.<signal>`.

## Out of Scope
- Migrating all persisted layout property names from PascalCase to snake_case.
- Removing `Request` command semantics beyond the existing separate workitem.
- Changing UDL module paths or CAN write behavior.
- Renaming docs/widget files whose names intentionally match persisted widget type values.

## Risks / Watchouts
- Documentation and persisted widget type names use `EnhancedSignals` as a widget identity; do not blindly rename those to `enhanced_signals`.
- Some old layouts or docs may reference `.EnhancedSignals.` as a registry path. Existing registry normalization is case-insensitive and segment-normalizing, but tests should confirm new output is canonical.
- `ExtendedSignalModule.Properties["kind"]` currently uses `ExtendedSignalModule` and is later overwritten to `EnhancedSignal`; changing this may affect type-based UI logic if any exists.
- Runtime child item names are already lowercase (`raw`, `read`, `set`, `out`, `state`, `alert`, `command`, `config`), so avoid unnecessary changes there.

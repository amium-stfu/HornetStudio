# DEBUG REPORT

## Problem
The multimaster self-test fails all `cross_write` cases because the sending node does not observe the value it wrote.

## Expected Behavior
When `server_a` writes `nodes.server_b.write_test`, all observers should be able to see the expected value, including `server_a`, unless the test intentionally excludes the writing observer.

## Actual Behavior
The value is distributed to other nodes, but the actor node reports the target path as missing.

Example from `logs/multimaster_self_test/latest_summary.md`:

- `cross_write_server_a_to_server_b`
- `server_b` sees the expected value.
- `server_c` sees the expected value.
- `server_a` reports `(missing)`.

The missing observers accumulate by actor:

- after `server_a` writes, `server_a` no longer sees the written paths.
- after `server_b` writes, `server_b` also hides written paths.
- after `server_c` writes, all actor-side writes are hidden for their writers.

## Error Messages / Logs
The report shows timeouts such as:

- `Timed out while waiting for all observers to reach the expected value.`
- `Missing observers: server_a`

The JSONL log also records successful publish actions before the failed case results:

- `Remote write published: nodes.server_b.write_test = ...`

## Relevant Code
`DemoNodeController.WriteRemoteValueAsync` currently writes through the actor's observer client:

```cs
var observerClient = GetRequiredObserverClient();
await observerClient.UpdateValueAsync(item, cancellationToken).ConfigureAwait(false);
```

`MqttRemoteItemClient.UpdateValueAsync` intentionally tracks locally published paths:

```cs
_publishedServerPaths.Add(ItemServerItemPath.Resolve(item));
```

`MqttRemoteItemClient.GetRemoteItemSnapshots` then removes those paths from visible remote snapshots:

```cs
foreach (var publishedPath in _publishedServerPaths)
{
    RemoveDescendant(visibleRoot, publishedPath);
}
```

## Attempted Fixes
- No code fix attempted yet.
- Existing logs and source inspection identify the failing behavior as observer-side hiding after writes through `MqttRemoteItemClient`.

## Current Hypothesis
- `cross_write` uses the wrong client role for diagnostic writes.
- `MqttRemoteItemClient` is designed to hide paths it publishes itself, which is useful for local-owner echo prevention.
- The self-test uses that same observer client as a remote writer, so the actor hides the exact path the test expects it to observe.
- Runtime remote-write tests currently avoid this by excluding actors/hidden observers from required observers, so they pass even when actor-side visibility is missing.

## Suggested Fix
- Do not perform cross-write diagnostic writes through the observer client.
- Prefer writing through a separate writer/publisher session that does not mutate the observer client's `_publishedServerPaths`.
- Minimal demo-side direction:
  - Change `DemoNodeController.WriteRemoteValueAsync` to use `_publisherSession.UpdateValueAsync(...)` or a dedicated writer session instead of `_observerClient.UpdateValueAsync(...)`.
  - Keep `_observerClient` read-only for observation and UI snapshots.
  - Re-run `--self-test`.
- If actor-side hiding is intentional for all local writes, then the self-test criteria must explicitly exclude the actor for remote writes. That is weaker and does not match the desired human-visible behavior described for `A -> B`.

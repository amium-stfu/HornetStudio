# DEBUG REPORT

## Problem
The multimaster demo `cross_write` self-test fails because remote writes are sent through the observer client instead of the normal publisher session write path.

## Expected Behavior
Demo and self-test write operations should use the normal `MqttItemClientSession` methods:

- `PublishSnapshotAsync`
- `UpdateValueAsync`
- `UpdatePropertyAsync`

The observer client should remain read-only and should only observe remote snapshots.

## Actual Behavior
`DemoNodeController.WriteRemoteValueAsync` used `MqttRemoteItemClient.UpdateValueAsync`.

That method marks the written path as locally published in `_publishedServerPaths`. `GetRemoteItemSnapshots` then hides those paths from the actor's visible remote snapshots. As a result, the actor node does not observe the value it just wrote during `cross_write` checks.

## Error Messages / Logs
Known failing self-test symptom:

- `Timed out while waiting for all observers to reach the expected value.`
- `Missing observers: server_a` or the current actor node

## Relevant Code
Changed `src/amium_item_server_multimaster_demo/Controllers/DemoNodeController.cs`.

`WriteRemoteValueAsync` now resolves the existing publisher session and calls:

```cs
await session.UpdateValueAsync(item, retained: true, cancellationToken: cancellationToken).ConfigureAwait(false);
```

## Attempted Fixes
- Replaced observer-side writes with publisher-session writes in `DemoNodeController.WriteRemoteValueAsync`.

## Current Hypothesis
- The failed `cross_write` cases were caused by observer-side local publish hiding, not by MQTT propagation or server write handling.
- Keeping `_observerClient` read-only avoids mutating `_publishedServerPaths` during self-test writes.

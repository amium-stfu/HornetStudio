# IMPLEMENTATION HANDOFF

## Goal
Extend the standalone Item.Server multimaster WinForms demo with a reproducible self-test mode that validates cross-node writes, runtime item creation, runtime item propagation, and remote writes to runtime-created items. The self-test must produce log files that can be reviewed by humans and by Codex without using the UI.

## Scope
Included:
- Extend the existing `src/amium_item_server_multimaster_demo/` project only.
- Add a headless self-test mode that can be started with `--self-test`.
- Keep the normal WinForms UI mode intact.
- Add an optional UI command to run the same self-test while the UI is open if this can be done without broad UI restructuring.
- Test all cross-node writes for existing `write_test` items.
- Test runtime item creation on each node.
- Test propagation of runtime-created items to all nodes.
- Test value updates on runtime-created items.
- Test remote writes to runtime-created writable items.
- Write machine-readable JSONL logs and a human-readable Markdown summary.
- Return a non-zero process exit code from headless mode when the self-test fails.

Not included:
- No changes to `Item.Server.Monitor`.
- No new production multimaster architecture, ownership model, routing, conflict resolution, TTL, hop count, vector clocks, persistence, or external broker configuration.
- No separate test project unless the existing app cannot support headless execution cleanly.
- No new external dependencies.
- No release or publish work.

## Starting Point
- The multimaster demo already exists under `src/amium_item_server_multimaster_demo/`.
- `Program.cs` starts the WinForms UI.
- `MainForm.cs` starts the demo automatically when shown and displays three node panels.
- `Controllers/MultimasterDemoController.cs` coordinates three nodes and owns the embedded MQTT host.
- `Controllers/DemoNodeController.cs` owns one node's MQTT publisher session, observer client, dynamic value publisher, writable `write_test` item, snapshots, and event reporting.
- Current node IDs are `server_a`, `server_b`, and `server_c`.
- Current item paths are:
  - `nodes.<node>.dynamic_value`
  - `nodes.<node>.write_test`
- Current observations are read through `MqttRemoteItemClient.GetRemoteItemSnapshots()` and the shared remote client id.

## Tasks
1. Inspect the current `Program.cs`, `MultimasterDemoController`, and `DemoNodeController` implementation to identify the smallest reusable entry point for headless execution.
2. Add a self-test runner that can start the existing demo controller, run test cases, wait for expected observations, write logs, and stop the controller.
3. Extend `DemoNodeController` with targeted diagnostic operations:
   - Write a value to an arbitrary existing item path where supported by the current client/session APIs.
   - Create and publish a runtime item path owned by the node.
   - Create runtime items with and without writable metadata as needed by the test cases.
   - Read observed value state for a specific path and observing node.
4. Keep existing UI behavior and dynamic value publishing unchanged except where needed to share self-test infrastructure.
5. Add test cases for existing item cross writes:
   - `server_a` writes `server_a.write_test`, `server_b.write_test`, `server_c.write_test`
   - `server_b` writes `server_a.write_test`, `server_b.write_test`, `server_c.write_test`
   - `server_c` writes `server_a.write_test`, `server_b.write_test`, `server_c.write_test`
6. Add runtime item creation test cases:
   - each node creates `nodes.<node>.runtime.created_<sequence>`
   - all nodes must observe the new item and initial value.
7. Add runtime item update test cases:
   - each creator updates its own runtime-created item
   - all nodes must observe the updated value.
8. Add runtime writable item remote-write test cases:
   - each node creates `nodes.<node>.runtime.remote_write_probe_<sequence>` with writable metadata.
   - every other node writes a distinct value to that runtime-created item.
   - all nodes must observe the final expected value.
9. Add timeout-based verification helpers:
   - wait until all observing nodes see a target path/value.
   - record propagation duration.
   - record missing observers and last observed values on timeout.
10. Add duplicate/echo suspicion indicators:
   - compare observed update counters before and after a write.
   - flag excessive update count growth within the wait window.
   - do not fail only because of suspicion unless the growth is clearly pathological and the threshold is intentionally defined.
11. Write logs to `logs/multimaster_self_test/`:
   - timestamped JSONL file.
   - timestamped Markdown summary.
   - `latest.jsonl` copy or overwrite.
   - `latest_summary.md` copy or overwrite.
12. Add `--self-test` handling in `Program.cs`:
   - no WinForms UI is shown.
   - run the test.
   - write summary path to console.
   - return exit code `0` on success and non-zero on failure.
13. If feasible, add a `Run Self-Test` button to the UI toolbar:
   - run asynchronously.
   - keep UI responsive.
   - append summary result to node/global event output or show a compact message box.
14. Build the project and full solution.
15. Run the headless self-test and inspect the produced summary/log.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible UI text and error messages must be English.
- New path-relevant names must use `snake_case`.
- Use only existing project dependencies and .NET APIs.
- Keep the self-test code inside the demo project.
- Do not change `Item.Server.Monitor`.
- Do not hide missing propagation with test-only shortcuts.
- Do not introduce a reusable production multimaster framework.
- Use async/await and cancellation correctly.
- Avoid `.Result`, `.Wait()`, blocking sleeps, and UI-thread blocking.
- Ensure headless mode disposes the controller and all node resources even when a test fails.
- Public classes and public methods must have XML documentation if they are public; prefer `internal` for demo-only helpers.

## Acceptance Criteria
- Running the demo normally still opens the WinForms UI and starts the three-node demo.
- Running the executable with `--self-test` performs no UI startup.
- Headless self-test starts three nodes and stops them cleanly.
- Existing item cross-write matrix is executed and summarized.
- Runtime item creation is executed for all three nodes.
- Runtime-created item propagation is verified on all three nodes.
- Runtime-created item updates are verified on all three nodes.
- Runtime-created writable items can be written from other nodes and the result is verified on all three nodes, if supported by the current Item.Server/MQTT write path.
- Self-test writes JSONL and Markdown logs under `logs/multimaster_self_test/`.
- `latest.jsonl` and `latest_summary.md` point to the latest run content.
- Headless mode returns exit code `0` only when all required checks pass.
- Failures include enough context to identify actor node, target path, expected value, observed values, timeout, and suspicious duplicate/update behavior.
- The full solution builds.

## Risks / Watchouts
- Arbitrary remote writes may not currently be supported by the available MQTT client/session API. If unsupported, document the limitation in the summary and keep the failing/blocked test category explicit instead of bypassing it.
- Runtime-created writable metadata may not be imported by all observers if metadata propagation is incomplete. The test should expose this clearly.
- Dynamic value publishing may add noise to duplicate/update counters. Use exact target paths and before/after counters for the write-test paths.
- Retained MQTT state from a previous run can contaminate results. Prefer unique runtime item names per run and consider a unique run id suffix in test values.
- If port `18830` is already in use, headless self-test should fail with a clear message and still write a summary when possible.
- Writing `latest.*` files is acceptable because they are generated diagnostic artifacts, but timestamped files should also be kept for comparison.

## Notes
- The self-test should be a diagnostic truth source. If a scenario fails because the current server/client behavior does not support it yet, the implementation should report that honestly rather than adding demo-specific workaround behavior.
- Keep generated log content stable and simple enough that Codex can inspect it with normal file reads.
- Prefer deterministic test values such as `<run_id>|<actor>|<target>|<case>` so retained or stale values are easy to identify.

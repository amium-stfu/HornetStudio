# IMPLEMENTATION HANDOFF

## Goal
Create a standalone WinForms demo application that starts three local Item.Server nodes and makes initial multimaster behavior visible to a human user.

## Scope
Included:
- Add a new standalone WinForms project under `src/amium_item_server_multimaster_demo/`.
- Start three local Item.Server nodes in one process: `ServerA`, `ServerB`, and `ServerC`.
- Connect all three nodes through the existing MQTT-based item server/client infrastructure.
- Show each node in the UI with two values:
  - `dynamic_value` for cyclic live updates.
  - `write_test` for cross-node write verification.
- Show observed values from all nodes so propagation, missing updates, duplicate updates, and echo behavior are visible.
- Show basic connection/status and per-node event logs.
- Add the new project to `HornetStudio.sln`.
- Keep the demo focused on diagnostics, not as a product UI.

Not included:
- No changes to `Item.Server.Monitor`.
- No new multimaster architecture, routing layer, ownership model, conflict resolution, TTL, hop-count, vector clocks, or persistence.
- No release/publish work.
- No new external dependencies unless technically unavoidable and explicitly justified.

## Starting Point
- `src/Amium.Item.Client.MqttDemoWinForms/` already provides a WinForms project pattern using `net9.0-windows`, `UseWindowsForms`, and references to existing Amium item projects.
- `src/Amium.Item.Server/` contains the in-process server runtime and MQTT adapter/host infrastructure.
- `src/Amium.Item.Client.Mqtt/` contains the MQTT remote item client.
- `src/Amium.Item.Server.Tests/Program.cs` and `src/amium_itembroker_mqtt_stress_test/` may contain useful examples for subscriptions, writes, retained state, and MQTT stress behavior.
- Current related work is focused on MQTT inbound observed-state import and monitor visibility; this demo should not silently fix those behaviors.

## Tasks
1. Inspect the existing WinForms MQTT demo and reuse its project setup, lifecycle handling, and UI conventions where practical.
2. Inspect current Item.Server MQTT host/client APIs and identify the smallest way to run three local server nodes and connect them through MQTT.
3. Create `src/amium_item_server_multimaster_demo/amium_item_server_multimaster_demo.csproj` as a standalone WinForms application.
4. Add a `Program.cs` entry point with normal WinForms startup and exception handling consistent with existing demos.
5. Add a main form that displays three node panels side by side or in a clear responsive layout.
6. Add a small node runtime/controller model for the demo only, responsible for starting/stopping one server node, publishing the dynamic value, exposing the writable test value, and reporting events to the UI.
7. Configure the demo so each node publishes:
   - `nodes.<node>.dynamic_value`
   - `nodes.<node>.write_test`
8. Configure subscriptions so each node observes all three nodes' values.
9. Add UI controls for each node:
   - current local `dynamic_value`
   - current local `write_test`
   - writable input and write button
   - observed values from `ServerA`, `ServerB`, and `ServerC`
   - connection/status indicator
   - compact log list
10. Make the dynamic value update on a timer with clean async cancellation/disposal.
11. Ensure UI updates are marshalled to the UI thread.
12. Add simple counters or timestamps so duplicate updates and stale values are visible.
13. Add the project to `HornetStudio.sln` using existing solution conventions.
14. Build the new project and then build the full solution without starting monitor, service, or host apps.

## Implementation Order
1. Read `Amium.Item.Client.MqttDemoWinForms` to copy the established WinForms startup and controller style.
2. Read the MQTT server/client APIs needed to start a local MQTT-backed node and observe/write item values.
3. Create the new project with only the necessary project references.
4. Implement one node controller first and verify it can publish/read its own `dynamic_value` and `write_test`.
5. Extend the controller setup to three nodes.
6. Add full-mesh observation through the existing MQTT topic mapping/client APIs.
7. Build the UI with three node panels and event logs.
8. Wire write-test inputs so writing from one node becomes visible on the other nodes.
9. Add lifecycle cleanup for timers, subscriptions, adapters, servers, and clients.
10. Add the project to the solution.
11. Run project build and full solution build.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible UI text and error messages must be English.
- New path-relevant names must use `snake_case`.
- Use WinForms because this demo must be standalone and human-usable.
- Do not modify `Item.Server.Monitor`.
- Use existing Amium Item.Server, MQTT adapter, and client patterns.
- Do not introduce new dependencies without explicit justification.
- Keep public classes and public methods XML-documented.
- Use async/await correctly; avoid `.Result`, `.Wait()`, and blocking UI thread work.
- Dispose timers, subscriptions, adapters, clients, and servers explicitly.
- Keep the demo diagnostic and small; avoid creating a reusable multimaster framework in this workitem.

## Acceptance Criteria
- A new standalone WinForms app exists at `src/amium_item_server_multimaster_demo/`.
- The app starts three local nodes named `ServerA`, `ServerB`, and `ServerC`.
- The UI displays all three nodes at the same time.
- Each node publishes a changing `nodes.<node>.dynamic_value`.
- Each node exposes a writable `nodes.<node>.write_test`.
- The UI shows whether every node observes all six values from all nodes.
- Writing `write_test` from one node is visible on the other nodes.
- The UI makes duplicates, stale values, missing values, and echo-like behavior visible through timestamps, source labels, counters, or log entries.
- Closing the app stops all background work cleanly.
- `dotnet build src\amium_item_server_multimaster_demo\amium_item_server_multimaster_demo.csproj --no-restore` succeeds after restore/build prerequisites are available.
- `dotnet build HornetStudio.sln --no-restore` succeeds after restore/build prerequisites are available.

## Verification

  - `dotnet build HornetStudio.sln --no-restore`
- Manual verification:
  - Start only the new WinForms demo.
  - Confirm all three node panels show live dynamic values.
  - Write a value in `ServerA.write_test` and verify `ServerB` and `ServerC` show the updated value.
  - Repeat from `ServerB` and `ServerC`.
  - Watch logs for repeated echo updates or missing propagation.

## Out of Scope
- Monitor integration.
- Automated regression tests for multimaster behavior.
- Persisted configuration.
- External broker configuration UI.
- Ownership, routing, conflict resolution, federation, hop count, TTL, or message envelopes.
- Release versioning and publish output.

## Risks / Watchouts
- If MQTT inbound observed-state import is still incomplete, the demo may expose missing propagation instead of passing end-to-end. Do not hide this with demo-specific shortcuts.
- Running three server nodes in one process may require unique client IDs, ports, node IDs, or topic/source identifiers to avoid self-filtering.
- Echo prevention must remain visible and diagnosable; do not suppress logs that show repeated propagation.
- WinForms cross-thread updates must be marshalled correctly.
- Keep `dynamic_value` update intervals modest so the UI remains readable and logs do not flood.
- If the existing MQTT host expects a single embedded broker rather than multiple local brokers, use one shared broker with three distinct server/client identities unless the existing architecture clearly supports another arrangement.

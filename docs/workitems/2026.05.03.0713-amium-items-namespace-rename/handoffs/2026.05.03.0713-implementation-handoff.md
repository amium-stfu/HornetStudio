# IMPLEMENTATION HANDOFF

## Goal
Rename the item model namespace from `Amium.Item` to `Amium.Items` so code no longer needs to reference `Amium.Item.Item`, while keeping the type name `Item` unchanged.

## Scope
Included:
- Namespace declaration updates inside `src/Amium.Item`
- Consumer updates across solution projects, tests, and relevant docs
- Alias updates such as `ItemModel = Amium.Items.Item`
- Verification that no old namespace references remain in active code paths

Not included:
- Renaming the `Item` class
- Changing assembly name, csproj name, or folder name
- Introducing compatibility shims unless explicitly requested
- Functional refactoring unrelated to the namespace rename

## Starting Point
- The item model currently lives in namespace `Amium.Item`.
- Core type references include `Item`, `Parameter`, `ItemDictionary`, `ParameterDictionary`, and related extension helpers.
- Multiple projects already reference the type via `using Amium.Item;` or aliases such as `using ItemModel = Amium.Item.Item;`.
- At least one documentation file explicitly mentions `Amium.Item.Item`.

## Tasks
1. Change all namespace declarations in the `src/Amium.Item` project from `Amium.Item` to `Amium.Items`.
2. Update all `using Amium.Item;` statements and all fully qualified references to `Amium.Items`.
3. Update namespace-based type aliases in broker/client/demo code and any XML documentation references if present.
4. Search the repository for remaining `Amium.Item` references and either migrate or explicitly leave them only where intentional.
5. Build the solution and run the most relevant tests for item broker and host/editor usage.
6. Fix any compile or test fallout caused by reflection, serialization, or namespace-qualified docs.

## File-Level Changes
- `src/Amium.Item/Item.cs` -> change namespace declaration and keep public API behavior unchanged
- `src/Amium.Item/ItemExtension.cs` -> change namespace declaration and keep extension API stable
- `src/Amium.Item/ItemPathExtensions.cs` -> change namespace declaration
- `src/Amium.Item.Server/**/*` -> update `using` directives and aliases that reference `Amium.Item`
- `src/HornetStudio*/**/*` -> update imports and any fully qualified item type references
- `src/Amium.UdlClient/**/*` -> update aliases and imports
- `src/*Tests/**/*` -> update imports and qualified references
- `src/Amium.Item/README.md` and other Markdown docs -> update explicit namespace examples where relevant

## Implementation Order
1. Update the namespace declarations in the `Amium.Item` project files.
2. Run a repository search for `Amium.Item.Item`, `using Amium.Item;`, and `namespace Amium.Item`.
3. Update all consuming code and docs in a single migration pass.
4. Build the solution and fix any missed references.
5. Run relevant tests and perform a final leftover-reference search.

## Technical Constraints
- Follow existing project conventions and keep code/comments/documentation in English.
- Do not widen scope into structural refactors or package renames.
- Prefer small mechanical edits over opportunistic cleanup.
- Preserve public type names and behavior.

## Acceptance Criteria
- No active code needs to reference `Amium.Item.Item`.
- Item model types compile under `Amium.Items`.
- Consumers across the solution compile against the new namespace.
- Relevant docs no longer present the old namespace as the preferred usage.
- Build and targeted tests pass or any remaining failures are clearly documented.

## Verification
- Search: `rg -n "Amium\\.Item(\\.Item)?|namespace Amium\\.Item|using Amium\\.Item;" src docs`
- Build: solution build command used by the repository
- Tests: run at minimum the broker tests and at least one host/editor test set that exercises item usage

## Out of Scope
- Backward-compatible adapter namespaces
- NuGet/package rename
- Assembly identity changes
- Broader cleanup of item model implementation details

## Risks / Watchouts
- Serialized or reflected type names may include the old namespace.
- XML doc cref references can silently drift if not covered by compile/build validation.
- External consumers outside this repository will see a breaking namespace change unless a compatibility strategy is added later.
- Some docs may intentionally describe historical names; do not rewrite unrelated archival material without reason.

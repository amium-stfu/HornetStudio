# PLAN

## Title
Rename `Amium.Item` namespace to `Amium.Items`

## Goal
Eliminate the awkward `Amium.Item.Item` type reference by renaming the namespace of the item model package from `Amium.Item` to `Amium.Items` while keeping the public type name `Item`.

## Problem Framing
The current combination of namespace `Amium.Item` and type `Item` produces usages such as `Amium.Item.Item` and corresponding type aliases. This reduces readability, creates visual noise in code reviews, and makes the API feel less intentional.

The rename should improve namespace clarity without expanding scope into broader model refactoring. The work is primarily a coordinated namespace migration across all consuming projects, tests, and documentation.

## Scope
Included:
- Rename the namespace in the `Amium.Item` project from `Amium.Item` to `Amium.Items`
- Update all direct consumers to use the new namespace
- Update namespace-qualified aliases such as `ItemModel = Amium.Item.Item`
- Update relevant documentation and examples that reference the old namespace

Excluded:
- Renaming the `Item` class itself
- Changing assembly name, package name, or project folder unless required separately
- Behavioral refactoring inside item model classes
- Broader API redesign around `Item`, `Parameter`, or related helpers

## Architectural Notes
- Keep the domain type name `Item` unchanged to minimize API churn.
- Restrict the change to namespace usage unless a compiler issue forces an adjacent adjustment.
- Preserve existing project boundaries and file layout unless there is a concrete build reason to change them.
- Treat this as an API surface migration with broad usage impact but low expected runtime risk.

## Impact Areas
- `src/Amium.Item/*`
- Broker projects using explicit aliases to `Amium.Item.Item`
- Host, editor, demo, and client projects using `using Amium.Item;`
- Tests and Markdown docs with fully qualified namespace references

## Sequence
1. Update the namespace declarations in the `Amium.Item` project.
2. Update all consumers, aliases, and fully qualified references across `src` and tests.
3. Update documentation that explicitly mentions `Amium.Item.Item`.
4. Run a full solution build and targeted tests to catch missed references.
5. Review for any serialization, reflection, or documentation assumptions that still embed the old namespace string.

## Key Decisions
- Preferred target namespace: `Amium.Items`
- Keep class name: `Item`
- Prefer mechanical migration over conceptual redesign

## Risks
- Reflection or serialized type metadata may still expect `Amium.Item.*` names.
- XML documentation `<see cref="...">` references may break if not updated everywhere.
- Fully qualified aliases may compile in some projects while docs or tests still lag behind.
- External consumers outside this repository may experience a breaking source/API change.

## Verification Strategy
- Solution build after migration
- Search for remaining `Amium.Item.Item` and `namespace Amium.Item` references
- Run relevant broker/editor/host tests that exercise item snapshots and parameter serialization

## Notes
- The project folder `src/Amium.Item` can remain unchanged initially if the goal is limited to namespace cleanup.
- If compatibility for external consumers matters, a temporary compatibility layer would need a separate decision because that expands scope.

# IMPLEMENTATION HANDOFF

## Goal
Ensure CustomSignals created, edited, or deleted through the CustomSignals widget are persisted to the owning page YAML and load correctly after page reload/reopen.

## Scope
Included:
- Persist CustomSignals definition changes from the widget UI.
- Reuse the existing owning page YAML save pattern used by EnhancedSignals.
- Keep the existing CustomSignals runtime publishing behavior.

Not included:
- Redesigning CustomSignals evaluation or registry publication.
- Changing EnhancedSignals behavior.
- Adding new storage formats.

## Starting Point
- `CustomSignalsControl` rebuilds rows from `FolderItemModel.CustomSignalDefinitions`.
- Add/Edit/Delete currently assign `ownerItem.CustomSignalDefinitions` directly.
- `MainWindowViewModel` already serializes and deserializes `CustomSignals`.
- `EnhancedSignalsControl` has `ApplyEnhancedSignalDefinitions(...)`, which persists changes through `TrySaveOwningPageYaml(...)` and `QueueSaveOwningPageYaml(...)`.

## Tasks
1. Add a local apply helper in `CustomSignalsControl`.
2. Route Add/Edit/Delete through that helper.
3. Preserve existing row rebuild and registry publish behavior.
4. Add a focused regression test only if practical in the existing editor test harness.
5. Build the editor project.

## File-Level Changes
- `src/Hornetstudio.Editor/Widgets/CustomSignals/CustomSignalsControl.axaml.cs` -> add `ApplyCustomSignalDefinitions(...)` and use it in Add/Edit/Delete handlers.
- `src/Hornetstudio.Editor.Tests/Program.cs` -> optional focused test if save/persistence can be verified without large UI setup.

## Implementation Order
1. Inspect `EnhancedSignalsControl.ApplyEnhancedSignalDefinitions(...)` and mirror only the persistence pattern needed for CustomSignals.
2. Add `ApplyCustomSignalDefinitions(FolderItemModel ownerItem, string rawDefinitions, bool queuePersist)`.
3. In the helper, set `ownerItem.CustomSignalDefinitions = rawDefinitions`.
4. If `queuePersist` is true and `ViewModel` is available, call `TrySaveOwningPageYaml(ownerItem, out _)`; if it returns false, call `QueueSaveOwningPageYaml(ownerItem)`.
5. Replace direct assignments in `OnAddSignalClicked`, `OnEditSignalClicked`, and `OnDeleteSignalClicked`.
6. Build and perform a manual reload check.

## Technical Constraints
- Use existing project patterns and APIs.
- Do not introduce new dependencies.
- Keep code and user-visible messages in English.
- Avoid broad refactoring.
- Preserve public APIs unless strictly necessary.

## Acceptance Criteria
- Created CustomSignals definitions survive page reload/reopen.
- Edited CustomSignals definitions survive page reload/reopen.
- Deleted CustomSignals definitions stay deleted after page reload/reopen.
- Existing CustomSignals runtime values still publish into `HostRegistries.Data`.
- No unrelated files are changed.

## Verification
- `dotnet build src/Hornetstudio.Editor/Hornetstudio.Editor.csproj --no-restore -v:minimal`
- Manual editor verification:
  1. Open a page with a CustomSignals widget.
  2. Add a custom signal.
  3. Confirm the owning page YAML contains `CustomSignals`.
  4. Reload/reopen the page.
  5. Confirm the custom signal is listed and published.

## Out of Scope
- Changing signal path generation.
- Changing `CustomSignalDefinitionCodec`.
- Changing `ProjectUiLayoutLoader` or document persistence unless implementation reveals a concrete loader defect.
- Runtime formula evaluation changes.

## Risks / Watchouts
- If `ViewModel` is null, the helper cannot persist immediately; this should remain a no-op rather than throwing.
- Avoid duplicate rebuild suppression unless it is demonstrably needed.
- Do not remove transient registry cleanup on widget detach.

## Notes
- The loader and codec already appear to support persisted CustomSignals.
- The likely missing link is explicit persistence after widget UI mutations.

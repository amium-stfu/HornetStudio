# CustomSignals Created Items Not Loaded

## Problem
CustomSignals created through the widget UI are visible during the current editor session but are not reliably loaded after reopening/reloading the page.

## Current Structure
- `CustomSignalsControl` stores definitions in `FolderItemModel.CustomSignalDefinitions`.
- `RebuildSignalRows()` parses `CustomSignalDefinitions` and publishes runtime registry items.
- The YAML/document load and save paths already include `CustomSignals`.
- `EnhancedSignalsControl` explicitly persists definition changes through the owning page save path.
- `CustomSignalsControl` currently updates only the model property in Add/Edit/Delete handlers.

## Scope
- Fix CustomSignals definition persistence after Add/Edit/Delete.
- Keep the existing definition codec, registry publication, and UI behavior.
- Do not redesign CustomSignals runtime evaluation.
- Do not change EnhancedSignals behavior.

## Proposed Design
Add a small CustomSignals-specific apply method that mirrors the EnhancedSignals pattern:

1. Set `ownerItem.CustomSignalDefinitions`.
2. Let the existing property-change flow rebuild rows.
3. Persist the owning page YAML immediately when possible.
4. Queue owning page YAML save as fallback.

## Target Files
- `src/Hornetstudio.Editor/Widgets/CustomSignals/CustomSignalsControl.axaml.cs`
- `src/Hornetstudio.Editor.Tests/Program.cs` if a focused regression test is practical.

## Implementation Sequence
1. Add an `ApplyCustomSignalDefinitions(FolderItemModel ownerItem, string rawDefinitions, bool queuePersist)` helper to `CustomSignalsControl`.
2. Update Add/Edit/Delete handlers to use the helper instead of assigning `ownerItem.CustomSignalDefinitions` directly.
3. Persist through `ViewModel.TrySaveOwningPageYaml(ownerItem, out _)` with fallback to `ViewModel.QueueSaveOwningPageYaml(ownerItem)`.
4. Keep `RebuildSignalRows()` and `PublishSignals(...)` behavior unchanged unless the implementation reveals duplicate rebuilds.
5. Add or update a focused test only if the existing test harness can verify the persistence path without large UI setup.
6. Build the editor project.

## Acceptance Criteria
- A CustomSignals definition created in the widget is written to the owning page YAML.
- A CustomSignals definition edited in the widget is written to the owning page YAML.
- A CustomSignals definition deleted in the widget is removed from the owning page YAML.
- Reopening/reloading the page loads the previously created CustomSignals definitions.
- Existing EnhancedSignals behavior is unchanged.

## Verification
- Build: `dotnet build src/Hornetstudio.Editor/Hornetstudio.Editor.csproj --no-restore -v:minimal`
- Manual check: create a CustomSignals item, reload/reopen the page, and confirm the created signal is still listed.

## Risks
- Saving immediately from Add/Edit/Delete can expose existing save errors that were previously hidden.
- If the owning page cannot be resolved, the queued save fallback must still preserve the definition.
- CustomSignals runtime registry items are transient and should remain derived from persisted definitions.

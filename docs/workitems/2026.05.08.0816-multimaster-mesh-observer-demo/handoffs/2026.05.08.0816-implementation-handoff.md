# IMPLEMENTATION HANDOFF

## Goal
Build a real multimaster observer-sight demo and self-test with three local instances. Each instance owns its own MQTT Item Server/Broker on a separate port, publishes local items to its own server, and connects to every other instance as a remote client. The self-test must verify that all instances can observe all expected values across the mesh.

## Scope
Included:

- Add a mesh-style variant for the existing multimaster demo/self-test.
- Start three independent local `MqttItemServerHost` instances on separate ports.
- Give each node one local publisher session against its own server.
- Give each node remote observer clients to the two peer servers.
- Publish two initial values per node:
  - A: `dyn_1`, `stat_2`
  - B: `dyn_3`, `stat_4`
  - C: `dyn_5`, `stat_6`
- Implement a headless self-test for three phases:
  1. baseline visibility: every node sees all six initial values.
  2. cross_write: nodes write values across peers and every node observes the last written value.
  3. runtime_create: every node adds a new item and every node observes all new items.
- Keep conflict behavior simple: last write wins.
- Add or extend UI support so the user can inspect the same mesh behavior manually.

Explicitly not included:

- Mirroring remote items into each local server.
- Production-grade peer discovery.
- Distributed conflict resolution beyond last write wins.
- Multi-machine deployment.
- Authentication, TLS, or network security.

## Starting Point
- Current `amium_item_server_multimaster_demo` self-test uses one MQTT server on `127.0.0.1:18830` with three logical clients.
- The current self-test passes, but it does not validate the intended topology where each device owns its own server/broker.
- `DemoNodeController` already separates publisher-session writes from observer reads.
- `MqttRemoteItemClient` can observe a single remote broker and hide paths it publishes itself.
- The first mesh implementation should use observer-sight only: a node can see remote values through remote clients, but remote values are not copied into the node's local server.

## Tasks
1. Define a mesh node configuration model inside the demo project with node id, display name, local broker endpoint, base topic, and peer endpoints.
2. Add a mesh node controller that owns one local `MqttItemServerHost`, one local `MqttItemClientSession`, and one read-only `MqttRemoteItemClient` per peer.
3. Publish initial node items:
   - `nodes.server_a.dyn_1`
   - `nodes.server_a.stat_2`
   - `nodes.server_b.dyn_3`
   - `nodes.server_b.stat_4`
   - `nodes.server_c.dyn_5`
   - `nodes.server_c.stat_6`
4. Implement dynamic updates for the `dyn_*` items and keep `stat_*` stable unless the test writes them.
5. Implement observation APIs that combine local server-owned items and peer observer snapshots into one per-node visible state.
6. Implement cross-write routing for the mesh self-test:
   - If the target path belongs to the actor node, write through the actor's local publisher session.
   - If the target path belongs to a peer node, write through a dedicated session/client connected to that peer's broker, not through a read-only observer client.
7. Implement runtime item creation on each node through its local publisher session.
8. Add a headless mesh self-test command or option, for example `--mesh-self-test`, without removing the existing `--self-test`.
9. Write mesh self-test summaries and JSONL logs to a distinct log directory, for example `logs/multimaster_mesh_self_test`.
10. Add a UI path to start and inspect the mesh scenario manually. Prefer extending the existing demo UI only if the change stays small; otherwise add a clearly separated mode/view.
11. Keep the existing single-broker self-test intact as a lower-level integration test.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Use existing project libraries and patterns. Do not add dependencies.
- Keep changes scoped to the demo, self-test, and directly required MQTT client/server usage.
- Observer clients must remain read-only.
- Use normal `MqttItemClientSession` methods for write paths:
  - `PublishSnapshotAsync`
  - `UpdateValueAsync`
  - `UpdatePropertyAsync`
- Do not implement remote-to-local mirroring in this workitem.
- Avoid echo-loop prevention complexity by not mirroring remote values into local brokers.
- Last write wins for conflicts.
- Keep path-relevant newly created folders/files in snake_case where project rules require it. Preserve existing path names.

## Acceptance Criteria
- A headless mesh self-test can be started without launching the UI.
- The mesh self-test starts three separate local MQTT Item Servers on separate ports.
- Each node publishes exactly two initial items matching the requested names.
- Every node observes all six initial items.
- Cross-write cases pass and every node observes the final expected value.
- Each node can create a runtime item and every node observes all runtime-created items.
- Observer clients are not used for writes.
- Existing `--self-test` behavior remains available.
- The UI allows manual inspection of the mesh scenario or a clear reason is documented if UI support is deferred.

## Out of Scope
- Mirroring peer items into every local `MqttItemServerHost`.
- Testing every node's local server as a complete replicated database.
- Peer discovery, config files, or user-editable topology.
- Authentication, TLS, WAN behavior, or real multi-device deployment.
- Complex conflict resolution, vector clocks, ownership leasing, or transaction semantics.

## Risks / Watchouts
- Multiple local brokers can fail to start if ports are already in use; errors must be explicit and cleanup must stop already-started hosts.
- Reusing `MqttRemoteItemClient` for writes would reintroduce hidden observer paths.
- Dynamic values can change during assertions; use stable expected values for cross-write and runtime-create assertions or isolate paths from dynamic update loops.
- Retained MQTT messages can make stale data look valid if base topics or ports are reused incorrectly. Use run-specific values and clear expected paths.
- UI changes can grow quickly; keep the first UI support minimal and inspection-focused.
- If write routing to peer brokers is missing in the current API shape, implement the smallest demo-local writer/session approach rather than adding production abstractions.

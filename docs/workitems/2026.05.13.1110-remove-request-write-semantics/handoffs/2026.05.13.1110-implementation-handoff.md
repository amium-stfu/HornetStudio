# IMPLEMENTATION HANDOFF

## Goal
Remove the old `Request` write semantics and make item write behavior use one canonical rule across SignalWidget, custom/enhanced signals, UDL runtime items, templates, and tests.

Canonical behavior:
- Display prefers the `read` property.
- Display falls back to `Value`, then the first visible user property.
- Write sets the `write` property when it exists.
- Write sets the `read` property when no `write` property exists.
- `Request` child items and `*.request` write paths are no longer part of the write model.

## Scope
Included:
- Remove runtime routing that writes to `Request` child items.
- Remove UDL-related `*.set.request` assumptions from templates and tests.
- Make SignalWidget writes to a target such as `...m310.set` update `Properties["write"]`.
- Keep read/display behavior separate from write behavior.
- Keep UDL runtime schema canonical: lowercase channel items with `read` and optional `write` properties.
- Update focused tests for the canonical write rule.
- Update relevant widget documentation if user-facing behavior or examples change.

Not included:
- Release/version changes.
- NuGet/package publishing.
- Reintroducing compatibility aliases for old `Request` paths.
- Adding new dependencies.
- Broad registry redesign.
- Hardware protocol changes unless verification proves the CAN write frame itself is wrong after the write target fix.

## Starting Point
- The SignalWidget is configured against `.set`, not `.set.request`.
- Expected behavior was clarified:
  - Display reads `read` first.
  - Write uses `write` when present.
  - Write falls back to `read` when `write` is absent.
- Current code still contains old `Request` semantics:
  - `SignalWriteMode.Request`
  - `*.set.request` template paths
  - `EnhancedSignalRuntime` forwarding to `set.request`
  - write-target resolution that can prefer `Request` children
- Current `PageItemModel` write resolution likely maps an item with `Properties["write"]` back to its `read` property through declared write binding resolution.
- `UdlClient` and `HostUdlClient` already expect write requests on channel `Properties["write"]`.
- The observed hardware symptom is consistent with writing the displayed/read value instead of the write property: the ItemTree shows the edit briefly, then the bus readback restores the original value.

## Tasks
1. Audit all `Request`, `.request`, `SignalWriteMode.Request`, `write_path`, and write-target resolution references.
2. Define one shared write-resolution rule in the existing model layer without adding unnecessary abstractions.
3. Fix `PageItemModel` so widget writes to an item with `Properties["write"]` update `write`, while display still prefers `read`.
4. Align interaction write paths in `PageItemModel` with the same rule.
5. Align `CustomSignalsControl` write routing with the canonical property rule.
6. Remove or neutralize `EnhancedSignalRuntime` request-child routing for `set.request` and `Command.Request`.
7. Remove `Request`-mode selection or normalize it to the canonical direct/property-based behavior.
8. Update YAML templates and development examples from `*.set.request` to `*.set`.
9. Update tests that assert `request` child behavior to assert property-based `write` behavior.
10. Verify that UDL write changes reach `HostUdlClient.OnWriteItemChanged(...)` / `UdlClient.OnWriteItemChanged(...)`.
11. Update relevant widget documentation if examples or write semantics are documented.

## Implementation Order
1. Run a focused reference audit for `Request`, `.request`, and `SignalWriteMode.Request`.
2. Fix `PageItemModel` write resolution first, because SignalWidget-to-UDL writeback depends on it.
3. Add or update focused editor tests:
   - Target item has `read` and `write`; display uses `read`; writing updates `write`.
   - Target item has `read` only; writing updates `read`.
   - No `Request` child is created or required.
4. Update custom signal and enhanced signal write routing to use the same rule.
5. Remove `Request` mode from UI selection and defaults; if persisted data requires compatibility, normalize old `Request` to the canonical behavior during load without preserving runtime request routing.
6. Replace YAML/template paths from `*.set.request` to `*.set`.
7. Update host/editor tests that still model `request` children.
8. Inspect `UdlClient` and `HostUdlClient` diagnostics to ensure a changed `write` property still calls `TryWrite(...)`.
9. Update widget documentation if user-visible write configuration changed.
10. Run focused tests/builds, then solution build when restore state allows it.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Do not add new dependencies.
- Avoid overengineering; prefer local changes in existing write-resolution helpers.
- Do not reintroduce compatibility aliases such as `request` children for UDL.
- Keep UDL channel names lowercase: `read`, `set`, `out`, `state`, `alert`.
- Keep UDL values on properties: `read` for actual value, `write` for desired value.
- Avoid broad registry architecture changes unless a focused test proves the existing registry cannot support property writes.

## Acceptance Criteria
- A SignalWidget bound to `...m310.set` displays `...m310.set.Properties["read"]`.
- Editing that SignalWidget updates `...m310.set.Properties["write"]`.
- Editing does not directly overwrite `...m310.set.Properties["read"]` when `write` exists.
- For an item without `write`, editing updates `read`.
- No normal write path depends on `Request`, `request`, `.set.request`, or `.Command.Request`.
- UDL CAN writeback is triggered from `write` property changes.
- Existing templates no longer use UDL `*.set.request` targets.
- Tests cover both `write`-present and `write`-absent behavior.

## Verification
- Run focused editor tests after updating write behavior.
- Run focused host tests after removing `request` child assumptions.
- Run `dotnet build src/HornetStudio.Editor/HornetStudio.Editor.csproj --no-restore`.
- Run `dotnet build HornetStudio.sln --no-restore` when restore state permits.
- Manual hardware check:
  1. Bind SignalWidget target to `...m310.set`.
  2. Confirm displayed value follows `read`.
  3. Edit the widget.
  4. Confirm ItemTree shows `write` changing, not `read`.
  5. Confirm UDL diagnostic logs show `request changed`, `try write`, and `send write pdo`.
  6. Confirm CAN device receives the write and later updates `read` through real bus feedback.

## Out of Scope
- Runtime compatibility for old persisted `*.request` target paths beyond simple load-time normalization if absolutely necessary.
- New request/ack protocol.
- CAN protocol redesign.
- Release publishing.
- Unrelated widget refactoring.

## Risks / Watchouts
- Removing `SignalWriteMode.Request` can be a breaking change for persisted layouts. Prefer deterministic normalization to the canonical model over runtime dual semantics.
- Some non-UDL enhanced signal flows may still use `Command.Request`, `Kalman.Request`, or `Adjustment.Request`. Audit each occurrence before removal; if these are command-input items rather than generic write semantics, rename or isolate them so they do not conflict with the canonical item write model.
- `HostRegistryPropertyPolicy.CanUserWriteProperty("write")` must continue to allow user writes to the `write` property.
- Registry path updates usually target item values. Property writes must call the property update path, not `UpdateValue(...)`.
- `UdlClient` and `HostUdlClient` both exist. The app path likely uses `HostUdlClient`; keep both behaviorally aligned if either is changed.
- Do not solve the issue by writing to both `read` and `write`; that would hide the problem and reintroduce split truth.

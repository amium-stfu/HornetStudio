# DEBUG REPORT

## Problem
Editing a SignalWidget bound to a UDL `.set` item briefly changes the value in the TreeView, then the value returns to the previous bus value.

## Expected Behavior
For a target item with both `read` and `write` properties:
- Display uses `read`.
- Editing updates `write`.
- `read` remains unchanged until real runtime feedback updates it.

## Actual Behavior
The edit updated the displayed/read side of the target. The next CAN readback restored the previous value, making the UI change appear briefly and then disappear.

## Error Messages / Logs
Existing `SignalWrite` diagnostics were sufficient to locate the problem because they log `writeParam`.

Relevant expected log after the fix:
- `writeParam=write`

Problematic log before the fix would show:
- `writeParam=read`

## Relevant Code
- `src/HornetStudio.Editor/Models/PageItemModel.cs`
  - `ResolveWriteParameter()`
  - `ResolveInteractionWriteParameter(...)`

## Attempted Fixes
- Fix 1 -> Reordered write-parameter resolution so a target item with `Properties["write"]` returns the `write` property before declared write bindings can fall back to the value/read parameter. Applied the same correction to interaction writes.
  - Result: Focused editor and host self-tests pass.

## Current Hypothesis
- The root cause was not the CAN writeback loop.
- The root cause was the UI write-resolution layer still resolving `.set` writes to `read` because `TryResolveDeclaredWriteBinding(...)` returns the same item for write-capable targets and the caller then used `ResolveValueParameter(...)`.

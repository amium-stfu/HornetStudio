# IMPLEMENTATION HANDOFF

## Goal
Simplify the `amium_item` production structure to only the stable runtime projects:

- `Amium.Item`
- `Amium.Item.Client`
- `Amium.Item.Server`

Remove `Amium.Item.Mesh` from the production library set because it is currently only a speculative concept/demo layer. Also remove the obsolete `Amium.Item/Server` folder from the core item project by moving still-needed responsibilities into either `Amium.Item.Client` or `Amium.Item.Server`.

## Scope
Included:

1. Remove `amium_item/src/Amium.Item.Mesh` as a production project.
2. Remove `Amium.Item.Mesh` from `amium_item/Amium.Item.sln`.
3. Remove project references to `Amium.Item.Mesh` from samples/tools if any exist.
4. Move client-only server leftovers from `amium_item/src/Amium.Item/Server` into `Amium.Item.Client`.
5. Move server-only health/topic/path helpers from `amium_item/src/Amium.Item/Server` into `Amium.Item.Server`, or replace them with direct `ItemPath` usage where simpler.
6. Remove obsolete generic broker policy code from `Amium.Item.Server` if it is only kept alive by legacy tests.
7. Update tests and documentation to describe the new three-project structure.

Explicitly not included:

- No new `Abstractions`, `Protocol`, or shared helper project.
- No new runtime dependency.
- No feature additions to MQTT publishing, received item reconstruction, health publishing, or periodic publishing.
- No changes to `amium_item/src/Amium.Item/Item.cs`.

## Starting Point
- `Amium.Item` currently contains a `Server/` folder with server/client shared leftovers:
  - `Contracts.cs`
  - `Health.cs`
  - `ItemServerPath.cs`
  - `ItemServerValueCoercion.cs`
  - `Messages.cs`
  - `SubscriptionPolicies.cs`
  - `Mqtt/MqttTopicMapping.cs`
- `Amium.Item.Client` currently references only `Amium.Item`, but imports `Amium.Item.Server` namespaces because those types are compiled into `Amium.Item`.
- `Amium.Item.Server` is now mostly a self-hosted MQTT broker/runtime with health publishing.
- `Amium.Item.Mesh` contains demo/concept multimaster controllers and models, and should not remain a production library unless a concrete product requirement appears later.
- Documentation still mentions `Amium.Item.Server.Abstractions`; this should be removed or corrected.

## Tasks
1. Inventory current references to `Amium.Item.Mesh`, `Amium.Item.Server.*` message types, `ItemServerPath`, `ItemServerValueCoercion`, and `MqttItemTopicMapper`.
2. Remove `Amium.Item.Mesh` from the solution and delete or relocate its code out of `src/` if still needed by sample projects.
3. Decide for each type under `Amium.Item/Server` whether it is client-only, server-only, or obsolete.
4. Move client-only types into `Amium.Item.Client` and update namespaces/usings.
5. Move server-only types into `Amium.Item.Server` and update namespaces/usings.
6. Remove obsolete generic broker policy types and their tests when no longer part of the design.
7. Delete `amium_item/src/Amium.Item/Server`.
8. Update docs to describe only `Amium.Item`, `Amium.Item.Client`, and `Amium.Item.Server`.
9. Build and run focused console self-tests.

## Implementation Order
1. Remove `Amium.Item.Mesh` from the solution and project references, then build to identify remaining sample dependencies.
2. If samples still need Mesh demo logic, move that logic under `amium_item/apps/samples/amium_item_server_multimaster_demo/` instead of keeping a production `src` project.
3. Move or recreate the MQTT topic mapper separately in `Amium.Item.Client` and `Amium.Item.Server` to avoid a server-to-client or client-to-server project dependency.
4. Move client-only subscription/message/clock/value-coercion types into `Amium.Item.Client`.
5. Move health paths and any server-only helper needed by health publishing into `Amium.Item.Server`.
6. Replace `ItemServerPath` usage with `ItemPath.Normalize` where no server-specific behavior is required. If client subscription matching still needs a helper, keep it as an internal/client-owned helper.
7. Remove `Policies.cs` and its legacy tests if the MQTT-only server no longer uses retention/publish policy resolvers.
8. Delete the now-empty `Amium.Item/Server` folder.
9. Update docs and README references.
10. Run verification.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Do not introduce new dependencies.
- Do not create a new `Abstractions` or `Protocol` project.
- Keep `Amium.Item` free of MQTT, server messages, client subscription contracts, and server health concepts.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Preserve public XML documentation for public classes/methods that remain public.
- Prefer existing project patterns and keep the change structural, not feature-oriented.
- Treat namespace changes as breaking changes and call them out in final implementation summary.

## Acceptance Criteria
- `amium_item/src/` contains no `Amium.Item.Mesh` project.
- `amium_item/src/Amium.Item/Server` no longer exists.
- `Amium.Item` builds as a pure item model/path/property project.
- `Amium.Item.Client` owns all client callback message, subscription, topic mapping, clock, and received-value coercion types it uses.
- `Amium.Item.Server` owns only the MQTT host/runtime and server health publishing types it uses.
- No project depends on a removed `Amium.Item.Mesh` library.
- Documentation no longer describes `Amium.Item.Server.Abstractions` as an existing project.
- Existing MQTT client/server behavior remains covered by focused tests.

- `dotnet build HornetStudio.sln --no-restore`

If build artifacts are stale after project removal, run restore/build without `--no-restore` only if necessary and allowed by the current environment.

## Out of Scope
- Designing a future Mesh product module.
- Adding retained-state registry behavior to the server.
- Adding broker-side routing, ownership, or multimaster logic.
- Adding a shared protocol/abstractions assembly.
- Changing runtime MQTT topic format unless required by namespace/type relocation.

## Risks / Watchouts
- Moving public message/subscription types from `Amium.Item.Server` to `Amium.Item.Client` is a breaking API change for current callers.
- Keeping one shared `MqttItemTopicMapper` without a fourth project would force an undesirable dependency direction. Prefer duplicated client/server mapper implementations if needed.
- Deleting Mesh may break sample projects that currently reference `Amium.Item.Mesh`; either remove those sample flows or relocate required demo-only code into the sample project.
- Tests may currently validate obsolete broker policy behavior. Remove or rewrite those tests to match the MQTT-only server direction.
- Documentation and README files may contain old names such as `ItemBroker`, `Server.Abstractions`, or Mesh as a production layer.

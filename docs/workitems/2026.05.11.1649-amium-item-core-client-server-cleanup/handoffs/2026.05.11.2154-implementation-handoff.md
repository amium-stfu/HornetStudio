# IMPLEMENTATION HANDOFF

## Goal
Remove the embedded `amium_item` source tree from the HornetStudio repository and consume `Amium.Item`, `Amium.Item.Client`, and `Amium.Item.Server` as fixed-version internal NuGet packages from the configured `amium-at` package source.

## Scope
Included:

1. Replace any remaining local source/project references to `amium_item` with NuGet `PackageReference` entries.
2. Install only the Amium packages that each consuming project directly needs:
   - `Amium.Item`
   - `Amium.Item.Client`
   - `Amium.Item.Server`
3. Remove the `amium_item` source solution from HornetStudio after the package migration builds.
4. Remove stale `amium_item` solution items and documentation links from `HornetStudio.sln`.
5. Update repository documentation that still describes `amium_item` as embedded source.
6. Update `CHANGELOG.md` for the dependency migration.

Explicitly not included:

- No feature changes to Amium item behavior.
- No API redesign or namespace migration.
- No new third-party dependencies.
- No local absolute NuGet package paths in shared project files.
- No release creation or Git tag creation.

## Starting Point
- `NuGet.Config` already contains the internal package source `amium-at` at `https://nuget.pkg.github.com/amium-at/index.json`.
- The package source was already added locally by the user.
- `HornetStudio.sln` still contains `amium_item` documentation entries under the `docs` solution folder.
- `amium_item/` still exists inside the HornetStudio working tree.
- A quick repository scan did not find direct `Amium.Item` references in `.csproj`, `.sln`, `.props`, `.targets`, `.config`, or Markdown files, but the implementation must re-run this scan before deleting anything.
- The exact package version to pin must be confirmed from the published internal NuGet packages or supplied by the user before adding `PackageReference` entries.

## Tasks
1. Determine the fixed published version for all three Amium packages.
2. Search the full repository, including ignored files if necessary, for `amium_item`, `Amium.Item`, `Amium.Item.Client`, and `Amium.Item.Server`.
3. Identify which HornetStudio projects directly use Amium item model, client, or server APIs.
4. Add `PackageReference` entries only to projects that directly use the matching package.
5. If `Directory.Packages.props` exists or is intentionally introduced later, keep versions centralized there; otherwise keep versions in each changed `.csproj`.
6. Run restore and build before deleting `amium_item`.
7. Remove `amium_item/` only after the package-based build succeeds or after all remaining failures are proven unrelated to the source removal.
8. Remove stale `amium_item` solution items from `HornetStudio.sln`.
9. Update relevant documentation and `CHANGELOG.md`.
10. Run final validation and confirm no local `amium_item` paths remain.

## File-Level Changes
- `NuGet.Config` -> keep `amium-at`; do not add credentials or tokens.
- `*.csproj` under `src/`, `tools/`, `tests/`, or `samples/` -> add only required `PackageReference` entries for `Amium.Item`, `Amium.Item.Client`, and/or `Amium.Item.Server`.
- `Directory.Packages.props` -> use only if it already exists or the repository deliberately adopts central package management in this task.
- `HornetStudio.sln` -> remove stale `amium_item` solution items and any embedded Amium project entries if present.
- `amium_item/**` -> remove from the HornetStudio repository after successful package migration.
- `CHANGELOG.md` -> document the migration from embedded source to internal NuGet packages.
- Relevant README or setup documentation -> document that HornetStudio restores Amium packages from `amium-at`.
- `NOTICE.md` -> review whether the internal runtime packages need to be listed according to the repository notice policy; do not add test/development-only dependencies.

## Implementation Order
1. Confirm the concrete package version to pin, preferably the same timestamp version for all three packages.
2. Run a full reference inventory:
   - `rg -uu "amium_item|Amium\.Item|Amium\.Item\.Client|Amium\.Item\.Server"`
3. Inspect any matching `.csproj` files and determine direct package needs.
4. Add package references with a fixed version.
5. Run `dotnet restore HornetStudio.sln`.
6. Run `dotnet build HornetStudio.sln --no-restore`.
7. Fix only migration-related build failures, such as missing package references or stale paths.
8. Remove `amium_item/` from the repository.
9. Remove stale `amium_item` entries from `HornetStudio.sln`.
10. Update `CHANGELOG.md` and any relevant setup documentation.
11. Run final restore/build and a final reference scan.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Use the existing `amium-at` NuGet source and do not commit credentials.
- Pin exact package versions; do not use floating versions.
- Prefer existing repository patterns and keep changes focused on dependency migration.
- Do not introduce new dependencies beyond the three internal Amium packages.
- Do not modify public APIs unless a package migration build failure makes it unavoidable.
- Do not keep local machine-specific package paths in committed files.

## Acceptance Criteria
- HornetStudio no longer contains or builds the embedded `amium_item` source solution.
- Any project that uses Amium APIs references the correct internal NuGet package.
- All Amium package references use explicit fixed versions.
- `HornetStudio.sln` contains no stale `amium_item` solution item entries.
- Repository documentation no longer instructs developers to build embedded `amium_item` as part of normal HornetStudio builds.
- `CHANGELOG.md` documents the migration.
- No committed file contains a required local absolute package path.

## Verification
- `rg -uu "amium_item|Amium.Item.sln|Amium.Item.csproj|Amium.Item.Client.csproj|Amium.Item.Server.csproj"`
- `dotnet restore HornetStudio.sln`
- `dotnet build HornetStudio.sln --no-restore`
- If package restore fails, verify GitHub Packages credentials outside committed files and retry restore.

## Out of Scope
- Publishing new Amium packages.
- Changing Amium package contents.
- Changing MQTT protocol behavior.
- Creating a HornetStudio release.
- Adding a local file-system NuGet feed as the primary shared integration path.

## Risks / Watchouts
- GitHub Packages restore can fail when the local or CI token does not have package read access.
- If no HornetStudio project currently references Amium APIs, only solution/documentation cleanup and source removal may be required.
- Removing `amium_item/` before the package-based build succeeds can make migration errors harder to diagnose.
- `HornetStudio.sln` currently contains `amium_item` documentation solution items; deleting the folder before updating the solution leaves broken solution entries.
- If CI uses a clean environment, it must configure `amium-at` credentials through secrets, not checked-in config.

## Notes
- The preferred long-term integration is the online internal NuGet feed. Local `.nupkg` files are only a development fallback.
- Use one fixed timestamp version for all three packages when they were released together.
- If the package version is not known, stop and ask for the exact version instead of adding unpinned or guessed references.

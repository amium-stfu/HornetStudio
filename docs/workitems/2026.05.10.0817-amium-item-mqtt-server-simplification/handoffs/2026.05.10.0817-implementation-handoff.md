# IMPLEMENTATION HANDOFF

## Goal
Simplify the MQTT-only item server structure by removing obsolete generic server wrapper concepts, clarifying the MQTT server runtime role, and documenting a separate MQTT payload numeric normalization step where floating point runtime values use at most `float` precision while integer values remain integers.

## Scope
Included:
- Review and remove or rename obsolete server wrapper code around `ItemServerServiceHost`.
- Reduce the old adapter-style surface of `MqttItemServerAdapter`.
- Keep `MqttItemServerHost` as the primary public MQTT server host facade unless direct references require a compatibility path.
- Move server health publishing away from public adapter-style `PublishSnapshotAsync` / `UpdateValueAsync` usage when practical.
- Plan and implement MQTT payload numeric normalization for runtime floating point values with `float` precision.
- Update tests for the changed public/internal surface and numeric formatting behavior.

Explicitly not included:
- Reintroducing a non-MQTT server abstraction.
- Moving shared MQTT topic mapping namespaces unless required by compilation.
- Adding new dependencies.
- Release versioning, NuGet packaging, or Git tagging.

## Starting Point
- `amium_item/src/Amium.Item.Server/Mqtt/IMqttMessagePublisher.cs` was removed in the prior change.
- `MqttItemServerAdapter` now hosts the embedded MQTT server and publishes retained item/health topics directly.
- `MqttItemServerHost` currently owns health publishing but still exposes `Adapter` publicly and calls `Adapter.PublishSnapshotAsync` / `Adapter.UpdateValueAsync`.
- `ItemServerServiceHost` is a thin `RunAsync` wrapper around `MqttItemServerHost` and may be obsolete.
- The current build is blocked by an unrelated accessibility error in `amium_item/src/Amium.Item/Item.cs`.

## Tasks
1. Verify references to `ItemServerServiceHost.RunAsync`.
2. If there are no meaningful references, remove `ItemServerServiceHost.cs`; otherwise rename or relocate it to a clearly MQTT-specific entry-point helper.
3. Rename or reclassify `MqttItemServerAdapter` to reflect its current role as the MQTT server runtime, not a generic adapter.
4. Make the runtime class `internal` if external callers only need `MqttItemServerHost`.
5. Replace public `MqttItemServerHost.Adapter` exposure with targeted host-level members if callers need `ConnectedExternalClientCount`, `VisibleItemCount`, or topic mapping access.
6. Replace health publishing calls to public adapter-style `PublishSnapshotAsync` / `UpdateValueAsync` with internal runtime publish methods or host-owned helper methods.
7. Normalize MQTT payload parsing/formatting so integer payloads remain integer values and floating point runtime payloads use at most `float` precision.
8. Keep system/health metric formatting intentional and test-covered, especially fixed-point health values such as `$SYS/metrics/item_count`.
9. Update focused tests for renamed/kapsulated server runtime and numeric payload behavior.

## File-Level Changes
- `amium_item/src/Amium.Item.Server/ItemServerServiceHost.cs` -> remove if unused, or rename only if a real caller still needs a service-host convenience wrapper.
- `amium_item/src/Amium.Item.Server/Mqtt/MqttItemServerAdapter.cs` -> rename/reclassify to the MQTT server runtime; reduce public surface; keep MQTTnet server lifecycle and retained publish logic.
- `amium_item/src/Amium.Item.Server/Mqtt/MqttItemServerHost.cs` -> become the clear public server facade; own health orchestration without exposing obsolete adapter terminology.
- `amium_item/src/Amium.Item.Server/Mqtt/MqttItemServerOptions.cs` -> remove or mark obsolete legacy options only if they are truly unused and this does not widen the scope too much.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> update tests for host/runtime API and numeric payload contract.
- Relevant docs under `amium_item/docs/` -> update only if public server usage or protocol behavior changes.

## Implementation Order
1. Run reference searches for `ItemServerServiceHost`, `MqttItemServerAdapter`, `.Adapter`, `PublishSnapshotAsync`, and `UpdateValueAsync`.
2. Remove or rename `ItemServerServiceHost` based on actual references.
3. Rename `MqttItemServerAdapter` to the chosen runtime name and update direct internal references.
4. Hide or replace `MqttItemServerHost.Adapter` with host-level methods/properties needed by existing callers.
5. Refactor health publishing to use internal runtime publish methods without exposing old adapter-style APIs publicly.
6. Implement MQTT numeric normalization in the payload layer only.
7. Update tests and documentation.
8. Run focused build/tests, then solution-level build if the known unrelated `Item.cs` accessibility blocker is fixed.

## Technical Constraints
- Answer and documentation planning context is German, but code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Preserve existing MQTTnet usage; do not add dependencies.
- Keep changes minimally fragmented and avoid new abstraction layers unless a concrete caller requires them.
- Prefer `MqttItemServerHost` as the public facade.
- Use named arguments for unclear or multi-argument calls.
- Public classes and public methods must keep XML documentation.
- Do not change unrelated dirty worktree files.

## Acceptance Criteria
- `ItemServerServiceHost` is removed or clearly justified and renamed as MQTT-specific.
- No public API exposes obsolete `Adapter` terminology unless retained deliberately for compatibility.
- `MqttItemServerHost` is the primary public entry point for starting/stopping the MQTT-only server.
- Health publishing still emits the expected retained `$SYS` topics.
- MQTT runtime integer payloads parse as integer values.
- MQTT runtime floating point payloads parse/format with at most `float` precision.
- Existing demos/tests compile against the new server structure or are minimally updated.

## Out of Scope
- Moving `MqttItemTopicMapper` out of `Amium.Item.Server.Mqtt`.
- Changing client-side publishing APIs unless required by server API changes.
- Global changes to `Item` numeric storage semantics.
- Release/version/changelog work unless the implementation changes documented public behavior.

## Risks / Watchouts
- Renaming a public class can be a breaking API change for samples, tools, or host projects.
- Making the runtime internal may break tests that currently instantiate it directly; prefer testing through `MqttItemServerHost` where possible.
- Float formatting can accidentally remove meaningful integer/floating distinction; define expected examples in tests.
- Health metrics currently use fixed-point formatting and should not be accidentally converted to generic runtime float formatting.

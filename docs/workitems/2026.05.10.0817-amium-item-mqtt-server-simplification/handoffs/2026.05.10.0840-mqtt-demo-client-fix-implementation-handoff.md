# IMPLEMENTATION HANDOFF

## Goal
Fix the MQTT demo client structure so the sample works with the intended topic model: no global MQTT base topic, and each client/demo publishes its own root context through the item path.

## Scope
Included:
- Keep `BaseTopic = ""` for the CLI demo and the selfhosted server host.
- Make the CLI demo publish under a clear client-owned item path instead of relying on a global MQTT base topic.
- Align sample output text and relevant documentation with the no-base-topic model.
- Keep the fix minimal and avoid redesigning the demo.

Explicitly not included:
- Replacing the CLI demo with a new standalone application.
- Introducing new dependencies.
- Changing `Item.cs`.
- Changing the core MQTT topic mapper unless a focused bug is discovered during verification.
- Release versioning, NuGet packaging, Git tagging, or Notice file updates.

## Starting Point
- `amium_item/samples/Amium.Item.Client.MqttDemoClient` builds successfully.
- The CLI demo is a publisher only; it requires an MQTT host/broker on `127.0.0.1:1883`.
- `amium_item/tools/amium_item_server_host` starts `MqttItemServerHost` with `BaseTopic = string.Empty`.
- The CLI demo currently uses `BaseTopic = ""`, which is correct and intentional.
- The WinForms demo currently uses `BaseTopic = "hornet"` and therefore demonstrates a different model.
- Existing docs still mention `hornet/...` in places and may describe the old/prefixed demo behavior.

## Tasks
1. Verify the current CLI demo behavior against `amium_item_server_host`.
2. Update the CLI demo item paths so the client/demo root context is explicit in the item path.
3. Keep `MqttItemClientOptions.BaseTopic` empty in the CLI demo.
4. Improve CLI startup/status text so it states that an external/selfhosted MQTT host must already be running.
5. Review whether the WinForms demo should be changed to `BaseTopic = ""` or explicitly left as a separate prefixed demo.
6. Update documentation that describes the CLI demo topics and host setup.
7. Run focused build verification for the affected sample and, if practical, `amium_item/Amium.Item.sln`.

## File-Level Changes
- `amium_item/samples/Amium.Item.Client.MqttDemoClient/Program.cs` -> keep empty base topic, use explicit client-owned item paths, update console text.
- `amium_item/docs/amium-itembroker-usage.md` -> update demo instructions and topic examples so they match the no-base-topic model.
- `amium_item/docs/amium-item-communication-structure.md` -> update sample/demo topic wording only if it still implies a required global `hornet` prefix for this path.
- `amium_item/samples/Amium.Item.Client.MqttDemoWinForms/MainForm.cs` -> change only if the implementation decision is to align the WinForms demo with the no-base-topic model.
- `amium_item/tools/amium_item_server_host/Program.cs` -> change only if text needs clarification; do not change its `BaseTopic = string.Empty`.

## Implementation Order
1. Run the CLI demo build to confirm the baseline still compiles.

3. Change the CLI demo item root paths to an explicit demo/client context, for example `DemoClient.Sensors.Temperature` and `DemoClient.Sensors.Pressure`.
4. Update console text to make the prerequisite host clear and to print the effective topic/path model.
5. Decide the WinForms sample alignment:
   - Preferred minimal fix: leave WinForms unchanged if it is intentionally a separate prefixed demo.
   - If consistency is required: change its server, publisher, writable controller, and topic display mapper to `BaseTopic = ""`.
6. Update the docs to remove stale claims that the CLI demo publishes under `hornet/...`.
7. Run focused builds and report any unrelated blockers separately.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Prefer existing MQTT client/server APIs and project patterns.
- Do not introduce new dependencies.
- Use named arguments where calls have unclear or multiple similar parameters.
- Keep the demo fix narrowly scoped; do not refactor the MQTT runtime or mapper as part of this task unless required by a confirmed bug.

## Acceptance Criteria
- The CLI demo still builds.
- The CLI demo uses `BaseTopic = ""`.
- The CLI demo publishes topics whose root comes from the item path itself.
- The sample instructions no longer imply that a global `hornet` base topic is required for the CLI demo.
- The host tool still starts with `BaseTopic = string.Empty`.
- Any remaining difference between CLI and WinForms demos is either removed or documented as intentional.

## Out of Scope
- Adding CLI argument parsing unless explicitly requested.
- Adding an embedded host to the CLI demo.
- Changing runtime retained-topic behavior.
- Changing health topic behavior under `$SYS`.
- Updating release metadata.

## Risks / Watchouts
- Changing WinForms from `hornet` to empty base topic could affect screenshots, labels, and manual expectations.
- Docs contain older `hornet/...` examples that may still be valid for other scenarios; update only demo-specific statements unless the general architecture decision is broader.
- Empty base topic means unrelated MQTT topics can be visible to broad subscriptions; clients should rely on explicit item path roots to avoid collisions.
- Retained stale MQTT topics from old runs can make manual verification misleading; use a clean broker session or clear retained topics when checking topic output.

## Notes
- The implementation should treat empty `BaseTopic` as a required behavior, not a fallback.
- The item path is the client/demo namespace. The MQTT base topic should not be used to create that namespace in this demo.
- If a future all-in-one demo is desired, create it as a separate sample rather than changing this client publisher sample into a mixed host/client application.

# DEBUG REPORT

## Problem
MQTT health topics are one level deeper than needed and timestamp payloads include fractional seconds.

## Expected Behavior
MQTT-specific health values should be published under a flatter `$SYS/mqtt/*` topic layout.

Timestamp values should use ISO-style UTC timestamps without fractional seconds.

## Actual Behavior
MQTT-specific health values are published under `$SYS/mqtt/status/*`.

Core timestamps are formatted with `DateTimeOffset.ToString("O")`, which includes fractional seconds, for example `2026-05-11T08:56:32.0820199+00:00`.

## Error Messages / Logs
No runtime exception. The issue is visible in the MQTT topic tree.

## Relevant Code
- `amium_item/src/Amium.Item/Server/Health.cs`
  - `ItemServerHealthPaths.MqttStatusState = "sys.mqtt.status.state"`
  - `ItemServerHealthPaths.MqttStatusClientCount = "sys.mqtt.status.client_count"`
  - `ItemServerHealthPaths.MqttStatusEndpoint = "sys.mqtt.status.endpoint"`
  - `ItemServerHealthPaths.MqttStatusLastError = "sys.mqtt.status.last_error"`
- `amium_item/src/Amium.Item/Server/Mqtt/MqttTopicMapping.cs`
  - maps these paths to `$SYS/mqtt/status/*`
- `amium_item/src/Amium.Item.Server/ItemServerHealthPublisher.cs`
  - uses `startedAt.ToString("O")`
  - uses `now.ToString("O")`

## Attempted Fixes
- None yet.

## Current Hypothesis
- The extra `status` segment is contract-level path complexity, not an MQTT viewer artifact.
- Timestamp precision is too high because the roundtrip format intentionally preserves fractional ticks.

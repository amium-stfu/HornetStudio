# IMPLEMENTATION HANDOFF

## Goal
Reduce the item server health payload to a smaller, more useful core set while keeping the existing architecture and MQTT system topic model intact.

## Scope
Included:
- Reduce periodic core health publishing in `Amium.Item.Server`.
- Keep health publishing enabled through the existing `ItemServerHealthOptions` and `MqttItemServerOptions` flow.
- Keep MQTT-specific health values separate from core runtime health values.
- Update tests and documentation when affected by the changed standard health contract.

Not included:
- No new runtime dependencies.
- No new health abstraction layer.
- No changes to `amium_item/src/Amium.Item/Item.cs`.
- No release versioning or publish work.
- No unrelated MQTT client or demo client changes.

## Starting Point
- `ItemServerHealthPublisher` currently publishes eight core health values every publish interval:
  - `sys.status.state`
  - `sys.status.uptime_seconds`
  - `sys.status.started_at_utc`
  - `sys.status.last_updated_utc`
  - `sys.metrics.item_count`
  - `sys.metrics.memory_working_set_mb`
  - `sys.metrics.memory_managed_heap_mb`
  - `sys.metrics.cpu_usage_percent`
- `MqttItemServerHost` publishes MQTT-specific health separately:
  - `sys.mqtt.status.state`
  - `sys.mqtt.status.client_count`
  - `sys.mqtt.status.endpoint`
  - `sys.mqtt.status.last_error`
- `ItemServerHealthPaths` and `MqttItemTopicMapper` define the public standard paths and `$SYS` mappings.
- Health retention is path-root based under `sys`, so removing individual health values does not require retention policy changes.

## Tasks
1. Keep the recommended core runtime health set:
   - `sys.status.state`
   - `sys.status.started_at_utc`
   - `sys.status.uptime_seconds`
   - `sys.status.last_updated_utc`
   - `sys.metrics.memory_working_set_mb`
   - `sys.metrics.cpu_usage_percent`
2. Stop publishing these core values:
   - `sys.metrics.memory_managed_heap_mb`
   - `sys.metrics.item_count`
3. Publish static values only when needed:
   - `sys.status.started_at_utc` should be part of the startup snapshot.
   - Avoid periodically republishing `started_at_utc` unless the existing snapshot/update distinction requires it for client consistency.
4. Keep cyclic values periodic:
   - `state`
   - `uptime_seconds`
   - `last_updated_utc`
   - `memory_working_set_mb`
   - `cpu_usage_percent`
5. Keep MQTT health behavior unchanged unless tests reveal it depends on removed core values.
6. Update or remove affected tests for removed values and expected `$SYS` mappings.
7. Update Markdown documentation if any existing docs describe the old health value list.

## File-Level Changes
- `amium_item/src/Amium.Item.Server/ItemServerHealthPublisher.cs` -> reduce snapshot/update publishing and remove now-unused retained item count provider usage if no longer needed.
- `amium_item/src/Amium.Item/Server/Health.cs` -> update the standard health path contract and `ItemServerHealthSnapshot` shape if this type is still intended to mirror the published core values.
- `amium_item/src/Amium.Item/Server/Mqtt/MqttTopicMapping.cs` -> remove `$SYS` mappings for values no longer published if the public path constants are removed.
- `amium_item/src/Amium.Item.Server/Mqtt/MqttItemServerHost.cs` -> remove `retainedItemCountProvider` wiring only if `ItemServerHealthPublisher` no longer accepts or uses it.
- `amium_item/tests/Amium.Item.Server.Tests/Program.cs` -> update tests that expect removed paths or mappings.
- Relevant Markdown docs under `amium_item/` or repository docs -> update only if they mention the removed health values.

## Implementation Order
1. Adjust `ItemServerHealthPublisher` to publish the reduced set and separate startup-only values from periodic values.
2. Remove unused constructor parameters/fields only if they become dead code after the reduced publishing set.
3. Align `ItemServerHealthPaths`, `ItemServerHealthSnapshot`, and MQTT topic mappings with the final supported health contract.
4. Update tests for the reduced health contract.
5. Search for old path names and update docs or references that would become misleading.
6. Run focused build/tests for `amium_item`.

## Technical Constraints
- Use existing classes and patterns; do not introduce a new health service or metrics provider.
- Keep code, comments, XML documentation, UI text, file names, and technical identifiers in English.
- Chat responses must remain German.
- Prefer named parameters for method calls with ambiguous or multiple parameters.
- Public classes and public members need XML documentation updates when signatures or behavior change.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Avoid breaking public APIs more than necessary. If constants or record properties are removed, call it out as a breaking contract change in the implementation summary.

## Acceptance Criteria
- Core health no longer publishes `memory_managed_heap_mb`.
- Core health no longer publishes `item_count`.
- Core health still publishes state, started timestamp, uptime, last update timestamp, working set memory, and CPU usage.
- `started_at_utc` is not redundantly updated on every periodic health tick unless a deliberate compatibility reason is documented.
- MQTT-specific health values remain unchanged.
- Tests reflect the reduced standard health contract.
- No stale references to removed health values remain in code or relevant documentation.

## Verification
- Run `dotnet build amium_item/Amium.Item.sln --no-restore`.
- Run the focused console self-test if the build output exists:
  - `dotnet artifacts/amium_item/debug/Amium.Item.Server.Tests/Amium.Item.Server.Tests.dll`
- If artifacts use a different configuration/path, locate the built `Amium.Item.Server.Tests.dll` and run it directly.

## Out of Scope
- Changing health publish interval defaults.
- Adding configurable health field selection.
- Adding multi-rate publishing for CPU/memory.
- Introducing OpenTelemetry, diagnostics counters, or external metrics dependencies.
- Changing MQTT client behavior.

## Risks / Watchouts
- `ItemServerHealthPaths` and `$SYS` topic mappings are a public-ish contract. Removing constants or mappings may break external clients that already read those values.
- If compatibility matters, prefer stopping publication first while leaving constants/mappings temporarily available and mark them obsolete in a later cleanup.
- `ItemServerHealthSnapshot` may need either reduction or compatibility handling depending on whether it is consumed externally.
- Removing `item_count` means `MqttItemServerHost.VisibleItemCount` remains available on the host API but is no longer exposed as core health.

## Notes
- Recommended final core values intentionally keep `last_updated_utc` because it is useful as a low-cost liveness signal.
- `uptime_seconds` is redundant with `started_at_utc`, but useful for simple clients and dashboards that should not calculate duration themselves.
- `memory_working_set_mb` is preferred over managed heap because it better represents process memory as observed by the operating system.

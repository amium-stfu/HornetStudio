# IMPLEMENTATION HANDOFF

## Goal
Restructure the `UdlClient` widget UI so it mirrors the broker-style split layout with `Received Items` and `Attached Items`, using compact one-line rows and explicit per-row `Attach` / `Detach` actions.

## Scope
Included:
- `UdlClientControl` header/body UI restructuring
- compact list row rendering for received and attached items
- reuse of existing attach selection and synchronization logic

Not included:
- transport/runtime generation changes
- attach serialization changes
- broker widget changes
- broader editor theming refactors

## Starting Point
- The current `UdlClient` body in `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml` renders module cards.
- Existing attach-related logic already exists in `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs`, especially around:
  - attach option discovery
  - `AttachRows`
  - `SynchronizeAttachedItems()`
  - runtime item enumeration
- The broker widget in `src/HornetStudio.Editor/Widgets/Broker/BrokerClientControl.axaml` is the visual reference for compact row sections.

## Tasks
1. Replace the current module-focused body layout with two equal-height sections named `Received Items` and `Attached Items`.
2. Add row view models and rebuild methods for compact received-item rows and compact attached-item rows.
3. Wire `Attach` on received rows to the existing `AttachToUi` selection flow.
4. Wire `Detach` on attached rows to remove that path from `AttachToUi`.
5. Refresh both sections whenever runtime items or attached selections change.
6. Keep tooltips and status indicators meaningful and theme-safe.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml` -> replace module-card body with broker-style split sections and compact row templates
- `src/HornetStudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs` -> add row models/rebuild logic, attach/detach handlers, row status/tooltip mapping, and header action wiring if needed
- `src/HornetStudio.Editor.Tests/Program.cs` -> add or update focused tests for row-source mapping and attach/detach selection behavior if the existing test style supports it

## Implementation Order
1. Inspect the current broker compact row structure and mirror only the relevant UI pattern in `UdlClientControl.axaml`.
2. Introduce received-item and attached-item row data structures in `UdlClientControl.axaml.cs`.
3. Build `Received Items` from attach-capable runtime items.
4. Build `Attached Items` from selected `AttachToUi` entries and resolve live status from runtime items.
5. Add `Attach` and `Detach` click handlers that update `Item.UdlAttachedItemPaths`.
6. Trigger row rebuilds from existing refresh points that already react to runtime and attach changes.
7. Run build/tests relevant to `HornetStudio.Editor`.

## Technical Constraints
- Keep code, comments, and identifiers in English.
- Keep chat/documentation language separate from code language.
- Use existing project patterns and avoid introducing new abstractions unless clearly needed.
- Preserve current `AttachToUi` persistence and synchronization behavior.
- Keep changes scoped to the `UdlClient` widget and directly related tests only.

## Acceptance Criteria
- The `UdlClient` widget shows two sections: `Received Items` and `Attached Items`.
- Each section uses compact single-line rows with a left status dot and tooltip.
- Each received row has an `Attach` button.
- Each attached row has a `Detach` button.
- Clicking `Attach` adds the path to `AttachToUi`.
- Clicking `Detach` removes the path from `AttachToUi`.
- Existing mirrored UI attachment behavior still works after selection changes.

## Verification
- Build the relevant editor/test project successfully.
- Manually verify in the editor that:
  - received runtime paths appear in `Received Items`
  - attached selections appear in `Attached Items`
  - `Attach` moves or adds items into the attached configuration
  - `Detach` removes them from the attached configuration
  - status dots update when runtime availability changes

## Out of Scope
- Reintroducing the old module-card body as an additional section
- Changes to UDL diagnostics, transport, or host plugin behavior
- Changing naming or semantics of persisted attach paths

## Risks / Watchouts
- Avoid duplicating the same path in both sections with conflicting status text.
- Ensure row rebuilds happen on the UI thread where required.
- If there are hidden dependencies on the old module list being visible, document them before removing that presentation.

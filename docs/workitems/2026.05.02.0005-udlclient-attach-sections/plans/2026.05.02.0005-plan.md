# PLAN

## Topic
`UdlClient` body restructuring to mirror the broker-style two-section list layout with explicit `Received Items` and `Attached Items`.

## Problem
The current `UdlClient` widget body is centered around module cards. That makes `AttachToUi` hard to understand at a glance because available runtime items and already attached UI items are not separated visually. The desired interaction is closer to the `BrokerWidget`: one section that shows currently available items and one section that shows the selected attached items.

## Goals
- Replace the current module-card-focused body with two broker-style list sections.
- Show `Received Items` as compact one-line rows.
- Add an `Attach` action on each `Received Items` row.
- Show `Attached Items` as compact one-line rows based on `AttachToUi`.
- Add a `Detach` action on each `Attached Items` row.
- Use a left traffic-light indicator and row tooltip in both sections.
- Keep existing `AttachToUi` behavior and synchronization logic intact.

## Scope
Included:
- `UdlClientControl` body layout in AXAML
- `UdlClientControl` header action placement for attach-related interaction
- New compact row presentation for received and attached item lists
- Reuse of existing attach option and attach synchronization logic

Excluded:
- Changes to UDL transport, connection handling, or runtime item generation
- Changes to broker widgets
- New attach dialog behavior beyond wiring existing logic into the new UI
- Changes to persisted `AttachToUi` serialization format

## Current State
- `UdlClientControl` currently renders module entries as bordered cards in `src/Hornetstudio.Editor/Widgets/UdlClient/UdlClientControl.axaml`.
- The widget already maintains attach option discovery and `AttachToUi` synchronization in `src/Hornetstudio.Editor/Widgets/UdlClient/UdlClientControl.axaml.cs`.
- `AttachRows` already represent configured attach selections, but they are not presented as a dedicated broker-style section.
- The broker widget already provides the target interaction model for compact rows, row tooltip usage, and split body sections.

## Target Structure
- Header:
  - Keep compact status actions.
  - Add or retain an explicit attach-oriented action in the widget header.
- Body:
  - Top summary remains only if still useful and visually small.
  - Two equal-height sections:
    - `Received Items`
    - `Attached Items`
- `Received Items` row format:
  - left status dot
  - single-line runtime path text
  - tooltip with full context
  - right-side `Attach` button
- `Attached Items` row format:
  - left status dot
  - single-line attached path text
  - tooltip with full context
  - right-side `Detach` button

## Design Decisions
1. Separate available runtime data from selected UI attachments instead of mixing both into module cards.
2. Use the same visual list grammar in both sections to reduce cognitive load.
3. Keep `AttachToUi` as the single source of truth for attached selections.
4. Derive row status from current runtime resolvability instead of introducing new persisted state.
5. Treat this as a UI restructuring, not as a business logic rewrite.

## Data Mapping
- `Received Items` should be built from current UDL runtime items that are valid attach candidates.
- `Attached Items` should be built from the currently selected `AttachToUi` entries.
- Attached-row status should reflect whether the selected path still resolves to a runtime item.
- Tooltips should carry the detail that no longer fits into the one-line compact row.

## Sequencing
1. Replace the module-card body structure with two compact section containers in AXAML.
2. Introduce dedicated row view models for received and attached item presentation.
3. Reuse existing attach selection methods for `Attach` and `Detach` actions.
4. Refresh row rebuilding and status updates when runtime items or attached paths change.
5. Verify empty states, tooltip content, and attach/detach interaction.

## Risks
- If runtime-to-row mapping is inconsistent, `Received Items` and `Attached Items` may drift apart visually.
- If attached-row status is not refreshed on runtime changes, the traffic-light state can become stale.
- If the old module list is removed without preserving necessary signals elsewhere, some UDL diagnostics may become less discoverable.

## Acceptance Direction
- `UdlClient` renders two broker-style list sections: `Received Items` and `Attached Items`.
- `Received Items` exposes per-row `Attach`.
- `Attached Items` exposes per-row `Detach`.
- Both sections use a compact single-line row with status dot and tooltip.
- Existing `AttachToUi` persistence and synchronization continue to work.

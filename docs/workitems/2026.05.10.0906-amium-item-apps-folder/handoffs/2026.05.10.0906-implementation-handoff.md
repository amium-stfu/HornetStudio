# IMPLEMENTATION HANDOFF

## Goal
Restructure runnable non-test projects in `amium_item/` so demo, tool, and host applications live below one clear parent folder while production libraries and tests stay separate.

Target layout:

```text
amium_item/
  src/
  tests/
  apps/
    samples/
    tools/
    hosts/
  docs/
```

## Scope
- Move runnable non-test projects currently under `amium_item/samples/` and `amium_item/tools/` into `amium_item/apps/`.
- Keep test projects under `amium_item/tests/`.
- Keep production libraries under `amium_item/src/`.
- Update solution entries, project references, documentation, and relevant command examples that point to the moved paths.
- Do not change runtime behavior, public APIs, namespaces, assembly names, package metadata, UI behavior, or demo logic.

## Starting Point
- `amium_item/` currently contains `src/`, `tests/`, `samples/`, `tools/`, and `docs/`.
- `amium_item/Amium.Item.sln` references projects under `samples/` and `tools/`.
- Documentation contains runnable command examples that point to `amium_item/samples/...` and `amium_item/tools/...`.
- Tests are executable projects too, but they are validation assets and must remain under `tests/`.

## Tasks
1. Create `amium_item/apps/` with purpose folders `samples/`, `tools/`, and `hosts/`.
2. Move sample/demo projects into `amium_item/apps/samples/`.
3. Move stress/test utility applications into `amium_item/apps/tools/`.
4. Move the server host executable into `amium_item/apps/hosts/`.
5. Update `amium_item/Amium.Item.sln` project paths and solution folder grouping.
6. Update relative project references inside moved `.csproj` files.
7. Update relevant Markdown documentation and runnable command examples.
8. Search for stale path references and update only current user-facing or build-relevant references.
9. Build and run focused validation.

## File-Level Changes
- `amium_item/Amium.Item.sln` -> update project paths and solution folders to match `apps/samples`, `apps/tools`, and `apps/hosts`.
- `amium_item/samples/Amium.Item.Client.MqttDemoClient/` -> move to `amium_item/apps/samples/Amium.Item.Client.MqttDemoClient/`.
- `amium_item/samples/Amium.Item.Client.MqttDemoWinForms/` -> move to `amium_item/apps/samples/Amium.Item.Client.MqttDemoWinForms/`.
- `amium_item/samples/amium_item_server_multimaster_demo/` -> move to `amium_item/apps/samples/amium_item_server_multimaster_demo/`.
- `amium_item/tools/amium_itembroker_mqtt_stress_test/` -> move to `amium_item/apps/tools/amium_itembroker_mqtt_stress_test/`.
- `amium_item/tools/amium_item_server_host/` -> move to `amium_item/apps/hosts/amium_item_server_host/`.
- `amium_item/docs/amium-itembroker-usage.md` -> update `dotnet run --project` examples.
- `amium_item/docs/amium-item-communication-structure.md` -> update documented repository layout and references to the moved demos/tools.
- Moved `.csproj` files -> update project references from old relative paths to the unchanged `amium_item/src/...` projects.

## Implementation Order
1. Audit all current references with `rg` for `amium_item/samples`, `amium_item/tools`, `samples\`, `tools\`, and the specific project folder names.
2. Move directories into the new `amium_item/apps/` layout.
3. Update `.csproj` relative `ProjectReference` paths after the move.
4. Update `amium_item/Amium.Item.sln` paths and solution folder nesting.
5. Update current documentation in `amium_item/docs/`.
6. Run stale-reference search again and fix build-relevant leftovers.
7. Verify with build commands and focused executable/test commands where practical.

## Technical Constraints
- Follow existing repository rules: chat in German; code, comments, file names, technical identifiers, and UI text in English.
- Do not introduce new dependencies.
- Do not rename projects, assemblies, namespaces, or classes as part of this structural move.
- Do not move `amium_item/tests/Amium.Item.Server.Tests`.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.
- Prefer minimal, mechanical path changes over refactoring.
- Preserve existing casing and snake_case folder names where already established.

## Acceptance Criteria
- `amium_item/apps/samples/` contains all sample/demo application projects previously under `amium_item/samples/`.
- `amium_item/apps/tools/` contains `amium_itembroker_mqtt_stress_test`.
- `amium_item/apps/hosts/` contains `amium_item_server_host`.
- `amium_item/samples/` and `amium_item/tools/` are removed if empty.
- `amium_item/tests/` remains unchanged as the test-project root.
- `amium_item/Amium.Item.sln` loads and builds the moved projects from their new paths.
- Current Amium.Item documentation no longer instructs users to run projects from the old `samples/` or `tools/` locations.

## Verification
- From `amium_item/`: `dotnet build Amium.Item.sln --no-restore`
- If the solution build fails because restore artifacts are missing, run the same command without `--no-restore` only if dependency restore is acceptable in the environment.
- Run stale reference searches after the move:
  - `rg -n "amium_item/samples|amium_item/tools|samples\\\\|tools\\\\" amium_item docs`
  - `rg -n "samples/|tools/" amium_item/docs`
- Focused self-test, if build artifacts are available: run the `Amium.Item.Server.Tests` executable from its built output as required by `agents/solution.md`.

## Out of Scope
- Renaming assemblies, namespaces, project names, or executable names.
- Changing demo behavior, host arguments, MQTT behavior, server/client APIs, or tests.
- Moving HornetStudio-owned tools outside `amium_item/`.
- Updating historical workitem handoffs unless a stale path is actively misleading for current execution.
- Creating release versions, tags, packages, or notice files.

## Risks / Watchouts
- Moved projects likely contain relative `ProjectReference` paths that must gain one extra `../` level.
- `.sln` solution folder hierarchy may contain old virtual folders that need careful cleanup.
- Documentation has both current docs and historical workitem docs; update current user-facing docs first and avoid noisy churn in old handoffs unless needed.
- Build output paths may change because project directory depth changes; do not compensate with custom output paths unless a real build or documentation issue appears.
- Open IDE tabs may point to old file paths after the move.

## Notes
- Use `apps` rather than `executables` because it describes the role of these projects without pulling test executables into the same category.
- Keep the conceptual split: `src` is reusable production code, `tests` is validation, `apps` is runnable non-test applications, and `docs` is documentation.

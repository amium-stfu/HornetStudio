# IMPLEMENTATION HANDOFF

## Goal
Remove `IItemClientSession` completely from `Amium.Item.Client` so the client SDK keeps only the currently required MQTT-specific API surface.

## Scope
Included:
- Delete the public `IItemClientSession` interface.
- Update `MqttItemClientSession` so it no longer implements the removed interface.
- Keep all existing `MqttItemClientSession` public methods that are currently used by Mesh, samples, tools, and tests.
- Keep `MqttRemoteItemClient` as the normal application-facing MQTT client facade.
- Update project files, docs, tests, samples, and tools only where they reference `IItemClientSession`.
- Verify that no compile-time or documentation references to `IItemClientSession` remain, except historical workitem notes if intentionally left untouched.

Not included:
- Do not remove `MqttItemClientSession`.
- Do not redesign `MqttRemoteItemClient`.
- Do not introduce a new transport abstraction.
- Do not introduce a replacement interface.
- Do not split MQTT code into a separate project.
- Do not change MQTT topic semantics, retained-state behavior, self-published path hiding, write behavior, or subscription behavior.

## Starting Point
- `IItemClientSession` exists at `amium_item/src/Amium.Item.Client/IItemClientSession.cs`.
- `MqttItemClientSession` is the only implementation.
- Current production and test code primarily uses `MqttRemoteItemClient` or `MqttItemClientSession` directly.
- The project direction is to keep MQTT as the concrete transport for now and avoid speculative transport-neutral APIs.
- Existing docs already position `MqttRemoteItemClient` as the primary facade and `MqttItemClientSession` as a lower-level MQTT building block.

## Tasks
1. Run a fresh reference audit for `IItemClientSession`.
2. Remove `IItemClientSession.cs` from `Amium.Item.Client`.
3. Remove `IItemClientSession` from the `MqttItemClientSession` base list.
4. Replace `<inheritdoc />` documentation on `MqttItemClientSession` members that inherited from `IItemClientSession` with explicit XML documentation where needed.
5. Check whether `Amium.Item.Client.csproj` or generated compatibility metadata references the removed file and update only if required.
6. Update non-historical documentation if it presents `IItemClientSession` as a current API.
7. Re-run reference audit and ensure remaining matches are either gone or only historical workitem records.
8. Build the relevant solution.

## File-Level Changes
- `amium_item/src/Amium.Item.Client/IItemClientSession.cs` -> Delete.
- `amium_item/src/Amium.Item.Client/Mqtt/MqttItemClientSession.cs` -> Remove interface implementation and replace inherited XML documentation with explicit XML documentation.
- `amium_item/src/Amium.Item.Client/Amium.Item.Client.csproj` -> Update only if the deleted file is explicitly included.
- `amium_item/docs/amium-itembroker-architecture.md` -> Update only if it mentions `IItemClientSession` as current API.
- `amium_item/docs/amium-itembroker-usage.md` -> Update only if it mentions `IItemClientSession` as current API.
- `docs/workitems/**` -> Do not rewrite old historical handoffs unless a current active handoff would mislead the implementation.

## Implementation Order
1. Run `rg -n "IItemClientSession" amium_item src docs --glob "!**/bin/**" --glob "!**/obj/**"`.
2. Delete `amium_item/src/Amium.Item.Client/IItemClientSession.cs`.
3. Change `MqttItemClientSession` from `public sealed class MqttItemClientSession : IItemClientSession, IAsyncDisposable` to a concrete MQTT session that only implements `IAsyncDisposable`.
4. Replace every affected `<inheritdoc />` in `MqttItemClientSession` with explicit XML summaries, parameters, and return documentation.
5. Build `amium_item/Amium.Item.sln --no-restore`.
6. If the build exposes downstream compile failures, update only direct `IItemClientSession` references to use `MqttItemClientSession` or `MqttRemoteItemClient`, whichever is already the local pattern.
7. Re-run the reference audit.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Do not add dependencies.
- Prefer existing MQTT-specific types over speculative abstractions.
- Keep public API changes scoped to removing `IItemClientSession`.
- Public classes and public methods must keep valid XML documentation.
- Do not modify `amium_item/src/Amium.Item/Item.cs`.

## Acceptance Criteria
- `IItemClientSession.cs` no longer exists.
- `MqttItemClientSession` no longer implements `IItemClientSession`.
- No active source file depends on `IItemClientSession`.
- `MqttRemoteItemClient` remains the normal application-facing MQTT facade.
- `MqttItemClientSession` remains available as a concrete lower-level MQTT session.
- Existing MQTT publish, update, write, subscribe, reconnect, and remote registry behavior is unchanged.
- Any remaining `IItemClientSession` references are limited to historical workitem notes or are removed.

## Verification
- `rg -n "IItemClientSession" amium_item src docs --glob "!**/bin/**" --glob "!**/obj/**"`
- `dotnet build amium_item/Amium.Item.sln --no-restore`
- If the broader repository is affected, run `dotnet build HornetStudio.sln --no-restore` after the `amium_item` build succeeds.

## Out of Scope
- New client base class.
- New `IItemClient` or transport-neutral interface.
- Named pipe or future transport preparation.
- Mesh orchestration refactoring.
- MQTT batching, throttling, or retained-state policy changes.
- Documentation rewrites unrelated to the removed interface.

## Risks / Watchouts
- `MqttItemClientSession` uses `<inheritdoc />` heavily because of the interface; deleting the interface without replacing those comments can degrade public API documentation.
- Removing `MqttItemClientSession` itself is too broad because Mesh, tests, samples, and tools still use it directly.
- If external package compatibility matters, this is a public API removal and should be called out as a breaking change in release notes later.
- Historical handoff files may still mention `IItemClientSession`; do not churn old planning records unless they are part of the current active scope.

- Prefer direct concrete MQTT naming because MQTT is the only current transport.
- If a future second transport becomes real, introduce an abstraction at that time based on two concrete implementations and proven caller needs.

# IMPLEMENTATION HANDOFF

## Goal
Change the MQTT item value topic shape so the primary item value is published directly on the item topic instead of the `/value` subtopic.

Current:
`hornet/clients/DummyClient1/Edm1/Temperature/value = 123`

Target:
`hornet/clients/DummyClient1/Edm1/Temperature = 123`
`hornet/clients/DummyClient1/Edm1/Temperature/params = ...`

## Scope
Included:
- Update `Amium.ItemBroker.Mqtt.Client` topic mapping for item value publish/subscribe.
- Keep `/params` as the parameter/configuration subtopic.
- Update demo client behavior/output if it references `/value`.
- Update tests that assert MQTT topic names.
- Update relevant documentation if the topic contract is documented.

Not included:
- No unrelated broker architecture changes.
- No new MQTT dependency or protocol abstraction.
- No redesign of parameter payloads.
- No changes to non-MQTT ItemBroker paths unless required by the mapper contract.

## Tasks
1. Inspect MQTT topic mapping and identify where `/value` is appended or parsed.
2. Change the value topic mapping so `hornet/clients/{client}/{path}` is the value topic.
3. Preserve parameter mapping as `hornet/clients/{client}/{path}/params`.
4. Ensure inbound topic parsing distinguishes item values from reserved subtopics such as `/params`.
5. Update `src/Amium.ItemBroker.MqttDemoClient/Program.cs` if displayed or published topics still use `/value`.
6. Update or add tests in `src/Amium.ItemBroker.Tests/Program.cs` for:
   - value topic without `/value`
   - params topic still ending in `/params`
   - topic parsing for both cases
7. Run the relevant build/test command.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- Use existing project patterns and naming.
- Do not introduce new dependencies.
- Public methods/classes changed during implementation need XML documentation.
- Keep changes limited to the MQTT topic contract.
- User-visible error and validation messages must remain English.

## Notes
- MQTT allows a topic to have both its own retained/value payload and child topics.
- Reserve suffix subtopics such as `/params` so they are not mistaken for item values.
- Consider whether old `/value` topics need temporary compatibility. Do not add compatibility unless explicitly requested.

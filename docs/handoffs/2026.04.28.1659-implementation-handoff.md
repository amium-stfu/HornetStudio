# IMPLEMENTATION HANDOFF

## Goal
Make `Amium.Item` the consistent central communication structure across Host, UDL client/runtime, widgets, signals, logging, scripting, and future MQTT-style clients by introducing one canonical item path resolution model.

## Scope
Included:
- Add central item path normalization and resolving in the Host data layer.
- Ensure root items and sub-items are resolved consistently by path.
- Update SignalRegistry and UI/widget item lookup paths to use the central resolver.
- Preserve existing layouts and path formats using `.`, `/`, and `\` separators.
- Keep `Value`, `Unit`, `Format`, `Kind`, `Writable`, `WritePath`, and similar metadata as item parameters, not child items.
- Consider UDL, EnhancedSignals, CustomSignals, RealtimeChart, CsvLogger, Signal widgets, Item/Value widgets, Python client access, and attach/folder workflows.

Not included:
- Do not implement MQTT transport yet.
- Do not redesign `Amium.Item` serialization unless required for resolver correctness.
- Do not rename existing public paths unless compatibility aliases are provided.
- Do not add new external dependencies.

## Tasks
1. Inspect current path usage and categorize all call sites that resolve `HostRegistries.Data` items.
2. Add a central public resolver API to `IDataRegistry`, for example `TryResolve(string path, out Item? item)`.
3. Normalize path comparison consistently for `.`, `/`, and `\`, while preserving original item `Path` casing for display and metadata.
4. Implement resolver behavior:
   - exact root key match first,
   - normalized exact match fallback,
   - longest matching root prefix,
   - child traversal by remaining path segments.
5. Decide whether to add a path index:
   - If added, index all root and descendant items on `UpsertSnapshot`.
   - Remove or refresh affected index entries on `Remove` and `pruneMissingMembers`.
   - Keep fallback traversal available for safety.
6. Update `DataRegistry.UpdateValue` and `UpdateParameter` to use the central resolver.
7. Update `SignalRegistry.TryGetBySourcePath` and signal update event handling so sub-items work reliably.
8. Replace duplicated resolver logic in UI/viewmodels/widgets with the central resolver where possible:
   - `MainWindowViewModel`
   - `PageItemModel`
   - `RealtimeChartControl`
   - `UdlClientControl`
   - `CustomSignalsControl`
   - `EnhancedSignalsControl`
   - target selection/dialog helpers
9. Keep `TargetPathHelper` for UI candidate generation and layout-relative path expansion, but delegate final Host item lookup to `IDataRegistry.TryResolve`.
10. Verify UDL module behavior:
    - `Read`, `Set`, `Out`, `State`, `Alert`, `Command`, and nested `Request` channels must resolve by path.
    - `Writable`, `WritePath`, and `WriteMode` remain parameters.
11. Verify EnhancedSignals behavior:
    - Raw/filter/statistics/kalman/dynamic branches resolve consistently.
    - Nested statistics sub-items such as `Min.TimeStamp` remain accessible.
12. Verify CustomSignals behavior:
    - input paths, write targets, and generated signal items use central resolving.
13. Verify RealtimeChart and CsvLogger:
    - series/logged paths resolve once where possible,
    - runtime sampling does not repeatedly scan all registry keys.
14. Verify Python client and scripting access:
    - public item path access resolves descendants consistently.
15. Add focused tests for resolver behavior:
    - exact root,
    - descendant under root,
    - mixed separators,
    - case-insensitive lookup,
    - ambiguous roots where longest root wins,
    - missing child,
    - update value on descendant,
    - signal creation from descendant.
16. Add or update architecture documentation describing:
    - `Amium.Item` as central communication structure,
    - root vs descendant items,
    - parameters vs child items,
    - canonical path rules,
    - future MQTT mapping expectations.
17. Run build/tests and fix warnings or regressions in changed areas.

## Technical Constraints
- Always answer user-facing chat in German.
- Code, comments, XML docs, UI text, file names, and technical identifiers must be English.
- Public methods/classes changed or added require XML documentation.
- Follow existing architecture and avoid unrelated refactoring.
- Do not add dependencies.
- Preserve existing layout path compatibility.
- `Value`, `Unit`, and similar metadata are item parameters, not sub-items.
- Prefer named parameters for unclear method calls.
- Update relevant Markdown docs when behavior or architecture changes.

## Notes
- Longest matching root should win when paths are ambiguous.
- Keep direct object references or signals in widgets after initial resolve where possible.
- A path index can improve UI/runtime access, but correctness must not depend only on cache state.
- Future MQTT mapping should treat MQTT topics as transport and map them into item path + parameter updates.
- Example target model:
  - `Runtime.Mqtt.Device01.Read` with `Params["Value"] = 23`, `Params["Unit"] = "ABC"`.
  - `Runtime.Mqtt.Device01.Set` with `Params["Value"] = 24`, `Params["Unit"] = "ABC"`.
  - `Runtime.Mqtt.Device01.Raw` with `Params["Value"] = 0.23`, `Params["Unit"] = "mA"`.

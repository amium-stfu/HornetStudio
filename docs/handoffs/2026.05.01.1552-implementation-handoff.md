# IMPLEMENTATION HANDOFF

## Goal
Establish one global rule for runtime registry visibility: `HostRegistries.Data` may contain public, internal, status, diagnostic, and broker-received items, but UI consumers must query items by `DataRegistryItemMetadata` capabilities instead of using raw `GetAllKeys()` lists.

## Scope
Included:
- Hide widget-internal registry entries from normal runtime item displays.
- Make `ItemTree` display only items with `DataRegistryItemCapabilities.Display`.
- Audit UI consumers that call `HostRegistries.Data.GetAllKeys()` and route each usage through the correct capability or explicit internal-discovery path.
- Preserve internal discovery paths such as `Status.AttachOptions` for attach dialogs.
- Keep BrokerWidget received MQTT items visible under `Studio.<FolderName>.<BrokerWidgetName>.Mqtt.<RemoteItemPath>`.
- Add focused tests for metadata-based visibility.
- Update relevant Markdown documentation.

Not included:
- Changing MQTT topic mapping.
- Changing persisted BrokerWidget attach identity format.
- Removing internal registry items from `HostRegistries.Data`.
- Large architectural refactoring beyond a small shared query/helper layer if needed.

## Tasks
1. Inspect all `HostRegistries.Data.GetAllKeys()` usages in `src/HornetStudio`, `src/HornetStudio.Editor`, and `src/HornetStudio.Host`.
2. Classify each usage by purpose: display, broker attach, broker publish, UDL attach, target selection, logging, debug inspection, or internal discovery.
3. Add a small shared query helper if it reduces duplication, for example methods that return keys by capability:
   - display keys
   - broker attach keys
   - broker publish keys
   - debug inspect keys
4. Update `ItemTreeWindowViewModel` so normal item tree display uses only `DataRegistryItemCapabilities.Display`.
5. Ensure `ItemTree` still resolves and renders descendant item data for display-capable root keys.
6. Keep BrokerWidget `Status.AttachOptions` readable by attach dialogs, but prevent those internal keys from appearing in normal item tree/runtime displays.
7. Review target selection and property dialog option sources so they use explicit capabilities or documented internal discovery rules instead of unfiltered `GetAllKeys()`.
8. Verify BrokerWidget received data remains registered with `DataRegistryItemMetadata.BrokerReceivedData()`.
9. Add or update tests proving:
   - `WidgetInternal()` entries do not appear in normal item tree data.
   - `BrokerReceivedData()` entries do appear in normal item tree data.
   - Broker attach options can still be discovered from `Status.AttachOptions`.
10. Update `src/HornetStudio/docs/data-flow-and-signals.md` or another relevant architecture document with the global registry visibility rule.
11. Update `src/HornetStudio/docs/widgets/BrokerWidget.md` only if wording must clarify internal discovery versus visible received MQTT items.
12. Run relevant build/tests after implementation.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- Prefer existing registry metadata patterns in `HostRegistries`.
- Do not introduce new dependencies.
- Do not remove or repurpose `Status.AttachOptions`; it is an internal discovery namespace.
- Do not use path-name filters such as `Status.AttachOptions` as the primary global visibility mechanism.
- Use `DataRegistryItemMetadata` and `DataRegistryItemCapabilities` as the source of truth for UI visibility.
- Keep changes local and focused.
- Document changed public methods/classes with XML summaries if public API is touched.

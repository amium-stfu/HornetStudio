# IMPLEMENTATION HANDOFF

## Goal
Change `BrokerWidget` so attached broker items are shown in the widget body like `UdlClient` modules, with per-item `Edit` and `Delete` actions. `Edit` opens exposure editing for the selected item directly. `ItemExposures` must remain an internal persisted configuration value and must not appear as a separate properties entry.

## Scope
Included:
- Remove the visible `ItemExposures` editor property from `BrokerWidget`.
- Keep internal `ItemExposures` persistence and runtime publishing support.
- Add a UDL-like attached item list to the `BrokerWidget` body.
- Add per-attached-item `Edit` and `Delete` actions.
- Change broker exposure editing from a global property list dialog to a single-item dialog launched from the item row.
- Remove stale exposure definitions when an attached broker item is deleted.

Not included:
- Removing the internal `ItemExposureDefinition` model or persistence.
- Migrating UDL exposure handling.
- Changing MQTT topic formats or broker protocol behavior.
- Adding new dependencies.
- Redesigning unrelated widgets.

## Tasks
1. Remove `BindItemExposureList("ItemExposures", ...)` from the `BrokerWidget` property bindings in `MainWindowViewModel`.
2. Remove `ItemExposures` from `BrokerWidget.yaml` visible/default properties unless the project has a non-visible default mechanism.
3. Remove the generic property-dialog path for `EditorPropertyType.ItemExposureList` if no other widget uses it:
   - enum value
   - `EditorDialogField` item exposure list helpers
   - `EditorPropertyDialog` routing to `ItemExposureDialogWindow`
   - `BindItemExposureList(...)`
4. Keep `FolderItemModel.ItemExposures`, layout persistence, YAML import/export, clone/copy paths, and runtime publishing logic.
5. Extend `BrokerClientControl` with an attached item row collection, similar to `UdlClientControl.Modules`.
6. Render the `BrokerWidget` body as a scrollable list of attached broker item rows.
7. Each row must show the attached item name/path, a compact summary, an `Edit` button, and a `Delete` button.
8. Build broker rows from `BrokerAttachedItemPaths`, current live broker items, and matching `ItemExposures`.
9. Implement `Edit` so it opens exposure editing for exactly the selected attached item.
10. Refactor or replace `ItemExposureDialogWindow` so it edits a single item path and merges the result back into the full `ItemExposures` JSON.
11. Implement `Delete` so it removes the selected path from `BrokerAttachedItemPaths`, removes the matching exposure definition, refreshes rows, and republishes runtime items.
12. Keep exposure matching tolerant for existing saved data by accepting both full flat broker paths and relative remote item paths.
13. Add or update focused tests for single-item exposure merge/delete behavior where practical.
14. Build the affected editor project and run relevant tests.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public changed or added methods/classes must have XML documentation.
- Use existing Avalonia patterns from `UdlClientControl`.
- Do not introduce new dependencies.
- Do not change runtime broker path format: `Runtime.ItemBroker.{WidgetName}.{RemoteClientId}.{ItemPath}`.
- `BrokerAttachedItemPaths` remains the gate for publishing broker items into `HostRegistries.Data`.
- `ItemExposures` is internal persisted state, not a visible properties field.
- Keep changes scoped to BrokerWidget exposure UI and required property-dialog cleanup.

## Notes
- The current implementation already applies `ItemExposures` during broker runtime publishing; preserve that behavior.
- The main correction is UI ownership: exposure editing belongs to each item row in the widget body, not to the widget properties dialog.
- The visual structure should follow `UdlClientControl`: cards/rows in the body with `Edit` and `Delete`.
- Saved exposure definitions without matching attached items should not be shown as rows.
- Deleting an attached item should remove its exposure definition to avoid hidden stale configuration.

# IMPLEMENTATION HANDOFF

## Goal
Finish the `Amium.Item` central communication structure cleanup so Host, UI, signals, UDL, EnhancedSignals, CustomSignals, logging, Python, and future MQTT clients use one consistent item path model without known rest points.

## Scope
Included:
- Complete central item resolving behavior in `DataRegistry`.
- Add a path index if it can be implemented safely.
- Remove or justify remaining direct `TryGet(...)` usages.
- Update item selection/projection paths to include descendant items where needed.
- Add focused resolver and signal tests.
- Update duplicated or outdated documentation.
- Verify the full solution build.

Not included:
- Do not implement MQTT transport.
- Do not redesign `Amium.Item` serialization.
- Do not introduce external dependencies.
- Do not change existing public path semantics without compatibility handling.

## Tasks
1. Re-check all `HostRegistries.Data.TryGet(...)` usages.
   - Keep only usages that intentionally access root keys.
   - Replace final item lookups with `TryResolve(...)`.
   - Add short code comments only where root-only access is intentional and otherwise unclear.
2. Add or complete a central path index in `DataRegistry`.
   - Index root items and all descendants by normalized path.
   - Preserve original `Item.Path` casing and metadata.
   - Use `.`, `/`, and `\` as equivalent separators.
   - Keep case-insensitive lookup.
3. Define index update rules.
   - On `UpsertSnapshot(...)`, index the resulting root and all descendants.
   - On merge with `pruneMissingMembers: true`, remove stale descendant paths from the index.
   - On `Remove(rootKey)`, remove the root and all indexed descendants.
   - Keep traversal fallback if an index lookup misses.
4. Confirm ambiguity behavior.
   - Exact path match wins.
   - If multiple roots match, longest normalized root path wins.
   - Child traversal must be case-insensitive.
5. Ensure `DataRegistry.UpdateValue(...)` and `UpdateParameter(...)` raise events using a useful key.
   - Prefer canonical resolved item path when available.
   - Ensure `SignalRegistry` still receives updates for signals registered via alias or canonical path.
6. Harden `SignalRegistry`.
   - Ensure `TryGetBySourcePath(...)` always stores canonical and requested aliases.
   - Ensure updates fire when `UpdateValue(...)` is called with either alias or canonical path.
   - Infer `IsWritable` from item metadata if available, instead of always `true`, if this is low-risk.
7. Update UI item selection/projection where descendant visibility is expected.
   - `MainWindowViewModel`
   - `ItemTreeWindowViewModel`
   - `EditorDialogField`
   - UDL exposure dialogs
   - Attach item dialogs
   - CustomSignals and EnhancedSignals selectors
8. Keep `TargetPathHelper` responsible for layout-relative candidate generation only.
   - Final host item lookup must use `HostRegistries.Data.TryResolve(...)`.
   - Remove duplicated prefix-scanning resolver logic if still present.
9. Verify UDL behavior.
   - `Read`, `Set`, `Out`, `State`, `Alert`, `Command`, and nested `Request` resolve consistently.
   - Existing exposed modules and bit items remain compatible.
10. Verify EnhancedSignals behavior.
    - Branches like `Raw`, `Read`, `Set`, `Config`, `Statistics`, `Kalman`, `Dynamic`, and nested items such as `Statistics.Min.TimeStamp` resolve consistently.
    - Reset/request write paths still work.
11. Verify CustomSignals behavior.
    - Input signal value preservation uses correct resolved item.
    - Computed signal source paths resolve through the central resolver.
    - Write targets resolve through the central resolver.
12. Verify logger/chart/widget behavior.
    - RealtimeChart resolves configured series correctly.
    - CsvLogger and SqlLogger resolve logged paths correctly.
    - CircleDisplay runtime paths resolve correctly.
    - Widgets do not perform repeated full registry scans for runtime value reads when avoidable.
13. Verify Python host value projection.
    - Projection should enumerate descendants intentionally.
    - Runtime updates should use `TryResolve(...)`.
    - Root-only `TryGet(...)` should remain only where enumerating root keys from `GetAllKeys()`.
14. Add focused tests.
    - Exact root resolve.
    - Descendant resolve.
    - Mixed separators.
    - Case-insensitive resolve.
    - Longest root wins.
    - Missing child returns false.
    - `UpdateValue(...)` updates descendant item.
    - `UpdateParameter(...)` updates descendant parameter.
    - `Remove(...)` clears indexed descendants.
    - `pruneMissingMembers` clears stale descendants.
    - `SignalRegistry.TryGetBySourcePath(...)` works for descendants.
    - Signal update fires for descendant updates.
15. Update documentation.
    - Update both `docs/data-flow-and-signals.md` and `src/HornetStudio/docs/data-flow-and-signals.md`.
    - Document `TryGet(...)` as root-only.
    - Document `TryResolve(...)` as canonical final lookup.
    - Document parameter vs child item rules.
    - Document path normalization and ambiguity rules.
    - Document future MQTT mapping expectations.
16. Run verification.
    - `dotnet build b:\HornetStudio\HornetStudio.sln --configfile b:\HornetStudio\NuGet.Config -m:1`
    - Run any added test project/test command.
    - Record remaining warnings if unrelated.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML docs, UI text, file names, and technical identifiers must be English.
- Public methods and changed public classes require XML documentation.
- Do not add external dependencies.
- Preserve compatibility with existing layout paths.
- Treat `Value`, `Unit`, `Format`, `Kind`, `Writable`, `WritePath`, and similar fields as parameters, not child items.
- Prefer existing project patterns and small scoped changes.
- Avoid broad refactoring outside resolver cleanup.

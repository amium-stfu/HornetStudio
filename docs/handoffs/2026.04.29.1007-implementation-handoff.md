# IMPLEMENTATION HANDOFF

## Goal
Add a first MQTT transport for `Amium.ItemBroker` so the broker can be started through `Amium.ItemBroker.Service` and inspected with MQTT Explorer.

## Scope
Included:
- Add an MQTT adapter project for broker inspection and basic publish/write testing.
- Start the MQTT adapter from `Amium.ItemBroker.Service`.
- Publish broker values to MQTT topics.
- Map MQTT publishes back to broker value or write messages where allowed.
- Add minimal health/status topics for the MQTT transport.
- Add focused tests for topic/path mapping and adapter behavior where practical.
- Update broker usage/architecture documentation.

Not included:
- Replacing `HornetStudio.Host` registries.
- Building a full MQTT broker from scratch.
- Adding UI widgets.
- Adding HTTP, WebSocket, TCP, or Named Pipe transports.
- Implementing full authentication/authorization beyond safe local defaults.

## Tasks
1. Create `src/Amium.ItemBroker.Mqtt` as a transport adapter project.
2. Add a suitable MQTT library dependency only to the MQTT adapter project, not to `Amium.ItemBroker`.
3. Define a small configuration model for MQTT host, port, base topic, client id, and enable/disable state.
4. Implement topic mapping between MQTT topics and broker paths/parameters.
5. Subscribe the adapter to broker changes and publish retained broker values to MQTT.
6. Handle incoming MQTT publishes and translate them into broker value updates or write requests.
7. Extend `Amium.ItemBroker.Service` to create the broker, start the MQTT adapter, and publish basic health values.
8. Add a simple demo publisher or startup health heartbeat so MQTT Explorer shows visible data immediately.
9. Add tests for topic mapping, path normalization, retained publish behavior, and incoming write mapping.
10. Update `docs/amium-itembroker-usage.md` and `docs/amium-itembroker-architecture.md` with MQTT test steps.
11. Build the solution and run `src/Amium.ItemBroker.Tests`.

## Technical Constraints
- Keep `Amium.ItemBroker` transport-neutral.
- MQTT-specific code must stay outside `Amium.ItemBroker`.
- Prefer existing project patterns and .NET 9.
- Do not introduce more dependencies than needed.
- Topic mapping must respect the project rule: MQTT topics are transport paths; broker data remains `Item path + parameter + value`.
- Use English for code, comments, identifiers, user-visible error messages, and file names.
- Chat responses remain German.
- Public classes and public methods changed or added during implementation need XML documentation.

## Notes
- Recommended first topic model:
  - Broker output: `hornet/items/{path segments}/{parameter}`
  - Example: `Runtime.Health.ItemBroker.Heartbeat` parameter `Value` maps to `hornet/items/Runtime/Health/ItemBroker/Heartbeat/Value`.
- Parameter names such as `Value`, `Unit`, `Format`, `Kind`, `Writable`, `WritePath`, and `WriteMode` are MQTT topic suffixes, not child items.
- Start with local, unauthenticated MQTT for easy testing only if bound to local development defaults.
- MQTT Explorer should be able to subscribe to `hornet/items/#`.
- The service should clearly log when MQTT is enabled, where it is listening/connected, and which base topic is used.

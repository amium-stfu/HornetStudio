# IMPLEMENTATION HANDOFF

## Goal
Add a generic `ItemExposures` configuration model and use it first for `BrokerWidget` item exposure editing, so broker-discovered items can define common value metadata such as bitmask publishing without duplicating UDL-specific exposure logic.

## Scope
Included:
- Introduce a source-neutral item exposure definition model for attached/discovered runtime items.
- Persist generic `ItemExposures` on page items.
- Add BrokerWidget editing for broker item exposures based on discovered/attachable broker items.
- Support generic bitmask metadata: enabled exposure, bit count, optional labels, format/unit where already supported by the project patterns.
- Apply `ItemExposures` while publishing attached broker items to `HostRegistries.Data`.
- Keep Python out of scope; Python defines item metadata in code.

Not included:
- Full UDL migration to `ItemExposures`.
- SignalWidget redesign.
- MQTT protocol/topic changes.
- New external dependencies.
- Large UI redesign beyond BrokerWidget exposure editing.

## Tasks
1. Add a generic `ItemExposureDefinition` model and codec, similar in shape to `UdlModuleExposureDefinitionCodec`, but source-neutral.
2. Add `ItemExposures` persistence to `FolderItemModel`, `LayoutDocument`, project/book layout loading, YAML export/import, and cloning/copy paths.
3. Add `ItemExposures` to `BrokerWidget.yaml` with an empty default value.
4. Add an editor property type and dialog entry for broker item exposures, reusing existing property dialog patterns.
5. Build a Broker exposure dialog that lists discovered or attachable broker items and edits generic metadata per item.
6. In `BrokerClientControl`, apply matching exposure metadata when publishing selected broker items into `HostRegistries.Data`.
7. For bitmask exposures, create/update helper bit items through generic helper logic instead of Broker-only logic.
8. Keep existing `BrokerAttachedItemPaths` behavior unchanged: only attached broker items are published globally.
9. Add focused tests where practical for codec parsing/serialization and exposure application.
10. Build the affected projects and run relevant tests.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public changed or added methods/classes must have XML documentation.
- Use existing project patterns and dependencies.
- Do not introduce new dependencies.
- Keep source-specific discovery in the source widget.
- Keep bitmask metadata and helper publishing generic.
- `SignalWidget` should consume resulting metadata; it should not own global helper item publishing.
- `BrokerAttachedItemPaths` remains the gate for publishing broker items into `HostRegistries.Data`.
- Runtime broker path format must stay `Runtime.ItemBroker.{WidgetName}.{RemoteClientId}.{ItemPath}`.

## Notes
- The first implementation should add generic `ItemExposures` and use it for BrokerWidget only.
- UDL can remain on `UdlModuleExposureDefinitions` until a separate migration is planned.
- Broker-specific UI should handle item discovery and selection; the metadata shape should remain generic.
- Python is excluded because Python item semantics should be declared by script/API code.
- Avoid storing exposure truth in `SignalWidget`; SignalWidget should render and interact with metadata from the target item.

# IMPLEMENTATION HANDOFF

## Goal
Unify item paths around `Studio` so project runtime items, target selection, broker publishing, and item tree display use one consistent structure without duplicate `Project` and `Studio.Project` roots.

## Scope
Included:
- Define `Studio` as the canonical top-level root for user-facing project/runtime item paths.
- Prevent duplicate registry/display paths such as `Project.Folder1...` and `Studio.Project.Folder1...`.
- Normalize target resolution and target tree display to the canonical `Studio.<Folder>...` structure.
- Keep BrokerWidget publish paths compatible with the `Studio` namespace.
- Add or update tests for path normalization, target options, and broker publish defaults.

Not included:
- No unrelated widget behavior changes.
- No large architectural rewrite of `DataRegistry`.
- No new dependencies.
- No release/versioning work.

## Tasks
1. Define the canonical path model:
   - User-facing/local runtime paths should be `Studio.<Folder>.<Item>...`.
   - `Project.<Folder>...` is legacy input only and should be normalized or resolved to `Studio.<Folder>...`.
   - Broker publish paths should not become `Studio.Project.<Folder>...`.
2. Update central path helpers:
   - Extend `TargetPathHelper` with `Studio` root handling.
   - Add normalization from legacy `Project.<Folder>...` to canonical `Studio.<Folder>...` where appropriate.
   - Ensure `Studio.Project...` is treated as a legacy/invalid doubled form and normalized to `Studio...` when safe.
3. Update runtime path builders:
   - Change widget/runtime path builders currently returning `Project.<Folder>...` to return `Studio.<Folder>...`.
   - Relevant builders include custom signals, enhanced signals, logger runtime, display runtime, UDL attach/status paths, Python/application runtime paths, and BrokerWidget status/attach option paths.
4. Update BrokerWidget publish defaults:
   - `BrokerPublishedItemDefinitionCodec.BuildDefaultBrokerPath(...)` should avoid `Studio.Project...`.
   - For a canonical local path `Studio.Folder1.UdlClient1.m400`, the default broker path should be `Studio.Folder1.UdlClient1.m400`.
   - For a legacy local path `Project.Folder1.UdlClient1.m400`, normalize the broker path to `Studio.Folder1.UdlClient1.m400`.
5. Update target selection and option filtering:
   - Target tree display should prefer `Studio.<Folder>.` as the folder scope.
   - Assignment dialogs should show canonical `Studio` paths and avoid duplicate legacy paths.
   - Broker publish item options should expose canonical local paths only.
6. Preserve compatibility:
   - Existing stored `Project.<Folder>...` target paths must still resolve.
   - Existing explicit broker paths should not be destructively rewritten during parse unless they are the doubled `Studio.Project...` default form.
   - Runtime resolution candidates should try canonical `Studio...` first, then legacy `Project...`.
7. Update tests:
   - Path helper normalization for `Project.Folder.X`, `Studio.Folder.X`, and `Studio.Project.Folder.X`.
   - Broker default path generation.
   - Target option de-duplication.
   - Registry resolution compatibility for legacy `Project` paths.
8. Update documentation:
   - Update architecture/path docs and BrokerWidget examples to show `Studio.<Folder>...`.
   - Mention `Project.<Folder>...` as legacy-compatible input only if needed.
9. Run verification:
   - Build `HornetStudio.sln`.
   - Run relevant editor/host tests.
   - Manually verify item tree no longer shows parallel `Project` and `Studio.Project` branches for the same project items.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- Use existing helpers and patterns; do not add dependencies.
- Keep `DataRegistry` neutral where possible; prefer canonicalization at path-helper, publisher, and UI-option boundaries.
- Public classes or public members changed during implementation need XML documentation.
- Do not make broad unrelated refactorings.
- Preserve existing saved layouts as much as possible through compatibility resolution.

## Notes
- The current duplicate tree likely comes from a mix of local `Project...` registry paths and BrokerWidget `Studio.Project...` paths.
- The important structural decision is that `Studio` replaces `Project` as the canonical user-facing root instead of wrapping it.
- Avoid solving this with display-only hiding; the underlying path builders and defaults should stop creating duplicate semantic roots.

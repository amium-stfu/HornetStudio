# IMPLEMENTATION HANDOFF

## Goal
Add an optional MQTT retain flag to the MQTT item broker client publish methods so callers can choose retained publishing per publish call. The default must remain non-retained.

## Scope
Included:
- Add `retained: false` support to MQTT publish methods in `Amium.ItemBroker.Client.Mqtt`.
- Forward the flag from item, value, and parameter publish methods to the MQTT message builder.
- Preserve existing behavior for all existing callers.
- Add or update focused tests if the project already has MQTT publish coverage for this client.

Not included:
- Do not add retained support to the transport-neutral `IItemBrokerClientSession` unless required by compilation.
- Do not change `WriteAsync` behavior unless explicitly needed.
- Do not introduce new dependencies.
- Do not change topic mapping, payload formatting, reconnect behavior, or publish policy behavior.

## Tasks
1. Update `MqttItemBrokerClientSession.PublishItemAsync` with an optional `bool retained = false` parameter.
2. Update MQTT-specific `PublishValueAsync` overloads with optional `bool retained = false`.
3. Update MQTT-specific `PublishParameterAsync` overloads with optional `bool retained = false`.
4. Pass `retained` through all internal publish calls made by `PublishItemAsync`.
5. Update `PublishParameterTopicAsync` to accept `bool retained` and set the MQTT application message retain flag.
6. Keep existing call sites source-compatible by placing the optional `retained` parameter before `CancellationToken` only if needed for named-argument compatibility; otherwise prefer adding it before `correlationId` only after checking interface constraints.
7. Build the affected project and run relevant tests.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and identifiers must be English.
- Use existing project patterns and MQTTnet APIs.
- Default retained behavior must be `false`.
- Avoid adding MQTT-specific concepts to transport-neutral abstractions unless compilation or existing design requires it.
- Document changed public methods with XML documentation where explicit documentation exists.
- Use named parameters for unclear multi-argument calls.

## Relevant Files
- `src/Amium.ItemBroker.Client.Mqtt/MqttItemBrokerClientSession.cs`
- `src/Amium.ItemBroker.Client/IItemBrokerClientSession.cs`
- `src/Amium.ItemBroker.Client/ItemBrokerClientSession.cs`
- `src/Amium.ItemBroker.Tests/Program.cs`
- `src/Amium.ItemBroker.MqttDemoClient/Program.cs`

## Notes
- MQTT retained messages are appropriate for current state values, not transient commands or events.
- `WriteAsync` should remain non-retained by default because it behaves more like a write/command path.
- Be careful with `null` payloads: retained deletion semantics may overlap with the current empty payload formatting, but retained deletion is not part of this task.
- If the interface prevents adding MQTT-only optional parameters cleanly, prefer adding explicit MQTT-specific overloads on `MqttItemBrokerClientSession` while keeping interface implementation compatible.

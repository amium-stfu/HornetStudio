# IMPLEMENTATION HANDOFF

## Goal
Plan and scaffold `Amium.ItemBroker` as an independent, transport-neutral broker for `Amium.Item` communication, with a future-capable standalone service host.

## Scope
Included:
- Create the architecture and initial project structure for `Amium.ItemBroker`.
- Plan a standalone service project that can run without HornetStudio UI.
- Define broker responsibilities, message contracts, client/session model, subscriptions, retained item state, and Host bridge boundaries.
- Keep MQTT as a future transport adapter, not part of the first implementation unless explicitly requested.
- Keep `HornetStudio.Host.DataRegistry` as the local live registry inside HornetStudio.

Not included:
- Do not implement MQTT transport in the first step.
- Do not replace `HostRegistries.Data`.
- Do not move UI logic into the broker.
- Do not add external dependencies unless explicitly justified.
- Do not implement security/auth in depth yet; reserve extension points only.

## Tasks
1. Add architecture documentation for `Amium.ItemBroker`.
   - Explain the difference between `Amium.Item`, `DataRegistry`, `ItemBroker`, service host, and transports.
   - Document embedded, local service, and network broker modes.
2. Create or plan solution projects:
   - `src/Amium.ItemBroker/Amium.ItemBroker.csproj`
   - `src/Amium.ItemBroker.Service/Amium.ItemBroker.Service.csproj`
   - Optional later: `src/Amium.ItemBroker.Mqtt/Amium.ItemBroker.Mqtt.csproj`
3. Define dependency direction:
   - `Amium.ItemBroker` depends on `Amium.Item`.
   - `Amium.ItemBroker.Service` depends on `Amium.ItemBroker`.
   - `HornetStudio.Host` may depend on `Amium.ItemBroker`.
   - `Amium.ItemBroker` must not depend on `HornetStudio.Host`, Editor, UI, or widgets.
4. Define broker domain contracts.
   - `IItemBroker`
   - `IItemBrokerClient`
   - `IItemBrokerTransport`
   - `IItemBrokerStore`
   - `IItemSubscription`
   - `IItemBrokerClock` if timestamps need testability.
5. Define message contracts.
   - `ItemSnapshotMessage`
   - `ItemValueChangedMessage`
   - `ItemParameterChangedMessage`
   - `ItemWriteRequestMessage`
   - `ItemRemoveMessage`
   - `ItemSubscribeMessage`
   - `ItemUnsubscribeMessage`
   - `ItemBrokerAckMessage`
6. Define addressing.
   - Use canonical item paths.
   - Normalize `.`, `/`, and `\` for comparison.
   - Keep `Value`, `Unit`, `Format`, `Kind`, `Writable`, `WritePath`, and `WriteMode` as parameters.
   - Treat transport topics as mapping inputs, not as canonical broker paths.
7. Define retained state behavior.
   - Broker keeps the latest known item snapshots/deltas by path.
   - New subscribers can receive retained snapshots.
   - Updates route to matching subscriptions.
   - Remove/prune behavior must clear stale retained descendants.
8. Define subscription semantics.
   - Exact path subscription.
   - Recursive subtree subscription.
   - Optional wildcard syntax only if kept small and documented.
   - Case-insensitive path matching.
9. Define write semantics.
   - Writes target `Item path + parameter`.
   - Default parameter is `Value`.
   - Write requests can carry correlation id, timestamp, source client id, and optional reply target.
   - Broker routes writes to owning publisher/adapter or Host bridge.
10. Define ownership/source semantics.
    - Track which client last published a path.
    - Allow Host bridge to publish local Host items.
    - Avoid hard ownership enforcement in the first version unless needed.
11. Plan Host bridge.
    - `DataRegistryItemBrokerBridge` or equivalent in `HornetStudio.Host`.
    - `DataRegistry.ItemChanged` -> broker publish.
    - broker snapshot/delta/write -> `DataRegistry.UpsertSnapshot`, `UpdateValue`, or `UpdateParameter`.
    - Prevent event echo loops with source ids or bridge-local suppression.
12. Plan service host.
    - Console/worker style host.
    - Reads config.
    - Starts broker core.
    - Starts enabled transports.
    - Logs lifecycle and client events.
    - Can later become Windows Service/systemd.
13. Plan embedded mode.
    - HornetStudio can run broker in-process for local setups.
    - Same broker core as service mode.
14. Plan transport adapters.
    - First transport can be in-process for tests and Host integration.
    - MQTT transport is a later adapter.
    - Transport adapters convert external protocol shape into broker messages.
15. Add focused tests.
    - Path normalization.
    - Snapshot publish and retained state.
    - Value update routing.
    - Parameter update routing.
    - Recursive subscription.
    - Remove clears descendants.
    - Write request routing.
    - Echo suppression in Host bridge if implemented.
16. Update documentation.
    - Extend `docs/amium-item-communication-structure.md`.
    - Add a dedicated `docs/amium-itembroker-architecture.md` if the broker details become too large.
    - Update `CHANGELOG.md` for relevant structural additions.
17. Verify build.
    - Add projects to `HornetStudio.sln`.
    - Run solution build serially with `-m:1`.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML docs, UI text, file names, and technical identifiers must be English.
- Public methods/classes require XML documentation.
- Prefer existing .NET project patterns.
- Do not introduce external dependencies without justification.
- Broker core must be UI-free and Host-free.
- `Amium.ItemBroker` must remain transport-neutral.
- MQTT is a future adapter, not broker core.
- Preserve existing Host `DataRegistry` behavior until the bridge is explicitly implemented.

## Notes
- `Amium.ItemBroker` is not a replacement for `HostRegistries.Data`.
- `HostRegistries.Data` remains the local live registry in HornetStudio.
- The broker is the distributed communication and routing layer around `Amium.Item`.
- Future ESP32/MQTT mapping should become `MQTT topic -> broker message -> Item path + parameter + value`.
- A standalone broker service must run without HornetStudio Editor or UI assemblies.

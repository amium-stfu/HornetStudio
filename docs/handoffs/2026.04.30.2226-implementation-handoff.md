# IMPLEMENTATION HANDOFF

## Goal
Introduce a central Host registry parameter protection policy so user-facing pickers do not show protected system parameters and user-/remote-driven writes cannot overwrite them.

## Scope
Included:
- Add a central policy for protected Host registry parameters.
- Hide protected parameters from SignalWidget and other user-facing target/parameter pickers where applicable.
- Guard user-/remote-driven parameter writes with the central policy.
- Preserve unrestricted internal runtime writes for system components.
- Keep the first blacklist small and limited to clearly system-relevant parameters.

Not included:
- No BrokerWidget writable write-back implementation.
- No MQTT shadow/write-back client.
- No broad redesign of registry storage.
- No blocking of internal runtime code that legitimately maintains system parameters.
- No retained broker cleanup behavior.

## Tasks
1. Add a central policy class in the Host layer, e.g. `HostRegistryParameterPolicy`.
2. Implement policy methods such as:
   - `IsProtectedParameter(string parameterName)`
   - `CanShowInUserPicker(string parameterName)`
   - `CanUserWriteParameter(string parameterName)`
3. Start with a small protected blacklist:
   - `Writable`
   - `IsWritable`
   - `WritePath`
   - `BrokerPath`
   - `LocalPath`
   - `Active`
   - `PublishMode`
   - `PublishIntervalMs`
4. Do not initially blacklist general metadata such as `Title`, `Text`, `Unit`, `Format`, `Kind`, or `Status` unless implementation review shows they are clearly system-critical.
5. Update SignalWidget/Signal target selection so protected parameters are not shown or selectable.
6. Update other existing user-facing pickers or target selection flows that expose registry parameters, if they share the same helper/source.
7. Add a guarded user-write path or helper, e.g. `TryUpdateUserParameter`, or apply `CanUserWriteParameter` at existing user-/remote-driven write call sites.
8. Keep low-level `HostRegistries.Data.UpdateParameter(...)` unrestricted for internal runtime use.
9. Ensure old layouts/YAML that reference protected parameters do not silently overwrite them; treat them as protected/invalid/missing in user-facing execution paths.
10. Add logging or clear rejection behavior for blocked writes without throwing disruptive UI exceptions.
11. Add focused tests for:
    - protected parameter detection
    - picker filtering
    - guarded write rejection
    - internal unrestricted update path still working
12. Update relevant documentation to explain protected system parameters.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- Code identifiers and user-visible validation/error messages must be English.
- Keep the blacklist central; do not duplicate protected parameter lists across widgets.
- Internal runtime writes must remain possible.
- User-/remote-driven writes must use the guarded policy.
- Do not add new external dependencies.
- Public classes or public members changed during implementation need XML documentation.
- Keep changes narrowly scoped to registry parameter protection.

## Notes
- The protection model has two layers:
  - User-facing picker filtering prevents bad configurations.
  - Guarded write paths protect against old layouts, YAML edits, remote input, and other non-UI paths.
- The policy should be used by future BrokerWidget writable write-back.
- `Value` is not protected.
- If a protected parameter appears in old saved configuration, do not delete the configuration automatically; prevent execution/write and surface it as protected where practical.

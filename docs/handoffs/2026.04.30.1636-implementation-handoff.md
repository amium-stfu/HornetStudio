# IMPLEMENTATION HANDOFF

## Goal
Refine BrokerWidget publishing UI and behavior so local publish roots are shown separately from attached remote items, and each publish root can be configured through an edit dialog before one-way broker publishing is enabled.

## Scope
Included:
- Split the BrokerWidget body into two sections: `Attached To UI` and `Published Items`.
- Keep `AttachToUI` behavior unchanged.
- Keep `PublishItems` as the TreeView selection entry for local registry roots.
- Add selected publish roots to a separate `Published Items` body list.
- Add an edit action for each publish root.
- In the edit dialog, list the selected root and its subitems.
- Configure each listed entry with `Active`, `PublishMode`, `PublishIntervalMs`, and `Writable`.
- Publish only active entries from HornetStudio to the broker.
- Send retained snapshots for initial publish and interval publish.

Not included:
- No external write-back into `HostRegistries.Data`.
- No active behavior for `Writable`; it is stored for future write-back support only.
- No changes to existing attached item exposure behavior beyond moving it into its own body section.
- No custom broker path editing unless already trivial in the existing UI patterns.
- No broad BrokerWidget redesign.

## Tasks
1. Update the BrokerWidget body layout to show two compact sections:
   - `Attached To UI`
   - `Published Items`
2. Keep existing attached item rows and actions under `Attached To UI`.
3. Add a published root row model for `Published Items`, including local root path, display name, status summary, and visual status state.
4. Make the `PublishItems` TreeView selection add local registry roots to the published root list instead of enabling publishing immediately.
5. Default newly selected publish root entries and subentries to:
   - `Active = false`
   - `PublishMode = OnChanged`
   - `PublishIntervalMs = 1000`
   - `Writable = false`
   - `BrokerPath = HornetStudio.<LocalPath>`
6. Store publish configuration as structured definitions, grouped by root in UI or stored flat with `LocalRootPath`.
7. Add a published-root edit dialog that resolves the current root and subitems from `HostRegistries.Data`.
8. In the edit dialog, list root and subitems with controls for `Active`, `PublishMode`, `PublishIntervalMs`, and `Writable`.
9. Preserve existing per-entry settings when the local registry tree changes or when `PublishItems` is reopened.
10. Add delete/remove behavior for published roots without affecting local registry items.
11. Update runtime publishing to publish only entries with `Active = true`.
12. On BrokerWidget connect, send retained snapshots for all active entries that exist.
13. For `OnChanged`, publish matching local changes for active entries only.
14. For `Interval`, publish retained snapshots for active entries at the configured interval.
15. Stop publisher timers/subscriptions on disconnect and visual detach.
16. Add tests for section data models, default inactive publish definitions, configuration persistence, and active-only publishing where feasible.
17. Update BrokerWidget documentation/YAML sample to describe the two sections and publish configuration flow.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- UI labels should use `Attached To UI`, `Published Items`, `PublishItems`, `Active`, `OnChanged`, `Interval`, and `Writable`.
- Default broker paths must stay flat: `HornetStudio.<LocalRegistryKey>`.
- Do not add a `Projects` segment.
- Do not implement external write-back in this step.
- `Writable` must be persisted but not acted on.
- New publish roots must be inactive by default to avoid surprising outbound traffic.
- Use existing Avalonia/editor patterns and existing ItemBroker/MQTT APIs.
- Do not add new external dependencies.
- Public classes or public members changed during implementation need XML documentation.

## Notes
- Direction must remain visually and structurally clear:
  - `Attached To UI`: broker to HornetStudio
  - `Published Items`: HornetStudio to broker
- A body row represents a selected local publish root, not necessarily an active publisher.
- The edit dialog is where root and subitem entries become active.
- Snapshot scope is exactly the configured entry root. A selected parent can publish a subtree; a selected leaf publishes only that item.
- Retained snapshots should be used for initial and interval publishing so external clients can reconstruct current state.

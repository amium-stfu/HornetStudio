# IMPLEMENTATION HANDOFF

## Goal
Add a general MQTT ItemBroker consumer/write concept and integrate it into HornetStudio through a Broker widget similar to the existing UdlClient workflow.

## Scope
Included:
- Add MQTT subscribe support to the general ItemBroker MQTT client layer.
- Rebuild remote `Amium.Item.Item` trees from MQTT retained and live topics.
- Allow direct writes by publishing to the same MQTT item parameter topics.
- Keep the concept general so HornetStudio is only one client among many.
- Add a HornetStudio host-side broker client with runtime items.
- Add an editor Broker widget that connects, displays status, exposes attach options, and publishes runtime items to `HostRegistries.Data`.

Not included:
- Owner-based write routing.
- Write request/ack topics.
- Conflict resolution beyond MQTT last-retained/live-message behavior.
- Removing offline remote items automatically.
- Large refactoring of UdlClient or existing widgets.

## Tasks
1. Align MQTT topic mapping in `Amium.ItemBroker.Mqtt.Client` with the shared mapper behavior, including `BaseTopic` handling.
2. Implement MQTT subscription support for item topics, subscribing to `clients/+/items/#` below the configured base topic.
3. Add parsing from MQTT topics to `RemoteClientId`, item path, and parameter name.
4. Add a remote item registry that reconstructs item trees keyed by `RemoteClientId + ItemPath`.
5. Apply retained and live MQTT messages to the registry:
   - `.../value` updates item `Value`.
   - `.../params/{ParameterName}` updates item parameters.
   - Missing ancestors are created as needed.
6. Expose registry change notifications for snapshots, value updates, parameter updates, client status changes, and diagnostics.
7. Keep direct write behavior simple: writing from any client publishes to the same item parameter topic with retain enabled where appropriate.
8. Add status metadata for remote client roots:
   - `ConnectionStatus`
   - `LastSeenUtc`
   - `Stale`
9. Keep offline items in the registry and mark the remote client/root as offline instead of removing items.
10. Add `HostItemBrokerClient` in `HornetStudio.Host` similar in shape to `HostUdlClient`.
11. Publish HornetStudio runtime items under `Runtime.ItemBroker.{WidgetName}.{RemoteClientId}.{ItemPath}`.
12. Add a `BrokerWidget`/`BrokerClientControl` similar to `UdlClientControl`:
   - Host
   - Port
   - BaseTopic
   - ClientId
   - AutoConnect
   - Connect/Disconnect
   - Status display
   - Attach option list
13. Integrate attached broker items with existing `HostRegistries.Data` / AttachToUi behavior.
14. Add focused tests for:
   - topic mapping with base topic
   - topic parsing
   - retained value/parameter rebuild
   - child item reconstruction
   - direct write topic publishing
15. Update relevant widget documentation if a new user-facing Broker widget is added.
16. Build the affected projects with the repository `NuGet.Config`.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public methods/classes changed or added must have XML documentation.
- Use existing project patterns and dependencies.
- Do not introduce new dependencies unless clearly justified.
- Keep HornetStudio as one generic client; do not bake HornetStudio-specific behavior into the ItemBroker MQTT library.
- Runtime path format must be `Runtime.ItemBroker.{WidgetName}.{RemoteClientId}.{ItemPath}`.
- Retained state is expected for snapshots, last values, and last parameters.
- Offline remote items must remain available and be marked through metadata.
- Writes are direct MQTT publishes to item parameter topics; no request/ack protocol in this version.

## Notes
- The MQTT bus is intentionally simple: every client may write to any item topic.
- If the original publisher continues publishing, it may overwrite another client's write.
- Do not overwrite item `Value` with `"offline"` automatically; preserve the last known value and expose offline state via metadata.
- UI widgets may choose to display `"offline"` or `"n/a"` based on status metadata.
- Prefer a small, working V1 over a large ownership/routing protocol.

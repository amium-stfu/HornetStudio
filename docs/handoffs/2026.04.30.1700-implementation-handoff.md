# IMPLEMENTATION HANDOFF

## Goal
Refine BrokerWidget publishing semantics after the initial implementation so publishing is scoped to active item entries, saving a publish-root dialog only snapshots that root, and change-triggered publishing updates only affected active entries.

## Scope
Included:
- Keep the existing two-section BrokerWidget body and publish-root edit dialog.
- Treat the published root body row as UI grouping only.
- Treat each active published item definition entry as the actual publishing boundary.
- On edit-dialog save, publish retained snapshots only for active entries under the edited root when the widget is connected.
- Avoid restarting/publishing all active definitions solely because one root was saved.
- Refine `OnChanged` matching so only affected active entries are published.
- Keep connect and interval snapshot behavior.

Not included:
- No external write-back into `HostRegistries.Data`.
- No active behavior for `Writable`.
- No retained broker delete when an entry is deactivated.
- No broad UI redesign.
- No custom broker path editing.

## Tasks
1. Add or expose a focused publisher operation that can publish retained snapshots for active definitions belonging to a specific `LocalRootPath`.
2. Update `PublishedItemDialogWindow` save handling integration so saving root `X` causes snapshots only for active entries where `LocalRootPath == X`, and only when the BrokerWidget is connected.
3. Remove duplicate full publisher restarts on edit-dialog save if `BrokerPublishedItemPaths` property change already restarts the publisher.
4. Ensure changing publish definitions still updates timers/subscriptions for future `OnChanged` and `Interval` behavior without unnecessarily snapshotting unrelated roots.
5. Document and enforce that the body root row is a grouping/navigation concept only; individual active entries are the publish units.
6. Refine `HostItemBrokerPublisher.OnDataItemChanged` matching:
   - Exact active entry change publishes that entry.
   - Descendant change publishes the active ancestor entry only when the active entry is a subtree root.
   - Ancestor snapshot/upsert may publish active descendant entries if needed to refresh created/updated subitems.
   - Ancestor value/parameter updates must not publish unrelated active descendants.
7. Preserve connect behavior: on connect, publish retained snapshots for all active entries.
8. Preserve interval behavior: interval timers publish retained snapshots only for active interval entries.
9. Preserve inactive behavior: inactive entries do not publish and are not removed from the broker.
10. Add tests for save-root snapshot scope, duplicate restart avoidance if testable, and `OnChanged` matching edge cases.
11. Update BrokerWidget documentation to describe snapshot triggers and active-entry scope.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- Default broker paths remain flat: `HornetStudio.<LocalRegistryKey>`.
- Do not add a `Projects` segment.
- `Writable` remains persisted only.
- Do not delete retained broker state for inactive or removed definitions in this step.
- Use existing BrokerWidget classes and ItemBroker/MQTT APIs.
- Do not add new external dependencies.
- Public classes or public members changed during implementation need XML documentation.

## Notes
- Snapshot triggers:
  - Connect: all active entries.
  - Save: active entries of the saved root only.
  - Interval: active interval entries only.
  - OnChanged: affected active entries only.
- A single active subitem must publish only that subitem, not siblings or the whole root.
- Deactivating an entry means no further publishing; it must not remove retained broker state automatically.

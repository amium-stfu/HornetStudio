# IMPLEMENTATION HANDOFF

## Goal
Improve BrokerWidget AttachToUi handling so MQTT ItemBroker items are presented as a compact tree, stale saved attachments can be removed, and filtering stays controlled by AttachToUi instead of BrokerClientId.

## Scope
Included:
- Keep BrokerWidget collecting all remote MQTT clients/items from the configured broker/base topic.
- Keep the persisted attach value unambiguous.
- Change the Attach Items editor for BrokerWidget to a tree-oriented view.
- Display compact item labels instead of repeated full runtime prefixes.
- Show saved but currently missing BrokerWidget attachments and allow deleting them from the saved configuration.
- Clarify BrokerClientId as the local MQTT client id, not a remote client filter.

Not included:
- No MQTT topic contract redesign.
- No change to non-BrokerWidget attach editors unless shared infrastructure requires a minimal extension.
- No new external dependencies.
- No broad UI redesign outside the Attach Items dialog.

## Tasks
1. Inspect current BrokerWidget attach option publishing and dialog binding:
   - `BrokerClientControl.EnumerateAttachOptions(...)`
   - `BrokerClientControl.PublishAttachOptionItems(...)`
   - `MainWindowViewModel.GetBrokerAttachItemOptions(...)`
   - `EditorDialogField` attach-list serialization
   - `AttachItemsEditorDialogWindow`
2. Keep BrokerWidget runtime registry paths internally unique, but do not show the full `Runtime.ItemBroker...` prefix in the dialog.
3. For BrokerWidget AttachToUi, build a tree model from full persisted paths:
   - Root should normally start at remote client id, because the dialog is already scoped to the BrokerWidget.
   - Example display: `DummyClient1 -> Edm1 -> Pressure`.
   - Leaf nodes represent attachable items.
4. Preserve full paths in `BrokerAttachedItemPaths` for storage and runtime matching.
5. Add stale/missing attachment handling:
   - Merge live attach options with saved paths.
   - Mark saved paths that are not currently live as missing.
   - Provide a delete/remove action for missing saved paths.
   - Deleting removes the path from `BrokerAttachedItemPaths`.
6. Avoid using `BrokerClientId` as a remote item filter.
7. Update visible label for `BrokerClientId` if practical:
   - Prefer `LocalClientId` or `MqttClientId` in the property dialog label.
   - Keep persisted property name `BrokerClientId` unless a migration is explicitly needed.
8. Ensure attach save behavior only serializes selected leaf paths that remain attached.
9. Add or update focused tests if existing test structure supports the changed attach serialization/tree model.
10. Build the affected editor project and, if practical, the solution.

## Technical Constraints
- Chat responses must be in German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be in English.
- User-visible error and validation messages must be in English.
- Use existing Avalonia patterns and current editor view model structure.
- Do not introduce new dependencies.
- Public classes and public methods changed during implementation need XML documentation.
- Keep changes limited to BrokerWidget attach UX and directly required shared attach editor support.

## Notes
- The runtime path may remain `Runtime.ItemBroker.{WidgetName}.{RemoteClientId}.{ItemPath}` for collision-free registry storage.
- The dialog should hide the machine prefix and optimize for human scanning.
- If several remote clients are present, group by remote client id.
- Structure nodes should not have to be attachable initially; leaf selection is enough.
- Tri-state parent selection can be added only if it stays simple and consistent with existing patterns.

# IMPLEMENTATION HANDOFF

## Goal
Fix MQTT/BrokerWidget write-back so numeric payloads are converted to the existing target value type before assignment, e.g. MQTT payload `30` parsed as `long` can update a local `double` value as `30.0`.

## Scope
Included:
- Add centralized value coercion for host registry writes.
- Apply coercion in `DataRegistry.UpdateValue`.
- Apply coercion in `DataRegistry.UpdateParameter`.
- Ensure `TryUpdateUserParameter` benefits through `UpdateParameter`.
- Keep protected parameter blocking unchanged.
- Add focused tests for numeric write conversion.

Not included:
- No changes to MQTT topic mapping.
- No changes to retained topic format.
- No broad rewrite of `Amium.Item.Parameter.Value`.
- No new external dependencies.

## Tasks
1. Add a private helper in `DataRegistry` to coerce incoming write values to the existing target runtime type.
2. In `UpdateValue`, resolve the target item, coerce the incoming value against `item.Value`, then assign the converted value.
3. In `UpdateParameter`, resolve the target parameter, coerce the incoming value against `parameter.Value`, then assign the converted value.
4. Support safe conversions for numeric-to-numeric values, string target values, enum target values, and other `IConvertible` target types.
5. On conversion failure, log a warning and return `false` without throwing.
6. Add host registry tests for `long` payloads updating existing `double` item values.
7. Add host registry tests for `long` payloads updating existing `double` parameters.
8. Add BrokerWidget write-back coverage for an incoming `long` value updating a local `double` value.
9. Run focused Host and Editor tests.
10. Update `Debug.md` with the current root cause, fix, and verification state.
11. Update `CHANGELOG.md` with the write-back numeric conversion fix.

## Technical Constraints
- Chat and planning are German; code, comments, XML documentation, UI text, file names, and identifiers are English.
- Follow existing `.editorconfig` and project patterns.
- Do not introduce new dependencies.
- Keep `Parameter.Value` strict; normalize external inputs before assignment.
- Do not weaken `HostRegistryParameterPolicy`.
- Conversion failures must not disrupt the UI; log and ignore the write.
- Avoid broad refactors outside `DataRegistry` and focused tests.

## Notes
- Root cause: MQTT payload `30` is parsed as `System.Int64`, while `30.1` is parsed as `System.Double`.
- Existing `Amium.Item.Parameter.Value` rejects assignments when the new runtime type differs from the existing runtime type.
- The clean boundary is the registry write path because MQTT, BrokerWidget write-back, and guarded parameter writes converge there.
- Fast path should return immediately when incoming and existing runtime types already match.
- Existing old broker paths such as `HornetStudio...` are not part of this fix.

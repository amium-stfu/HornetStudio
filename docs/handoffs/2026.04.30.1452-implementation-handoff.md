# IMPLEMENTATION HANDOFF

## Goal
Add an explicit BrokerWidget mode that either connects to an existing MQTT ItemBroker endpoint or creates and owns a local in-process ItemBroker with MQTT adapter before connecting.

## Scope
Included:
- Add a persisted BrokerWidget mode/property for external connection vs own broker.
- Keep existing external connection behavior as the default.
- Start and stop an owned broker only when the widget is explicitly configured for own broker mode.
- Preserve current Failed behavior when the configured broker endpoint cannot be used.
- Update property editor, layout persistence, YAML sample, tests, and widget documentation.

Not included:
- No automatic fallback when no external broker is available.
- No unrelated BrokerWidget UI redesign.
- No new broker transport beyond the existing MQTT adapter.

## Tasks
1. Add an explicit BrokerWidget mode/property to `FolderItemModel`, defaulting to external connection for backward compatibility.
2. Persist and restore the new property in layout import/export paths.
3. Expose the property in the BrokerWidget property editor as an either-or choice.
4. Update `BrokerClientControl` connection flow so own-broker mode creates an `InMemoryItemBroker` and starts `MqttItemBrokerAdapter` before creating `HostItemBrokerClient`.
5. Ensure disconnect, visual detach, connect cancellation, and connect failure dispose the owned adapter and broker state correctly.
6. Preserve current `Failed` status and logging when external connect fails or owned broker startup fails.
7. Add or update tests for default external mode, own-broker mode startup/disposal, and failure behavior.
8. Update BrokerWidget YAML sample and widget documentation.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, and identifiers are English.
- Prefer existing projects and patterns.
- `Amium.ItemBroker` remains transport-neutral.
- The widget may reference/use existing `Amium.ItemBroker` and `Amium.ItemBroker.Mqtt` components only if project references already permit it or are added narrowly where required.
- Public classes or public members changed during implementation need XML documentation.
- Do not add new external dependencies.
- No implicit fallback: external mode connects to the configured endpoint; own-broker mode starts the configured local endpoint.

## Notes
- Prefer a string/choice property such as `BrokerMode` with values like `External` and `Own` over combining multiple booleans.
- Existing saved layouts without the new property must behave exactly like current external broker mode.
- If own-broker mode binds to an occupied port, startup should fail and the widget should show `Failed`.
- The owned broker lifecycle belongs to the widget instance and must not dispose any externally managed broker or client.

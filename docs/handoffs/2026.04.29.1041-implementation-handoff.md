# IMPLEMENTATION HANDOFF

## Goal
Change the MQTT topic model so `hornet` only exposes HornetStudio/ItemBroker state, while publishing clients expose their own items in separate client-owned topic areas.

## Scope
Included:
- Replace the current MQTT inspection mapping `hornet/items/{path segments}/{parameter}` with a flatter external MQTT view.
- Expose broker/runtime health below `hornet/broker/...`.
- Expose client-published items below `clients/{clientId}/items/...`.
- Keep canonical broker item paths unchanged internally.
- Update tests and documentation for the new topic model.

Not included:
- Changing the internal `ItemBrokerPath` model.
- Replacing `Amium.ItemBroker` routing, retained state, or subscription behavior.
- Adding authentication, authorization, UI changes, or non-MQTT transports.
- Introducing new runtime dependencies.

## Tasks
1. Update `MqttItemTopicMapper` so MQTT topics are an external view instead of a direct mirror of broker paths.
2. Map broker health paths such as `Runtime.Health.ItemBroker.Heartbeat`, `Runtime.Health.ItemBroker.Uptime`, and `Runtime.Health.ItemBroker.Transport.Mqtt.Status` to `hornet/broker/...` topics.
3. Map client-owned item values to `clients/{clientId}/items/{item path}/value`.
4. Map non-value parameters to `clients/{clientId}/items/{item path}/params/{parameter}`.
5. Ensure incoming MQTT publishes map back to the correct broker path plus parameter.
6. Define and document how `clientId` is derived for MQTT topics, using the broker message source id where available.
7. Update tests in `src/Amium.ItemBroker.Tests/Program.cs` for topic mapping, retained publish behavior, and incoming publish/write mapping.
8. Update `docs/amium-itembroker-architecture.md`, `docs/amium-itembroker-usage.md`, and `docs/amium-item-communication-structure.md` to describe the new topic structure.
9. Build the affected projects and run the ItemBroker test project.

## Technical Constraints
- `Amium.ItemBroker` must remain transport-neutral and must not depend on MQTT.
- MQTT-specific mapping stays in `src/Amium.ItemBroker.Mqtt`.
- Internal canonical paths remain dot-separated broker paths.
- MQTT topic names should be lowercase for structural/static segments: `hornet`, `broker`, `clients`, `items`, `value`, `params`.
- Preserve client and item names without unnecessary case changes unless existing normalization already requires it.
- Use URL escaping for MQTT path segments as currently done.
- Use English for code, comments, identifiers, user-visible error messages, and file names.
- Public classes and public methods changed during implementation need XML documentation.

## Notes
- Recommended MQTT shape:
  - `hornet/broker/status`
  - `hornet/broker/uptime`
  - `hornet/broker/heartbeat`
  - `hornet/broker/clients/count`
  - `hornet/broker/subscriptions/count`
  - `hornet/broker/retained/count`
  - `hornet/broker/messages/per-second`
  - `hornet/broker/mqtt/status`
  - `clients/{clientId}/items/{itemPath}/value`
  - `clients/{clientId}/items/{itemPath}/params/{parameter}`
- `hornet/...` is reserved for HornetStudio/ItemBroker state.
- `clients/...` is reserved for data and status published by clients.
- The MQTT adapter may need a fallback client id such as `unknown` for messages without a source id, but this should be explicit and tested.

# IMPLEMENTATION HANDOFF

## Goal
Introduce registry item roles and capabilities so BrokerWidget publish selection shows only explicitly publishable user/domain items, while widget-internal status/options items stay hidden from normal publish pickers. Also prevent a BrokerWidget/host broker client from listing its own published items as newly received remote items.

## Scope
Included:
- Add metadata for items stored in `HostRegistries.Data`.
- Make metadata easy to assign and adjust at item producer locations.
- Filter BrokerWidget `PublishItems` by metadata capability instead of path blacklist.
- Classify existing registry producers in broad groups.
- Filter broker self-echo by client id.
- Add or update focused tests.

Not included:
- Do not add `Internal` to `Amium.Item.Item`.
- Do not change the MQTT protocol unless required for existing client id filtering.
- Do not redesign all picker UIs.
- Do not perform unrelated refactoring.

## Tasks
1. Add `DataRegistryItemRole` with roles such as `Data`, `Command`, `Configuration`, `Status`, `Diagnostic`, and `System`.
2. Add `[Flags] DataRegistryItemCapabilities` with capabilities such as `Display`, `BrokerPublish`, `BrokerAttach`, `UdlAttach`, `Log`, and `DebugInspect`.
3. Add `DataRegistryItemMetadata` with factory helpers such as `PublicData()`, `PublicCommand()`, `WidgetInternal()`, `WidgetStatus()`, `Diagnostic()`, and `System()`.
4. Extend `IDataRegistry` / `DataRegistry` so `UpsertSnapshot` can receive optional metadata and store it per normalized registry key.
5. Add registry query helpers to retrieve keys by capability and/or role, e.g. broker-publishable keys and debug-inspectable keys.
6. Replace BrokerWidget `PublishItems` option generation with a metadata capability query for `BrokerPublish`.
7. Classify existing `HostRegistries.Data.UpsertSnapshot(...)` producers:
   - Received external/user data from Broker/UDL/Python exposed data: public data where appropriate.
   - EnhancedSignals and CustomSignals generated from user-defined lists: public data.
   - User-created commands/signals: public command/data as appropriate.
   - Widget-created initialization/status/options items such as `Status.AttachOptions`: widget internal/status/debug only.
   - Diagnostics and connection/status items: diagnostic/status/debug only unless explicitly needed elsewhere.
8. Update `HostItemBrokerClient` rebuild/change handling so roots from the same `ClientId` are not exposed as remote received items.
9. Add tests for:
   - Broker publish option filtering excludes widget internal/status/options items.
   - Public data still appears in Broker publish options.
   - Self-published broker items are not listed as remote received items for the same client id.
   - Metadata defaults do not accidentally grant `BrokerPublish`.
10. Run relevant editor/host tests and a build when feasible.

## Technical Constraints
- Keep `Amium.Item.Item` neutral; do not add Studio-specific visibility state to it.
- Store role/capability metadata in the host registry layer, not in the item transport model.
- Role assignment should be visible at item producer sites through metadata helper methods.
- Capability defaults must not automatically include `BrokerPublish`.
- Preserve existing public APIs where possible by adding overloads or optional parameters.
- Use existing project patterns, nullable conventions, XML documentation, and formatting.
- User-visible messages must remain English.

## Notes
- The first classification rule is creation context:
  - Items received from outside or generated from user-defined signal lists are public candidates.
  - Items created by widgets during initialization for their own operation are internal/status/debug candidates.
- Keep metadata factories centralized so role/capability decisions can be adjusted later without hunting flag combinations throughout the codebase.
- For backward compatibility, existing `UpsertSnapshot` callers without metadata should remain valid, but their default metadata must be conservative.
- Broker self-echo is a transport/source concern and should be solved with client id filtering, not item metadata.

# IMPLEMENTATION HANDOFF

## Goal
Fix BrokerWidget registration/display paths for received MQTT items so attached broker items appear under the widget's MQTT transport branch instead of the internal attach-options discovery branch.

## Scope
Included:
- Register received/attached MQTT items under `BrokerWidgetName.Mqtt.<RemoteItemPath>`.
- Hide internal `Status.AttachOptions` discovery paths from normal item tree/runtime item display.
- Omit the MQTT shared client/root marker from visible received item paths.
- Keep `PublishedItems` visible only in the BrokerWidget publish UI, not as redundant normal item-tree entries.
- Preserve existing attach selection behavior and compatibility where possible.

Not included:
- No MQTT topic format change.
- No publish/write-back feature changes.
- No new transport implementation.
- No broad path canonicalization beyond BrokerWidget received item registration/display.
- No release/versioning work.

## Tasks
1. Inspect current BrokerWidget receive/attach flow:
   - `EnumerateAttachOptions(...)`
   - `PublishAttachOptionItems(...)`
   - `PublishRuntimeItems(...)`
   - attach path parsing and saved `BrokerAttachedItemPaths`
   - item tree filtering/display behavior for registry paths.
2. Define path roles explicitly:
   - Discovery path: `Studio.<Folder>.<BrokerWidget>.Status.AttachOptions...`
   - Saved attach identity: keep compatible with existing stored selections, but map to runtime registration cleanly.
   - Visible received runtime path: `Studio.<Folder>.<BrokerWidget>.Mqtt.<RemoteItemPath>` or the project's current canonical local root equivalent.
3. Add or adjust a BrokerWidget helper for received MQTT item path construction:
   - Input: widget item, remote root/client key, relative remote item path.
   - Remove/ignore `shared` when it is only the MQTT shared namespace marker.
   - Include `Mqtt` as the transport segment.
   - Ensure normalized path segments and no duplicated `Status.AttachOptions`.
4. Update `PublishRuntimeItems()`:
   - Match selected attach options using existing saved identities.
   - Register cloned received snapshots at the new visible MQTT runtime path.
   - Track stale runtime keys using the new path shape.
   - Avoid publishing received items below `Status.AttachOptions`.
5. Update stale cleanup:
   - Remove stale old BrokerWidget runtime registrations where necessary.
   - Avoid deleting unrelated widget/status/config paths.
6. Update attach UI compatibility:
   - Existing selections stored with old `Runtime.ItemBroker...` or attach-option-based identities should still resolve where practical.
   - New selections should lead to the new `Mqtt` visible runtime path.
7. Ensure normal item tree excludes internal BrokerWidget discovery items:
   - `Status.AttachOptions` should remain available to the attach dialog but not appear as normal attachable/runtime data.
   - `PublishedItems` should not be added as normal registry items unless already required for the publish editor internals.
8. Update documentation:
   - Update BrokerWidget docs to describe received MQTT items under `BrokerWidgetName.Mqtt...`.
   - Clarify that `Status.AttachOptions` is internal discovery and `PublishedItems` are managed in the BrokerWidget publish UI.
9. Add or update focused tests:
   - Received item path construction includes `Mqtt`.
   - `shared` is omitted from visible received paths.
   - `Status.AttachOptions` is not part of received runtime registration.
   - Published local items are not duplicated as normal received item-tree entries.
10. Run verification:
   - Build relevant projects or full solution when feasible.
   - Run relevant editor/host tests.
   - Manually verify item tree shows examples like:
     - `BrokerWidget1.Mqtt.Edm1.Pressure`
     - `BrokerWidget1.Mqtt.Edm1.Temperature`

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- Use existing helpers and project path conventions.
- Do not add dependencies.
- Keep changes local to BrokerWidget path handling, attach/runtime registration, item tree filtering, tests, and documentation.
- Public classes or public members changed during implementation need XML documentation.
- Preserve compatibility with saved layouts and existing `BrokerAttachedItemPaths` where possible.

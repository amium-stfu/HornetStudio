# IMPLEMENTATION HANDOFF

## Goal
Hide the normal `Parameter` selection from user-facing widget property dialogs and default/fallback `TargetParameterPath` to `Value`.

## Scope
Included:
- Keep `TargetParameterPath` in models and persisted layouts.
- Hide the `Parameter` editor field for normal target-based widgets.
- Normalize empty, protected, or non-user-visible `TargetParameterPath` values to `Value`.
- Preserve runtime compatibility with older layouts.
- Add focused tests for visibility and fallback behavior.

Not included:
- Do not remove `TargetParameterPath` from layout documents.
- Do not introduce an advanced parameter picker.
- Do not redesign widget property generation.
- Do not broaden Broker/UDL/debug-specific editor behavior unless they share the normal target-parameter field.

## Tasks
1. Find the binding definition(s) that expose `TargetParameterPath` as the `Parameter` property in the editor.
2. Hide that field for normal user-facing widget dialogs while keeping the backing model property unchanged.
3. Ensure new target-based widgets default `TargetParameterPath` to `Value`.
4. Ensure target changes normalize `TargetParameterPath` to `Value` when it is empty, protected, or not user-picker-visible.
5. Ensure loaded layouts with protected/invalid `TargetParameterPath` do not execute against that parameter and fall back to `Value`.
6. Keep any existing internal/runtime direct parameter handling intact.
7. Add or update editor tests for hidden `Parameter` field behavior and `Value` fallback.
8. Update documentation/changelog if behavior is user-visible.

## Technical Constraints
- Chat and planning are German; code, comments, XML documentation, UI text, file names, and identifiers are English.
- Do not remove persisted properties or break existing layout documents.
- Use `HostRegistryParameterPolicy.CanShowInUserPicker(...)` or the existing policy equivalent for visibility/fallback decisions.
- Keep changes narrowly scoped to parameter selection visibility and fallback.
- Do not add dependencies.

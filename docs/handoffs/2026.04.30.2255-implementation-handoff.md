# IMPLEMENTATION HANDOFF

## Goal
Implement BrokerWidget writable write-back so external broker writes can update local HornetStudio registry items for active published entries with `Writable = true`, guarded by the central Host registry parameter policy.

## Scope
Included:
- Add a BrokerWidget write-back runtime component, e.g. `HostItemBrokerWriteBackClient`.
- Subscribe only to active published entries where `Writable = true`.
- Apply external broker value/parameter updates back to the configured local registry path.
- Guard parameter writes with the central Host registry parameter policy.
- Protect against echo loops from HornetStudio's own publishes.
- Keep conflict behavior simple: last writer wins.

Not included:
- No per-entry parameter whitelist UI.
- No custom broker path editing.
- No retained broker delete behavior.
- No conflict resolution UI/history.
- No write-back for inactive or non-writable entries.
- No broad redesign of ItemBroker/MQTT transport.

## Tasks
1. Review the existing MQTT client/session subscribe API and extend it only if needed to receive broker updates with path, parameter name, value, and source client id.
2. Add a write-back runtime class owned by `BrokerClientControl`, e.g. `HostItemBrokerWriteBackClient`.
3. Start the write-back runtime when the BrokerWidget is connected and stop it on disconnect, visual detach, or publisher restart.
4. Build write-back subscriptions from parsed `BrokerPublishedItemDefinition` entries where:
   - `Active = true`
   - `Writable = true`
   - `LocalPath` and `BrokerPath` are not empty
5. Subscribe to exact broker paths for writable entries.
6. On received broker update, find the matching writable definition by `BrokerPath`.
7. Ignore updates from the BrokerWidget's own publish client id where source client id information is available.
8. Add internal last-state/echo suppression per `BrokerPath + ParameterName` to avoid applying immediate self-publish echoes.
9. Apply `Value` updates with `HostRegistries.Data.UpdateValue(LocalPath, value)`.
10. Apply non-`Value` parameter updates only when `HostRegistryParameterPolicy.CanUserWriteParameter(parameterName)` allows it.
11. Block/log protected parameter write attempts without throwing disruptive UI exceptions.
12. Do not apply updates when the target local item/path is missing unless existing registry APIs clearly support safe creation for this scenario.
13. Let normal publishing observe the resulting local registry update and republish retained state if configured.
14. Preserve last-writer-wins behavior; do not add conflict resolution.
15. Add focused tests for:
   - non-writable entries are ignored
   - inactive entries are ignored
   - writable `Value` update writes back to `LocalPath`
   - protected parameters are blocked by policy
   - own-source/echo updates are ignored
   - cleanup unsubscribes/stops write-back on disconnect
16. Update BrokerWidget documentation to describe `Writable` behavior and the central protected-parameter guard.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- `Writable = false` remains the safe default.
- Write-back must be entry-scoped; a writable subitem must not make siblings or its root writable.
- Use the central Host registry parameter policy; do not duplicate protected parameter lists.
- `Value` is writable when the entry is active and `Writable = true`.
- Protected parameters must not be written through Broker write-back.
- Internal runtime registry writes must remain possible.
- Do not add external dependencies.
- Public classes or public members changed during implementation need XML documentation.

## Notes
- The write-back component is a shadow runtime client, not a UI concept.
- Keep subscriptions narrow and exact to configured writable broker paths.
- If the MQTT layer cannot reliably expose source client id for received updates, rely on internal last-state suppression and equality checks as the fallback echo guard.
- Broker write-back should be prepared to handle retained messages on subscribe; retained self-state must not create unnecessary local writes.

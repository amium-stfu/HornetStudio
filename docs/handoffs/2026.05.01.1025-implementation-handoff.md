# IMPLEMENTATION HANDOFF

## Goal
Prevent BrokerWidget `PublishItems` from offering items that were received from the broker, because publishing received broker items back through the same publish mechanism can create loops.

## Scope
Included:
- Adjust registry metadata for broker-received items.
- Keep broker-received items visible/attachable/debug-inspectable where appropriate.
- Keep locally produced/user-defined items publishable.
- Add focused tests for broker-received item filtering.

Not included:
- Do not redesign `PublishItems`.
- Do not remove received broker items from normal display/attach/debug flows.
- Do not change `Amium.Item.Item`.
- Do not add forwarding/bridge behavior between brokers.

## Tasks
1. Add a metadata helper for broker-received data, e.g. `DataRegistryItemMetadata.BrokerReceivedData()`.
2. Define broker-received metadata as `Role = Data` with capabilities for visibility/inspection/attach as appropriate, but without `BrokerPublish`.
3. Update broker receive registry writes to use the new helper instead of `PublicData()`.
   - Primary known location: `src/HornetStudio.Editor/Widgets/Broker/BrokerClientControl.axaml.cs` around the `HostRegistries.Data.UpsertSnapshot(...)` call for received broker snapshots.
4. Keep `PublicData()` unchanged for locally generated or user-defined data that should remain publishable.
5. Verify `GetBrokerPublishItemOptions(...)` continues to query `DataRegistryItemCapabilities.BrokerPublish`; no path blacklist should be reintroduced.
6. Add/update tests:
   - Broker-received item gets metadata without `BrokerPublish`.
   - Broker-received item does not appear in BrokerWidget `PublishItems`.
   - Broker-received item remains visible/attachable/debug-inspectable according to chosen capabilities.
   - Local public data, CustomSignals, and EnhancedSignals still appear in `PublishItems`.
7. Run relevant editor/host tests and a build when feasible.

## Technical Constraints
- `Public` or `Display` must not imply `BrokerPublish`.
- Broker-received data is public for viewing, but not publishable by default.
- Preserve existing registry metadata structure and helper style.
- Do not solve this with path-prefix filtering.
- Keep source/loop handling conceptually separate from item role.

## Notes
- The intended distinction is:
  - Local/user-defined data: `PublicData()` and may include `BrokerPublish`.
  - Broker-received data: public/visible but no `BrokerPublish`.
- Future explicit broker forwarding/bridging should be a separate feature with loop protection, not implicit via `PublishItems`.

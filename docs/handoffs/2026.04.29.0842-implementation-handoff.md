# IMPLEMENTATION HANDOFF

## Goal
Extend the current `Amium.ItemBroker` concept into a complete runtime architecture with high-throughput publishing, central retention rules, a client SDK, external inspection support, health publishing, and clear usage documentation.

## Scope
Included:
- Define and scaffold a policy-aware `Amium.ItemBroker.Client` project.
- Extend broker architecture documentation for performance, retention, external transports, client SDK usage, and health data.
- Add usage documentation that explains the current and planned way to use the broker from applications, services, and external tools.
- Define retention and publish policy contracts.
- Define subscription options needed for fast values, throttling, batching, and slow-client handling.
- Define health/diagnostic message structure.
- Prepare MQTT as the first external inspection transport conceptually, without turning the core broker into an MQTT broker.

Not included:
- Do not implement a full MQTT transport unless explicitly requested later.
- Do not build a custom explorer UI in this step.
- Do not refactor unrelated Host, Editor, UI, widget, or plugin code.
- Do not change public APIs outside the ItemBroker-related projects unless required for compilation.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public classes and public methods must have XML documentation.
- Prefer existing project patterns and `.editorconfig`.
- Avoid new runtime dependencies unless clearly justified.
- `Amium.ItemBroker` must remain transport-neutral.
- `Amium.ItemBroker.Client` may provide convenient publishing behavior, but the broker must remain the central authority for final retention decisions.
- MQTT is a transport adapter, not the broker core.
- Slow external clients must not block high-frequency publishers.
- For 100 Hz measurement values, avoid full item snapshots on every update; prefer deltas, batching, throttling, or latest-value behavior according to policy.

## Notes
- The current service prints `No external transports are enabled.` and cannot be inspected with MQTT Explorer yet.
- MQTT Explorer becomes useful after an MQTT adapter exposes broker data as topics.
- A custom ItemBroker Explorer may become useful later because MQTT Explorer cannot show broker-specific semantics such as retention policy, write ownership, slow-client status, or dropped-message behavior.
- Recommended package split:
  - `Amium.ItemBroker` for core contracts, messages, routing, retained state, and central policies.
  - `Amium.ItemBroker.Client` for convenient publishing, local diffing, throttling, batching, and reconnect-ready client behavior.
  - `Amium.ItemBroker.Mqtt` later for MQTT topic mapping and external inspection.
- `PublishItemAsync(Item)` should not always send a full snapshot. The client SDK should compare state where possible and send snapshots, value deltas, parameter deltas, or throttled latest-value updates according to policy.

# IMPLEMENTATION HANDOFF

## Goal
Implement one-way BrokerWidget publishing from local HornetStudio registry items to the MQTT ItemBroker, using structured published item definitions with trigger mode, interval, broker path, and a stored future `Writable` flag.

## Scope
Included:
- Replace or extend the existing publish item path configuration with structured published item definitions.
- Publish local `HostRegistries.Data` items to the broker when the BrokerWidget is connected.
- Support `OnChanged` and `Interval` publishing modes.
- Store `Writable` per published item for future write-back support, but do not act on it yet.
- Use a flat default broker path format: `HornetStudio.<LocalRegistryKey>`.
- Keep existing `AttachToUI` behavior unchanged.

Not included:
- No external write-back into `HostRegistries.Data`.
- No conflict handling for simultaneous local and external writes.
- No per-parameter writable rules.
- No custom broker path editor unless needed by existing UI structure.
- No unrelated BrokerWidget redesign.

## Tasks
1. Add a structured published item definition model, e.g. `BrokerPublishedItemDefinition`, with `LocalPath`, `BrokerPath`, `PublishMode`, `PublishIntervalMs`, and `Writable`.
2. Add a codec, e.g. `BrokerPublishedItemDefinitionCodec`, to parse and serialize definitions as JSON.
3. Migrate or tolerate the existing newline path list from the first `PublishItems` preparation step by converting paths to definitions with defaults.
4. Update the `PublishItems` TreeView selection flow so selected local paths create or retain full definitions.
5. Default new definitions to:
   - `LocalPath = selected registry key`
   - `BrokerPath = HornetStudio.<LocalPath>`
   - `PublishMode = OnChanged`
   - `PublishIntervalMs = 1000`
   - `Writable = false`
6. Add UI/property editing for `PublishMode`, `PublishIntervalMs`, and `Writable` only if it fits the existing property editor pattern; otherwise persist defaults and keep editable UI for a follow-up.
7. Add a runtime publisher owned by `BrokerClientControl` or a focused helper class, e.g. `HostItemBrokerPublisher`.
8. On BrokerWidget connect, publish initial snapshots for configured items that currently exist in `HostRegistries.Data`.
9. For `OnChanged` definitions, subscribe to `HostRegistries.Data.ItemChanged` and publish value/parameter/snapshot updates for matching local paths.
10. For `Interval` definitions, publish snapshots on a timer using the configured interval.
11. On disconnect or visual detach, stop timers/subscriptions and remove only runtime publisher resources; do not remove local registry items.
12. Prevent self-trigger loops for local registry changes caused by publishing if any registry writes are introduced during implementation.
13. Add focused tests for definition codec defaults/migration, broker path derivation, on-changed publishing, and interval publishing where test infrastructure allows it.
14. Update BrokerWidget documentation/YAML sample to describe one-way publishing and the definition fields accurately.

## Technical Constraints
- Chat and planning are German; code, comments, XML docs, UI text, file names, and identifiers are English.
- Default broker paths must stay as flat as possible.
- Default broker path format is exactly `HornetStudio.<LocalRegistryKey>`.
- Do not add a `Projects` segment.
- Do not implement external write-back in this step.
- `Writable` is persisted for future behavior only.
- Use existing ItemBroker/MQTT client APIs and existing editor patterns.
- Do not add new external dependencies.
- Public classes or public members changed during implementation need XML documentation.
- Keep changes narrowly scoped to BrokerWidget publishing.

## Notes
- The existing registry key is the source of truth for path shape. Example: `Project.DefaultLayout.UdlClient1.m300` publishes as `HornetStudio.Project.DefaultLayout.UdlClient1.m300`.
- If a configured local path does not exist at connect time, skip publishing it and publish it later if an `OnChanged` event creates it.
- `Interval` publishing should avoid one timer per item when practical; group definitions by interval if that fits the local style.
- Keep retained MQTT publishing consistent with the existing ItemBroker client/session behavior.
- A later follow-up should define active `Writable` handling and external write-back semantics.
